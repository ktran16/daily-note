## 00:03 - dbt Project Structure: The Staging → Intermediate → Mart Layered Architecture

Well-organized dbt projects follow a three-layer convention that separates concerns,
promotes reuse, and makes lineage immediately readable. The pattern — popularized by
dbt Labs and the community — divides models into **staging**, **intermediate**, and
**mart** layers, each with a distinct role and materialization strategy.

### The Three Layers

**`models/staging/`** — One model per source table. Clean and rename raw columns,
cast types, and apply no business logic. Every downstream model should reference
staging, never raw sources directly. Typically materialized as `view`.

```sql
-- models/staging/stg_orders.sql
select
    id            as order_id,
    customer_id,
    cast(created_at as timestamp) as created_at,
    status,
    total_amount_cents / 100.0  as total_amount
from {{ source('ecommerce', 'orders') }}
where _fivetran_deleted = false
```

**`models/intermediate/`** — Business logic that doesn't belong in a mart but is
shared across multiple marts. Joins, aggregations, and derived entities live here.
Prefix with `int_` to signal that these are not final outputs. Materialized as
`ephemeral` or `view` to avoid unnecessary storage.

```sql
-- models/intermediate/int_orders_with_customer_metrics.sql
select
    o.order_id,
    o.customer_id,
    o.total_amount,
    c.customer_segment,
    row_number() over (partition by o.customer_id order by o.created_at) as order_sequence
from {{ ref('stg_orders') }} o
left join {{ ref('stg_customers') }} c using (customer_id)
```

**`models/marts/`** — Final, business-facing models organized by domain (e.g.,
`marts/finance/`, `marts/marketing/`). These are the tables BI tools and analysts
query. Materialized as `table` or `incremental`.

```sql
-- models/marts/finance/fct_orders.sql
select
    order_id,
    customer_id,
    customer_segment,
    order_sequence,
    case when order_sequence = 1 then true else false end as is_first_order,
    total_amount
from {{ ref('int_orders_with_customer_metrics') }}
```

### Configuring Layers in `dbt_project.yml`

Apply materializations by folder so individual models don't need config blocks:

```yaml
models:
  my_project:
    staging:
      +materialized: view
      +schema: staging
    intermediate:
      +materialized: ephemeral
    marts:
      +materialized: table
      finance:
        +schema: finance
      marketing:
        +schema: marketing
```

### Why This Matters

- **Lineage is self-documenting**: `source → stg_ → int_ → fct_/dim_` tells you
  exactly where a model sits in the DAG at a glance.
- **DRY logic**: Intermediate models are computed once and reused across many marts
  instead of duplicating joins in every mart.
- **Blast radius control**: A broken source only requires fixing one staging model,
  not hunting down duplicated logic across a dozen marts.
- **Schema isolation**: Each layer can land in its own database schema, giving
  analysts clean namespaces and enabling role-based access at the schema level.

This architecture is not mandatory, but projects that skip it quickly accumulate
spaghetti lineage and duplicated logic. Start with it from day one.

## 05:02 - dbt Snapshots: Tracking Historical Changes with SCD Type 2

dbt **snapshots** solve one of the hardest problems in analytics engineering: source
tables overwrite rows when records change, destroying the history you need for
trend analysis. Snapshots implement **Slowly Changing Dimension Type 2 (SCD2)** —
they detect changed rows and write new versioned records with validity timestamps,
leaving the old rows intact.

### How Snapshots Work

dbt compares the current source data against the previously snapshotted state. For
each row it finds three cases:

- **New row** — inserts with `dbt_valid_from = now()`, `dbt_valid_to = NULL`
- **Changed row** — closes the old record (`dbt_valid_to = now()`), inserts the new
  version with `dbt_valid_from = now()`
- **Deleted row** — closes the record (only with `invalidate_hard_deletes: true`)

### Snapshot Syntax

Snapshots live in the `snapshots/` directory and use the `{% snapshot %}` block:

```sql
-- snapshots/snap_customers.sql
{% snapshot snap_customers %}

{{
    config(
        target_schema = 'snapshots',
        unique_key    = 'customer_id',
        strategy      = 'timestamp',
        updated_at    = 'updated_at',
        invalidate_hard_deletes = true
    )
}}

select
    customer_id,
    email,
    plan_tier,
    updated_at
from {{ source('app_db', 'customers') }}

{% endsnapshot %}
```

Run with: `dbt snapshot`

### Two Detection Strategies

**`timestamp`** (preferred) — uses an `updated_at` column. Fast and reliable when
your source reliably updates that column.

**`check`** — compares a list of columns and marks a row changed if any differ.
Use when there is no reliable `updated_at`:

```sql
{{
    config(
        strategy  = 'check',
        check_cols = ['email', 'plan_tier']
    )
}}
```

### Querying Snapshot History

The snapshot table gains four metadata columns you can filter on:

| Column | Meaning |
|---|---|
| `dbt_scd_id` | Unique surrogate key per version |
| `dbt_updated_at` | When this version was loaded |
| `dbt_valid_from` | When this version became active |
| `dbt_valid_to` | When this version was superseded (`NULL` = current) |

```sql
-- Current state only
select * from snapshots.snap_customers
where dbt_valid_to is null;

-- Point-in-time: what did the record look like on Jan 1 2026?
select * from snapshots.snap_customers
where customer_id = 42
  and dbt_valid_from <= '2026-01-01'
  and (dbt_valid_to > '2026-01-01' or dbt_valid_to is null);
```

### Best Practices

- **Run snapshots before other models** in your pipeline so marts always join
  against the freshest historical state.
- **Snapshot only what you need** — don't snapshot tables that are append-only
  (use incremental models instead) or that never change.
- **Don't transform inside a snapshot** — keep the select simple; push logic
  into a downstream model that references `ref('snap_customers')`.
- **Grant access carefully** — snapshot schemas accumulate PII over time; apply
  column-level masking policies at the warehouse layer.

## 10:02 - dbt Testing: Generic Tests, Singular Tests, and Custom Test Macros

Data quality in dbt is enforced through **tests** — SQL assertions that run after
your models build and fail the pipeline when expectations are violated. dbt offers
two flavors: **generic tests** (reusable, declared in YAML) and **singular tests**
(one-off SQL files for bespoke logic).

### Generic Tests

Four built-in generic tests cover the most common assertions. Declare them in a
`schema.yml` file alongside your model definitions:

```yaml
# models/staging/schema.yml
models:
  - name: stg_orders
    columns:
      - name: order_id
        tests:
          - unique
          - not_null
      - name: status
        tests:
          - accepted_values:
              values: ['placed', 'shipped', 'returned', 'cancelled']
      - name: customer_id
        tests:
          - relationships:
              to: ref('stg_customers')
              field: customer_id
```

Run all tests with `dbt test`, or target a single model: `dbt test --select stg_orders`.

### Singular Tests

For logic that doesn't fit a generic test, write a plain SQL file in `tests/`.
The test **fails if the query returns any rows**:

```sql
-- tests/assert_no_negative_order_amounts.sql
-- Fails if any order has a negative total — indicates a bad ETL transform.
select
    order_id,
    total_amount
from {{ ref('stg_orders') }}
where total_amount < 0
```

### Extending with dbt-utils

The `dbt-utils` package ships a library of additional generic tests that cover
gaps in the built-in four:

```yaml
# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.2.0
```

```yaml
# Composite uniqueness — the combination must be unique, not individual columns
- name: fct_order_items
  tests:
    - dbt_utils.unique_combination_of_columns:
        combination_of_columns: ['order_id', 'line_item_id']

# Expression-level assertion
columns:
  - name: discount_pct
    tests:
      - dbt_utils.expression_is_true:
          expression: ">= 0 and <= 1"
```

### Custom Generic Tests

Write your own reusable generic test as a Jinja macro in `macros/`:

```sql
-- macros/test_is_positive.sql
{% test is_positive(model, column_name) %}

select {{ column_name }}
from {{ model }}
where {{ column_name }} <= 0

{% endtest %}
```

Use it just like a built-in test:

```yaml
columns:
  - name: unit_price
    tests:
      - is_positive
```

### Severity and Thresholds

By default, a failing test errors the run. Downgrade to a warning for non-critical
checks, or allow a small number of failures with `error_if` / `warn_if`:

```yaml
- name: order_id
  tests:
    - unique:
        severity: warn          # warn, don't fail
    - not_null:
        error_if: ">10"         # error only if >10 failures
        warn_if: ">0"           # warn on any failure
```

### Best Practices

- **Test at the staging layer** — validate raw data contracts where it enters your
  DAG so errors surface immediately and don't propagate downstream.
- **Use `store_failures: true`** in `dbt_project.yml` to persist failing rows to a
  table for debugging: `tests: +store_failures: true`.
- **Don't over-test marts** — marts are derived; if staging tests pass the mart
  data is trustworthy. Reserve mart tests for critical business rules only.
- **Run tests in CI** — `dbt build` (which runs models + tests together) is the
  canonical CI command; a test failure blocks the merge before bad data reaches prod.

## 15:03 - dbt Macros & Jinja Templating: Writing Reusable SQL Logic

dbt runs your SQL through the **Jinja2** templating engine before sending it to the
warehouse. This unlocks Python-style control flow, variables, and reusable functions
called **macros** — letting you eliminate copy-paste SQL and encode business logic
once, in one place.

### Jinja Basics in dbt

Three delimiters govern Jinja in dbt:

| Delimiter | Purpose | Example |
|---|---|---|
| `{{ }}` | Output an expression | `{{ ref('stg_orders') }}` |
| `{% %}` | Control flow / statements | `{% if condition %}` |
| `{# #}` | Comment (not sent to DB) | `{# TODO: revisit #}` |

### Writing a Macro

Macros live in the `macros/` directory. They're called with `{{ macro_name(...) }}`
anywhere in your project — models, tests, other macros, `schema.yml`.

```sql
-- macros/cents_to_dollars.sql
{% macro cents_to_dollars(column_name, precision=2) %}
    round({{ column_name }} / 100.0, {{ precision }})
{% endmacro %}
```

Use it in a model:

```sql
-- models/staging/stg_orders.sql
select
    order_id,
    {{ cents_to_dollars('total_amount_cents') }}  as total_amount,
    {{ cents_to_dollars('tax_cents', precision=4) }} as tax_amount
from {{ source('ecommerce', 'orders') }}
```

### Control Flow: `if`, `for`, `set`

Use `{% if %}` to branch on variables or environment targets:

```sql
-- models/marts/fct_orders.sql
select
    order_id,
    {% if target.name == 'prod' %}
        -- Full PII in production
        customer_email
    {% else %}
        -- Mask in dev / staging
        md5(customer_email) as customer_email
    {% endif %}
from {{ ref('stg_orders') }}
```

Use `{% for %}` to generate repetitive column logic without copy-paste:

```sql
{% set event_types = ['click', 'view', 'add_to_cart', 'purchase'] %}

select
    session_id,
    {% for event in event_types %}
    sum(case when event_type = '{{ event }}' then 1 else 0 end) as {{ event }}_count
    {{ "," if not loop.last }}
    {% endfor %}
from {{ ref('stg_events') }}
group by 1
```

### The `run_query` Macro

Sometimes you need to fetch data from the warehouse at compile time — for example,
to dynamically build a pivot. `run_query` executes SQL and returns a relation object:

```sql
{% macro get_active_markets() %}
    {% set query %}
        select distinct market_code from {{ ref('stg_markets') }} where is_active
    {% endset %}

    {% set results = run_query(query) %}

    {% if execute %}
        {{ return(results.columns[0].values()) }}
    {% else %}
        {{ return([]) }}
    {% endif %}
{% endmacro %}
```

Then reference it in a model:

```sql
{% set markets = get_active_markets() %}

select
    order_id,
    {% for market in markets %}
    sum(case when market_code = '{{ market }}' then revenue else 0 end) as {{ market }}_revenue
    {{ "," if not loop.last }}
    {% endfor %}
from {{ ref('fct_orders') }}
group by 1
```

### `var()` and `env_var()` for Configuration

Pass values in from `dbt_project.yml` or the CLI without hardcoding them:

```yaml
# dbt_project.yml
vars:
  lookback_days: 30
  payment_methods: ['credit_card', 'paypal', 'bank_transfer']
```

```sql
-- Use a project variable; override at runtime: dbt run --vars 'lookback_days: 7'
where order_date >= dateadd('day', -{{ var('lookback_days') }}, current_date)
```

```sql
-- Read from an environment variable (second arg is a default)
where warehouse_region = '{{ env_var("WAREHOUSE_REGION", "us-east-1") }}'
```

### Best Practices

- **`{% if execute %}`** — always guard `run_query` results behind this block;
  dbt parses templates twice (parse + execute) and the result set only exists on
  the second pass. Skipping this guard causes `NoneType` errors at parse time.
- **Keep macros small and named clearly** — a macro named `cents_to_dollars` is
  self-documenting; a 200-line macro called `transform` is a maintenance trap.
- **Prefer `dbt-utils` before rolling your own** — the package provides battle-tested
  macros for `surrogate_key`, `safe_divide`, `date_spine`, `pivot`, and many more.
- **Use `dispatch` for cross-adapter macros** — if your macro needs adapter-specific
  SQL (e.g. BigQuery vs Snowflake date syntax), use `adapter.dispatch` so the right
  implementation is chosen at runtime without if/else chains.

## 19:02 - dbt Incremental Models: Building Large Tables Without Full Refreshes

**Incremental models** are dbt's answer to tables that are too large or too
expensive to rebuild from scratch on every run. Instead of truncating and
re-inserting all rows, dbt computes only the *new or changed* rows and merges
them into the existing table — dramatically cutting warehouse compute costs and
run times for fact tables that grow by millions of rows per day.

### How It Works

On the first run dbt builds the table normally. On subsequent runs it wraps
your model in a filter that restricts the scan to recent data, then uses the
warehouse's native `MERGE` (or `INSERT OVERWRITE`) to fold the new rows in:

```sql
-- models/marts/fct_events.sql
{{
    config(
        materialized = 'incremental',
        unique_key   = 'event_id'
    )
}}

select
    event_id,
    user_id,
    event_type,
    occurred_at
from {{ ref('stg_events') }}

{% if is_incremental() %}
    -- On incremental runs, only process events from the last 3 days
    -- (overlap guards against late-arriving data)
    where occurred_at >= (
        select dateadd('day', -3, max(occurred_at)) from {{ this }}
    )
{% endif %}
```

`{{ this }}` refers to the existing target table, and `is_incremental()` is
`true` only when the table already exists and `--full-refresh` was not passed.

### Incremental Strategies

dbt exposes four strategies; which ones are available depends on your adapter:

| Strategy | Behavior | Best For |
|---|---|---|
| `merge` (default) | MERGE on `unique_key` — upserts rows | Mutable fact tables with updates |
| `append` | INSERT only — no dedup | Immutable event logs where rows never change |
| `delete+insert` | DELETE matching keys, then INSERT | Warehouses without MERGE support |
| `insert_overwrite` | Overwrite entire partitions | Partitioned tables on BigQuery/Spark |

```sql
{{
    config(
        materialized = 'incremental',
        unique_key   = 'event_id',
        incremental_strategy = 'merge',
        -- Merge only the columns that can change (skips expensive full-row comparison)
        merge_update_columns = ['event_type', 'properties', 'updated_at']
    )
}}
```

### Handling Late-Arriving Data

The biggest pitfall with incremental models is late-arriving rows: events that
land in your source days after they occurred. A filter on `max(occurred_at)`
misses them. The two standard mitigations:

1. **Lookback window** — always scan the last N days rather than just since the
   last max timestamp (as shown above with `dateadd('day', -3, ...)`).
2. **Partition-based overwrite** — on partitioned tables, overwrite the
   affected partitions entirely:

```sql
{{
    config(
        materialized        = 'incremental',
        incremental_strategy = 'insert_overwrite',
        partition_by        = {'field': 'event_date', 'data_type': 'date'},
        partitions          = ["date_sub(current_date, interval 3 day)",
                               "date_sub(current_date, interval 2 day)",
                               "date_sub(current_date, interval 1 day)",
                               "current_date"]
    )
}}
```

### Full Refresh

When the model logic changes significantly, force a rebuild with:

```bash
dbt run --select fct_events --full-refresh
```

This drops and recreates the table from scratch, bypassing `is_incremental()`.
Schedule periodic full refreshes (weekly/monthly) as a safety net to correct
any accumulated drift from late-arriving data the lookback window missed.

### Best Practices

- **Always set `unique_key`** when using `merge` — without it dbt appends
  duplicates on every run, silently corrupting the table.
- **Keep the incremental filter on an indexed/partitioned column** — filtering
  on `occurred_at` on a clustered or partitioned column is orders of magnitude
  faster than a full scan with a WHERE clause on an unindexed string column.
- **Test with `--full-refresh` in CI** — your CI pipeline should occasionally
  run full refreshes on incremental models to catch schema drift that the
  incremental path wouldn't surface.
- **Use `on_schema_change`** to control what dbt does when you add or drop
  columns: `'ignore'` (default), `'fail'`, `'append_new_columns'`, or
  `'sync_all_columns'`. Set `'append_new_columns'` for most production tables
  to avoid silent data loss when columns are added.
