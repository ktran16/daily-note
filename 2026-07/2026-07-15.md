## 00:02 - dbt Semantic Layer & MetricFlow: Define Metrics Once, Query Everywhere

The **dbt Semantic Layer** (powered by MetricFlow) lets you define business metrics in your dbt project and serve them consistently to any downstream BI tool — Looker, Tableau, Mode, or a custom app — without duplicating logic.

### Why It Matters

Without a semantic layer, "revenue" might be calculated differently in five dashboards. The semantic layer creates a single source of truth: define the metric once in YAML, and every tool queries the same definition.

### Core Concepts

**Semantic Models** map a dbt model to entities, dimensions, and measures:

```yaml
# models/semantics/orders.yml
semantic_models:
  - name: orders
    model: ref('fct_orders')
    entities:
      - name: order_id
        type: primary
      - name: customer_id
        type: foreign
    dimensions:
      - name: order_date
        type: time
        type_params:
          time_granularity: day
      - name: status
        type: categorical
    measures:
      - name: order_total
        agg: sum
        expr: amount
      - name: order_count
        agg: count_distinct
        expr: order_id
```

**Metrics** reference those measures and add business logic:

```yaml
metrics:
  - name: revenue
    type: simple
    label: "Total Revenue"
    type_params:
      measure: order_total

  - name: avg_order_value
    type: derived
    label: "Average Order Value"
    type_params:
      expr: revenue / order_count
      metrics:
        - name: revenue
        - name: order_count
```

### Querying with the Semantic Layer

Use the `mf` CLI (MetricFlow) to test queries locally:

```bash
mf query --metrics revenue --group-by order_date__month
mf query --metrics revenue,avg_order_value --group-by customer__region --limit 10
```

Or via dbt Cloud's JDBC/ADBC endpoint, BI tools hit a single interface and receive consistent results regardless of the underlying warehouse.

### Metric Types

| Type | Use Case |
|------|----------|
| `simple` | Direct aggregation of a measure |
| `ratio` | Numerator / denominator (e.g., conversion rate) |
| `derived` | Expression combining other metrics |
| `cumulative` | Running totals (e.g., cumulative revenue) |

### Best Practices

- Keep semantic models 1:1 with grain-level fact/dimension models — don't layer semantic models on top of wide marts.
- Use `label` fields liberally; they surface in BI tool dropdowns.
- Validate with `mf validate-configs` before deploying — misconfigured joins silently produce wrong numbers.
- Pin metric definitions in version control so changes go through code review, not a BI admin panel.

The semantic layer shifts metric governance from BI tools back into dbt, making your data team the true owners of business logic.

## 05:02 - Debugging dbt: compile, show, print(), and Reading Error Messages

When something breaks in dbt, knowing the right debugging tool for the situation saves enormous time. Here's a practical toolkit for diagnosing issues at every layer.

### `dbt compile` — See the Generated SQL

Before running a model, compile it to inspect the SQL dbt will actually execute:

```bash
dbt compile --select my_model
```

The rendered SQL lands in `target/compiled/<project>/<path>/my_model.sql`. This is the fastest way to catch Jinja rendering errors, wrong `ref()` paths, or macros expanding unexpectedly.

### `dbt show` — Run a Quick Preview

`dbt show` compiles and executes a model (or an ad-hoc query), printing the first rows to your terminal without materializing anything:

```bash
# Preview a model
dbt show --select fct_orders --limit 10

# Run an inline query against your warehouse
dbt show --inline "select count(*) from {{ ref('fct_orders') }}"
```

This is invaluable for sanity-checking logic mid-development without waiting for a full `dbt run`.

### `{{ print() }}` and `{{ log() }}` — Debug Jinja at Runtime

Use these built-in Jinja functions to inspect macro variables during compilation:

```sql
{% set my_list = ['a', 'b', 'c'] %}
{{ print("my_list =", my_list) }}   -- prints to stdout during dbt compile/run

{% if execute %}
  {{ log("Running model: " ~ this, info=true) }}
{% endif %}
```

`print()` always emits output; `log()` with `info=true` appears in the console log. Wrap in `{% if execute %}` to avoid firing during parsing when `execute` is `False`.

### `--debug` Flag — Full SQL and Adapter Calls

```bash
dbt run --select my_model --debug
```

This logs every SQL statement sent to the warehouse, including DDL for creating tables/views and `describe` calls. Useful when a model silently fails or produces unexpected schema changes.

### Reading dbt Error Messages

dbt error messages follow a consistent structure:

```
Compilation Error in model my_model (models/marts/my_model.sql)
  'orders' is undefined
  
  > in macro ref (macros/cross_db_utils/...)
  > called from model my_model (models/marts/my_model.sql)
```

Key things to check:
- **Compilation Error** → Jinja issue; use `dbt compile` to isolate
- **Database Error** → SQL issue; check the compiled SQL in `target/compiled/`
- **Runtime Error in test** → Check `target/compiled/<project>/tests/` for the rendered test SQL

### `target/run_results.json` — Post-Run Forensics

After any `dbt run` or `dbt test`, inspect `target/run_results.json` for timing, row counts, and failure details on every node — useful for scripted CI checks or diagnosing which step in a long run failed.

The fastest debugging loop: `dbt compile` → inspect `target/compiled/` → `dbt show` → fix → repeat. Reserve `--debug` for adapter-level mysteries.

## 10:02 - dbt Model Contracts: Enforcing Schema Guarantees at Build Time

Introduced in dbt 1.5, **model contracts** let you declare the expected shape of a model — column names, data types, and constraints — and have dbt enforce them before or during materialization. Think of it as a typed interface between the models that produce data and the models (or BI tools) that consume it.

### Why Contracts Matter

Without contracts, a downstream model silently breaks when an upstream column is renamed or its type changes. Contracts make that failure loud and early — at compile time or before the DDL runs — rather than when an analyst notices wrong numbers in a dashboard.

### Enabling a Contract

Set `contract.enforced: true` in the model's config block or `schema.yml`:

```yaml
# models/marts/schema.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: unique
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: amount
        data_type: numeric
      - name: status
        data_type: varchar
```

When `enforced: true`, dbt will:
1. **Verify column names and types** match what the model actually produces before materializing (for table/incremental models).
2. **Apply constraints** (NOT NULL, UNIQUE, PRIMARY KEY, FOREIGN KEY) as DDL-level database constraints on supported adapters (Snowflake, BigQuery, Redshift, Postgres).

### Constraint Types

| Constraint | DDL Effect | Checked By |
|------------|-----------|------------|
| `not_null` | `NOT NULL` column | Database or dbt |
| `unique` | `UNIQUE` constraint | Database |
| `primary_key` | `PRIMARY KEY` | Database |
| `foreign_key` | `FOREIGN KEY` | Database |
| `check` | `CHECK (expr)` | Database |
| `custom` | Raw DDL string | Database |

### Table vs. Incremental Behavior

For **table** models, dbt creates the table with constraints in the DDL. For **incremental** models, dbt applies constraints only on the initial build; incremental runs don't re-validate the full table. Use dbt tests (`not_null`, `unique`) as a complement to catch bad incremental data.

### Contracts and Versioned Models

Contracts pair naturally with **model versioning** (`versions:` in `schema.yml`). When you publish `fct_orders v2` with a breaking schema change, consumers can pin to `ref('fct_orders', v=1)` while migrating. The contract on each version documents exactly what that version guarantees.

```yaml
models:
  - name: fct_orders
    versions:
      - v: 1
      - v: 2
    config:
      contract:
        enforced: true
```

### Best Practices

- Enforce contracts on **public models** — those depended on by other teams or BI tools. Internal staging models rarely need them.
- Pair `data_type` declarations with `accepted_values` or `not_null` tests for defense in depth; contracts catch schema drift, tests catch data quality issues.
- Run `dbt compile` locally after changing a contracted model's SELECT to catch type mismatches before CI does.
- Document `data_type` using warehouse-native types (`bigint`, `varchar`, `timestamp_ntz`) — dbt maps them per-adapter.

Model contracts shift schema governance left: the broken interface is caught in the PR, not in production.

## 15:02 - dbt Packages & dbt-utils: The Essential Macro Toolkit

**dbt packages** are reusable collections of models, macros, and tests published to the [dbt Package Hub](https://hub.getdbt.com/). The most widely used is **dbt-utils**, a utility belt that covers common SQL patterns so you don't reinvent them in every project.

### Installing Packages

Add packages to `packages.yml` in the project root, then run `dbt deps`:

```yaml
# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0
  - package: dbt-labs/codegen
    version: 0.13.0
  - package: calogica/dbt_date
    version: 0.10.1
```

```bash
dbt deps   # installs into dbt_packages/
```

Commit `packages.yml` but add `dbt_packages/` to `.gitignore` — treat it like `node_modules`.

### Most Useful dbt-utils Macros

**`generate_surrogate_key`** — produces a consistent hashed surrogate key from one or more columns, handling NULLs safely:

```sql
select
    {{ dbt_utils.generate_surrogate_key(['order_id', 'line_item_id']) }} as surrogate_key,
    order_id,
    line_item_id,
    amount
from {{ ref('stg_order_items') }}
```

**`date_spine`** — generates a complete date series, invaluable for filling gaps in time-series data:

```sql
with spine as (
    {{ dbt_utils.date_spine(
        datepart="day",
        start_date="cast('2024-01-01' as date)",
        end_date="current_date"
    ) }}
)
select date_day from spine
```

**`union_relations`** — unions a list of tables with identical structure, adding a `_dbt_source_relation` column to track origin:

```sql
{{ dbt_utils.union_relations(
    relations=[
        ref('orders_us'),
        ref('orders_eu'),
        ref('orders_apac')
    ]
) }}
```

**`get_column_values`** — returns distinct values of a column at compile time, often used to pivot dynamically:

```sql
{% set payment_methods = dbt_utils.get_column_values(
    table=ref('stg_payments'),
    column='payment_method'
) %}

select
    order_id,
    {% for method in payment_methods %}
    sum(case when payment_method = '{{ method }}' then amount else 0 end) as {{ method }}_amount
    {% if not loop.last %},{% endif %}
    {% endfor %}
from {{ ref('stg_payments') }}
group by 1
```

### codegen: Scaffold YAML Instantly

The `codegen` package generates `schema.yml` stubs from existing tables, saving hours of hand-writing column definitions:

```bash
# Generate a source definition for a raw schema
dbt run-operation codegen.generate_source --args '{"schema_name": "raw_stripe"}'

# Generate a model YAML block for an existing model
dbt run-operation codegen.generate_model_yaml --args '{"model_names": ["stg_orders"]}'
```

Paste the output into your YAML files and fill in descriptions — a much faster starting point than writing from scratch.

### Generic Tests from dbt-utils

dbt-utils ships extra generic tests beyond dbt Core's `not_null`, `unique`, `accepted_values`, and `relationships`:

```yaml
columns:
  - name: created_at
    tests:
      - dbt_utils.not_null_proportion:
          at_least: 0.95          # 95% must be non-null
  - name: event_timestamp
    tests:
      - dbt_utils.recency:
          datepart: hour
          interval: 24            # data must have arrived in the last 24h
```

### Best Practices

- Pin exact package versions in `packages.yml` — a surprise major version bump can silently change macro behavior.
- Run `dbt deps` in CI before `dbt build` to ensure the environment is reproducible.
- Prefer `generate_surrogate_key` over hand-rolled MD5/SHA expressions — it handles cross-warehouse NULL coalescing correctly.
- Use `codegen` as a scaffolding accelerator, not a substitute for writing meaningful descriptions.

Packages multiply your productivity by putting battle-tested patterns at your fingertips — invest 30 minutes exploring the dbt Package Hub when starting a new project.
