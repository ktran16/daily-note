## 00:02 - dbt Sources & Freshness: Validating That Data Arrives On Time

In dbt, **sources** are the raw tables in your warehouse that dbt does _not_ manage — they're loaded by external pipelines (Airbyte, Fivetran, Kafka, custom ETL). Declaring them in a `sources.yml` file unlocks three big wins: lineage tracking through `source()`, documentation, and **freshness checks** that alert you when data stops flowing.

### Declaring a Source

```yaml
# models/staging/sources.yml
version: 2

sources:
  - name: raw_shopify
    database: analytics
    schema: raw
    tables:
      - name: orders
        description: "Raw orders from the Shopify connector"
        freshness:
          warn_after: { count: 6, period: hour }
          error_after: { count: 24, period: hour }
        loaded_at_field: _airbyte_emitted_at   # timestamp column to check

      - name: customers
        freshness:
          warn_after: { count: 12, period: hour }
          error_after: { count: 48, period: hour }
        loaded_at_field: updated_at
```

The `loaded_at_field` is the timestamp column dbt compares against `now()`. If the most-recent row is older than your threshold, dbt emits a warning or error — without running any models.

### Referencing Sources in Models

Instead of hardcoding `raw.orders`, use the `source()` function:

```sql
-- models/staging/stg_orders.sql
select
    id          as order_id,
    customer_id,
    total_price,
    created_at
from {{ source('raw_shopify', 'orders') }}
where created_at >= '2020-01-01'
```

This tells dbt that `stg_orders` depends on `raw_shopify.orders`, making it visible in the lineage DAG and allowing `dbt source freshness` to gate downstream builds.

### Running Freshness Checks

```bash
# Check all sources
dbt source freshness

# Check a specific source table
dbt source freshness --select source:raw_shopify.orders
```

Output shows each source's `max(loaded_at_field)`, the age of that row, and a pass/warn/error status. In CI, run freshness _before_ `dbt build` to catch stale data before it corrupts downstream models.

### Connecting Freshness to CI/CD

A common pattern in GitHub Actions:

```yaml
- name: Check source freshness
  run: dbt source freshness --profiles-dir .
  # Non-zero exit on ERROR status; warnings do not fail the step
```

Combine this with `dbt build --select +stg_orders` (note the `+` to include upstream dependents) so that a freshness error aborts the pipeline before wasted compute.

### Key Takeaway

Declaring sources and configuring freshness transforms dbt from a pure transformation tool into an **observable data pipeline** — you get lineage, documentation, and proactive staleness alerts all from a single YAML block.

## 05:03 - dbt Model Contracts: Enforcing Schema Guarantees Across Teams

As dbt projects scale across multiple teams, a critical risk emerges: a model's schema silently changes and breaks downstream consumers. **Model contracts** (introduced in dbt 1.5) solve this by letting you declare the expected columns and data types for a model — and having dbt enforce them at build time.

### What a Contract Does

When `contract.enforced: true` is set on a model, dbt will:
1. Compare the model's SQL output columns against the declared `columns` list.
2. Raise a compile-time error if any declared column is missing or has the wrong data type.
3. (For table/incremental materializations) issue `CREATE TABLE` DDL with explicit column types rather than inferring them from the SELECT.

### Declaring a Contract

```yaml
# models/marts/fct_orders.yml
version: 2

models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key  # documentation-only on most adapters
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_total
        data_type: numeric
      - name: ordered_at
        data_type: timestamp
      - name: status
        data_type: varchar
```

The `data_type` values must match your warehouse's native types (e.g., `bigint` on Snowflake/BigQuery, `int64` on BigQuery, `integer` on Postgres).

### Column-Level Constraints

Beyond type enforcement, dbt supports several constraint types:

| Constraint | Enforced by DB? | Notes |
|---|---|---|
| `not_null` | Yes (most adapters) | Adds `NOT NULL` to DDL |
| `unique` | Yes (most adapters) | Adds `UNIQUE` |
| `primary_key` | Doc-only (Snowflake, BQ) | Enforced only on Postgres/Redshift |
| `foreign_key` | Varies | Requires `to` and `to_columns` |
| `check` | Varies | Custom SQL expression |

```yaml
- name: status
  data_type: varchar
  constraints:
    - type: check
      expression: "status in ('pending', 'shipped', 'delivered', 'cancelled')"
```

### Contracts + Versioning

Model contracts are designed to pair with **model versioning**. When you bump a version (`v2`), consumers can pin to `v1` while migrating — and the contract guarantees that `v1` keeps its promised shape:

```yaml
models:
  - name: fct_orders
    versions:
      - v: 1
      - v: 2
    config:
      contract:
        enforced: true
```

### When to Use Contracts

Contracts are most valuable for **public models** — those exposed to other teams, BI tools, or downstream dbt projects. For internal staging models, the overhead rarely pays off. A good rule: if a schema change would page someone at 2am, that model should have a contract.
