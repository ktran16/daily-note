## 00:02 - dbt Sources & Freshness: Validating That Data Arrives On Time

In dbt, **sources** are the raw tables in your warehouse that dbt does _not_ manage — they're loaded by external pipelines (Airbyte, Fivetran, Kafka, custom ETL). Declaring them in a `sources.yml` file unlocks three big wins: lineage tracking through `source()`, documentation, and **freshness checks** that alert you when data stops flowing.

### Declaring a Source

```yaml
# models/staging/sources.yml
version: 2

sources:
  - name: raw_shopify
    database: analytics
    schema: raw
    tables:
      - name: orders
        description: "Raw orders from the Shopify connector"
        freshness:
          warn_after: { count: 6, period: hour }
          error_after: { count: 24, period: hour }
        loaded_at_field: _airbyte_emitted_at   # timestamp column to check

      - name: customers
        freshness:
          warn_after: { count: 12, period: hour }
          error_after: { count: 48, period: hour }
        loaded_at_field: updated_at
```

The `loaded_at_field` is the timestamp column dbt compares against `now()`. If the most-recent row is older than your threshold, dbt emits a warning or error — without running any models.

### Referencing Sources in Models

Instead of hardcoding `raw.orders`, use the `source()` function:

```sql
-- models/staging/stg_orders.sql
select
    id          as order_id,
    customer_id,
    total_price,
    created_at
from {{ source('raw_shopify', 'orders') }}
where created_at >= '2020-01-01'
```

This tells dbt that `stg_orders` depends on `raw_shopify.orders`, making it visible in the lineage DAG and allowing `dbt source freshness` to gate downstream builds.

### Running Freshness Checks

```bash
# Check all sources
dbt source freshness

# Check a specific source table
dbt source freshness --select source:raw_shopify.orders
```

Output shows each source's `max(loaded_at_field)`, the age of that row, and a pass/warn/error status. In CI, run freshness _before_ `dbt build` to catch stale data before it corrupts downstream models.

### Connecting Freshness to CI/CD

A common pattern in GitHub Actions:

```yaml
- name: Check source freshness
  run: dbt source freshness --profiles-dir .
  # Non-zero exit on ERROR status; warnings do not fail the step
```

Combine this with `dbt build --select +stg_orders` (note the `+` to include upstream dependents) so that a freshness error aborts the pipeline before wasted compute.

### Key Takeaway

Declaring sources and configuring freshness transforms dbt from a pure transformation tool into an **observable data pipeline** — you get lineage, documentation, and proactive staleness alerts all from a single YAML block.

## 05:03 - dbt Model Contracts: Enforcing Schema Guarantees Across Teams

As dbt projects scale across multiple teams, a critical risk emerges: a model's schema silently changes and breaks downstream consumers. **Model contracts** (introduced in dbt 1.5) solve this by letting you declare the expected columns and data types for a model — and having dbt enforce them at build time.

### What a Contract Does

When `contract.enforced: true` is set on a model, dbt will:
1. Compare the model's SQL output columns against the declared `columns` list.
2. Raise a compile-time error if any declared column is missing or has the wrong data type.
3. (For table/incremental materializations) issue `CREATE TABLE` DDL with explicit column types rather than inferring them from the SELECT.

### Declaring a Contract

```yaml
# models/marts/fct_orders.yml
version: 2

models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key  # documentation-only on most adapters
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_total
        data_type: numeric
      - name: ordered_at
        data_type: timestamp
      - name: status
        data_type: varchar
```

The `data_type` values must match your warehouse's native types (e.g., `bigint` on Snowflake/BigQuery, `int64` on BigQuery, `integer` on Postgres).

### Column-Level Constraints

Beyond type enforcement, dbt supports several constraint types:

| Constraint | Enforced by DB? | Notes |
|---|---|---|
| `not_null` | Yes (most adapters) | Adds `NOT NULL` to DDL |
| `unique` | Yes (most adapters) | Adds `UNIQUE` |
| `primary_key` | Doc-only (Snowflake, BQ) | Enforced only on Postgres/Redshift |
| `foreign_key` | Varies | Requires `to` and `to_columns` |
| `check` | Varies | Custom SQL expression |

```yaml
- name: status
  data_type: varchar
  constraints:
    - type: check
      expression: "status in ('pending', 'shipped', 'delivered', 'cancelled')"
```

### Contracts + Versioning

Model contracts are designed to pair with **model versioning**. When you bump a version (`v2`), consumers can pin to `v1` while migrating — and the contract guarantees that `v1` keeps its promised shape:

```yaml
models:
  - name: fct_orders
    versions:
      - v: 1
      - v: 2
    config:
      contract:
        enforced: true
```

### When to Use Contracts

Contracts are most valuable for **public models** — those exposed to other teams, BI tools, or downstream dbt projects. For internal staging models, the overhead rarely pays off. A good rule: if a schema change would page someone at 2am, that model should have a contract.

## 10:03 - dbt Snapshots: Tracking Slowly Changing Dimensions (SCD Type 2)

Most source tables only store the _current_ state of a record. When a customer changes their email, the old value is overwritten and history is lost. **dbt Snapshots** solve this by automatically maintaining a historical record of how rows change over time — implementing SCD Type 2 without manual bookkeeping.

### How Snapshots Work

On the first run, dbt copies the source rows into a snapshot table and stamps each row with:
- `dbt_scd_id` — a unique hash of the row's natural key + data
- `dbt_valid_from` — when this version of the row became active
- `dbt_valid_to` — `NULL` for the current row; a timestamp for expired rows
- `dbt_updated_at` — when dbt last processed this row

On every subsequent run, dbt compares the source to the snapshot. Changed rows get their `dbt_valid_to` filled in, and a new row is inserted with the updated values.

### Defining a Snapshot

Snapshots live in the `snapshots/` directory and use their own `{% snapshot %}` block:

```sql
-- snapshots/snap_customers.sql
{% snapshot snap_customers %}

{{
  config(
    target_schema='snapshots',
    unique_key='customer_id',
    strategy='timestamp',
    updated_at='updated_at',
    invalidate_hard_deletes=True
  )
}}

select
    customer_id,
    email,
    plan_tier,
    updated_at
from {{ source('raw_app', 'customers') }}

{% endsnapshot %}
```

**Key config options:**

| Option | Description |
|---|---|
| `strategy` | `timestamp` (uses an `updated_at` column) or `check` (hashes specified columns) |
| `updated_at` | Required for `timestamp` strategy; the column dbt compares to detect changes |
| `unique_key` | The natural key identifying a record (can be a column name or list) |
| `invalidate_hard_deletes` | When `true`, rows deleted from the source get `dbt_valid_to` set — prevents ghost records |

### The `check` Strategy (No `updated_at` Column)

When your source lacks a reliable `updated_at`:

```sql
{{
  config(
    strategy='check',
    check_cols=['email', 'plan_tier'],   -- or check_cols='all'
    unique_key='customer_id'
  )
}}
```

dbt hashes the listed columns and inserts a new version whenever the hash changes. The trade-off: every snapshot run must read the entire source table.

### Querying Snapshot History

```sql
-- Current state only
select * from {{ ref('snap_customers') }}
where dbt_valid_to is null

-- What was a customer's plan_tier on a specific date?
select *
from {{ ref('snap_customers') }}
where customer_id = 42
  and '2025-06-01' between dbt_valid_from and coalesce(dbt_valid_to, current_timestamp)
```

### Running Snapshots

```bash
dbt snapshot                          # all snapshots
dbt snapshot --select snap_customers  # one snapshot
```

Snapshots are **not** included in `dbt run` or `dbt build` by default — they need their own command, which is why snapshot runs are typically a separate step in your CI/CD schedule (e.g., hourly or daily).

### Key Takeaway

Snapshots turn a mutable source table into an append-only history log with zero application-layer changes. They're essential for auditing, slowly changing dimensions in a dimensional warehouse, and any use case where "what did this record look like last quarter?" needs a reliable answer.

## 15:02 - dbt Macros & Jinja: Writing Reusable SQL Logic

Macros are dbt's mechanism for reusable code — think of them as SQL functions written in Jinja2. They eliminate copy-paste logic, enforce consistency across models, and can generate entire SQL blocks dynamically.

### Defining a Macro

Macros live in the `macros/` directory. A simple example that generates a `cents_to_dollars` conversion:

```sql
-- macros/cents_to_dollars.sql
{% macro cents_to_dollars(column_name, precision=2) %}
    round({{ column_name }} / 100.0, {{ precision }})
{% endmacro %}
```

Call it in any model:

```sql
-- models/marts/fct_orders.sql
select
    order_id,
    {{ cents_to_dollars('subtotal_cents') }}       as subtotal,
    {{ cents_to_dollars('tax_cents', precision=4) }} as tax
from {{ ref('stg_orders') }}
```

### Macros with Conditional Logic

Macros can branch on arguments using Jinja's `{% if %}` blocks:

```sql
-- macros/date_trunc.sql
{% macro date_trunc_to_period(column, period) %}
    {% if target.type == 'bigquery' %}
        date_trunc({{ column }}, {{ period }})
    {% elif target.type == 'snowflake' %}
        date_trunc('{{ period }}', {{ column }})
    {% else %}
        date_trunc('{{ period }}', {{ column }}::timestamp)
    {% endif %}
{% endmacro %}
```

This pattern — branching on `target.type` — is the basis of dbt's **dispatch** system and is how `dbt-utils` ships cross-adapter macros.

### `run_query()`: Executing SQL at Compile Time

The `run_query()` built-in runs a query during compilation and returns the result as an [Agate table](https://agate.readthedocs.io/). Use it to drive dynamic SQL generation:

```sql
-- macros/get_payment_methods.sql
{% macro get_payment_methods() %}
    {% set query %}
        select distinct payment_method
        from {{ ref('stg_payments') }}
        order by 1
    {% endset %}

    {% set results = run_query(query) %}

    {% if execute %}
        {{ return(results.columns['payment_method'].values()) }}
    {% else %}
        {{ return([]) }}
    {% endif %}
{% endmacro %}
```

Then use it to pivot dynamically:

```sql
-- models/marts/fct_payments_pivoted.sql
{% set payment_methods = get_payment_methods() %}

select
    order_id,
    {% for method in payment_methods %}
    sum(case when payment_method = '{{ method }}' then amount else 0 end)
        as {{ method | replace(' ', '_') }}_amount
    {% if not loop.last %},{% endif %}
    {% endfor %}
from {{ ref('stg_payments') }}
group by 1
```

The `{% if execute %}` guard is critical — during parsing dbt calls macros without running queries, so you must short-circuit to avoid errors.

### Variable Passing with `var()`

Pass values from the CLI or `dbt_project.yml` into macros and models:

```yaml
# dbt_project.yml
vars:
  start_date: '2024-01-01'
  environment: 'production'
```

```sql
-- In any model or macro
where created_at >= '{{ var("start_date") }}'
```

Override at runtime:
```bash
dbt run --vars '{"start_date": "2025-01-01", "environment": "staging"}'
```

### Key Takeaways

| Use Case | Tool |
|---|---|
| Reusable expressions | Macro with arguments |
| Cross-adapter SQL | Macro with `target.type` branches |
| Dynamic column lists | `run_query()` + loop |
| Runtime configuration | `var()` with CLI override |

Keep macros focused on a single concern. If a macro exceeds ~30 lines, consider whether it's doing too much — or whether it belongs in a package you can share across projects.

## 19:03 - dbt Tests: Generic vs Singular and When to Use Each

Testing is a first-class citizen in dbt. Out of the box, dbt ships with four **generic tests** and lets you write unlimited **singular tests** — together they form a lightweight but powerful data quality layer that runs without leaving your dbt project.

### Generic Tests (YAML-Driven)

Generic tests are parameterized assertions declared in `.yml` files. dbt compiles each one to a `SELECT` query; a test **passes** when zero rows are returned.

The four built-in generics:

```yaml
# models/staging/stg_orders.yml
version: 2

models:
  - name: stg_orders
    columns:
      - name: order_id
        tests:
          - not_null
          - unique

      - name: status
        tests:
          - accepted_values:
              values: ['pending', 'shipped', 'delivered', 'cancelled']

      - name: customer_id
        tests:
          - relationships:
              to: ref('stg_customers')
              field: customer_id
```

| Test | What it checks |
|---|---|
| `not_null` | Column has no `NULL` values |
| `unique` | All values are distinct |
| `accepted_values` | Every value is in the declared list |
| `relationships` | Every FK value exists in the referenced model |

### Configuring Test Severity & Thresholds

Not every test failure should block a pipeline. You can demote a failure to a warning or allow a percentage of bad rows:

```yaml
- name: order_id
  tests:
    - not_null:
        config:
          severity: warn          # warn instead of error
    - unique:
        config:
          severity: error
          error_if: ">10"         # error only if >10 failing rows
          warn_if: ">0"           # warn for any failure first
```

### Singular Tests (Custom SQL)

Singular tests are plain `.sql` files inside the `tests/` directory. Like generic tests, they **pass when they return zero rows**. Use them for business logic that doesn't fit a generic template:

```sql
-- tests/assert_orders_total_positive.sql
-- Every order must have a positive total after discounts.
select
    order_id,
    order_total
from {{ ref('fct_orders') }}
where order_total <= 0
```

```sql
-- tests/assert_no_future_orders.sql
-- Orders cannot be created in the future.
select order_id, created_at
from {{ ref('stg_orders') }}
where created_at > current_timestamp
```

### Custom Generic Tests (Macros)

You can turn any singular test into a reusable generic by wrapping it in a macro inside `tests/`:

```sql
-- tests/generic/is_positive.sql
{% test is_positive(model, column_name) %}
select {{ column_name }}
from {{ model }}
where {{ column_name }} <= 0
{% endtest %}
```

Now use it like a built-in:

```yaml
- name: order_total
  tests:
    - is_positive
```

### Running Tests

```bash
dbt test                                 # all tests
dbt test --select stg_orders             # tests for one model
dbt test --select stg_orders,fct_orders  # multiple models
dbt test --select tag:critical           # tests by tag
dbt build                                # run models + tests together
```

### Storing Test Results

In dbt 1.7+, you can persist test results as tables for auditing:

```yaml
# dbt_project.yml
tests:
  +store_failures: true
  +store_failures_as: table   # or 'view'
  +schema: test_failures       # dedicated schema
```

This writes failing rows into `<schema>.not_null_stg_orders_order_id`, making it trivial to inspect what broke and when — essential for data SLAs.

### Key Takeaway

Use **generic tests** for common, column-level assertions (nulls, uniqueness, FK integrity, allowed values). Use **singular tests** for business rules too complex for a parameter list. Promote frequently reused singular tests to **custom generic tests** to keep your YAML DRY. Aim for at minimum `not_null` + `unique` on every primary key in your project.
