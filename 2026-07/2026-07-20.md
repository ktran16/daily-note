## 00:03 - dbt Node Selection Syntax: Surgical Control Over What Gets Built

One of dbt's most powerful (and underused) features is its **node selection syntax** — the language behind `--select` and `--exclude` flags. Mastering it turns clunky "run everything" jobs into fast, targeted executions.

### Basic Selectors

```bash
# By model name (exact match)
dbt run --select orders

# By directory path (all models in folder)
dbt run --select models/marts/finance/

# By tag
dbt run --select tag:pii

# By config property
dbt run --select config.materialized:incremental

# By source
dbt run --select source:salesforce
dbt run --select source:salesforce.opportunities
```

### Graph Operators

These are where selection gets powerful. Graph operators walk your DAG:

```bash
# + prefix: include all upstream ancestors
dbt run --select +orders          # orders plus everything it depends on

# + suffix: include all downstream descendants
dbt run --select orders+          # orders plus everything that depends on it

# Both: a model and its full neighborhood
dbt run --select +orders+

# Limit depth with a number
dbt run --select 2+orders         # only 2 levels upstream
dbt run --select orders+3         # only 3 levels downstream

# @ operator: model + its ancestors + all descendants of those ancestors
# Useful for testing: ensures all models that share upstream data are run
dbt test --select @orders
```

### Set Operations

Combine selectors with spaces (union), commas (intersection), and `--exclude`:

```bash
# Union: run both
dbt run --select orders customers

# Intersection: run models that match BOTH criteria
dbt run --select tag:finance,config.materialized:table

# Exclude: run everything except
dbt run --select marts/ --exclude tag:deprecated

# Complex: union of two sets, minus exclusions
dbt run --select +orders tag:marketing --exclude path:models/staging/
```

### State Selectors (the CI/CD superpower)

State selectors compare your current project against a saved manifest — the engine behind Slim CI:

```bash
# Only models that changed since last run
dbt run --select state:modified --state /path/to/prod/

# Only newly-added models
dbt run --select state:new --state /path/to/prod/

# Modified models AND their downstream dependents
dbt run --select state:modified+ --state /path/to/prod/

# Models whose upstream sources changed
dbt run --select state:modified.sources --state /path/to/prod/
```

### Result Selectors (rerun failures)

After a failed run, target only what went wrong:

```bash
# Re-run models that errored last time
dbt run --select result:error --state ./

# Re-run errors and their downstreams
dbt run --select result:error+ --state ./

# Run tests only on models that were skipped
dbt test --select result:skipped --state ./
```

### Putting It Together: A Practical CI Job

```bash
# In CI: fetch prod manifest, run only what changed + downstream, test the same
dbt run \
  --select state:modified+ \
  --state s3://my-bucket/prod/manifest.json \
  --defer \
  --favor-state

dbt test \
  --select state:modified+ \
  --state s3://my-bucket/prod/manifest.json
```

The `--defer` flag tells dbt to use production relations for unselected upstream models — so your modified model can reference `prod.dim_customers` without rebuilding it from scratch in CI.

### Key Takeaways

- Use `+model` / `model+` to walk the DAG; limit depth with a number prefix/suffix.
- Combine selectors with spaces (union) and commas (intersection).
- `state:modified+` is the correct default for Slim CI — catch regressions in downstream models too.
- `result:error+` lets you surgically rerun only failures without touching green models.
- Bookmark the [dbt selection syntax docs](https://docs.getdbt.com/reference/node-selection/syntax) — there are more method types (package, fqn, exposure, metric) worth knowing.

## 05:03 - dbt Model Contracts: Enforcing Your Data Schema at Build Time

**Model contracts** (introduced in dbt Core 1.5) let you declare a binding schema for a model — column names, data types, and constraints — and have dbt enforce that contract before any downstream consumer ever touches the data. Think of them as compile-time type checking for your warehouse.

### Why Contracts Matter

Without contracts, a refactor that renames a column silently breaks every downstream dashboard and API that depended on it. Contracts make that breakage loud and immediate, at CI time rather than at 2 AM when an alert fires.

### Enabling a Contract

In your model's `.yml` file, set `contract.enforced: true` and declare every column:

```yaml
# models/marts/finance/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric
      - name: status
        data_type: varchar
```

When `enforced: true`, dbt will:
1. **Verify** that every declared column exists in the model's SQL output with the correct data type.
2. **Fail the run** if the shape doesn't match — before writing data.
3. **Apply constraints** (NOT NULL, PRIMARY KEY, FOREIGN KEY) in warehouses that support them (Snowflake, Redshift, BigQuery, etc.).

### Supported Constraint Types

| Constraint       | Enforced by warehouse | Enforced by dbt test |
|------------------|-----------------------|----------------------|
| `not_null`       | Yes (some platforms)  | Yes                  |
| `primary_key`    | Yes (some platforms)  | No (advisory only)   |
| `foreign_key`    | Yes (some platforms)  | No (advisory only)   |
| `unique`         | Yes (some platforms)  | No                   |
| `check`          | Yes (some platforms)  | No                   |

For warehouses that don't enforce constraints natively (e.g. BigQuery), dbt still validates data types — constraints become documentation rather than guardrails.

### Custom Constraints

You can pass warehouse-specific constraint expressions via `expression`:

```yaml
constraints:
  - type: check
    expression: "total_amount >= 0"
    name: chk_positive_amount
```

### Contracts + Versioned Models

Contracts pair naturally with **model versioning** (dbt 1.5+). When you bump a version, the old contract remains stable for existing consumers while the new version can introduce breaking changes safely:

```yaml
models:
  - name: fct_orders
    versions:
      - v: 1           # old contract — still served to legacy consumers
      - v: 2           # new contract with a renamed column
    latest_version: 2
```

### Practical Workflow

```bash
# 1. Add contract to your YAML, then compile to catch type mismatches immediately
dbt compile --select fct_orders

# 2. In CI, run with --fail-fast so a contract violation stops the build early
dbt run --select fct_orders --fail-fast

# 3. Check what constraints were applied in the warehouse
dbt show --select fct_orders
```

### Key Takeaways

- Set `contract.enforced: true` in the model config to activate enforcement.
- Every column must be declared with a `data_type` when a contract is active — partial declarations will error.
- Constraints that the warehouse can't enforce are still valuable as machine-readable documentation.
- Combine contracts with model versioning to give downstream consumers stable references while you iterate.
- Contracts work with `table` and `incremental` materializations; they are **not supported** for `view` or `ephemeral` models (views lack a persistent schema to enforce against).
