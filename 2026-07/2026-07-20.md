## 00:03 - dbt Node Selection Syntax: Surgical Control Over What Gets Built

One of dbt's most powerful (and underused) features is its **node selection syntax** — the language behind `--select` and `--exclude` flags. Mastering it turns clunky "run everything" jobs into fast, targeted executions.

### Basic Selectors

```bash
# By model name (exact match)
dbt run --select orders

# By directory path (all models in folder)
dbt run --select models/marts/finance/

# By tag
dbt run --select tag:pii

# By config property
dbt run --select config.materialized:incremental

# By source
dbt run --select source:salesforce
dbt run --select source:salesforce.opportunities
```

### Graph Operators

These are where selection gets powerful. Graph operators walk your DAG:

```bash
# + prefix: include all upstream ancestors
dbt run --select +orders          # orders plus everything it depends on

# + suffix: include all downstream descendants
dbt run --select orders+          # orders plus everything that depends on it

# Both: a model and its full neighborhood
dbt run --select +orders+

# Limit depth with a number
dbt run --select 2+orders         # only 2 levels upstream
dbt run --select orders+3         # only 3 levels downstream

# @ operator: model + its ancestors + all descendants of those ancestors
# Useful for testing: ensures all models that share upstream data are run
dbt test --select @orders
```

### Set Operations

Combine selectors with spaces (union), commas (intersection), and `--exclude`:

```bash
# Union: run both
dbt run --select orders customers

# Intersection: run models that match BOTH criteria
dbt run --select tag:finance,config.materialized:table

# Exclude: run everything except
dbt run --select marts/ --exclude tag:deprecated

# Complex: union of two sets, minus exclusions
dbt run --select +orders tag:marketing --exclude path:models/staging/
```

### State Selectors (the CI/CD superpower)

State selectors compare your current project against a saved manifest — the engine behind Slim CI:

```bash
# Only models that changed since last run
dbt run --select state:modified --state /path/to/prod/

# Only newly-added models
dbt run --select state:new --state /path/to/prod/

# Modified models AND their downstream dependents
dbt run --select state:modified+ --state /path/to/prod/

# Models whose upstream sources changed
dbt run --select state:modified.sources --state /path/to/prod/
```

### Result Selectors (rerun failures)

After a failed run, target only what went wrong:

```bash
# Re-run models that errored last time
dbt run --select result:error --state ./

# Re-run errors and their downstreams
dbt run --select result:error+ --state ./

# Run tests only on models that were skipped
dbt test --select result:skipped --state ./
```

### Putting It Together: A Practical CI Job

```bash
# In CI: fetch prod manifest, run only what changed + downstream, test the same
dbt run \
  --select state:modified+ \
  --state s3://my-bucket/prod/manifest.json \
  --defer \
  --favor-state

dbt test \
  --select state:modified+ \
  --state s3://my-bucket/prod/manifest.json
```

The `--defer` flag tells dbt to use production relations for unselected upstream models — so your modified model can reference `prod.dim_customers` without rebuilding it from scratch in CI.

### Key Takeaways

- Use `+model` / `model+` to walk the DAG; limit depth with a number prefix/suffix.
- Combine selectors with spaces (union) and commas (intersection).
- `state:modified+` is the correct default for Slim CI — catch regressions in downstream models too.
- `result:error+` lets you surgically rerun only failures without touching green models.
- Bookmark the [dbt selection syntax docs](https://docs.getdbt.com/reference/node-selection/syntax) — there are more method types (package, fqn, exposure, metric) worth knowing.

## 05:03 - dbt Model Contracts: Enforcing Your Data Schema at Build Time

**Model contracts** (introduced in dbt Core 1.5) let you declare a binding schema for a model — column names, data types, and constraints — and have dbt enforce that contract before any downstream consumer ever touches the data. Think of them as compile-time type checking for your warehouse.

### Why Contracts Matter

Without contracts, a refactor that renames a column silently breaks every downstream dashboard and API that depended on it. Contracts make that breakage loud and immediate, at CI time rather than at 2 AM when an alert fires.

### Enabling a Contract

In your model's `.yml` file, set `contract.enforced: true` and declare every column:

```yaml
# models/marts/finance/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric
      - name: status
        data_type: varchar
```

When `enforced: true`, dbt will:
1. **Verify** that every declared column exists in the model's SQL output with the correct data type.
2. **Fail the run** if the shape doesn't match — before writing data.
3. **Apply constraints** (NOT NULL, PRIMARY KEY, FOREIGN KEY) in warehouses that support them (Snowflake, Redshift, BigQuery, etc.).

### Supported Constraint Types

| Constraint       | Enforced by warehouse | Enforced by dbt test |
|------------------|-----------------------|----------------------|
| `not_null`       | Yes (some platforms)  | Yes                  |
| `primary_key`    | Yes (some platforms)  | No (advisory only)   |
| `foreign_key`    | Yes (some platforms)  | No (advisory only)   |
| `unique`         | Yes (some platforms)  | No                   |
| `check`          | Yes (some platforms)  | No                   |

For warehouses that don't enforce constraints natively (e.g. BigQuery), dbt still validates data types — constraints become documentation rather than guardrails.

### Custom Constraints

You can pass warehouse-specific constraint expressions via `expression`:

```yaml
constraints:
  - type: check
    expression: "total_amount >= 0"
    name: chk_positive_amount
```

### Contracts + Versioned Models

Contracts pair naturally with **model versioning** (dbt 1.5+). When you bump a version, the old contract remains stable for existing consumers while the new version can introduce breaking changes safely:

```yaml
models:
  - name: fct_orders
    versions:
      - v: 1           # old contract — still served to legacy consumers
      - v: 2           # new contract with a renamed column
    latest_version: 2
```

### Practical Workflow

```bash
# 1. Add contract to your YAML, then compile to catch type mismatches immediately
dbt compile --select fct_orders

# 2. In CI, run with --fail-fast so a contract violation stops the build early
dbt run --select fct_orders --fail-fast

# 3. Check what constraints were applied in the warehouse
dbt show --select fct_orders
```

### Key Takeaways

- Set `contract.enforced: true` in the model config to activate enforcement.
- Every column must be declared with a `data_type` when a contract is active — partial declarations will error.
- Constraints that the warehouse can't enforce are still valuable as machine-readable documentation.
- Combine contracts with model versioning to give downstream consumers stable references while you iterate.
- Contracts work with `table` and `incremental` materializations; they are **not supported** for `view` or `ephemeral` models (views lack a persistent schema to enforce against).

## 10:02 - dbt Snapshots: Tracking Historical Change with Type 2 SCDs

**dbt Snapshots** solve a classic data warehousing problem: your source system only keeps the current state of a record, but you need to know what it looked like last Tuesday. Snapshots implement **Slowly Changing Dimensions Type 2 (SCD2)** automatically — capturing every version of a row with valid-from/valid-to timestamps.

### How Snapshots Work

Every time `dbt snapshot` runs, dbt compares the current source rows against the last captured state. Rows that changed get their `dbt_valid_to` timestamp closed off, and a new open row is inserted. Unchanged rows are untouched.

### Snapshot File Structure

Snapshots live in the `snapshots/` directory (configurable) and use a special `{% snapshot %}` block:

```sql
-- snapshots/orders_snapshot.sql
{% snapshot orders_snapshot %}

  {{
    config(
      target_schema='snapshots',
      unique_key='order_id',
      strategy='timestamp',
      updated_at='updated_at',
      invalidate_hard_deletes=True
    )
  }}

  select * from {{ source('ecommerce', 'orders') }}

{% endsnapshot %}
```

### Strategy Options

dbt supports two detection strategies:

**`timestamp`** — the recommended approach when a reliable `updated_at` column exists:
```sql
strategy='timestamp',
updated_at='updated_at'
```

**`check`** — compares a list of columns directly; used when there's no reliable updated-at:
```sql
strategy='check',
check_cols=['status', 'amount', 'customer_id']
-- or: check_cols='all'  -- compares every column (expensive on wide tables)
```

### Columns dbt Adds Automatically

| Column            | Description                                               |
|-------------------|-----------------------------------------------------------|
| `dbt_scd_id`      | Unique surrogate key for each snapshot row                |
| `dbt_updated_at`  | When dbt last touched this row                            |
| `dbt_valid_from`  | When this version of the record became active             |
| `dbt_valid_to`    | When this version was superseded (NULL = current version) |

### Querying Snapshot Data

```sql
-- Current state only
select * from {{ ref('orders_snapshot') }}
where dbt_valid_to is null;

-- State as of a point in time
select * from {{ ref('orders_snapshot') }}
where dbt_valid_from <= '2026-06-01'
  and (dbt_valid_to > '2026-06-01' or dbt_valid_to is null);

-- Full history for a specific order
select order_id, status, dbt_valid_from, dbt_valid_to
from {{ ref('orders_snapshot') }}
where order_id = 12345
order by dbt_valid_from;
```

### Hard Deletes

By default, deleting a source row leaves the snapshot row open (valid_to stays NULL). Setting `invalidate_hard_deletes=True` marks deleted rows with a valid_to timestamp — but dbt can only detect deletions if the source query returns all rows, not a pre-filtered subset.

### Best Practices

- **Snapshot raw source data**, not transformed models — you want the full fidelity of the source, and transformations can always be applied downstream via `ref()`.
- **Run snapshots before other models** in your orchestration so downstream models see the freshest history.
- **Use `timestamp` strategy over `check`** wherever possible — it's more performant and semantically clearer.
- **Don't snapshot everything** — only tables where historical tracking genuinely matters (orders, subscriptions, SLA statuses). Snapshotting high-churn lookup tables creates noise.
- Snapshots always use `table` materialization under the hood; they cannot be ephemeral or views.

## 15:02 - dbt Packages & dbt-utils: Reusable Macros for Everyday Transformations

**dbt packages** let you pull in pre-built macros, models, tests, and seeds from the community — similar to npm or pip for your data project. The most widely used is **dbt-utils**, a utility belt that covers patterns you'd otherwise rewrite project after project.

### Installing Packages

Declare dependencies in `packages.yml` at the project root:

```yaml
# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0
  - package: dbt-labs/audit_helper
    version: 0.11.0
  - package: calogica/dbt_expectations
    version: 0.10.4
```

Then install them:

```bash
dbt deps
```

Installed packages land in `dbt_packages/` (add this to `.gitignore`).

### Essential dbt-utils Macros

**`generate_surrogate_key`** — creates a hashed surrogate key from a list of columns, handling NULLs safely:

```sql
select
  {{ dbt_utils.generate_surrogate_key(['order_id', 'line_item_id']) }} as surrogate_key,
  order_id,
  line_item_id,
  product_id
from {{ ref('stg_order_items') }}
```

**`star`** — selects all columns except the ones you exclude, avoiding `SELECT *` anti-patterns:

```sql
select
  {{ dbt_utils.star(from=ref('stg_orders'), except=['_fivetran_synced', '_fivetran_deleted']) }}
from {{ ref('stg_orders') }}
```

**`date_spine`** — generates a continuous date series, useful for filling gaps in time-series data:

```sql
{{ dbt_utils.date_spine(
    datepart="day",
    start_date="cast('2024-01-01' as date)",
    end_date="cast('2026-12-31' as date)"
) }}
```

**`get_column_values`** — fetches distinct values from a column at compile time, enabling dynamic pivots:

```sql
{% set order_statuses = dbt_utils.get_column_values(
    table=ref('stg_orders'),
    column='status'
) %}

select
  customer_id,
  {% for status in order_statuses %}
    sum(case when status = '{{ status }}' then 1 else 0 end) as {{ status }}_orders
    {% if not loop.last %},{% endif %}
  {% endfor %}
from {{ ref('stg_orders') }}
group by 1
```

**`union_relations`** — unions a list of tables with schema alignment, padding missing columns with NULL:

```sql
{{ dbt_utils.union_relations(
    relations=[
        ref('stg_orders_us'),
        ref('stg_orders_eu'),
        ref('stg_orders_apac')
    ],
    include=['order_id', 'customer_id', 'amount', 'region']
) }}
```

### Generic Tests from dbt-utils

Beyond macros, dbt-utils ships reusable generic tests:

```yaml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - dbt_utils.not_empty_string
          - dbt_utils.expression_is_true:
              expression: "amount >= 0"
      - name: order_date
        tests:
          - dbt_utils.not_null_proportion:
              at_least: 0.99   # 99% of rows must be non-null
```

### Other Useful Packages

| Package | What it adds |
|---|---|
| `dbt-labs/audit_helper` | Compare model outputs between branches or environments |
| `calogica/dbt_expectations` | 50+ data quality tests (inspired by Great Expectations) |
| `dbt-labs/dbt_project_evaluator` | Flags DAG structure, naming, and documentation anti-patterns |
| `brooklyn-data/dbt_artifacts` | Persists run results & manifest data to your warehouse |

### Key Takeaways

- Always pin package versions in `packages.yml` — unpinned packages can silently introduce breaking changes.
- Run `dbt deps` as part of every CI job before `dbt run` or `dbt test`.
- `generate_surrogate_key` is the safe way to build join keys across sources; it normalises NULLs to `''` before hashing so NULL + valid ≠ valid + NULL.
- `dbt_project_evaluator` is worth adding to any mature project — it surfaces architectural debt you didn't know you had.
- Check [hub.getdbt.com](https://hub.getdbt.com) for the full package registry.

## 19:03 - dbt Exposures: Documenting Downstream Consumers of Your Models

**Exposures** are dbt's way of declaring what consumes your data — dashboards, applications, ML models, reverse-ETL pipelines, anything downstream of your warehouse. They don't run any SQL; they add metadata that makes your DAG complete and enables impact analysis when models change.

### Why Exposures Matter

Without exposures, your lineage graph ends at the warehouse. You can see that `fct_orders` feeds `mart_revenue`, but you have no visibility into which Tableau dashboards, which Jupyter notebooks, or which customer-facing APIs actually read from those tables. When you need to rename a column or change a model's grain, you're flying blind on impact.

Exposures close that gap.

### Defining an Exposure

Exposures are declared in `.yml` files (typically in a dedicated `models/exposures/` folder or alongside related models):

```yaml
# models/exposures/finance_exposures.yml
version: 2

exposures:
  - name: monthly_revenue_dashboard
    label: "Monthly Revenue Dashboard"
    type: dashboard
    maturity: high
    url: https://company.tableau.com/views/MonthlyRevenue
    description: >
      Executive dashboard tracking monthly recurring revenue, churn, and cohort
      retention. Refreshed nightly. Source of truth for the finance QBR deck.
    depends_on:
      - ref('fct_orders')
      - ref('fct_subscriptions')
      - ref('dim_customers')
    owner:
      name: Finance Analytics Team
      email: finance-analytics@company.com
    tags:
      - finance
      - executive
```

### Exposure Types

dbt recognises these `type` values (purely documentary — no enforcement):

| Type | Use case |
|---|---|
| `dashboard` | BI tools (Tableau, Looker, Metabase, Mode) |
| `notebook` | Jupyter, Hex, Observable |
| `analysis` | Ad-hoc SQL scripts, one-off reports |
| `ml` | ML training pipelines, feature stores |
| `application` | Product features reading directly from the warehouse |

### Maturity Levels

The `maturity` field signals reliability expectations:

```yaml
maturity: low     # experimental / exploratory
maturity: medium  # in use but not mission-critical
maturity: high    # production, SLA-backed, executives rely on this
```

### Querying Exposures in Selection Syntax

Once defined, exposures are first-class nodes in the DAG. You can build everything that feeds a specific exposure:

```bash
# Build all upstream models for a specific exposure
dbt run --select +exposure:monthly_revenue_dashboard

# Test all models that feed any exposure tagged 'executive'
dbt test --select +exposure:monthly_revenue_dashboard

# Build and test only what changed that affects this exposure
dbt run --select state:modified+,exposure:monthly_revenue_dashboard \
        --state ./prod-artifacts/
```

This is especially useful in CI: instead of running `state:modified+` globally, you can scope to exposures tagged `high` maturity — the things that break SLAs if they regress.

### Exposures in the Docs Site

Run `dbt docs generate && dbt docs serve` and every exposure appears as a node in the lineage DAG. Clicking on it shows its owner, URL, description, and the full upstream dependency chain. This makes it easy for non-engineers to trace where a dashboard number comes from.

### Practical Workflow: Change Impact Analysis

```bash
# You're about to rename a column in fct_orders.
# First, see every exposure that depends on it:
dbt ls --select fct_orders+ --resource-type exposure

# Output:
# exposure:monthly_revenue_dashboard
# exposure:churn_prediction_model
# exposure:customer_portal_api

# Now you know exactly who to notify before you merge.
```

### Key Takeaways

- Exposures are documentation-only — they don't execute SQL or create database objects.
- Declare an `owner` on every exposure so there's a clear contact when upstream models change.
- Use `maturity: high` sparingly to mark genuinely SLA-backed consumers; it helps prioritise incident response.
- The `+exposure:name` selector is the cleanest way to scope CI jobs to "only what this dashboard needs."
- Keep exposure YAML close to the models they consume, or in a dedicated `exposures/` folder — avoid scattering them across unrelated model directories.
