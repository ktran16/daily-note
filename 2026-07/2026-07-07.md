## 00:02 - dbt Project Structure: Staging, Intermediate, and Mart Layers

A well-structured dbt project is the foundation of a maintainable analytics codebase. The most widely adopted pattern — popularized by dbt Labs — divides models into three distinct layers: **staging**, **intermediate**, and **marts**. Each layer has a clear contract, a specific materialization, and one job to do.

### The Three-Layer Architecture

```
models/
├── staging/          # One-to-one with raw sources
│   ├── stripe/
│   │   ├── _stripe__sources.yml
│   │   ├── stg_stripe__customers.sql
│   │   └── stg_stripe__invoices.sql
│   └── salesforce/
│       ├── _salesforce__sources.yml
│       └── stg_salesforce__accounts.sql
├── intermediate/     # Business logic, joins, aggregations
│   └── int_revenue_by_customer.sql
└── marts/            # Audience-facing, wide, denormalized
    ├── finance/
    │   └── fct_monthly_revenue.sql
    └── core/
        └── dim_customers.sql
```

### Staging Layer — Clean Once, Reuse Everywhere

Staging models are **ephemeral or views**, one per source table. Their only job is to clean and rename raw data — no joins, no business logic.

```sql
-- models/staging/stripe/stg_stripe__customers.sql
with source as (
    select * from {{ source('stripe', 'customers') }}
),

renamed as (
    select
        id                          as customer_id,
        email                       as email_address,
        created                     as created_at,
        metadata:company_name::text as company_name
    from source
)

select * from renamed
```

Key conventions: prefix with `stg_`, use `source_name__entity` naming, cast data types, rename to snake_case, and filter out deleted records if needed.

### Intermediate Layer — Isolate Complexity

Intermediate models capture reusable business logic that multiple marts will share. Materializing these as **ephemeral** (compiled inline) keeps the warehouse clean; use **views** if you need to debug them.

```sql
-- models/intermediate/int_revenue_by_customer.sql
with customers as (
    select * from {{ ref('stg_stripe__customers') }}
),

invoices as (
    select * from {{ ref('stg_stripe__invoices') }}
    where status = 'paid'
)

select
    c.customer_id,
    c.email_address,
    sum(i.amount_due / 100.0) as total_revenue_usd
from customers c
left join invoices i using (customer_id)
group by 1, 2
```

### Marts Layer — Purpose-Built for Consumers

Marts are **tables** (or incremental models for large datasets), denormalized for the tools and teams that consume them. Organize by business domain (`finance/`, `marketing/`, `core/`).

```sql
-- models/marts/finance/fct_monthly_revenue.sql
select
    date_trunc('month', paid_at)  as revenue_month,
    customer_id,
    sum(total_revenue_usd)        as monthly_revenue_usd
from {{ ref('int_revenue_by_customer') }}
group by 1, 2
```

### Why This Structure Works

| Layer | Materialization | Sources | Purpose |
|-------|----------------|---------|---------|
| Staging | `view` / `ephemeral` | `source()` only | Clean & rename raw data |
| Intermediate | `ephemeral` / `view` | `ref()` staging/other int. | Reusable logic |
| Marts | `table` / `incremental` | `ref()` only | Business-ready outputs |

**The golden rule:** downstream models only ever `ref()` their immediate upstream layer. Marts never directly reference raw sources — if you need to change a source column name, you fix it once in staging and every downstream model inherits the fix automatically.

This layered approach dramatically reduces duplication, makes debugging straightforward (each layer has a single responsibility), and keeps your `graph` clean and predictable for `dbt docs generate`.

## 05:02 - dbt Snapshots: Tracking Slowly Changing Dimensions

dbt **snapshots** let you capture point-in-time state of mutable source data — the canonical use case for Slowly Changing Dimensions (SCD Type 2). Without snapshots, when a row in your source changes, the old value is gone forever. With snapshots, every historical state is preserved automatically.

### How Snapshots Work

Snapshots live in the `snapshots/` directory and use the `{% snapshot %}` block. On each `dbt snapshot` run, dbt:

1. Queries the source table.
2. Compares each row against the current snapshot using a **unique key**.
3. Closes out changed rows by stamping `dbt_valid_to` with the current timestamp.
4. Inserts the new row with `dbt_valid_from` set and `dbt_valid_to = null` (meaning "current").

### Two Strategies: `timestamp` vs `check`

**`timestamp` strategy** — fastest; relies on an `updated_at` column in the source:

```sql
-- snapshots/snap_customers.sql
{% snapshot snap_customers %}

{{
    config(
        target_schema = 'snapshots',
        unique_key    = 'customer_id',
        strategy      = 'timestamp',
        updated_at    = 'updated_at',
    )
}}

select * from {{ source('crm', 'customers') }}

{% endsnapshot %}
```

**`check` strategy** — slower but works when there's no reliable `updated_at`; dbt hashes the columns you specify and detects any change:

```sql
{{
    config(
        strategy   = 'check',
        unique_key = 'customer_id',
        check_cols = ['email', 'plan_tier', 'account_status'],
    )
}}
```

### What dbt Adds to Each Row

| Column | Description |
|---|---|
| `dbt_scd_id` | Surrogate key unique to this version of the row |
| `dbt_updated_at` | When dbt last processed this row |
| `dbt_valid_from` | When this version became active |
| `dbt_valid_to` | When this version expired (`null` = still current) |

### Querying the Snapshot

```sql
-- Current state only
select * from {{ ref('snap_customers') }}
where dbt_valid_to is null;

-- State as of a specific date
select * from {{ ref('snap_customers') }}
where '2025-01-01' between dbt_valid_from and coalesce(dbt_valid_to, '9999-12-31');
```

### Key Gotchas

- **Run snapshots before other models** in your pipeline — they are the source of truth for historical data that marts depend on.
- Snapshots do **not** support incremental-style filters; they always full-scan the source table, so keep source tables lean or pre-filter in the `select`.
- Use `invalidate_hard_deletes = true` in config to mark rows as expired when a source row disappears entirely.
- Snapshot tables are built in `target_schema`, separate from your main models schema — make sure your warehouse permissions allow writes there.

Snapshots are one of dbt's most underused features. Pairing them with a `dim_customers` mart that filters to `dbt_valid_to is null` gives you clean SCD Type 2 history with zero custom ETL code.

## 10:02 - dbt `store_failures`: Persisting Test Failures for Investigation

When a dbt test fails, the default output is a row count and a log line — useful for CI gates, but useless for debugging *which* rows failed. `store_failures` fixes this by materializing the failing rows into a table in your warehouse, so you can query them directly.

### Enabling `store_failures`

Set it globally in `dbt_project.yml` or per-test in your `.yml` config:

```yaml
# dbt_project.yml — enable for all tests in the project
tests:
  +store_failures: true
  +store_failures_as: table   # or 'view' (default when store_failures is true)
  +schema: dbt_test__audit    # dedicated schema to keep test tables separate
```

Or scope it to a single model's tests:

```yaml
# models/marts/finance/_finance__models.yml
models:
  - name: fct_monthly_revenue
    columns:
      - name: customer_id
        tests:
          - not_null:
              config:
                store_failures: true
          - relationships:
              to: ref('dim_customers')
              field: customer_id
              config:
                store_failures: true
                severity: warn   # fail loudly but don't break the run
```

### What Gets Stored

When a test fails, dbt creates (or replaces) a table named after the test — e.g., `dbt_test__audit.not_null_fct_monthly_revenue_customer_id`. The table contains exactly the rows that violated the assertion, with all columns from the model, so you can immediately inspect them:

```sql
select *
from dbt_test__audit.not_null_fct_monthly_revenue_customer_id
limit 50;
```

### Controlling Severity: `warn` vs `error`, `warn_if`, `error_if`

By default, any test failure exits with a non-zero status and breaks the run. You can degrade that to a warning — the run continues, but the failure is logged:

```yaml
- name: order_total_usd
  tests:
    - dbt_utils.accepted_range:
        min_value: 0
        max_value: 100000
        config:
          severity: warn          # log it, don't stop the run
```

For nuanced thresholds, use `warn_if` and `error_if` with a count expression:

```yaml
- name: event_id
  tests:
    - unique:
        config:
          warn_if: ">= 10"        # warn when 10+ duplicates found
          error_if: ">= 100"      # hard failure only at 100+ duplicates
```

This lets you set a "tolerable noise floor" for tests on messy upstream sources while still hard-failing on genuine data disasters.

### Practical Workflow

1. Run tests in CI with `store_failures: true` and `schema: dbt_test__audit`.
2. On a failure, analysts query the audit table directly — no digging through logs.
3. In production, set `severity: warn` for known-noisy upstream sources and `severity: error` for your own mart-layer guarantees.
4. Add a downstream model or dbt exposure that reads from audit tables to build a **data quality dashboard** tracking failure counts over time.

`store_failures` turns test failures from a binary CI gate into an observable, queryable data quality layer — a significant upgrade for any team doing serious data validation.
