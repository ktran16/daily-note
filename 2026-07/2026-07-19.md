## 00:02 - dbt Ephemeral Models: Reusable Logic Without Materializing Tables

Ephemeral models are dbt's "invisible" materialization — they exist purely as Common Table Expressions (CTEs) injected into the models that depend on them. They never create a table or view in the database, which makes them ideal for intermediate transformation logic you want to reuse without paying storage or query-planning costs.

### When to Use Ephemeral Models

- **Shared intermediate logic** used by multiple downstream models (e.g., a cleaned `stg_events` that feeds both `fct_sessions` and `fct_pageviews`)
- **Logic too complex for a single SQL file** but not worth persisting as its own object
- **Avoiding namespace pollution** in your data warehouse (no extra tables cluttering the schema)

### Configuration

Set the materialization in `dbt_project.yml` or in the model file itself:

```yaml
# dbt_project.yml
models:
  my_project:
    staging:
      +materialized: ephemeral
```

Or in the model's config block:

```sql
-- models/staging/stg_orders_cleaned.sql
{{ config(materialized='ephemeral') }}

select
    order_id,
    customer_id,
    lower(trim(status))          as status,
    cast(created_at as date)     as order_date,
    amount / 100.0               as amount_usd
from {{ source('raw', 'orders') }}
where order_id is not null
```

Any downstream model that `ref()`s this ephemeral model will have the CTE inlined automatically:

```sql
-- models/marts/fct_orders.sql
with orders as (
    {{ ref('stg_orders_cleaned') }}   -- ← CTE injected here at compile time
),

enriched as (
    select
        o.*,
        c.country
    from orders o
    left join {{ ref('dim_customers') }} c using (customer_id)
)

select * from enriched
```

At compile time, dbt replaces `ref('stg_orders_cleaned')` with the full SQL of that model as a CTE — no database round-trip needed.

### Key Limitations to Know

| Concern | Detail |
|---|---|
| **Not queryable directly** | You can't `dbt run --select stg_orders_cleaned` and then `SELECT * FROM stg_orders_cleaned` — it doesn't exist in the warehouse |
| **Compiled into every dependent** | If 10 models reference the same ephemeral model, its SQL is duplicated 10 times; warehouse optimizers may not share the scan |
| **No `dbt test` on its own** | Tests on ephemeral models must be run indirectly via their parents, or the tests are injected as CTEs too |
| **Circular references** | The same as any other dbt model — circular `ref()` chains will cause a compile error |

### Practical Tip: Ephemeral vs. View

Choose **ephemeral** when the intermediate result is never queried standalone and you want to avoid database objects. Choose **view** when you want the intermediate layer queryable for debugging. A good rule of thumb: start with a view during development, switch to ephemeral once the model is stable and you're sure it's only used as a building block.

Ephemeral models shine in staging layers where you define common cleaning logic once and share it freely — letting your mart models stay focused on business logic rather than data wrangling.

## 05:02 - dbt Semantic Layer & MetricFlow: Define Business Logic Once, Query Everywhere

The dbt Semantic Layer (powered by **MetricFlow**) lets you define metrics and dimensions in YAML alongside your dbt models — and then query them consistently from any BI tool, notebook, or API. Instead of every analyst writing their own revenue calculation, you define it once in dbt and every consumer gets the same number.

### The Problem It Solves

Without a semantic layer, the same metric gets re-implemented in Looker LookML, a Tableau calculated field, a Jupyter notebook, and a one-off SQL query — all slightly differently. The semantic layer is the single source of truth for *what* a metric means.

### Core Concepts

| Concept | Purpose |
|---|---|
| **Semantic Model** | Maps a dbt model to entities, dimensions, and measures |
| **Metric** | A named, reusable calculation built from measures |
| **Entity** | A join key (e.g., `customer_id`) that links semantic models |
| **Dimension** | An attribute you can group or filter by (e.g., `status`, `created_date`) |

### Defining a Semantic Model

```yaml
# models/marts/fct_orders.yml
semantic_models:
  - name: orders
    model: ref('fct_orders')

    entities:
      - name: order
        type: primary
        expr: order_id
      - name: customer
        type: foreign
        expr: customer_id

    dimensions:
      - name: status
        type: categorical
      - name: order_date
        type: time
        type_params:
          time_granularity: day

    measures:
      - name: order_count
        agg: count
        expr: order_id
      - name: revenue
        agg: sum
        expr: amount_usd
```

### Defining Metrics

```yaml
# models/marts/metrics.yml
metrics:
  - name: total_revenue
    label: Total Revenue
    type: simple
    type_params:
      measure: revenue

  - name: revenue_per_customer
    label: Revenue per Customer
    type: derived
    type_params:
      expr: total_revenue / count_distinct_customers
      metrics:
        - name: total_revenue
        - name: count_distinct_customers

  - name: monthly_active_customers
    label: Monthly Active Customers
    type: simple
    type_params:
      measure: order_count
    filter: |
      {{ Dimension('order__order_date') }} >= dateadd('day', -30, current_date)
```

### Querying with the dbt CLI

```bash
# Query revenue grouped by status and date grain
dbt sl query \
  --metrics total_revenue \
  --group-by order__status,metric_time__month \
  --limit 10
```

### Querying via Python SDK

```python
from dbtsl import SemanticLayerClient

client = SemanticLayerClient(
    environment_id=12345,
    auth_token="your_token",
    host="semantic-layer.cloud.getdbt.com"
)

with client.session():
    df = client.query(
        metrics=["total_revenue"],
        group_by=["order__status", "metric_time__month"],
    )
    print(df)
```

### Key Benefits

- **BI tool agnostic**: Tableau, Looker, Mode, Hex, and custom apps all query the same definitions
- **Automatic join logic**: MetricFlow resolves multi-hop joins between semantic models using entity relationships — no manual join SQL
- **Time intelligence built-in**: Use `metric_time` to slice any metric by day/week/month/year without extra SQL

### When to Adopt It

Start with the semantic layer once you have stable mart models and notice metric definitions diverging across tools. Even one canonical `total_revenue` metric that all stakeholders trust is worth the initial setup investment.

## 10:02 - dbt Model Contracts: Enforce Schema Guarantees at Build Time

dbt Model Contracts (available since dbt v1.5) let you declare the exact columns, data types, and constraints a model must produce — and dbt will fail the run if reality doesn't match the declaration. This shifts schema-breaking bugs from "discovered in production" to "caught at build time."

### Why Contracts Matter

Without contracts, a developer can rename a column, drop a field, or change a data type in a mart model that dozens of downstream BI dashboards depend on. The model runs successfully, but consumers silently break. Contracts make these changes a compile- or run-time error instead.

### Enabling a Contract

Add a `contract` block inside the model's YAML definition:

```yaml
# models/marts/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true

    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: status
        data_type: varchar
      - name: amount_usd
        data_type: numeric
```

When `enforced: true`, dbt will compare the contract definition against what the model's SQL actually produces. If a column is missing, has the wrong type, or a constraint is violated, the run fails with a clear error.

### Supported Constraint Types

| Constraint | Enforced by dbt? | Enforced by warehouse? |
|---|---|---|
| `not_null` | At build time (dbt) | Passed as DDL |
| `primary_key` | At build time (dbt) | Passed as DDL (if supported) |
| `unique` | At build time (dbt) | Passed as DDL |
| `foreign_key` | Schema check only | Passed as DDL (if supported) |
| `check` | Not enforced by dbt | Passed as DDL |

Support varies by warehouse — Snowflake enforces `not_null` but not `primary_key` at write time; BigQuery does not enforce any at the DML level but still validates the column list.

### Contracts + Versioned Models

Contracts pair naturally with **model versions** (also v1.5+). When you bump `fct_orders` to `v2`, the old version remains available under its versioned alias, and both have their own contracts:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1
        config:
          contract:
            enforced: true
      - v: 2
        config:
          contract:
            enforced: true
        columns:
          - name: order_id
            data_type: bigint
          # ... updated schema
```

Downstream models pin to `ref('fct_orders', v=1)` until they're ready to migrate — giving a safe, breaking-change-free upgrade path.

### Practical Tips

- **Start with public-facing mart models**: Contracts add the most value at the boundary between the data team and consumers (BI tools, reverse ETL, APIs). Staging models that only serve internal transformations rarely need them.
- **Use `data_type` values your warehouse recognizes**: `varchar`, `bigint`, `timestamp_ntz` (Snowflake), `INT64` (BigQuery) — not generic SQL aliases.
- **Contracts don't replace tests**: A `not_null` constraint tells dbt the column should be non-null; a `not_null` test tells dbt to *verify* it. Use both for maximum coverage.

Model contracts are the handshake between your data team and your data consumers — make it explicit, make it enforceable, and catch breaking changes before they reach production.

## 15:02 - dbt Hooks: Running SQL Before and After Model Materialization

dbt Hooks let you execute arbitrary SQL statements immediately before (`pre-hook`) or after (`post-hook`) a model materializes. They're the escape hatch for anything dbt doesn't handle natively: granting permissions, creating indexes, logging build metadata, calling stored procedures, or dropping temporary objects.

### Where to Configure Hooks

Hooks can be set at three scopes — project-wide, directory-level, or per-model:

```yaml
# dbt_project.yml — applies to ALL models in the project
models:
  my_project:
    +post-hook:
      - "grant select on {{ this }} to role reporter"

    marts:
      +post-hook:
        - "grant select on {{ this }} to role analyst"
        - "analyze {{ this }}"   # Snowflake: update table statistics
```

Or inline in a model's config block:

```sql
-- models/marts/fct_orders.sql
{{
  config(
    materialized='table',
    pre_hook=[
      "delete from audit.model_log where model_name = 'fct_orders'"
    ],
    post_hook=[
      "insert into audit.model_log (model_name, built_at) values ('fct_orders', current_timestamp)",
      "grant select on {{ this }} to role bi_reader"
    ]
  )
}}

select * from {{ ref('stg_orders') }}
```

### The `this` Variable

Inside a hook, `{{ this }}` refers to the model's fully-qualified relation — `database.schema.model_name`. It resolves correctly whether the model is a table, view, or incremental table, so you don't need to hardcode identifiers.

```sql
-- Grant on the exact object that was just built
grant select on {{ this }} to role reporter;

-- Use this.identifier, this.schema, this.database individually
insert into logs.build_events values ('{{ this.schema }}', '{{ this.identifier }}', current_timestamp);
```

### Practical Use Cases

| Use Case | Hook | SQL Pattern |
|---|---|---|
| Grant read access post-build | `post-hook` | `grant select on {{ this }} to role ...` |
| Create index after table build | `post-hook` | `create index if not exists ...` (Postgres/Redshift) |
| Log build metadata | `post-hook` | `insert into audit.log ...` |
| Drop staging temp table | `post-hook` | `drop table if exists tmp_staging_orders` |
| Set search path before model runs | `pre-hook` | `set search_path = my_schema` (Redshift) |
| Truncate audit table before re-insert | `pre-hook` | `truncate table audit.model_log` |

### Hook Execution Order

When multiple hooks are defined at different scopes, they merge and execute in this order:

1. Project-level `pre-hook` → Directory-level `pre-hook` → Model-level `pre-hook`
2. Model SQL runs (the actual materialization)
3. Model-level `post-hook` → Directory-level `post-hook` → Project-level `post-hook`

Hooks at the same scope run in the order they're listed in the array.

### Transactions and Failure Behavior

By default, dbt wraps the model SQL and its hooks in a single transaction. If a post-hook fails, the entire transaction rolls back — the model is not persisted. You can opt out of the transaction wrapper:

```yaml
+post-hook:
  - sql: "grant select on {{ this }} to role reporter"
    transaction: false   # runs outside the transaction; survives even if the model fails
```

Use `transaction: false` for operations like `grant` that can't run inside a transaction on some warehouses (Redshift), or for audit logging that should persist even on failure.

### Tip: Prefer `grants` Config for Permissions

For straightforward access control, dbt v1.2+ has a dedicated `grants` config that's cleaner than a `post-hook`:

```yaml
models:
  - name: fct_orders
    config:
      grants:
        select: ['reporter', 'bi_reader']
```

Reserve hooks for everything that `grants` can't express — custom indexes, stored procedure calls, metadata logging, and multi-step DDL sequences.
