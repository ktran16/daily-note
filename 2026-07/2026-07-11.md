## 00:02 - dbt Defer: Run Only What You Changed Using Production Artifacts

One of the biggest pain points in dbt development is waiting for upstream models to rebuild before you can test a downstream change. The `--defer` flag solves this by letting you reference production-built artifacts for models you haven't modified, so you only rebuild what actually changed.

### How It Works

When you run dbt, it produces a `manifest.json` and `run_results.json` in the `target/` directory. In dbt Cloud, the "production" job's artifacts are stored and made available. Locally or in CI you can download them with the dbt Cloud API or pass a path via `--state`.

```bash
# Download production artifacts to a local folder
mkdir -p prod-artifacts
# (download manifest.json + run_results.json into prod-artifacts/)

# Run only modified models, referencing production for everything else
dbt run --select state:modified+ --defer --state prod-artifacts/
```

The `state:modified+` selector finds every model that changed (or whose upstream changed) since the production run. `--defer` makes dbt resolve `ref()` calls for *unselected* models against the production schema rather than your dev schema, so you never need to rebuild `dim_customers` just to test a small change in `fct_orders`.

### Practical CI/CD Pattern

```yaml
# .github/workflows/dbt-ci.yml
- name: Run modified models
  run: |
    dbt run \
      --select state:modified+ \
      --defer \
      --state prod-artifacts/ \
      --target ci
    dbt test \
      --select state:modified+ \
      --defer \
      --state prod-artifacts/ \
      --target ci
```

This makes CI pipelines dramatically faster — a change to one mart model no longer triggers a full project rebuild.

### Selecting by State

| Selector | Meaning |
|---|---|
| `state:modified` | Models whose SQL or config changed |
| `state:modified+` | Modified models plus all downstream |
| `state:new` | Models that don't exist in the production manifest |
| `state:modified.body` | Only SQL body changed (not config/deps) |

### Key Caveats

- **Schema must exist**: `--defer` still queries the production *tables*, so your dev role needs read access to the prod schema.
- **Seeds and snapshots**: `--defer` works for models and tests; seeds require a separate `dbt seed --select state:modified` call.
- **dbt Core users**: You manage artifact storage yourself (S3, GCS, or CI artifact uploads). dbt Cloud handles this automatically via the "Slim CI" feature.

`--defer` pairs well with `--favor-state` (dbt 1.6+), which prefers production artifacts even when a dev version of the table exists, ensuring your tests always compare against production data quality.
