## 00:02 - dbt Defer: Run Only What You Changed Using Production Artifacts

One of the biggest pain points in dbt development is waiting for upstream models to rebuild before you can test a downstream change. The `--defer` flag solves this by letting you reference production-built artifacts for models you haven't modified, so you only rebuild what actually changed.

### How It Works

When you run dbt, it produces a `manifest.json` and `run_results.json` in the `target/` directory. In dbt Cloud, the "production" job's artifacts are stored and made available. Locally or in CI you can download them with the dbt Cloud API or pass a path via `--state`.

```bash
# Download production artifacts to a local folder
mkdir -p prod-artifacts
# (download manifest.json + run_results.json into prod-artifacts/)

# Run only modified models, referencing production for everything else
dbt run --select state:modified+ --defer --state prod-artifacts/
```

The `state:modified+` selector finds every model that changed (or whose upstream changed) since the production run. `--defer` makes dbt resolve `ref()` calls for *unselected* models against the production schema rather than your dev schema, so you never need to rebuild `dim_customers` just to test a small change in `fct_orders`.

### Practical CI/CD Pattern

```yaml
# .github/workflows/dbt-ci.yml
- name: Run modified models
  run: |
    dbt run \
      --select state:modified+ \
      --defer \
      --state prod-artifacts/ \
      --target ci
    dbt test \
      --select state:modified+ \
      --defer \
      --state prod-artifacts/ \
      --target ci
```

This makes CI pipelines dramatically faster — a change to one mart model no longer triggers a full project rebuild.

### Selecting by State

| Selector | Meaning |
|---|---|
| `state:modified` | Models whose SQL or config changed |
| `state:modified+` | Modified models plus all downstream |
| `state:new` | Models that don't exist in the production manifest |
| `state:modified.body` | Only SQL body changed (not config/deps) |

### Key Caveats

- **Schema must exist**: `--defer` still queries the production *tables*, so your dev role needs read access to the prod schema.
- **Seeds and snapshots**: `--defer` works for models and tests; seeds require a separate `dbt seed --select state:modified` call.
- **dbt Core users**: You manage artifact storage yourself (S3, GCS, or CI artifact uploads). dbt Cloud handles this automatically via the "Slim CI" feature.

`--defer` pairs well with `--favor-state` (dbt 1.6+), which prefers production artifacts even when a dev version of the table exists, ensuring your tests always compare against production data quality.

## 05:02 - dbt Documentation: doc() Blocks, Schema Descriptions, and Generating a Self-Updating Data Catalog

dbt's built-in documentation system turns your YAML descriptions into a searchable, auto-generated data catalog — complete with column-level lineage, test coverage indicators, and rich markdown. The secret to keeping docs maintainable at scale is the `doc()` block.

### The Problem with Inline Descriptions

Inline descriptions in `schema.yml` become painful when the same column appears in dozens of models:

```yaml
# schema.yml — brittle at scale
models:
  - name: fct_orders
    columns:
      - name: status
        description: "Order status: pending, shipped, returned, or completed"
```

Every model that exposes `status` must copy-paste this string. When the definition changes, you have to update it everywhere.

### doc() Blocks: Single Source of Truth

Store reusable descriptions in `.md` files inside your project:

```markdown
{% docs order_status %}
The current state of the order in the fulfillment lifecycle.

Possible values:
- `pending` — payment received, not yet picked
- `shipped` — left the warehouse, in transit
- `returned` — customer initiated a return
- `completed` — delivered and return window closed
{% enddocs %}
```

Then reference it from any model or source:

```yaml
models:
  - name: fct_orders
    columns:
      - name: status
        description: "{{ doc('order_status') }}"

  - name: stg_orders
    columns:
      - name: status
        description: "{{ doc('order_status') }}"
```

Now there is exactly one place to update when the definition changes.

### Model-Level Descriptions with Jinja

Model descriptions also support `doc()` blocks and full markdown:

```markdown
{% docs fct_orders %}
One row per order. Joins `stg_orders` with fulfillment events to produce
the final order status and revenue metrics used by the Finance dashboard.

**Grain:** `order_id`
**Owner:** Data Engineering
**SLA:** Refreshed every 15 minutes via dbt Cloud
{% enddocs %}
```

### Generating and Serving the Catalog

```bash
# Compile docs (generates target/catalog.json)
dbt docs generate

# Serve locally on http://localhost:8080
dbt docs serve
```

The catalog UI shows the full DAG, column descriptions, data types, row counts (if you ran `dbt docs generate` after a run), and which tests are applied to each column.

### Best Practices

| Practice | Why |
|---|---|
| Store `doc()` blocks in `models/docs/` | Keeps `.md` files out of model SQL directories |
| Name doc blocks after the *concept*, not the model | `doc('customer_id')` reuses across staging, intermediate, and mart |
| Add `description` to every source column | Sources feed lineage — undocumented sources create dead ends in the catalog |
| Commit `target/manifest.json` in CI | Enables dbt Cloud to display lineage without running the full project |

### Automating Catalog Freshness

In dbt Cloud, enable "Generate docs on run" in your production job settings. This keeps `catalog.json` current after every production run, so stakeholders always see up-to-date metadata without a manual `dbt docs generate` step.

## 10:02 - dbt Performance Tuning: Partitioning, Clustering, and Incremental Strategies

Warehouse compute costs and query latency are directly tied to how much data the engine scans. dbt lets you control this at the model level through partition/cluster configs and by choosing the right incremental strategy — without leaving your SQL files.

### Partitioning (BigQuery) and Clustering (Snowflake / BigQuery)

**BigQuery** supports partition pruning on a single `DATE`/`TIMESTAMP`/`INT64` column and up to four clustering columns:

```sql
-- models/marts/fct_events.sql
{{ config(
    materialized = 'incremental',
    partition_by = {
        "field": "event_date",
        "data_type": "date",
        "granularity": "day"
    },
    cluster_by = ["user_id", "event_type"]
) }}

SELECT
    DATE(occurred_at)   AS event_date,
    user_id,
    event_type,
    properties
FROM {{ ref('stg_events') }}

{% if is_incremental() %}
WHERE occurred_at >= (SELECT MAX(event_date) FROM {{ this }})
{% endif %}
```

BigQuery only scans matching partitions, and clustering orders micro-blocks so high-cardinality filters skip irrelevant data without a full scan.

**Snowflake** uses virtual warehouse automatic clustering, but you can declare a `cluster_by` config to define a custom clustering key:

```sql
{{ config(
    materialized = 'incremental',
    unique_key = 'order_id',
    cluster_by = ['DATE_TRUNC("MONTH", order_date)', 'region']
) }}
```

### Incremental Strategies: Which to Pick

| Strategy | How it works | Best for |
|---|---|---|
| `append` | INSERT only — never deletes | immutable event logs with no late-arriving data |
| `merge` | MERGE on `unique_key` — upserts | slowly changing facts, deduplicated keys |
| `delete+insert` | DELETE matching keys, then INSERT | Redshift (no native MERGE) |
| `insert_overwrite` | Replace entire partitions | BigQuery; late-arriving data within a day/month |

`insert_overwrite` is the BigQuery sweet spot for event data: instead of a costly full-table MERGE, dbt drops and rewrites only the affected partitions:

```sql
{{ config(
    materialized = 'incremental',
    incremental_strategy = 'insert_overwrite',
    partition_by = {"field": "event_date", "data_type": "date"}
) }}
```

### The `on_schema_change` Safety Net

When your source adds a column, dbt's default is to error. Set `on_schema_change` to control the behavior:

```sql
{{ config(
    materialized = 'incremental',
    on_schema_change = 'sync_all_columns'  -- append new cols, drop removed cols
) }}
```

Options: `ignore` (default), `fail`, `append_new_columns`, `sync_all_columns`.

### Key Takeaways

- Combine partitioning + clustering for compound pruning — partitioning narrows the row groups, clustering orders within them.
- Prefer `insert_overwrite` over `merge` on BigQuery when your unique key is naturally partition-aligned; it avoids full-table scans.
- Always filter on the partition column in your `{% if is_incremental() %}` block — otherwise the engine scans every partition even if it only writes to one.
- Use `dbt compile` to inspect the rendered SQL and confirm partition filters are present before running in production.

## 15:02 - dbt Model Contracts: Enforcing Column-Level Guarantees at Build Time

dbt Model Contracts (introduced in dbt 1.5) let you declare a formal schema agreement on a model — column names, data types, and constraints — that dbt validates *before* writing data downstream. Think of it as a compile-time type system for your data pipeline.

### Why Contracts Matter

Without contracts, a developer can rename a column or change its type and dbt will silently rebuild the model. Downstream consumers — dashboards, BI tools, reverse ETL — break at runtime, often hours later. Contracts make that break happen at the dbt build step instead.

### Enabling a Contract

Add `contract: enforced: true` under the model's `config` block in `schema.yml`:

```yaml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: unique
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric
      - name: status
        data_type: varchar
```

When `enforced: true`, dbt will:
1. **Verify column names** — every column declared in YAML must exist in the model SQL, and no extra columns are allowed.
2. **Cast to declared types** — dbt wraps the model in a `CREATE TABLE ... AS SELECT CAST(col AS data_type)` so type mismatches surface immediately.
3. **Apply constraints** — `not_null`, `unique`, `primary_key`, and `foreign_key` constraints are pushed down to the warehouse DDL where supported (Snowflake, BigQuery, Redshift, Postgres).

### Constraints by Warehouse

| Constraint | Snowflake | BigQuery | Redshift | Postgres |
|---|---|---|---|---|
| `not_null` | enforced | enforced | enforced | enforced |
| `unique` | unenforced (informational) | unenforced | unenforced | enforced |
| `primary_key` | unenforced | unenforced | unenforced | enforced |
| `foreign_key` | unenforced | unenforced | unenforced | enforced |

Even "unenforced" constraints are valuable — query optimizers on Snowflake and Redshift use them for join elimination and query rewrites, improving performance without adding runtime checks.

### Column-Level Constraints in SQL (Alternative Syntax)

You can also declare constraints inline in the model SQL using dbt's `constraints` meta:

```sql
-- models/marts/fct_orders.sql
SELECT
    order_id,      -- declared not_null + unique in schema.yml
    customer_id,
    order_date,
    total_amount,
    status
FROM {{ ref('stg_orders') }}
```

The YAML declaration is the canonical place; dbt compiles it into the DDL.

### Contracts + Versioning

Contracts pair with dbt model versioning (also 1.5+) to give you safe, breaking-change-free schema evolution:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1
        config:
          contract:
            enforced: true
      - v: 2
        config:
          contract:
            enforced: true
```

Consumers reference `ref('fct_orders', version=1)` while you develop v2 in parallel — no breaking changes until consumers migrate.

### When to Use Contracts

- **Public models** shared across teams or consumed by external tools (BI, reverse ETL, ML features).
- **Mart layer** — staging models change frequently; marts should be stable.
- **Regulated data** — enforce `not_null` on PII-sensitive keys to prevent silent NULLs reaching compliance reports.

Start by contracting your most-used marts and gradually expand inward as your schema stabilizes.
