# Daily Notes - 2026-07-14

## 00:02 - dbt Model Contracts: Enforcing Schemas as Documented Agreements

dbt model contracts (introduced in dbt 1.5) let you declare the exact columns and data types a model must produce — and dbt will **fail the build** if the output doesn't match. Think of them as a formal API contract between upstream producers and downstream consumers.

This is especially valuable in:
- Cross-team pipelines where downstream teams rely on stable schemas
- Public models exposed via the semantic layer or BI tools
- Multi-project setups using dbt Mesh, where cross-project `ref()` requires a contract

### Enabling a Contract

Add `contract: {enforced: true}` in the model's config inside your `.yml` file:

```yaml
models:
  - name: orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric
      - name: status
        data_type: varchar
```

When `enforced: true`, dbt will:
1. Verify every declared column exists in the model's output
2. Verify column data types match (within adapter tolerances)
3. Apply database-level constraints (NOT NULL, PRIMARY KEY) where the adapter supports them

### Why Contracts Beat Silent Drift

Without contracts, removing a column from a model may not fail until a downstream query runs — and only for consumers that happen to reference that column. A contract catches the breakage at the **producer**, not the consumer.

### Column Constraints by Type

| Constraint Type | SQL Equivalent             | Enforcement       |
|-----------------|---------------------------|-------------------|
| `not_null`      | `NOT NULL`                | dbt + some DBs    |
| `unique`        | `UNIQUE`                  | dbt test only     |
| `primary_key`   | `PRIMARY KEY`             | informational*    |
| `foreign_key`   | `FOREIGN KEY REFERENCES`  | informational*    |
| `check`         | `CHECK (expression)`      | adapter-dependent |

*Most analytical warehouses (Snowflake, BigQuery, Redshift) accept constraints as metadata but don't enforce them at insert time. dbt's contract check is the real safeguard.

### Contracts + Model Versioning

Contracts pair naturally with **model versioning**. When you need a breaking schema change, release a new version so downstream consumers can migrate at their own pace:

```yaml
models:
  - name: orders
    versions:
      - v: 1       # existing consumers keep using this
      - v: 2       # new schema, defined in models/orders_v2.sql
        defined_in: orders_v2
```

Consumers pin to `ref('orders', v=1)` while you migrate them to `v=2` — no overnight breakage.

### Best Practices

- Enforce contracts on **public models** that other teams or projects consume
- Keep `data_type` values adapter-consistent (`TEXT` vs `VARCHAR` vs `STRING` vary by warehouse)
- Combine with `meta.owner` to make schema ownership explicit
- In dbt Mesh, contracts are **required** for any model referenced cross-project

Model contracts shift schema drift from a runtime surprise to a build-time failure — catching it at the producer is always cheaper than debugging it in a consumer.
