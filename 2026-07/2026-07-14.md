## 00:02 - dbt Model Contracts: Enforcing Schemas as Documented Agreements

dbt model contracts (introduced in dbt 1.5) let you declare the exact columns and data types a model must produce — and dbt will **fail the build** if the output doesn't match. Think of them as a formal API contract between upstream producers and downstream consumers.

This is especially valuable in:
- Cross-team pipelines where downstream teams rely on stable schemas
- Public models exposed via the semantic layer or BI tools
- Multi-project setups using dbt Mesh, where cross-project `ref()` requires a contract

### Enabling a Contract

Add `contract: {enforced: true}` in the model's config inside your `.yml` file:

```yaml
models:
  - name: orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric
      - name: status
        data_type: varchar
```

When `enforced: true`, dbt will:
1. Verify every declared column exists in the model's output
2. Verify column data types match (within adapter tolerances)
3. Apply database-level constraints (NOT NULL, PRIMARY KEY) where the adapter supports them

### Why Contracts Beat Silent Drift

Without contracts, removing a column from a model may not fail until a downstream query runs — and only for consumers that happen to reference that column. A contract catches the breakage at the **producer**, not the consumer.

### Column Constraints by Type

| Constraint Type | SQL Equivalent             | Enforcement       |
|-----------------|---------------------------|-------------------|
| `not_null`      | `NOT NULL`                | dbt + some DBs    |
| `unique`        | `UNIQUE`                  | dbt test only     |
| `primary_key`   | `PRIMARY KEY`             | informational*    |
| `foreign_key`   | `FOREIGN KEY REFERENCES`  | informational*    |
| `check`         | `CHECK (expression)`      | adapter-dependent |

*Most analytical warehouses (Snowflake, BigQuery, Redshift) accept constraints as metadata but don't enforce them at insert time. dbt's contract check is the real safeguard.

### Contracts + Model Versioning

Contracts pair naturally with **model versioning**. When you need a breaking schema change, release a new version so downstream consumers can migrate at their own pace:

```yaml
models:
  - name: orders
    versions:
      - v: 1       # existing consumers keep using this
      - v: 2       # new schema, defined in models/orders_v2.sql
        defined_in: orders_v2
```

Consumers pin to `ref('orders', v=1)` while you migrate them to `v=2` — no overnight breakage.

### Best Practices

- Enforce contracts on **public models** that other teams or projects consume
- Keep `data_type` values adapter-consistent (`TEXT` vs `VARCHAR` vs `STRING` vary by warehouse)
- Combine with `meta.owner` to make schema ownership explicit
- In dbt Mesh, contracts are **required** for any model referenced cross-project

Model contracts shift schema drift from a runtime surprise to a build-time failure — catching it at the producer is always cheaper than debugging it in a consumer.

## 05:02 - dbt Sources: Declaring, Testing, and Freshness-Checking Raw Data

`source()` is how dbt knows about data you didn't build — raw tables landed by Fivetran, Airbyte, Kafka, or any other loader. Declaring sources unlocks lineage, auto-documentation, built-in tests, and freshness alerts.

### Declaring a Source

Create a `_sources.yml` (or any `.yml` file) in your models directory:

```yaml
sources:
  - name: stripe          # logical name used in {{ source() }}
    database: raw         # override if different from your target DB
    schema: stripe_raw
    description: "Raw Stripe event data loaded by Fivetran"
    tables:
      - name: charges
        description: "One row per charge attempt"
        loaded_at_field: _fivetran_synced   # column dbt checks for freshness
        freshness:
          warn_after: {count: 6, period: hour}
          error_after: {count: 24, period: hour}
        columns:
          - name: id
            tests:
              - unique
              - not_null
          - name: customer_id
            tests:
              - not_null
              - relationships:
                  to: source('stripe', 'customers')
                  field: id
      - name: customers
        loaded_at_field: _fivetran_synced
        freshness:
          warn_after: {count: 12, period: hour}
```

### Referencing a Source in a Model

Use `{{ source('source_name', 'table_name') }}` instead of a hard-coded schema path:

```sql
-- models/staging/stg_stripe_charges.sql
with source as (
    select * from {{ source('stripe', 'charges') }}
),

renamed as (
    select
        id            as charge_id,
        customer_id,
        amount        as amount_cents,
        currency,
        status,
        created       as charged_at,
        _fivetran_synced
    from source
)

select * from renamed
```

This gives you:
- **Lineage** — the DAG shows `stripe.charges → stg_stripe_charges`
- **Refactoring safety** — change the schema in `_sources.yml` once; all models update
- **Documentation** — source appears in `dbt docs generate` automatically

### Running Freshness Checks

```bash
# Check all sources
dbt source freshness

# Check a specific source
dbt source freshness --select source:stripe
```

dbt queries `MAX(loaded_at_field)` and compares it to the current timestamp. Results are also written to `target/sources.json` — useful for dashboards or alerting pipelines.

### Source Tests vs Model Tests

| Scope          | Where defined          | When to run                        |
|----------------|------------------------|------------------------------------|
| Source tests   | `_sources.yml`         | `dbt test --select source:stripe`  |
| Model tests    | model `.yml` files     | `dbt test --select stg_*`          |
| All tests      | —                      | `dbt test`                         |

Run source tests **early in your pipeline** — failing fast on raw data prevents bad records from propagating downstream.

### Best Practices

- Keep source `.yml` files colocated with your staging models, not scattered across the project
- Always declare `loaded_at_field` and `freshness` for operational tables — it turns data latency from a mystery into a measurable SLA
- Use `source_status:fresher_than_x_hours` in node selection (`dbt build --select source_status:fresher+`) to skip unchanged sources in CI
- Tag sources by domain (`tags: [stripe, finance]`) for selective testing in large projects

Treating sources as first-class citizens in your schema files is the difference between a DAG that starts at your staging models and one that traces all the way back to the ingestion layer.
