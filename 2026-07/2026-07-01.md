## 00:03 - dbt Packages: Extending dbt with dbt-utils and the Package Ecosystem

dbt packages let you import reusable macros, models, and tests from the community into your project. They are the primary way to avoid reinventing the wheel and to standardize common patterns across teams.

### Installing Packages

Packages are declared in `packages.yml` at the project root and installed with `dbt deps`:

```yaml
# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0
  - package: calogica/dbt_expectations
    version: 0.10.4
  - package: dbt-labs/audit_helper
    version: 0.12.0
```

```bash
dbt deps   # downloads packages into the dbt_packages/ directory
```

Always pin to an exact or constrained version (`[">=1.2.0", "<2.0.0"]`) to keep builds reproducible.

### dbt-utils: The Essential Toolkit

`dbt-labs/dbt_utils` is the most widely used package. Key highlights:

**Surrogate keys** — generate a consistent hash-based primary key from multiple columns:
```sql
select
    {{ dbt_utils.generate_surrogate_key(['order_id', 'line_item_id']) }} as surrogate_key,
    order_id,
    line_item_id,
    quantity
from {{ ref('stg_order_items') }}
```

**Date spine** — generate a complete sequence of dates, useful for filling gaps in time-series data:
```sql
{{ dbt_utils.date_spine(
    datepart="day",
    start_date="cast('2024-01-01' as date)",
    end_date="current_date"
) }}
```

**Pivot / unpivot** — reshape data without hand-coding CASE WHEN blocks:
```sql
{{ dbt_utils.pivot('payment_method', dbt_utils.get_column_values(ref('orders'), 'payment_method')) }}
```

**Generic tests from dbt-utils:**
```yaml
models:
  - name: orders
    columns:
      - name: order_id
        tests:
          - dbt_utils.not_empty_string
          - dbt_utils.expression_is_true:
              expression: "amount > 0"
```

### dbt-expectations: Data Quality at Scale

Inspired by Great Expectations, `calogica/dbt_expectations` adds rich column-level assertions:

```yaml
- name: revenue
  tests:
    - dbt_expectations.expect_column_values_to_be_between:
        min_value: 0
        max_value: 1000000
    - dbt_expectations.expect_column_to_exist
    - dbt_expectations.expect_column_values_to_not_be_null:
        row_condition: "status = 'completed'"
```

### audit_helper: Validating Refactors

`dbt-labs/audit_helper` is invaluable when refactoring an existing model — it compares row-level output between two queries so you can confirm the rewrite is identical:

```sql
{% set old_query %}
    select * from {{ ref('orders_legacy') }}
{% endset %}

{% set new_query %}
    select * from {{ ref('orders') }}
{% endset %}

{{ audit_helper.compare_queries(
    a_query=old_query,
    b_query=new_query,
    primary_key="order_id"
) }}
```

### Best Practices

- **Commit `packages.yml`, not `dbt_packages/`** — add `dbt_packages/` to `.gitignore` and run `dbt deps` in CI.
- **Check the dbt Hub** at hub.getdbt.com for available packages before writing custom macros.
- **Namespace your macros** — if you write project-level macros with the same name as a package macro, dbt will prefer your version; be intentional.
- **Update periodically** — run `dbt deps --upgrade` in a branch and review the changelog before bumping versions in production.

Packages dramatically reduce boilerplate and bring battle-tested logic into your project with a single `dbt deps`.

## 05:02 - dbt Model Contracts: Enforcing Schema at Build Time

dbt model contracts let you declare the expected schema of a model — column names, data types, and constraints — and have dbt enforce those declarations every time the model runs. This turns implicit assumptions into explicit, machine-checked guarantees.

### Why Contracts Matter

Without contracts, a model's schema is inferred from whatever SQL produces. If an upstream change adds or removes a column, downstream consumers break silently at query time. Contracts catch schema drift at **build time**, before bad data ever reaches consumers.

### Enabling a Contract

Contracts are configured per-model in the `schema.yml` (or `models/` YAML) file. Set `enforced: true` under `contract`:

```yaml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric
      - name: status
        data_type: varchar
```

With `enforced: true`, dbt will raise a compile-time error if:
- Any declared column is missing from the model's SELECT
- A column's data type doesn't match the declaration
- The SELECT contains columns not listed in the contract (order matters too — columns must appear in the same sequence)

### Supported Constraints

Beyond `not_null` and `primary_key`, dbt supports:

```yaml
constraints:
  - type: unique
  - type: not_null
  - type: primary_key
  - type: foreign_key
    to: ref('dim_customers')
    to_columns: [customer_id]
  - type: check
    expression: "total_amount >= 0"
    name: chk_positive_amount   # optional but recommended for debugging
```

Constraint support varies by warehouse — Snowflake enforces `not_null` natively but treats `primary_key`/`unique` as informational; BigQuery and Redshift have their own support profiles. Check the dbt docs for your adapter.

### Model-Level vs. Column-Level Constraints

Constraints can be declared at the column level (as above) or at the model level for multi-column cases:

```yaml
models:
  - name: fct_order_items
    config:
      contract:
        enforced: true
    constraints:
      - type: primary_key
        columns: [order_id, line_item_id]
    columns:
      - name: order_id
        data_type: bigint
      - name: line_item_id
        data_type: int
```

### Contracts and Versioned Models

Contracts become especially powerful combined with **model versions**. When you bump a model to `v2`, you can evolve the contract independently of `v1`, letting downstream consumers migrate on their own schedule while the old contract remains stable:

```yaml
models:
  - name: fct_orders
    versions:
      - v: 1
      - v: 2
        defined_in: fct_orders_v2
    config:
      contract:
        enforced: true
```

### Best Practices

- **Start with `enforced: false`** on existing models to audit mismatches without breaking CI, then flip to `true` once clean.
- **Declare `data_type` explicitly** — omitting it skips type-checking and weakens the contract.
- **Use contracts on public models** (those used by other teams or downstream tools) where interface stability is critical.
- **Combine with `meta` tags** to document ownership alongside the structural guarantee.

Model contracts are a foundational tool for data mesh architectures, where cross-team dependencies need stable, versioned interfaces just like software APIs.

## 10:02 - dbt Macros & Jinja: Writing Reusable SQL Logic

Macros are dbt's mechanism for reusable logic — they are Jinja2 functions defined in `.sql` files inside the `macros/` directory that compile down to plain SQL at runtime. They let you eliminate copy-paste SQL, enforce naming conventions, and encapsulate adapter-specific logic behind a single call.

### Anatomy of a Macro

```sql
-- macros/cents_to_dollars.sql
{% macro cents_to_dollars(column_name, scale=2) %}
    round({{ column_name }} / 100.0, {{ scale }})
{% endmacro %}
```

Call it from any model:

```sql
-- models/marts/fct_orders.sql
select
    order_id,
    {{ cents_to_dollars('amount_cents') }}         as amount_usd,
    {{ cents_to_dollars('shipping_cents', 4) }}    as shipping_usd
from {{ ref('stg_orders') }}
```

dbt compiles this to:

```sql
select
    order_id,
    round(amount_cents / 100.0, 2)    as amount_usd,
    round(shipping_cents / 100.0, 4)  as shipping_usd
from "analytics"."staging"."stg_orders"
```

### Control Flow with Jinja

Jinja's `if`, `for`, and `set` blocks are first-class inside macros:

```sql
-- macros/generate_surrogate_key.sql
{% macro surrogate_key(columns) %}
    md5(
        concat_ws('|',
            {% for col in columns %}
                coalesce(cast({{ col }} as varchar), '')
                {%- if not loop.last %}, {% endif %}
            {% endfor %}
        )
    )
{% endmacro %}
```

Usage:

```sql
select
    {{ surrogate_key(['order_id', 'line_item_id']) }} as sk_order_line,
    order_id,
    line_item_id
from {{ ref('stg_order_lines') }}
```

### The `run_query` Macro

Macros can execute SQL at *compile time* and use the results to drive further SQL generation. This is useful for dynamic column lists:

```sql
{% macro get_column_values(table, column) %}
    {% set query %}
        select distinct {{ column }}
        from {{ table }}
        order by 1
    {% endset %}
    {% set results = run_query(query) %}
    {% if execute %}
        {{ return(results.columns[0].values()) }}
    {% else %}
        {{ return([]) }}
    {% endif %}
{% endmacro %}
```

### Calling Macros from `dbt_project.yml` Hooks

Macros can be invoked as pre/post hooks or via `on-run-start` / `on-run-end`:

```yaml
# dbt_project.yml
on-run-end:
  - "{{ grant_select(database=target.database, schema=target.schema) }}"

models:
  my_project:
    marts:
      +post-hook:
        - "{{ audit_log(this, 'model_run') }}"
```

### Adapter Dispatch: Cross-Platform Macros

When a macro needs different SQL per warehouse, use `adapter.dispatch`:

```sql
-- macros/safe_divide.sql
{% macro safe_divide(numerator, denominator) %}
    {{ return(adapter.dispatch('safe_divide')(numerator, denominator)) }}
{% endmacro %}

{% macro default__safe_divide(numerator, denominator) %}
    case when {{ denominator }} = 0 then null
         else {{ numerator }} / {{ denominator }}
    end
{% endmacro %}

{% macro bigquery__safe_divide(numerator, denominator) %}
    safe_divide({{ numerator }}, {{ denominator }})
{% endmacro %}
```

dbt automatically selects the right implementation based on the active adapter — `bigquery__safe_divide` on BigQuery, `default__safe_divide` everywhere else. This is exactly how `dbt-utils` and `dbt-core` ship cross-platform macros.

### Key Built-in Variables & Context

| Variable | What it gives you |
|---|---|
| `{{ target.name }}` | The active target (`dev`, `prod`) |
| `{{ target.schema }}` | The target schema |
| `{{ this }}` | The current model's fully-qualified relation |
| `{{ model.name }}` | The current model's name |
| `{{ env_var('VAR') }}` | Read an environment variable at compile time |

### Best Practices

- **Keep macros small and single-purpose** — a macro that does one thing is easy to test and reuse.
- **Document with `{% docs %}` blocks** — macros can have their own doc entries, which appear in `dbt docs generate`.
- **Prefer `dbt-utils` macros before writing your own** — `generate_surrogate_key`, `star`, `pivot`, `union_relations` cover the most common patterns.
- **Guard with `{% if execute %}`** when using `run_query` — the macro runs twice (parse + execute); without the guard you'll get errors during parsing.
- **Test macros via models** — create a thin model in your `tests/` or a dedicated `integration_tests/` package that calls the macro with known inputs and assert with `dbt test`.

## 15:02 - dbt `generate_schema_name`: Controlling Where Models Land in Your Warehouse

One of the most misunderstood parts of dbt is how it decides which schema (or dataset, in BigQuery terms) a model materializes into. By default, dbt appends a custom schema suffix to your target schema — but this behavior is fully overridable via the `generate_schema_name` macro. Getting this right is essential for clean multi-environment setups.

### How Schema Resolution Works by Default

When you set a custom schema for a model:

```yaml
# dbt_project.yml
models:
  my_project:
    staging:
      +schema: staging
    marts:
      +schema: marts
```

dbt's default behavior produces:

| `target.schema` | `custom_schema` | Resulting schema |
|---|---|---|
| `dbt_alice` | `staging` | `dbt_alice_staging` |
| `dbt_alice` | `marts` | `dbt_alice_marts` |
| `analytics` (prod) | `staging` | `analytics_staging` |

The `_` concatenation keeps dev schemas isolated but creates ugly production names like `analytics_staging` instead of just `staging`.

### Overriding with `generate_schema_name`

Create `macros/generate_schema_name.sql` to override the default:

```sql
-- macros/generate_schema_name.sql
{% macro generate_schema_name(custom_schema_name, node) -%}
    {%- set default_schema = target.schema -%}

    {%- if custom_schema_name is none -%}
        {{ default_schema }}

    {%- elif target.name == 'prod' -%}
        {# In production, use the custom schema exactly as declared #}
        {{ custom_schema_name | trim }}

    {%- else -%}
        {# In dev/staging, prefix with developer's schema to avoid collisions #}
        {{ default_schema }}_{{ custom_schema_name | trim }}

    {%- endif -%}
{%- endmacro %}
```

Now results are:

| `target.name` | `target.schema` | `custom_schema` | Resulting schema |
|---|---|---|---|
| `dev` | `dbt_alice` | `staging` | `dbt_alice_staging` |
| `prod` | `analytics` | `staging` | `staging` |
| `prod` | `analytics` | `marts` | `marts` |

Production models land in clean, predictable schemas (`staging`, `marts`) while each developer gets their own prefixed namespace.

### Controlling the Database Too

For warehouses that support multiple databases (Snowflake, BigQuery projects), use `generate_database_name`:

```sql
-- macros/generate_database_name.sql
{% macro generate_database_name(custom_database_name, node) -%}
    {%- set default_database = target.database -%}

    {%- if custom_database_name is none -%}
        {{ default_database }}
    {%- elif target.name == 'prod' -%}
        {{ custom_database_name | trim }}
    {%- else -%}
        {{ default_database }}
    {%- endif -%}
{%- endmacro %}
```

Combined with model config:

```yaml
models:
  my_project:
    marts:
      finance:
        +database: finance_db
        +schema: reporting
```

In production this lands models in `finance_db.reporting`; in dev they fall into the developer's default database.

### Per-Model Schema Override

You can also set schema directly in a model's config block for one-off cases:

```sql
-- models/marts/fct_revenue.sql
{{ config(schema='finance') }}

select ...
```

### The `node` Argument: Per-Model Logic

The `node` argument gives you access to metadata about each model, enabling schema routing by layer or tag:

```sql
{% macro generate_schema_name(custom_schema_name, node) -%}
    {%- if 'pii' in (node.config.tags | default([])) -%}
        {# Route PII models to a restricted schema regardless of config #}
        {{ 'restricted_pii' }}
    {%- elif custom_schema_name is not none and target.name == 'prod' -%}
        {{ custom_schema_name | trim }}
    {%- else -%}
        {{ target.schema }}{% if custom_schema_name %}_{{ custom_schema_name | trim }}{% endif %}
    {%- endif -%}
{%- endmacro %}
```

### Best Practices

- **Define `generate_schema_name` in every project** — the default suffix behavior almost always needs tuning for production.
- **Use `target.name` not `target.schema`** to branch logic — schema names can vary but target names (`dev`, `prod`, `ci`) are explicit and reliable.
- **Never hardcode schema strings in SQL** — always use `{{ ref() }}` or `{{ source() }}`; dbt resolves the fully-qualified name automatically based on your macro.
- **Test in `--dry-run` mode** — run `dbt compile` and inspect `target/compiled/` to verify schemas resolve as expected before running.
- **Document your macro** — add a comment explaining the routing logic so the next engineer doesn't have to reverse-engineer it.

## 19:02 - dbt Snapshots: Tracking Slowly Changing Dimensions Over Time

dbt snapshots capture the historical state of mutable source tables by recording when rows change. They implement Type 2 Slowly Changing Dimensions (SCD2) automatically, giving you a full audit trail of how records looked at any point in time without complex CDC pipelines.

### The Problem Snapshots Solve

Most source systems only store the *current* state of a record. If a customer changes their plan from `starter` to `enterprise`, the old value is simply overwritten. Snapshots intercept that change before it disappears, creating a timestamped history row for every version of every record.

### Defining a Snapshot

Snapshots live in the `snapshots/` directory and use a `{% snapshot %}` block:

```sql
-- snapshots/snp_customers.sql
{% snapshot snp_customers %}

{{
    config(
        target_schema='snapshots',
        unique_key='customer_id',
        strategy='timestamp',
        updated_at='updated_at',
        invalidate_hard_deletes=True
    )
}}

select
    customer_id,
    name,
    plan,
    status,
    updated_at
from {{ source('crm', 'customers') }}

{% endsnapshot %}
```

Run with:
```bash
dbt snapshot
```

### Snapshot Strategies

**Timestamp strategy** — relies on an `updated_at` column maintained by the source system. Fast and precise; dbt only re-evaluates rows whose `updated_at` changed since the last snapshot run.

```yaml
strategy: timestamp
updated_at: updated_at
```

**Check strategy** — no `updated_at` column? dbt hashes the values of specified columns and treats any hash change as a modification. More flexible but scans the entire source table each run.

```yaml
strategy: check
check_cols: ['plan', 'status', 'email']   # or 'all' to check every column
```

### What dbt Adds to the Table

Every snapshot table gets four system columns:

| Column | Meaning |
|---|---|
| `dbt_scd_id` | Unique ID for each row version (hash of key + `dbt_updated_at`) |
| `dbt_updated_at` | When this version was written by dbt |
| `dbt_valid_from` | When this version became active |
| `dbt_valid_to` | When this version was superseded (`null` = current row) |

Query the current state:
```sql
select * from {{ ref('snp_customers') }}
where dbt_valid_to is null
```

Query the state at a specific point in time:
```sql
select * from {{ ref('snp_customers') }}
where '2025-06-01'::date between dbt_valid_from and coalesce(dbt_valid_to, current_date)
```

### Hard Deletes

By default, rows deleted from the source table are NOT invalidated in the snapshot — they persist as if still current. Set `invalidate_hard_deletes=True` to close out deleted rows:

```sql
config(
    invalidate_hard_deletes=True
)
```

With this flag, dbt runs a `LEFT JOIN` from the snapshot back to the source. Any snapshot row with no matching source row gets its `dbt_valid_to` set to the snapshot run time.

### Downstream Use in Marts

Reference snapshots with `{{ ref() }}` just like a model — build a mart that reconstructs SCD2 dimensions from snapshot history:

```sql
-- models/marts/dim_customers.sql
select
    dbt_scd_id          as customer_version_key,
    customer_id         as customer_key,
    name,
    plan,
    status,
    dbt_valid_from      as valid_from,
    dbt_valid_to        as valid_to,
    dbt_valid_to is null as is_current
from {{ ref('snp_customers') }}
```

### Best Practices

- **Run snapshots on a separate schedule** from models — typically more frequent (every hour) since they capture point-in-time state and gaps mean lost history.
- **Prefer `timestamp` over `check`** when possible — it's faster and produces cleaner history since it only tracks genuine source changes, not re-runs.
- **Snapshot close to the source** — always snapshot raw/source tables, not downstream models; models are re-buildable, lost history is not.
- **Keep snapshots narrow** — only include columns you actually need for history; extra columns increase check-strategy hash churn and bloat the table.
- **Document `dbt_valid_from` / `dbt_valid_to` semantics** in your YAML so downstream consumers know how to query current vs. historical records.
