## 00:02 - dbt Model Contracts: Enforcing Data Promises

dbt **model contracts** (introduced in dbt 1.5) let you declare the expected structure of a model's output and have dbt enforce it at build time. Think of them as a typed interface: downstream consumers can rely on specific columns, data types, and constraints existing — and dbt will fail the build loudly if the model violates the contract.

### Why Contracts Matter

Without contracts, a model's column list and types are implicit — they emerge from the SQL. A refactor that renames `customer_id` to `id` silently breaks every downstream model or BI tool referencing the old name. Contracts make that breakage explicit at CI time instead of in production.

### Enabling a Contract

Contracts live in your YAML config:

```yaml
# models/core/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true   # turns contract enforcement on
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key   # DDL constraint (warehouse-dependent)
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric
      - name: status
        data_type: varchar
```

When `enforced: true`, dbt:

1. Compares the declared columns + data types against the compiled SQL's output columns.
2. Fails **before** writing to the warehouse if there is a mismatch (wrong name, wrong type, missing column).
3. Applies DDL-level constraints (e.g., `NOT NULL`, `PRIMARY KEY`) where the warehouse supports them (Snowflake, Redshift, BigQuery, and others).

### Constraints vs. Tests

| Feature | Constraints | dbt Tests |
|---|---|---|
| Enforced by | Warehouse DDL | dbt test runner |
| Timing | At `dbt run` | At `dbt test` |
| Failure mode | Build fails | Test fails (separate step) |
| Types | `not_null`, `primary_key`, `foreign_key`, `unique`, `check` | Any generic/singular test |

Use **constraints** for structural guarantees the warehouse can enforce cheaply. Use **tests** for business-logic assertions (value ranges, referential integrity across sources, etc.).

### Contracts and Versioned Models

Contracts pair naturally with **model versions** (`v1`, `v2`). When you bump a model version, the old contract stays in place for consumers still on `v1`, and the new contract documents what changed in `v2`:

```yaml
models:
  - name: fct_orders
    versions:
      - v: 1
        config:
          contract:
            enforced: true
      - v: 2
        config:
          contract:
            enforced: true
    columns:
      - name: order_id
        data_type: bigint
```

### Best Practices

- **Start with `enforced: false`** on existing models to document columns without breaking builds, then flip to `true` once you have full column coverage.
- **Only list columns that are part of the public interface** — internal staging models rarely need contracts.
- **Pair contracts with model access** (`access: public`) to signal that a model is a stable, consumable asset.
- **Use `data_type` values your warehouse understands** — `varchar` in Snowflake, `STRING` in BigQuery, etc. dbt does not translate types.
- **Contracts catch refactors early** — run `dbt compile` or `dbt build` in CI to surface violations before merge.

Model contracts shift dbt from a transformation tool into a platform with explicit data interfaces, which is foundational for teams scaling to many models, consumers, and contributors.

## 05:02 - dbt Variables: Parameterizing Pipelines for Dev, Prod, and Backfills

dbt **variables** let you pass values into your project at runtime using `var()` in Jinja. They're the cleanest way to make a single model behave differently across environments, support ad-hoc backfills, or wire feature flags into your SQL — without touching the model code.

### Declaring and Using Variables

Define defaults in `dbt_project.yml`:

```yaml
# dbt_project.yml
vars:
  # Default to the last 30 days in dev; prod overrides via CLI or env
  lookback_days: 30
  target_schema: "analytics"
  enable_pii_columns: false
```

Reference them inside any model or macro with `var()`:

```sql
-- models/fct_events.sql
select
    event_id,
    user_id,
    {% if var('enable_pii_columns') %}
    email,
    ip_address,
    {% endif %}
    event_type,
    created_at
from {{ ref('stg_events') }}
where created_at >= dateadd('day', -{{ var('lookback_days') }}, current_date)
```

### Overriding Variables at Runtime

Pass variables via `--vars` on the CLI (YAML syntax):

```bash
# Full historical load in production
dbt run --vars '{"lookback_days": 365, "enable_pii_columns": true}'

# Backfill a specific date range
dbt run --select fct_events --vars '{"lookback_days": 90}'
```

Variables passed on the CLI always win over `dbt_project.yml` defaults — useful for CI jobs, one-off backfills, and environment promotion scripts.

### Environment-Aware Logic with `target`

Combine `var()` with `target.name` for environment-specific behaviour without CLI flags:

```sql
-- models/stg_orders.sql
{% set days_back = var('lookback_days', 30) %}

select *
from {{ source('raw', 'orders') }}
where
    {% if target.name == 'prod' %}
        -- In prod, use the variable (could be 365 for full refresh)
        created_at >= dateadd('day', -{{ days_back }}, current_date)
    {% else %}
        -- In dev/CI, always limit to 7 days to keep queries cheap
        created_at >= dateadd('day', -7, current_date)
    {% endif %}
```

### var() vs. env_var()

| Function | Source | Use case |
|---|---|---|
| `var('key')` | CLI `--vars` or `dbt_project.yml` | Model-level parameters, feature flags |
| `env_var('KEY')` | OS environment variable | Secrets, connection strings, CI tokens |

```yaml
# dbt_project.yml — pull a secret from the environment, never hardcode it
vars:
  api_key: "{{ env_var('MY_API_KEY') }}"
```

`env_var()` also accepts a default: `env_var('DEBUG', 'false')`.

### Practical Patterns

- **Dev speed limit**: default `lookback_days: 7` in `dbt_project.yml`; override to `365` in prod CI.
- **Schema routing**: `var('target_schema', target.schema)` lets orchestrators inject a custom schema without changing model config.
- **Feature flags**: wrap unreleased columns in `{% if var('feature_new_revenue_calc', false) %}` — ship the model, enable the flag when ready.
- **Backfill windows**: instead of modifying SQL, run `dbt run --vars '{"start_date": "2024-01-01", "end_date": "2024-06-30"}'` and reference `var('start_date')` in a date filter.

Variables are a lightweight but powerful contract between your orchestration layer and your dbt project — they keep environment logic out of source control and make pipelines trivially parameterizable.
