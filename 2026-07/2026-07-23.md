## 00:03 - dbt Semantic Layer & MetricFlow: Define Metrics Once, Query Them Everywhere

The **dbt Semantic Layer** (powered by MetricFlow) lets you define business metrics in your dbt project and expose them through a consistent API — so every downstream tool (BI dashboards, notebooks, ad-hoc SQL) gets the same numbers, always.

### Why It Matters

Without a semantic layer, "monthly revenue" might be computed differently in Looker, Tableau, and a Jupyter notebook. MetricFlow makes the metric definition the single source of truth: one definition, many consumers.

### Core Concepts

**Semantic Models** describe a dbt model (or source) as a logical table with *entities*, *dimensions*, and *measures*:

```yaml
# models/semantic/fct_orders.yml
semantic_models:
  - name: orders
    model: ref('fct_orders')
    entities:
      - name: order
        type: primary
        expr: order_id
      - name: customer
        type: foreign
        expr: customer_id
    dimensions:
      - name: order_date
        type: time
        type_params:
          time_granularity: day
      - name: status
        type: categorical
    measures:
      - name: order_total
        agg: sum
        expr: amount
      - name: order_count
        agg: count_distinct
        expr: order_id
```

**Metrics** are derived from measures and can be simple, ratio, cumulative, or derived:

```yaml
metrics:
  - name: revenue
    type: simple
    label: "Total Revenue"
    type_params:
      measure: order_total

  - name: average_order_value
    type: ratio
    label: "Average Order Value"
    type_params:
      numerator: order_total
      denominator: order_count

  - name: cumulative_revenue
    type: cumulative
    label: "Cumulative Revenue (All Time)"
    type_params:
      measure: order_total
      window: unbounded
```

### Querying with the MetricFlow CLI

```bash
# List available metrics
mf list metrics

# Query revenue by month
mf query --metrics revenue --group-by metric_time__month

# Query AOV broken down by order status
mf query --metrics average_order_value --group-by status --order -metric_time__month
```

### Dimension Slicing with Entities

Because entities link semantic models, MetricFlow can automatically join across models. Defining `customer` as a foreign entity on the orders model means you can query `revenue` sliced by a `customer_segment` dimension that lives in a separate `customers` semantic model — no manual SQL join required.

### Key Takeaways

- Define metrics in YAML once; all BI tools consume the same definition.
- Semantic models describe *what the data means* (entities, measures, dimensions), not how to compute it.
- MetricFlow auto-generates optimized SQL at query time, including any necessary joins.
- Available in **dbt Cloud** via the Semantic Layer API; `mf` CLI works with dbt Core locally.
- Pair with **Saved Queries** (`saved_queries:` block) to pre-define common metric+dimension combos for faster BI tool integration.

## 10:02 - dbt Snapshots: Tracking Historical Changes with Type 2 SCDs

**Snapshots** are dbt's built-in mechanism for capturing row-level history over time. When a source record changes (a customer's email, an order's status, an employee's department), a snapshot records both the old and new versions with validity timestamps — creating a **Type 2 Slowly Changing Dimension (SCD2)** without any manual bookkeeping.

### When to Use Snapshots

Snapshots are the right tool when you need to answer questions like:
- "What was this customer's plan tier last quarter?"
- "Which account manager owned this deal when it was closed?"
- "What price did this product have on a specific date?"

Without snapshots, you only know the current state. With snapshots, you get the full history.

### Snapshot File Structure

Snapshots live in the `snapshots/` directory and use a `{% snapshot %}` block:

```sql
-- snapshots/snap_customers.sql
{% snapshot snap_customers %}

{{
    config(
        target_schema='snapshots',
        unique_key='customer_id',
        strategy='timestamp',
        updated_at='updated_at'
    )
}}

select
    customer_id,
    email,
    plan_tier,
    account_manager_id,
    updated_at
from {{ source('raw', 'customers') }}

{% endsnapshot %}
```

Running `dbt snapshot` adds four metadata columns to the snapshot table:

| Column | Meaning |
|---|---|
| `dbt_scd_id` | Unique surrogate key per historical row |
| `dbt_updated_at` | Timestamp when dbt processed the row |
| `dbt_valid_from` | When this version of the row became active |
| `dbt_valid_to` | When it was superseded (`NULL` = current record) |

### Two Snapshot Strategies

**`timestamp`** — dbt compares `updated_at` to detect changes. Requires a reliable `updated_at` column; efficient and precise.

```sql
config(
    strategy='timestamp',
    updated_at='updated_at'
)
```

**`check`** — dbt hashes the specified columns and flags a change if the hash differs. Useful when source tables have no reliable `updated_at`:

```sql
config(
    strategy='check',
    check_cols=['email', 'plan_tier', 'account_manager_id']
    -- or check_cols='all' to hash every column
)
```

### Querying Snapshot History

```sql
-- Current state of all customers (open records)
select * from {{ ref('snap_customers') }}
where dbt_valid_to is null;

-- Point-in-time: what did customer 42 look like on 2025-06-01?
select *
from {{ ref('snap_customers') }}
where customer_id = 42
  and dbt_valid_from <= '2025-06-01'
  and (dbt_valid_to > '2025-06-01' or dbt_valid_to is null);

-- Full history for a customer (all versions, newest first)
select *
from {{ ref('snap_customers') }}
where customer_id = 42
order by dbt_valid_from desc;
```

### Joining Snapshots to Fact Tables

The real power comes from joining snapshot history to facts at the time of the event:

```sql
-- Which plan tier was the customer on when they made each purchase?
select
    o.order_id,
    o.customer_id,
    o.order_date,
    o.amount,
    c.plan_tier   -- tier at the time of the order
from {{ ref('fct_orders') }} o
left join {{ ref('snap_customers') }} c
    on  o.customer_id = c.customer_id
    and o.order_date >= c.dbt_valid_from
    and (o.order_date < c.dbt_valid_to or c.dbt_valid_to is null)
```

### Configuration Best Practices

```yaml
# dbt_project.yml
snapshots:
  my_project:
    +target_schema: snapshots       # isolate from marts
    +invalidate_hard_deletes: true  # mark deleted source rows as expired
```

`invalidate_hard_deletes: true` is critical: if a row disappears from the source, dbt will close its open snapshot record (set `dbt_valid_to`) rather than leaving it permanently "current".

### Key Takeaways

- Run `dbt snapshot` on a schedule (e.g. every hour or nightly) — the more often you snapshot, the finer your historical granularity.
- Snapshots are append-only by design; dbt never updates or deletes historical rows.
- Use `timestamp` strategy when `updated_at` is trustworthy; fall back to `check` for sources without a reliable watermark.
- Always set `invalidate_hard_deletes: true` so soft-deleted source records don't stay "active" forever in your snapshot table.
- Pair snapshots with `dbt_valid_to is null` filters in downstream dimension models to serve the current state efficiently.

## 05:03 - dbt Unit Tests: Asserting SQL Logic Without Real Data

Introduced in **dbt 1.8**, unit tests let you validate the SQL transformation logic inside a model using hand-crafted input fixtures — no warehouse rows required. Think of them as the dbt equivalent of a Python `unittest`: you control the inputs and assert the exact outputs.

### Why Unit Tests vs. Data Tests?

| | **Data tests** (`dbt test`) | **Unit tests** (`dbt test --select`) |
|---|---|---|
| Input | Real warehouse data | Mocked fixture rows you define |
| Validates | Data quality (nulls, uniqueness, referential integrity) | SQL logic (calculations, CASE logic, joins, edge cases) |
| Speed | Slow (full table scans) | Fast (tiny in-memory datasets) |
| Catches | Bad data | Bad code |

Both are complementary — use data tests for quality assurance and unit tests for logic correctness.

### Anatomy of a Unit Test

Unit tests live in YAML alongside your model (or in a separate `tests/` file). The `unit_tests:` key is a top-level sibling of `models:`.

```yaml
# models/intermediate/int_orders.yml
unit_tests:
  - name: test_discount_is_applied_correctly
    model: int_orders
    given:
      - input: ref('stg_orders')
        rows:
          - { order_id: 1, subtotal: 100.00, discount_pct: 0.10 }
          - { order_id: 2, subtotal: 200.00, discount_pct: 0.00 }
          - { order_id: 3, subtotal: 50.00,  discount_pct: 0.25 }
    expect:
      rows:
        - { order_id: 1, final_amount: 90.00  }
        - { order_id: 2, final_amount: 200.00 }
        - { order_id: 3, final_amount: 37.50  }
```

dbt injects your `given` rows as CTEs, compiles the model SQL against them, and compares every row against `expect`. The test fails if any row doesn't match (by value and count).

### Mocking Sources & Seeds

You can also mock `source()` references and `ref()` calls to seeds:

```yaml
given:
  - input: source('raw', 'payments')   # mocks a source, not a model
    rows:
      - { payment_id: 'p1', amount: 50.00, status: 'completed' }
  - input: ref('seed_exchange_rates')  # mocks a seed
    rows:
      - { currency: 'EUR', rate: 1.08 }
```

### Overriding `this` for Incremental Models

Incremental models reference `this` (the existing table). Override it in unit tests so you can verify the merge/append logic:

```yaml
overrides:
  macros:
    is_incremental: true   # force incremental code path
  vars:
    execution_date: '2024-06-01'
given:
  - input: this            # rows already in the target table
    rows:
      - { event_id: 'e1', processed_at: '2024-05-31' }
  - input: ref('stg_events')   # new batch arriving
    rows:
      - { event_id: 'e1', processed_at: '2024-05-31' }  # duplicate — should be deduped
      - { event_id: 'e2', processed_at: '2024-06-01' }  # new — should be inserted
expect:
  rows:
    - { event_id: 'e1', processed_at: '2024-05-31' }
    - { event_id: 'e2', processed_at: '2024-06-01' }
```

### Running Unit Tests

```bash
# Run all unit tests
dbt test --select "test_type:unit"

# Run unit tests for a specific model only
dbt test --select "int_orders,test_type:unit"

# Run both unit and data tests together
dbt test --select "int_orders"
```

### Key Takeaways

- Unit tests validate **logic**, not data — they're ideal for complex CASE statements, date math, conditional joins, and business rules.
- Fixture rows are tiny, so unit tests run in seconds even against a live warehouse.
- The `overrides:` block lets you test incremental paths, variable-dependent branches, and macro flags without changing model code.
- Add unit tests to CI alongside `dbt build` to catch SQL regressions before they hit production.
