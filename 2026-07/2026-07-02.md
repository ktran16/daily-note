## 00:02 - dbt Sources & Freshness: Declaring Raw Data and Monitoring Pipeline SLAs

**Sources** are how dbt models the boundary between raw ingested data and your transformation layer. Instead of hard-coding raw table references in SQL, you declare them in YAML and reference them with `source()` — giving you lineage, documentation, and freshness checks in one place.

### Declaring Sources

```yaml
# models/staging/_sources.yml
version: 2

sources:
  - name: stripe
    database: raw
    schema: stripe
    loaded_at_field: _etl_loaded_at   # column dbt uses to check freshness
    freshness:
      warn_after: { count: 6, period: hour }
      error_after: { count: 24, period: hour }
    tables:
      - name: charges
        description: "Raw charges from Stripe webhook events"
        columns:
          - name: id
            tests:
              - unique
              - not_null
      - name: customers
        freshness:
          warn_after: { count: 12, period: hour }  # table-level override
```

### Referencing Sources in Models

```sql
-- models/staging/stg_stripe__charges.sql
select
    id                   as charge_id,
    amount / 100.0       as amount_usd,
    currency,
    status,
    created              as charged_at,
    customer_id
from {{ source('stripe', 'charges') }}
where status != 'failed'
```

Using `source()` instead of `ref()` or a raw `raw.stripe.charges` reference is critical — it:
- Generates lineage edges from raw tables into your DAG
- Makes the source appear in `dbt docs` with documentation and test status
- Enables freshness checks

### Running Freshness Checks

```bash
# Check all sources
dbt source freshness

# Check a specific source
dbt source freshness --select source:stripe

# Check only sources upstream of specific models
dbt source freshness --select source:stripe.charges+
```

dbt queries `max(loaded_at_field)` from each source table and compares it to the current timestamp. Results are written to `target/sources.json` for downstream use (e.g., in dbt Cloud or custom alerting).

### Why This Matters

Freshness failures catch broken ingestion pipelines **before** your transformations run, preventing stale dashboards or silent data gaps. Pair freshness checks with `dbt build` in CI so a late-arriving source blocks the downstream model run:

```bash
# In CI pipeline
dbt source freshness && dbt build
```

**Best practice:** define `loaded_at_field` at the source level and override `freshness` thresholds at the table level only when SLAs differ. Use the `warn_after` / `error_after` split to give your on-call team a grace window before a hard failure alerts.
