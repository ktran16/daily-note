## 00:02 - dbt Sources & Freshness: Declaring Raw Data and Monitoring Pipeline SLAs

**Sources** are how dbt models the boundary between raw ingested data and your transformation layer. Instead of hard-coding raw table references in SQL, you declare them in YAML and reference them with `source()` — giving you lineage, documentation, and freshness checks in one place.

### Declaring Sources

```yaml
# models/staging/_sources.yml
version: 2

sources:
  - name: stripe
    database: raw
    schema: stripe
    loaded_at_field: _etl_loaded_at   # column dbt uses to check freshness
    freshness:
      warn_after: { count: 6, period: hour }
      error_after: { count: 24, period: hour }
    tables:
      - name: charges
        description: "Raw charges from Stripe webhook events"
        columns:
          - name: id
            tests:
              - unique
              - not_null
      - name: customers
        freshness:
          warn_after: { count: 12, period: hour }  # table-level override
```

### Referencing Sources in Models

```sql
-- models/staging/stg_stripe__charges.sql
select
    id                   as charge_id,
    amount / 100.0       as amount_usd,
    currency,
    status,
    created              as charged_at,
    customer_id
from {{ source('stripe', 'charges') }}
where status != 'failed'
```

Using `source()` instead of `ref()` or a raw `raw.stripe.charges` reference is critical — it:
- Generates lineage edges from raw tables into your DAG
- Makes the source appear in `dbt docs` with documentation and test status
- Enables freshness checks

### Running Freshness Checks

```bash
# Check all sources
dbt source freshness

# Check a specific source
dbt source freshness --select source:stripe

# Check only sources upstream of specific models
dbt source freshness --select source:stripe.charges+
```

dbt queries `max(loaded_at_field)` from each source table and compares it to the current timestamp. Results are written to `target/sources.json` for downstream use (e.g., in dbt Cloud or custom alerting).

### Why This Matters

Freshness failures catch broken ingestion pipelines **before** your transformations run, preventing stale dashboards or silent data gaps. Pair freshness checks with `dbt build` in CI so a late-arriving source blocks the downstream model run:

```bash
# In CI pipeline
dbt source freshness && dbt build
```

**Best practice:** define `loaded_at_field` at the source level and override `freshness` thresholds at the table level only when SLAs differ. Use the `warn_after` / `error_after` split to give your on-call team a grace window before a hard failure alerts.

## 05:02 - dbt Model Contracts: Enforcing Schema Stability Across Teams

**Model contracts** (introduced in dbt 1.5) let you declare a model's expected output schema as a binding agreement — dbt will fail the build if the actual query result doesn't match the declared columns and data types. This turns an implicit "please don't break this" into a hard compile-time guarantee.

### Why Contracts Matter

Without contracts, a column rename or type change in an upstream model silently breaks downstream consumers. Contracts make schema drift a build failure, not a surprise in production.

### Enabling a Contract

```yaml
# models/marts/fct_orders.yml
version: 2

models:
  - name: fct_orders
    config:
      contract:
        enforced: true          # activate the contract
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key   # informational on most adapters
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount_usd
        data_type: numeric
      - name: status
        data_type: varchar
```

When `enforced: true`, dbt compares the compiled SQL's output columns against this YAML at **parse time** (not run time on most adapters). Any mismatch — wrong name, wrong type, missing column — fails before SQL is executed.

### Constraints

Constraints add database-level (or informational) rules on top of the type check:

```yaml
constraints:
  - type: not_null
  - type: unique
  - type: foreign_key
    to: ref('dim_customers')
    to_columns: [customer_id]
  - type: check
    expression: "total_amount_usd >= 0"
```

Support varies by adapter: Snowflake, BigQuery, Redshift, and Postgres each enforce different subsets. Check the dbt docs for your adapter's constraint support matrix.

### Versioned Models + Contracts

Contracts shine alongside **model versioning**:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1
        config:
          contract:
            enforced: true
      - v: 2
        config:
          contract:
            enforced: true
```

Downstream models reference `ref('fct_orders', version=1)` until they migrate. The contract guarantees the v1 interface stays stable even as v2 evolves.

### Best Practice

Apply contracts to **public models** — those consumed by other teams, BI tools, or reverse-ETL. Leave internal staging/intermediate models contract-free for flexibility. Pair contracts with `access: public` in model config to signal the boundary clearly:

```yaml
config:
  access: public
  contract:
    enforced: true
```

This combination is the cornerstone of dbt's **multi-project** and **mesh** architecture, where each project exposes a stable, typed API surface to others.
