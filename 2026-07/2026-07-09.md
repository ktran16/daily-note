## 00:03 - dbt Snapshots: Tracking Slowly Changing Dimensions (SCD Type 2)

Most tables in your warehouse only reflect the *current* state of the world. But sometimes you need a full history â€” "what was this customer's subscription tier last month?" That's where **dbt snapshots** come in. They implement **Slowly Changing Dimension Type 2 (SCD2)**: when a row changes, dbt closes the old version and inserts a new one, so you retain the full historical record.

### How Snapshots Work

Snapshots live in a `snapshots/` directory and use the `{% snapshot %}` Jinja block. dbt adds four system columns to the target table:

| Column | Purpose |
|---|---|
| `dbt_scd_id` | Unique key for each snapshot record |
| `dbt_updated_at` | Timestamp when the row was last evaluated |
| `dbt_valid_from` | When this version became active |
| `dbt_valid_to` | When this version was superseded (`NULL` = current) |

### Timestamp Strategy (Most Common)

Use this when your source table has a reliable `updated_at` column:

```sql
{% snapshot orders_snapshot %}

{{
  config(
    target_schema='snapshots',
    unique_key='order_id',
    strategy='timestamp',
    updated_at='updated_at',
    invalidate_hard_deletes=True
  )
}}

select * from {{ source('raw', 'orders') }}

{% endsnapshot %}
```

### Check Strategy (Fallback)

Use this when there's no `updated_at` column â€” dbt hashes the specified columns to detect changes:

```sql
{% snapshot customers_snapshot %}

{{
  config(
    target_schema='snapshots',
    unique_key='customer_id',
    strategy='check',
    check_cols=['email', 'plan', 'status']
  )
}}

select * from {{ source('raw', 'customers') }}

{% endsnapshot %}
```

### Querying Snapshot History

To get point-in-time state, filter on `dbt_valid_from` and `dbt_valid_to`:

```sql
-- What plan was customer 42 on during Q1 2026?
select plan
from snapshots.customers_snapshot
where customer_id = 42
  and dbt_valid_from <= '2026-03-31'
  and (dbt_valid_to > '2026-03-31' or dbt_valid_to is null)
```

To get only current rows, use:

```sql
where dbt_valid_to is null
```

### Key Best Practices

- **Run snapshots before your models** â€” schedule them first in your DAG so downstream models see the latest snapshot state.
- **Prefer `timestamp` over `check`** â€” it's more reliable and performant since it avoids full-row hashing.
- **Set `invalidate_hard_deletes=True`** if rows can be deleted from the source; otherwise deleted records stay open indefinitely.
- **Don't transform in the snapshot** â€” keep the snapshot select simple and do joins/renames in a downstream staging model that reads from it.
- **Snapshot raw sources, not staging models** â€” you want the unmodified source data captured; transformations applied later won't retroactively change history.

Snapshots are one of dbt's most powerful features for analytics engineering â€” they turn append-only logic into a first-class pattern without any custom orchestration code.
