## 00:03 - dbt Snapshots: Tracking Slowly Changing Dimensions (SCD Type 2)

Most tables in your warehouse only reflect the *current* state of the world. But sometimes you need a full history — "what was this customer's subscription tier last month?" That's where **dbt snapshots** come in. They implement **Slowly Changing Dimension Type 2 (SCD2)**: when a row changes, dbt closes the old version and inserts a new one, so you retain the full historical record.

### How Snapshots Work

Snapshots live in a `snapshots/` directory and use the `{% snapshot %}` Jinja block. dbt adds four system columns to the target table:

| Column | Purpose |
|---|---|
| `dbt_scd_id` | Unique key for each snapshot record |
| `dbt_updated_at` | Timestamp when the row was last evaluated |
| `dbt_valid_from` | When this version became active |
| `dbt_valid_to` | When this version was superseded (`NULL` = current) |

### Timestamp Strategy (Most Common)

Use this when your source table has a reliable `updated_at` column:

```sql
{% snapshot orders_snapshot %}

{{
  config(
    target_schema='snapshots',
    unique_key='order_id',
    strategy='timestamp',
    updated_at='updated_at',
    invalidate_hard_deletes=True
  )
}}

select * from {{ source('raw', 'orders') }}

{% endsnapshot %}
```

### Check Strategy (Fallback)

Use this when there's no `updated_at` column — dbt hashes the specified columns to detect changes:

```sql
{% snapshot customers_snapshot %}

{{
  config(
    target_schema='snapshots',
    unique_key='customer_id',
    strategy='check',
    check_cols=['email', 'plan', 'status']
  )
}}

select * from {{ source('raw', 'customers') }}

{% endsnapshot %}
```

### Querying Snapshot History

To get point-in-time state, filter on `dbt_valid_from` and `dbt_valid_to`:

```sql
-- What plan was customer 42 on during Q1 2026?
select plan
from snapshots.customers_snapshot
where customer_id = 42
  and dbt_valid_from <= '2026-03-31'
  and (dbt_valid_to > '2026-03-31' or dbt_valid_to is null)
```

To get only current rows, use:

```sql
where dbt_valid_to is null
```

### Key Best Practices

- **Run snapshots before your models** — schedule them first in your DAG so downstream models see the latest snapshot state.
- **Prefer `timestamp` over `check`** — it's more reliable and performant since it avoids full-row hashing.
- **Set `invalidate_hard_deletes=True`** if rows can be deleted from the source; otherwise deleted records stay open indefinitely.
- **Don't transform in the snapshot** — keep the snapshot select simple and do joins/renames in a downstream staging model that reads from it.
- **Snapshot raw sources, not staging models** — you want the unmodified source data captured; transformations applied later won't retroactively change history.

Snapshots are one of dbt's most powerful features for analytics engineering — they turn append-only logic into a first-class pattern without any custom orchestration code.

## 05:02 - dbt Model Contracts: Enforcing Schema Guarantees

As dbt projects grow, models become shared infrastructure consumed by many downstream users. **Model contracts** (introduced in dbt 1.5) let you declare an explicit, enforced schema for a model — column names, data types, and constraints — so that breaking changes are caught at build time rather than silently corrupting downstream queries.

### Why Contracts Matter

Without contracts, renaming a column or changing its type succeeds silently in the model but breaks every downstream model, dashboard, or BI tool that depends on it. Contracts turn those runtime surprises into explicit build failures.

### Enabling a Contract

Add a `contract` block to the model's config in `schema.yml`:

```yaml
models:
  - name: dim_customers
    config:
      contract:
        enforced: true
    columns:
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: email
        data_type: varchar
        constraints:
          - type: not_null
      - name: created_at
        data_type: timestamp
      - name: lifetime_value
        data_type: numeric
```

When `enforced: true`, dbt will:
1. Verify that every column in the contract exists in the model's SELECT output.
2. Check that data types match (adapter-dependent — some databases cast implicitly).
3. Apply any declared constraints (e.g., `NOT NULL`, `PRIMARY KEY`) as database-level constraints where supported.

### Supported Constraints

| Constraint | Notes |
|---|---|
| `not_null` | Enforced at the DB level where possible |
| `unique` | Applied as a DB unique constraint |
| `primary_key` | Implies uniqueness + not null |
| `foreign_key` | Declares referential integrity (DB support varies) |
| `check` | Custom SQL expression (e.g., `amount > 0`) |

### Contract + Versioning

Contracts pair naturally with **model versioning** (also dbt 1.5+). You can version a model while maintaining the same contract across versions — useful when refactoring a canonical model without breaking consumers:

```yaml
models:
  - name: dim_customers
    latest_version: 2
    versions:
      - v: 1
      - v: 2
    config:
      contract:
        enforced: true
```

### Materializations That Support Contracts

Contracts only work with **table** and **incremental** materializations — not `view` or `ephemeral`, because those don't write a physical schema that can be enforced. If you set `enforced: true` on a view, dbt will raise an error.

### Best Practices

- **Contract your "interface" models** — marts, dimension tables, and fact tables that external teams or tools consume; don't contract every intermediate staging model.
- **Define explicit `data_type` for every column** — dbt requires this when contracts are enforced, so it's a good forcing function for documentation hygiene.
- **Treat a contract change like a breaking API change** — bump the model version, communicate to consumers, and migrate deliberately.
- **Use contracts with `store_failures`** — pairing them with dbt tests and `store_failures: true` gives you both build-time and data-quality enforcement in one workflow.

Model contracts transform dbt models from loosely defined SQL files into first-class data interfaces with enforceable guarantees — essential for scaling analytics engineering across multiple teams.
