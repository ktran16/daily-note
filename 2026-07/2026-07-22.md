## 00:03 - dbt Access & Groups: Enforcing Boundaries Within Your dbt Project

Introduced in dbt 1.5, **model access** and **groups** let you declare ownership boundaries and restrict which models other developers (or projects) can reference. This is the foundation of dbt Mesh — but it's just as valuable inside a single project with multiple teams sharing one repo.

### The Three Access Levels

```yaml
# models/staging/stg_orders.yml
models:
  - name: stg_orders
    access: private   # only models in the same group can ref() this
  - name: int_order_items
    access: protected # default — any model in this project can ref() it
  - name: fct_orders
    access: public    # other dbt projects (via dbt Mesh) can ref() this
```

| Level       | Who can `ref()` it                              |
|-------------|--------------------------------------------------|
| `private`   | Only models in the **same group**               |
| `protected` | Any model in the **same project** (default)     |
| `public`    | Any model in **any project** (cross-project ref)|

### Defining Groups

Groups pair with access to define clear ownership. Each group has an `owner` field for accountability:

```yaml
# models/finance/finance_group.yml
groups:
  - name: finance
    owner:
      name: Finance Analytics Team
      email: finance-analytics@company.com
```

Then assign models to that group:

```yaml
models:
  - name: stg_invoices
    group: finance
    access: private   # only finance group models can ref this

  - name: fct_revenue
    group: finance
    access: public    # shared with other projects / teams
```

### Why This Matters in Practice

Without access controls, a staging model (`stg_orders`) built for internal cleanup might get `ref()`-ed by 15 downstream models across teams. When you need to change it, every downstream model breaks.

With `access: private`, dbt will **raise a compile-time error** if a model outside the group tries to reference it:

```
Error: Model 'analytics.fct_daily_summary' attempted to ref
private model 'finance.stg_invoices'.
```

This forces teams to consume only your stable, public interfaces (`fct_*` or `dim_*` models) rather than your internal implementation details.

### Best Practice Pattern

```
models/
  finance/
    _finance_group.yml     # defines the group + owner
    staging/
      stg_invoices.sql     # access: private
    intermediate/
      int_invoice_agg.sql  # access: private
    marts/
      fct_revenue.sql      # access: public   ← the stable contract
```

Groups + access give teams the freedom to refactor internals without fear — only the public surface is a contract. For dbt Mesh setups, public models also need `contract: {enforced: true}` so cross-project consumers get a schema guarantee alongside the access grant.

## 05:03 - dbt Unit Tests: Validating SQL Logic with Mock Data (dbt 1.8+)

dbt 1.8 introduced **unit tests** — a way to assert that your model's SQL logic is correct by providing mock inputs and expected outputs, all without touching real warehouse data. Unlike data tests (which validate rows in a built model), unit tests validate the *transformation logic itself*, making them fast, warehouse-free, and runnable in CI before any data is loaded.

### Why Unit Tests?

| | Data Tests (`dbt test`) | Unit Tests |
|---|---|---|
| What they validate | Data quality in built tables | SQL transformation logic |
| Need real data? | Yes | No — uses mocks |
| Speed | Slow (runs queries on full tables) | Fast (compiles to a CTE) |
| When they catch bugs | After `dbt run` | Before `dbt run` |

### Anatomy of a Unit Test

Unit tests live in `.yml` files alongside your models. You declare the model under test, mock inputs for its upstream `ref()` and `source()` calls, then specify the rows you expect the model to produce:

```yaml
# models/marts/fct_orders.yml
unit_tests:
  - name: test_discount_applied_correctly
    description: "Orders with a coupon should have discount subtracted from total."
    model: fct_orders
    given:
      - input: ref('stg_orders')
        rows:
          - {order_id: 1, subtotal: 100, coupon_discount: 10, status: 'shipped'}
          - {order_id: 2, subtotal: 200, coupon_discount: 0,  status: 'pending'}
      - input: ref('stg_customers')
        rows:
          - {customer_id: 42, tier: 'gold'}
    expect:
      rows:
        - {order_id: 1, order_total: 90,  status: 'shipped'}
        - {order_id: 2, order_total: 200, status: 'pending'}
```

dbt compiles the model's SQL with mock rows substituted for the real `ref()` calls, runs the resulting query in a lightweight CTE, then compares the output to `expect.rows`. A test fails when the output doesn't match — including extra or missing rows.

### Mocking Sources

Use the same `input:` syntax for `source()` references:

```yaml
given:
  - input: source('raw_shopify', 'orders')
    rows:
      - {id: 99, customer_id: 1, total_price: 5000, created_at: '2025-01-15'}
```

### Overriding Variables and the Incremental Flag

Unit tests can override `var()` values and macro behavior to exercise branches that are otherwise hard to isolate:

```yaml
unit_tests:
  - name: test_incremental_filter_excludes_old_events
    model: fct_events
    overrides:
      macros:
        is_incremental: true    # force the {% if is_incremental() %} branch
      vars:
        cutoff_days: 7
    given:
      - input: ref('stg_events')
        rows:
          - {event_id: 1, occurred_at: '2025-06-01'}  # old — should be excluded
          - {event_id: 2, occurred_at: '2025-07-20'}  # recent — should pass through
    expect:
      rows:
        - {event_id: 2, occurred_at: '2025-07-20'}
```

This is the primary reason to adopt unit tests for incremental models: the `{% if is_incremental() %}` block only executes on subsequent runs, making it untestable with regular data tests.

### Running Unit Tests

```bash
# Run only unit tests (skip data tests)
dbt test --select "test_type:unit"

# Run unit tests for one model
dbt test --select fct_orders,test_type:unit

# dbt build runs unit tests BEFORE the model materializes
dbt build --select fct_orders
```

With `dbt build`, unit tests act as a pre-build gate: if a unit test fails, the model never runs. This makes logic bugs surface at compile time, not after bad data has already propagated downstream.

### Best Practices

- **Test one business rule per test** — a failing test should immediately name the broken invariant.
- **Keep mocks minimal** — only include the columns your model's SQL actually references.
- **Cover edge cases**: nulls, zero values, empty joins, status boundary values.
- **Pair with data tests** — unit tests catch logic bugs; data tests catch data quality issues in real data. Both are necessary for a robust pipeline.

Unit tests shift dbt's quality assurance left, catching transformation errors at the authoring stage rather than discovering them after a production run has already materialized bad data into downstream models.
