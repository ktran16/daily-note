## 00:12 - dbt Tests: Generic vs Singular Data Quality Checks

Data pipelines break silently — a bad join doubles rows, a source drops a column, a NULL slips into a primary key. **dbt tests** are executable assertions that catch these issues at build time, before broken data reaches dashboards or downstream consumers.

dbt has two test categories: **generic tests** (reusable, configured in YAML) and **singular tests** (custom SQL files).

### Generic Tests

Generic tests are applied in `schema.yml` and run against model columns or tables. dbt ships four out of the box:

```yaml
models:
  - name: orders
    columns:
      - name: order_id
        tests:
          - not_null
          - unique
      - name: status
        tests:
          - accepted_values:
              values: ['placed', 'shipped', 'delivered', 'returned']
      - name: customer_id
        tests:
          - relationships:
              to: ref('customers')
              field: customer_id
```

| Test | What it checks |
|---|---|
| `not_null` | No NULLs in the column |
| `unique` | All values are distinct |
| `accepted_values` | Every value is in the allowed list |
| `relationships` | Every FK value exists in the referenced model |

`dbt-utils` and `dbt-expectations` packages extend this library with dozens more — `expect_column_values_to_be_between`, `expression_is_true`, `recency`, `not_constant`, and so on.

### Singular Tests

Singular tests are plain SQL files in the `tests/` directory. A test **fails if the query returns any rows** — so you write a SELECT that finds the bad data:

```sql
-- tests/assert_orders_total_is_positive.sql
select order_id, total_amount
from {{ ref('orders') }}
where total_amount <= 0
```

```sql
-- tests/assert_no_orphan_order_items.sql
-- Every order_item must have a matching order
select oi.order_item_id
from {{ ref('order_items') }} oi
left join {{ ref('orders') }} o using (order_id)
where o.order_id is null
```

Singular tests are ideal for business rules too complex to express as a generic test — multi-column constraints, cross-model assertions, or aggregate checks.

### Severity and Thresholds

By default a failed test blocks the run. You can soften this with `severity` and `error_if` / `warn_if` thresholds:

```yaml
- name: order_id
  tests:
    - not_null:
        severity: warn        # warn instead of error
    - unique:
        error_if: ">10"       # only error if >10 duplicates
        warn_if: ">0"         # warn on any duplicates
```

### Storing Failures for Debugging

Add `store_failures: true` to a test (or globally in `dbt_project.yml`) to write failing rows to a table in your warehouse:

```yaml
tests:
  - unique:
      store_failures: true
      schema: dbt_test_failures
```

dbt creates `dbt_test_failures.orders_order_id_unique` as a table you can query to inspect the exact rows that failed — far more useful than a count.

### Running Tests Selectively

```bash
# All tests
dbt test

# Tests for one model
dbt test --select orders

# Tests for a model and all its upstream dependencies
dbt test --select +orders

# Only tests tagged "critical"
dbt test --select tag:critical
```

### Best Practices

- **Test at sources too** — apply `not_null` and `unique` on source columns in `sources.yml`; raw data breaks differently than transformed data.
- **Tag critical tests** so CI can run a fast subset on every PR and a full suite on schedule.
- **Treat `warn` carefully** — warnings that never get fixed become noise; escalate to errors once a model is stable.
- **Use singular tests for business rules** — if a data team member can't explain what the YAML config means, it probably belongs in a readable SQL file.
- **Combine with `store_failures`** in staging environments so engineers can query failures directly rather than re-running models to reproduce them.

Generic tests cover the common cases with minimal config; singular tests handle anything custom. Together they form a continuous data quality layer that runs alongside your transformations — making "trust but verify" the default rather than the exception.

## 05:03 - dbt Model Contracts: Enforcing Schemas as Guarantees

dbt models typically infer their schema from the SQL SELECT — whatever columns the query produces become the model's schema. **Model contracts** (introduced in dbt v1.5) flip this around: you declare the schema first, and dbt verifies the model conforms to it at build time. The schema becomes a _promise_ to downstream consumers rather than an accidental byproduct.

### Why Model Contracts Matter

Without contracts, a column rename or type change silently propagates to every downstream model, dashboard, and API that depends on it. Contracts enforce a change-management discipline: you must explicitly version or update the contract before a breaking change can land.

### Defining a Contract

Add `contract: {enforced: true}` to your model's config, then declare every column with a `data_type`:

```yaml
# models/schema.yml
models:
  - name: orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: status
        data_type: varchar
      - name: total_amount
        data_type: numeric
      - name: created_at
        data_type: timestamp
```

When `enforced: true`, dbt will **fail the build** if the compiled SQL produces:
- A column that isn't listed in the contract
- A listed column missing from the SELECT
- A column whose data type doesn't match

### Constraints

Contracts also support database-level constraints:

| Constraint type | What it does |
|---|---|
| `not_null` | Adds a NOT NULL constraint (or equivalent check) |
| `primary_key` | Marks the column as PK |
| `unique` | Adds a UNIQUE constraint |
| `foreign_key` | References another model's column |
| `check` | Custom expression (adapter-dependent) |

```yaml
- name: order_id
  data_type: bigint
  constraints:
    - type: primary_key
- name: customer_id
  data_type: bigint
  constraints:
    - type: foreign_key
      to: ref('customers')
      to_columns: ['customer_id']
```

Not all adapters support every constraint type at the database level — check your adapter docs. dbt always validates types and column presence regardless of adapter support.

### Model Versions

Contracts integrate tightly with **model versioning**, which lets you run multiple versions of a model simultaneously during a migration:

```yaml
models:
  - name: orders
    latest_version: 2
    versions:
      - v: 1
      - v: 2
        defined_in: orders_v2   # points to models/orders_v2.sql
```

Downstream models reference `ref('orders')` and get the latest version; consumers that need stability can pin to `ref('orders', v=1)` until they're ready to migrate.

### Best Practices

- **Start with stable, high-traffic models** — mart tables consumed by BI tools and APIs are the highest-value contract targets.
- **Use contracts as API boundaries** — any model shared across teams or domains is a good candidate.
- **Pair with semantic layer / exposures** — contracts protect the schema; exposures document who consumes it.
- **Don't over-specify** — staging models that only you consume don't need contracts; reserve them for the "public interface" of your dbt project.
- **Treat contract failures like compile errors** — a failed contract means a breaking change is in flight; address it before merging, not after.

Model contracts turn implicit schema assumptions into explicit, machine-checked guarantees — a critical tool for any team where multiple squads share a dbt project or where downstream BI/API consumers need stability.
