## 00:03 - dbt Packages: Extending Your Project with dbt-utils and Beyond

dbt packages let you install and reuse community-built macros, models, and tests — think of them as npm packages for your data warehouse. The most popular is `dbt-utils`, maintained by dbt Labs, but a rich ecosystem exists for testing, auditing, and adapter-specific utilities.

### Installing Packages

Create a `packages.yml` file at the root of your project:

```yaml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0
  - package: calogica/dbt_expectations
    version: 0.10.0
  - package: dbt-labs/audit_helper
    version: 0.12.0
```

Then run:

```bash
dbt deps
```

This downloads packages into `dbt_packages/`. Add that directory to `.gitignore` — treat it like `node_modules`.

### dbt-utils Highlights

**Surrogate keys** — deterministic hash from multiple columns, portable across warehouses:

```sql
select
  {{ dbt_utils.generate_surrogate_key(['order_id', 'product_id']) }} as surrogate_key,
  order_id,
  product_id,
  quantity
from {{ ref('stg_order_items') }}
```

**Date spine** — generate a contiguous series of dates to fill reporting gaps:

```sql
with spine as (
  {{ dbt_utils.date_spine(
      datepart="day",
      start_date="cast('2024-01-01' as date)",
      end_date="cast('2025-01-01' as date)"
  ) }}
)
select date_day from spine
```

**star()** — select all columns except a blocklist (great for staging models):

```sql
select
  {{ dbt_utils.star(from=ref('stg_orders'), except=["_loaded_at", "_batch_id"]) }}
from {{ ref('stg_orders') }}
```

**get_column_values()** — dynamically pivot without hardcoding values:

```sql
{% set product_types = dbt_utils.get_column_values(
    table=ref('stg_products'), column='product_type'
) %}

select
  order_date,
  {% for product_type in product_types %}
    sum(case when product_type = '{{ product_type }}' then revenue end) as {{ product_type }}_revenue
    {% if not loop.last %},{% endif %}
  {% endfor %}
from {{ ref('fct_orders') }}
group by 1
```

### Generic Tests from dbt-utils

```yaml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - dbt_utils.not_empty_string
      - name: amount
        tests:
          - dbt_utils.expression_is_true:
              expression: ">= 0"
      - name: email
        tests:
          - dbt_utils.not_constant
```

### dbt-expectations for Advanced Column Assertions

Modeled after Great Expectations, useful for stricter data quality checks:

```yaml
columns:
  - name: email
    tests:
      - dbt_expectations.expect_column_values_to_match_regex:
          regex: "^[\\w.+-]+@[\\w-]+\\.[\\w.]+"
  - name: revenue
    tests:
      - dbt_expectations.expect_column_values_to_be_between:
          min_value: 0
          max_value: 1000000
```

### Best Practices

- **Pin versions** in `packages.yml` — floating versions break reproducibility in CI.
- **Run `dbt deps` in CI** before `dbt build` so packages are always installed fresh.
- **Use dispatch** in `dbt_project.yml` to override package macros cleanly without forking the package:

```yaml
dispatch:
  - macro_namespace: dbt_utils
    search_order: ['my_project', 'dbt_utils']
```

- **Check dbt Hub** (`hub.getdbt.com`) before writing custom macros — a well-tested package solution likely already exists.
- **Audit with audit_helper** when replacing models: it provides `compare_relations` to diff row counts and column values between old and new logic side-by-side.

Packages dramatically reduce boilerplate, standardize patterns across your team, and let you focus on modeling business logic instead of reinventing warehouse utilities.

## 05:14 - dbt Hooks: Running SQL Before and After Your Models

dbt hooks let you execute arbitrary SQL at well-defined points in your run — before or after individual models, or at the start/end of the entire `dbt run`. They're the escape hatch for warehouse operations that don't fit inside a model itself: granting privileges, managing transactions, updating audit tables, or setting session-level configuration.

### The Four Hook Types

| Hook | Scope | When it fires |
|------|-------|--------------|
| `pre-hook` | model / seed / snapshot | Before each node builds |
| `post-hook` | model / seed / snapshot | After each node builds |
| `on-run-start` | project | Once, before the first node |
| `on-run-end` | project | Once, after the last node |

### Granting Permissions with post-hook

The most common use case — automatically granting SELECT to a reporting role after every model build:

```yaml
# dbt_project.yml
models:
  my_project:
    +post-hook:
      - "grant select on {{ this }} to role reporter"
```

The `{{ this }}` context variable resolves to the fully qualified relation being built (e.g. `analytics.dbt_prod.fct_orders`), so this works for any materialization.

### Session Configuration with on-run-start

Run warehouse-level setup once before any models execute:

```yaml
# dbt_project.yml
on-run-start:
  - "alter session set query_tag = 'dbt_run'"
  - "alter session set timezone = 'UTC'"
```

This is useful for Snowflake query tags, BigQuery job labels, or any session parameter your models rely on being set.

### Writing Audit Records with on-run-end

Log a run summary to a table after every dbt run:

```yaml
on-run-end:
  - "{{ log_dbt_run_summary() }}"
```

```sql
-- macros/log_dbt_run_summary.sql
{% macro log_dbt_run_summary() %}
  insert into {{ target.schema }}.dbt_run_audit
    (run_started_at, target, env, status)
  values (
    '{{ run_started_at }}',
    '{{ target.name }}',
    '{{ target.schema }}',
    '{{ results | selectattr("status", "equalto", "error") | list | length }} errors'
  )
{% endmacro %}
```

The `results` variable (available only in `on-run-end`) is a list of node result objects — you can filter for errors, count successes, and build rich run logs.

### Model-Level Hooks with Config Blocks

Override hooks for a single model inline:

```sql
-- models/marts/fct_orders.sql
{{
  config(
    materialized='table',
    post_hook=[
      "grant select on {{ this }} to role analyst",
      "grant select on {{ this }} to role bi_tool"
    ]
  )
}}

select * from {{ ref('int_orders_enriched') }}
```

### Scoping Hooks by Directory

Apply hooks only to a subtree by nesting under the right path in `dbt_project.yml`:

```yaml
models:
  my_project:
    marts:
      +post-hook: "grant select on {{ this }} to role reporter"
    staging:
      +post-hook: "grant select on {{ this }} to role dbt_dev"
```

### Key Gotchas

- **Hooks run inside the same transaction** as the model on warehouses that support DDL transactions (Postgres, Redshift). On Snowflake and BigQuery they run as separate statements.
- **Failures in pre-hooks abort the model build**; failures in post-hooks mark the model as errored even if the DDL succeeded.
- **`on-run-end` always runs**, even when some models fail — making it a reliable place to flush audit records regardless of partial failures.
- **Avoid heavy computation** in hooks; they're not models and don't benefit from dbt's dependency tracking or result caching.

Hooks bridge the gap between dbt's declarative model layer and the imperative operations warehouses often require — use them to keep permission grants, audit trails, and session config in version control rather than scattered across manual runbooks.

## 10:03 - dbt Model Contracts: Enforcing Schema Guarantees Across Your Pipeline

Model contracts, introduced in dbt Core 1.5, let you declare an explicit schema agreement for a model — column names, data types, and constraints — that dbt will enforce at build time. They're the equivalent of a typed interface: downstream consumers and upstream producers both have a stable promise to rely on, and dbt refuses to build the model if the contract is violated.

### Why Contracts Matter

Without contracts, a model's columns are inferred from whatever SQL produces. If a developer renames `order_total` to `total_amount`, downstream models silently break at query time. A contract catches that mismatch during `dbt run`, before it reaches production.

### Defining a Contract

Contracts live in your model's `.yml` file under `config`:

```yaml
# models/marts/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: varchar
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_total
        data_type: numeric
      - name: ordered_at
        data_type: timestamp
```

When `enforced: true`, dbt compares the contract columns against what the model SQL actually produces. If a declared column is missing, has the wrong type, or an extra undeclared column appears, the build fails with a clear error.

### Supported Constraint Types

| Constraint | What it does | Warehouse support |
|------------|-------------|-------------------|
| `not_null` | Column must have no NULLs | All |
| `primary_key` | Unique non-null identifier | Metadata only on most warehouses |
| `unique` | All values distinct | Metadata only on most warehouses |
| `foreign_key` | References another relation | Snowflake, BigQuery (metadata) |
| `check` | Custom SQL expression | Postgres, Snowflake |

Most warehouses store `primary_key` and `foreign_key` as metadata without physically enforcing them — the real enforcement comes from dbt's schema comparison, not the warehouse engine.

### Inline Constraints in SQL

For warehouses that do enforce constraints (Postgres, Redshift), you can also declare them in the model SQL using the `{{ constraints() }}` helper — but the `.yml` approach is the recommended path as it's adapter-agnostic.

### Contracts and Versioned Models

Contracts are essential when using **model versions** (`v1`, `v2`). Each version can carry its own contract, giving consumers a stable interface to pin to while the latest version evolves:

```yaml
models:
  - name: fct_orders
    versions:
      - v: 1
        config:
          contract:
            enforced: true
      - v: 2
        config:
          contract:
            enforced: true
    columns:
      - name: order_id
        data_type: bigint
```

### Best Practices

- **Enforce contracts on mart-layer models** consumed by BI tools or external teams — these are the true interface boundaries.
- **Use `data_type` that matches your warehouse's type names** exactly (e.g. `varchar` vs `text` vs `string` differ by adapter).
- **Pair contracts with `on_schema_change: fail`** on incremental models so schema drift is caught from both directions.
- **Don't enforce contracts on staging models** — they change frequently and the contract overhead slows iteration without much benefit at that layer.
- **Add contracts incrementally**: start with your most-consumed models, measure the safety improvement, then expand coverage.

Model contracts shift schema governance left — from a runtime surprise to a build-time guarantee — and are the foundation of treating your dbt project as a product with versioned, reliable APIs rather than a collection of ad-hoc SQL files.
