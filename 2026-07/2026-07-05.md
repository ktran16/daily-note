## 00:02 - dbt Variables: Parameterizing Models with `var()` and `env_var()`

dbt variables let you inject runtime values into SQL and Jinja logic, making models
configurable across environments without code changes. There are two mechanisms:
`var()` for project-level and CLI-supplied values, and `env_var()` for secrets and
environment-specific settings pulled from the shell.

### `var()` — Project and Runtime Variables

Define defaults in `dbt_project.yml`:

```yaml
# dbt_project.yml
vars:
  start_date: '2020-01-01'
  is_full_refresh: false
  environment: 'dev'
```

Consume them in any model or macro:

```sql
-- models/fct_events.sql
select *
from {{ ref('stg_events') }}
where event_date >= '{{ var("start_date") }}'
```

Override at runtime without touching source files:

```bash
dbt run --vars '{"start_date": "2024-01-01", "is_full_refresh": true}'
```

This pattern is useful for **backfills**: keep the production default, then pass a
historical start date only when re-processing old data.

#### Scoping Variables to a Package

Variables are global by default, but you can scope them to a specific package to
avoid collisions when using third-party packages:

```yaml
vars:
  my_package:
    lookback_days: 7
  other_package:
    lookback_days: 14
```

Inside `my_package`, `var("lookback_days")` resolves to `7`; inside `other_package`
it resolves to `14`.

### `env_var()` — Shell Environment Variables

`env_var()` reads directly from the process environment, making it the right choice
for credentials and per-environment configuration that shouldn't be committed:

```yaml
# profiles.yml
my_project:
  target: prod
  outputs:
    prod:
      type: snowflake
      account: "{{ env_var('SNOWFLAKE_ACCOUNT') }}"
      user: "{{ env_var('SNOWFLAKE_USER') }}"
      password: "{{ env_var('SNOWFLAKE_PASSWORD') }}"
      database: "{{ env_var('DBT_DATABASE', 'ANALYTICS') }}"  # default value
```

The second argument is an optional default — without it, dbt raises a compile error
if the variable is unset, which is often the safer behavior for required secrets.

#### Using `env_var()` in Models

You can also reference environment variables inside SQL for runtime branching:

```sql
-- models/config_flags.sql
select
    '{{ env_var("DEPLOY_ENV", "dev") }}'  as environment,
    {{ env_var("MAX_ROWS", "1000000") }}  as row_limit
```

### Practical Patterns

**Limit dev data by default:**

```sql
-- models/stg_orders.sql
{% set max_rows = var('dev_row_limit', 100000) %}

select *
from {{ source('raw', 'orders') }}
{% if target.name != 'prod' %}
limit {{ max_rows }}
{% endif %}
```

**Feature flags via variables:**

```sql
{% if var('enable_new_logic', false) %}
  -- new experimental calculation
  {{ new_revenue_calculation() }}
{% else %}
  -- stable path
  {{ legacy_revenue_calculation() }}
{% endif %}
```

Toggle with `--vars '{"enable_new_logic": true}'` for testing before a full
rollout.

### Key Rules

- `var()` values are **strings** at the YAML level; cast explicitly in SQL if you
  need integers (`{{ var("days") | int }}`).
- Never store secrets in `dbt_project.yml` vars — use `env_var()` for anything
  sensitive so it stays out of version control.
- CI systems should set `env_var()` secrets via pipeline secrets managers (GitHub
  Actions secrets, GitLab CI variables, etc.) rather than `.env` files.

## 05:02 - dbt Materializations: Choosing the Right Build Strategy for Every Model

Materializations control *how* dbt writes model output into the warehouse. Choosing the right one is one of the highest-leverage decisions in a dbt project — it directly affects query performance, build time, and warehouse cost.

### The Four Core Materializations

| Materialization | What dbt creates | Data is fresh when… |
|---|---|---|
| `view` | A database view | Queried (always reads source) |
| `table` | A full table (DROP + INSERT) | dbt runs |
| `incremental` | A table updated partially | dbt runs (only new rows) |
| `ephemeral` | Nothing (inlined as CTE) | Queried |

---

### `view` — Default, Lightweight

```sql
-- models/staging/stg_orders.sql
{{ config(materialized='view') }}

select
    order_id,
    customer_id,
    cast(order_date as date) as order_date
from {{ source('raw', 'orders') }}
```

Best for: staging/light transformation layers. Zero storage cost; always current. Avoid when the view is queried frequently and the underlying source is large — the warehouse re-scans the source every time.

---

### `table` — Reliable, Predictable

```sql
{{ config(materialized='table') }}
```

dbt drops the old table and rebuilds it from scratch each run. Use for: reporting/mart models that need fast query performance. Downside: full rebuilds can be slow and expensive on large datasets.

---

### `incremental` — Efficient for High-Volume Fact Tables

```sql
{{ config(
    materialized='incremental',
    unique_key='event_id',
    incremental_strategy='merge'
) }}

select
    event_id,
    user_id,
    event_type,
    occurred_at
from {{ source('raw', 'events') }}

{% if is_incremental() %}
  -- only process rows newer than the latest row already in the table
  where occurred_at > (select max(occurred_at) from {{ this }})
{% endif %}
```

Key concepts:
- `{{ this }}` refers to the existing table in the warehouse.
- `is_incremental()` is `false` on a full-refresh (`dbt run --full-refresh`), so the model rebuilds completely when needed.
- `unique_key` tells dbt which column(s) to use for MERGE/UPSERT logic; without it, dbt only appends rows.
- Common strategies: `append`, `merge` (default on most adapters), `delete+insert`, `insert_overwrite` (BigQuery/Spark).

**Late-arriving data gotcha:** If source rows can arrive late (out-of-order events), widen your lookback window:

```sql
{% if is_incremental() %}
  where occurred_at > (
    select dateadd(day, -3, max(occurred_at)) from {{ this }}
  )
{% endif %}
```

---

### `ephemeral` — Pure Abstraction, No Storage

```sql
{{ config(materialized='ephemeral') }}

select order_id, sum(amount) as total
from {{ source('raw', 'line_items') }}
group by 1
```

dbt compiles ephemeral models as CTEs and injects them inline wherever they are referenced. No object is created in the warehouse. Use for small, reusable sub-queries you want to name and test without paying storage costs. Avoid for large intermediate datasets — they'll bloat every downstream query that uses them.

---

### Choosing the Right Materialization

```
Is the model a raw source cleanup?          → view
Is it a small reference/lookup table?       → view or table
Is it a large fact table (millions+ rows)?  → incremental
Is it a final mart queried by BI tools?     → table
Is it a tiny reusable sub-query?            → ephemeral
```

Set defaults per directory in `dbt_project.yml` and override at the model level only when needed:

```yaml
# dbt_project.yml
models:
  my_project:
    staging:
      +materialized: view
    intermediate:
      +materialized: ephemeral
    marts:
      +materialized: table
```

This layered approach keeps your project predictable and avoids over-engineering individual models.

## 10:02 - dbt Hooks: Running SQL Before and After Model Execution

dbt hooks let you execute arbitrary SQL immediately before or after a model is materialized. They're the escape hatch for anything dbt doesn't handle natively: granting permissions, setting session variables, logging run metadata, or swapping partitions.

### The Two Hook Types

| Hook | When it runs | Typical use |
|---|---|---|
| `pre-hook` | Before the model's SQL runs | Set session vars, create temp tables |
| `post-hook` | After the model is materialized | Grant permissions, register in a catalog |

### Configuring Hooks

**At the model level** (affects only this model):

```sql
-- models/marts/fct_orders.sql
{{ config(
    materialized='table',
    post_hook=[
        "grant select on {{ this }} to role REPORTER",
        "grant select on {{ this }} to role ANALYST"
    ]
) }}

select ...
```

**At the project level** (applies to every model in the path):

```yaml
# dbt_project.yml
models:
  my_project:
    marts:
      +post-hook:
        - "grant select on {{ this }} to role REPORTER"
```

Project-level hooks and model-level hooks are **additive** — both run in order.

### Practical Examples

**Log every model run to an audit table:**

```sql
-- Applied as a post-hook in dbt_project.yml
insert into analytics.audit.model_runs (model_name, run_at, row_count)
values (
    '{{ this }}',
    current_timestamp,
    (select count(*) from {{ this }})
)
```

**Set Snowflake query tags for cost attribution:**

```sql
{{ config(
    pre_hook="alter session set query_tag = 'dbt|{{ model.name }}|{{ invocation_id }}'"
) }}
```

**Revoke and re-grant after incremental updates** (useful when row-level security changes):

```sql
{{ config(
    materialized='incremental',
    post_hook=[
        "revoke all on {{ this }} from role RESTRICTED_ROLE",
        "grant select on {{ this }} to role RESTRICTED_ROLE where region = 'US'"
    ]
) }}
```

**Analyze a table on Redshift after rebuild** (updates query planner statistics):

```yaml
models:
  my_project:
    marts:
      +post-hook: "analyze {{ this }}"
```

### Hook Context Variables

Inside hook SQL you have access to the full Jinja context:

| Variable | Value |
|---|---|
| `{{ this }}` | The fully-qualified relation being built |
| `{{ model.name }}` | Model name string |
| `{{ invocation_id }}` | Unique ID for this dbt run |
| `{{ target.name }}` | Active target (`dev`, `prod`, etc.) |

### Key Gotchas

- **Hooks run inside the same transaction** as the model build (where your adapter supports transactions). If the hook fails, the entire model fails.
- **Order matters**: multiple hooks in an array run top-to-bottom. Put permission grants last so they only fire on successful builds.
- **`{{ this }}` in pre-hooks** refers to the *existing* relation (before dbt recreates it). If the table doesn't exist yet on first run, referencing it in a `pre-hook` `select` will error — guard with a `{% if adapter.get_relation(...) %}` check.
- Avoid heavy computation in hooks. A hook that re-scans a billion-row table on every run will silently bloat your pipeline costs.

## 15:02 - dbt Snapshots: Tracking Slowly Changing Dimensions

Snapshots capture the state of a mutable source table over time, letting you answer historical questions like "what was a customer's status on a given date?" without any changes to the source system. They implement **Type 2 Slowly Changing Dimension (SCD2)** logic automatically.

### How Snapshots Work

dbt compares each run's source rows against the existing snapshot table using a **strategy**, then:
- Inserts new rows that didn't exist before.
- Closes out changed rows by setting `dbt_valid_to` to the current timestamp.
- Appends a new open row (with `dbt_valid_to = null`) representing the current state.

### Defining a Snapshot

Snapshots live in the `snapshots/` directory and use the `{% snapshot %}` block:

```sql
-- snapshots/snap_customers.sql
{% snapshot snap_customers %}

{{
    config(
        target_schema='snapshots',
        unique_key='customer_id',
        strategy='timestamp',
        updated_at='updated_at'
    )
}}

select
    customer_id,
    name,
    email,
    plan_tier,
    updated_at
from {{ source('raw', 'customers') }}

{% endsnapshot %}
```

Run with:

```bash
dbt snapshot
```

### The Two Strategies

**`timestamp`** (preferred): dbt compares the `updated_at` column between runs. A row is considered changed if `updated_at` is newer than what's stored.

```yaml
strategy: timestamp
updated_at: updated_at
```

**`check`**: dbt hashes a set of columns and compares hashes. Use when the source has no reliable `updated_at` field.

```yaml
strategy: check
check_cols:
  - email
  - plan_tier
  - status
```

`check_cols: all` hashes every column, but is expensive on wide tables.

### Metadata Columns dbt Adds

| Column | Meaning |
|---|---|
| `dbt_scd_id` | Surrogate key unique per row version |
| `dbt_updated_at` | When dbt detected the change |
| `dbt_valid_from` | When this version became current |
| `dbt_valid_to` | When this version was superseded (`null` = current) |

### Querying Snapshot History

```sql
-- What plan was customer 42 on at 2025-06-01?
select plan_tier
from snapshots.snap_customers
where customer_id = 42
  and dbt_valid_from <= '2025-06-01'
  and (dbt_valid_to > '2025-06-01' or dbt_valid_to is null)
```

### Referencing Snapshots in Models

Use `{{ ref() }}` just like any other model:

```sql
-- models/fct_subscription_changes.sql
select
    customer_id,
    plan_tier                          as new_plan,
    lag(plan_tier) over (
        partition by customer_id
        order by dbt_valid_from
    )                                  as previous_plan,
    dbt_valid_from                     as changed_at
from {{ ref('snap_customers') }}
where dbt_valid_to is not null  -- only closed (changed) rows
```

### Key Gotchas

- **Snapshots don't support `--full-refresh`** — the history is the whole point. To rebuild, truncate the snapshot table manually and re-run.
- **Hard deletes are invisible**: if a row disappears from the source, the snapshot keeps its last known state open (`dbt_valid_to = null`) forever. Use `invalidate_hard_deletes: true` in config to auto-close deleted rows.
- **Run frequency matters**: snapshot granularity is limited by how often `dbt snapshot` runs. A record that changes and reverts between runs leaves no trace.
- Keep snapshot source queries simple — heavy transformations belong in downstream models, not in the snapshot itself.
