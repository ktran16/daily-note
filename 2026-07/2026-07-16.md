# Daily Notes - 2026-07-16

## 00:02 - dbt Mesh: Scaling Analytics Across Multiple dbt Projects

**dbt Mesh** (available in dbt Core 1.6+ and dbt Cloud) lets large organizations split a monolithic dbt project into multiple smaller, independently deployable projects that can still reference each other's models. Think of it as microservices for your data platform.

### The Problem It Solves

A single-project dbt repo with 500+ models becomes painful: long CI runs, too many contributors stepping on each other, and no clear ownership boundaries. dbt Mesh solves this by letting each domain team own their project while still sharing curated datasets cross-project.

### Core Concepts

**1. Public Models**

A model must be declared `access: public` to be referenced by another project:

```yaml
# models/marts/schema.yml
models:
  - name: fct_orders
    access: public          # visible to other projects
    config:
      contract:
        enforced: true      # required for cross-project ref()
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: customer_id
        data_type: bigint
      - name: order_total
        data_type: numeric
```

Models default to `access: protected` (only the same project can reference them) or `access: private` (only the same group within the project).

**2. Cross-Project `ref()`**

A downstream project references an upstream model using a two-argument `ref()`:

```sql
-- in the finance project, referencing the orders project
select
    o.order_id,
    o.order_total,
    c.customer_name
from {{ ref('orders', 'fct_orders') }} o
join {{ ref('customers', 'dim_customers') }} c
    on o.customer_id = c.customer_id
```

The first argument is the **project name** (from `dbt_project.yml`), the second is the model name.

**3. Dependencies File**

Declare upstream projects in `dependencies.yml` at the root of the consuming project:

```yaml
# dependencies.yml
projects:
  - name: orders
  - name: customers

packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0
```

### How dbt Resolves Cross-Project Refs

In dbt Cloud, the metadata service knows the latest compiled state of each project. When the finance project runs, dbt resolves `ref('orders', 'fct_orders')` to the actual warehouse relation (e.g., `analytics.orders.fct_orders`) from the orders project's last successful run — no re-compilation of upstream models needed.

In dbt Core, you'd use `dbt ls --select orders.fct_orders` with the `--project-dir` flag pointing to the upstream project to build and export its manifest, then reference that manifest in the downstream project.

### Access Levels at a Glance

| Level | Visible To |
|-------|-----------|
| `private` | Same group within the project |
| `protected` (default) | Same project only |
| `public` | Any project in the mesh |

### Best Practices

- **Pair public models with contracts** — a contract enforces that the schema won't silently change and break downstream consumers.
- **Keep public models in marts** — staging and intermediate models should stay `protected`. Only expose clean, stable mart models as the "API" of your project.
- **Version public models** — use dbt's model versioning (`versions:` block) before making breaking schema changes so consumers can migrate on their own schedule.
- **Own the interface, not the implementation** — upstream teams can refactor internals freely as long as public model contracts remain satisfied.

dbt Mesh is the path forward for organizations where a single monorepo has become a bottleneck — it brings software engineering's service ownership model to the data layer.
