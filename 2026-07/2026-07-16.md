## 00:02 - dbt Mesh: Scaling Analytics Across Multiple dbt Projects

**dbt Mesh** (available in dbt Core 1.6+ and dbt Cloud) lets large organizations split a monolithic dbt project into multiple smaller, independently deployable projects that can still reference each other's models. Think of it as microservices for your data platform.

### The Problem It Solves

A single-project dbt repo with 500+ models becomes painful: long CI runs, too many contributors stepping on each other, and no clear ownership boundaries. dbt Mesh solves this by letting each domain team own their project while still sharing curated datasets cross-project.

### Core Concepts

**1. Public Models**

A model must be declared `access: public` to be referenced by another project:

```yaml
# models/marts/schema.yml
models:
  - name: fct_orders
    access: public          # visible to other projects
    config:
      contract:
        enforced: true      # required for cross-project ref()
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: customer_id
        data_type: bigint
      - name: order_total
        data_type: numeric
```

Models default to `access: protected` (only the same project can reference them) or `access: private` (only the same group within the project).

**2. Cross-Project `ref()`**

A downstream project references an upstream model using a two-argument `ref()`:

```sql
-- in the finance project, referencing the orders project
select
    o.order_id,
    o.order_total,
    c.customer_name
from {{ ref('orders', 'fct_orders') }} o
join {{ ref('customers', 'dim_customers') }} c
    on o.customer_id = c.customer_id
```

The first argument is the **project name** (from `dbt_project.yml`), the second is the model name.

**3. Dependencies File**

Declare upstream projects in `dependencies.yml` at the root of the consuming project:

```yaml
# dependencies.yml
projects:
  - name: orders
  - name: customers

packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0
```

### How dbt Resolves Cross-Project Refs

In dbt Cloud, the metadata service knows the latest compiled state of each project. When the finance project runs, dbt resolves `ref('orders', 'fct_orders')` to the actual warehouse relation (e.g., `analytics.orders.fct_orders`) from the orders project's last successful run — no re-compilation of upstream models needed.

In dbt Core, you'd use `dbt ls --select orders.fct_orders` with the `--project-dir` flag pointing to the upstream project to build and export its manifest, then reference that manifest in the downstream project.

### Access Levels at a Glance

| Level | Visible To |
|-------|-----------|
| `private` | Same group within the project |
| `protected` (default) | Same project only |
| `public` | Any project in the mesh |

### Best Practices

- **Pair public models with contracts** — a contract enforces that the schema won't silently change and break downstream consumers.
- **Keep public models in marts** — staging and intermediate models should stay `protected`. Only expose clean, stable mart models as the "API" of your project.
- **Version public models** — use dbt's model versioning (`versions:` block) before making breaking schema changes so consumers can migrate on their own schedule.
- **Own the interface, not the implementation** — upstream teams can refactor internals freely as long as public model contracts remain satisfied.

dbt Mesh is the path forward for organizations where a single monorepo has become a bottleneck — it brings software engineering's service ownership model to the data layer.

## 05:02 - dbt Snapshots: Capturing Slowly Changing Dimensions (Type 2)

**dbt Snapshots** implement Type 2 Slowly Changing Dimensions (SCD2) — they track the full history of how a record changes over time, not just its current state. When a customer changes their email or a subscription changes its tier, a snapshot preserves both the old and new values with timestamps.

### The Problem Without Snapshots

Source tables typically reflect only current state. If a customer's `plan` changes from `starter` to `pro`, the row is overwritten and history is lost. Snapshots solve this by appending new rows whenever source data changes, each with `dbt_valid_from` and `dbt_valid_to` columns.

### Creating a Snapshot

Snapshots live in a `snapshots/` directory and use a special `snapshot` block:

```sql
-- snapshots/snap_customers.sql
{% snapshot snap_customers %}

{{
    config(
        target_schema='snapshots',
        unique_key='customer_id',
        strategy='timestamp',
        updated_at='updated_at',
    )
}}

select * from {{ source('raw', 'customers') }}

{% endsnapshot %}
```

Run with:

```bash
dbt snapshot
dbt snapshot --select snap_customers
```

### Two Snapshot Strategies

**`timestamp`** — uses an `updated_at` column to detect changes. Preferred when the source has a reliable update timestamp:

```sql
config(
    strategy='timestamp',
    updated_at='updated_at',
)
```

**`check`** — compares the values of specific columns; a row is considered changed if any of them differ. Use when the source has no `updated_at`:

```sql
config(
    strategy='check',
    check_cols=['plan', 'status', 'email'],
    -- or check_cols='all' to compare every column
)
```

### What dbt Adds to Each Row

| Column | Description |
|---|---|
| `dbt_scd_id` | Unique hash for each snapshot row |
| `dbt_updated_at` | When dbt last evaluated this row |
| `dbt_valid_from` | When this version became active |
| `dbt_valid_to` | When this version was superseded (`NULL` = current) |

### Querying Snapshot History

```sql
-- All historical versions of a customer
select *
from snapshots.snap_customers
where customer_id = 42
order by dbt_valid_from;

-- Point-in-time lookup: what was the customer's plan on 2025-06-01?
select customer_id, plan, dbt_valid_from, dbt_valid_to
from snapshots.snap_customers
where customer_id = 42
  and dbt_valid_from <= '2025-06-01'
  and (dbt_valid_to > '2025-06-01' or dbt_valid_to is null);
```

### Referencing Snapshots in Models

Use `ref()` just like any other model:

```sql
-- models/marts/fct_subscription_history.sql
select
    s.customer_id,
    s.plan,
    s.dbt_valid_from,
    s.dbt_valid_to,
    datediff('day', s.dbt_valid_from, coalesce(s.dbt_valid_to, current_date)) as days_on_plan
from {{ ref('snap_customers') }} s
```

### Best Practices

- **Run snapshots before your models** in your production job — snapshots capture the raw source state, so they should run against fresh source data before transformations consume it. Use `dbt snapshot && dbt run`.
- **Never filter rows in the snapshot query** — include all records so deletions can be detected. Filtering hides records from dbt's diff logic.
- **Prefer `timestamp` over `check`** when possible — `check` must compare every column value on every row, which is expensive on large tables.
- **Store snapshots in a dedicated schema** (`target_schema='snapshots'`) so they don't clutter your mart or staging schemas.
- **Hardcode the `target_schema`** rather than using `{{ target.schema }}` — you want snapshots to always land in the same place regardless of dev/prod environment, since they accumulate history across runs.

Snapshots are one of dbt's most unique capabilities: turning an overwrite-only OLTP source into a full historical audit trail with just a few lines of configuration.

## 10:03 - dbt Incremental Models: Building Efficient Pipelines That Only Process New Data

**Incremental models** are the workhorse of production dbt pipelines. Instead of rebuilding an entire table on every run, they append or merge only the rows that have changed since the last run — dramatically cutting warehouse compute costs and run time for large datasets.

### The Core Pattern

Add `materialized='incremental'` to your model config, then wrap a filter in an `is_incremental()` block:

```sql
-- models/marts/fct_events.sql
{{
    config(
        materialized='incremental',
        unique_key='event_id',
        on_schema_change='sync_all_columns'
    )
}}

select
    event_id,
    user_id,
    event_type,
    event_at,
    properties
from {{ source('raw', 'events') }}

{% if is_incremental() %}
    -- Only process rows newer than the latest row already in the table
    where event_at > (select max(event_at) from {{ this }})
{% endif %}
```

`{{ this }}` is a special dbt variable that refers to the model's own relation in the warehouse — safe to use inside the `is_incremental()` guard.

### Incremental Strategies

Different warehouses support different merge strategies. Set the strategy in `config()`:

| Strategy | How It Works | Best For |
|---|---|---|
| `append` | Only inserts new rows, never updates | Immutable event streams |
| `merge` (default on most DWs) | Upserts rows based on `unique_key` | Mutable records, deduplication |
| `delete+insert` | Deletes matching keys then re-inserts | Redshift, when merge is slow |
| `insert_overwrite` | Replaces entire partitions | BigQuery partitioned tables |

```sql
{{
    config(
        materialized='incremental',
        unique_key='order_id',
        incremental_strategy='merge',
        merge_update_columns=['status', 'updated_at']  -- only update these cols on match
    )
}}
```

### Handling Schema Changes with `on_schema_change`

When you add a new column to an incremental model, by default dbt raises an error if the existing table schema doesn't match. Control this with `on_schema_change`:

```sql
{{
    config(
        materialized='incremental',
        on_schema_change='sync_all_columns'  -- adds new columns, removes dropped ones
    )
}}
```

| Value | Behavior |
|---|---|
| `'ignore'` (default) | Ignore new/dropped columns; never alter the existing table |
| `'fail'` | Raise an error on any schema mismatch |
| `'append_new_columns'` | Add new columns to the existing table, never remove |
| `'sync_all_columns'` | Add new columns AND drop removed ones |

`sync_all_columns` is the most aggressive — use it only when you are confident old columns are truly gone from your query.

### Full Refresh

To rebuild the entire table from scratch (e.g., after a logic change that affects historical rows):

```bash
dbt run --select fct_events --full-refresh
```

This ignores the `is_incremental()` filter and re-runs the model as a `table` materialization, then switches back to incremental on the next run.

### Best Practices

- **Always filter on a partition column** (e.g., `event_at`, `created_at`) rather than an ID range — partition pruning makes the scan orders of magnitude cheaper on columnar stores.
- **Include a small lookback window** to catch late-arriving data: `where event_at > (select max(event_at) from {{ this }}) - interval '3 hours'`. This re-processes a small overlap but ensures you don't miss events that arrive slightly out of order.
- **Set `unique_key` whenever records can be updated** — without it, the `append` strategy will silently create duplicates if the same `event_id` appears in two runs.
- **Test idempotency** by running your model twice in a row against the same source data; row counts should not change on the second run.
- **Avoid `select *`** in incremental models — if a new source column appears, it can silently change your model's schema and cause `on_schema_change='fail'` to halt your pipeline.

Incremental models are the single biggest lever for improving pipeline performance in dbt — master them and you can scale to billions of rows without rethinking your architecture.

## 15:02 - dbt Generic and Singular Tests: Building a Data Quality Safety Net

dbt has a built-in testing framework that lets you assert expectations about your data directly alongside your model definitions. Tests run with `dbt test` and fail loudly when data violates the rules you set — making them the last line of defense before bad data reaches analysts or downstream systems.

### Two Types of Tests

**Generic tests** are parameterized tests defined in YAML and applied to columns or models. dbt ships with four out of the box:

```yaml
# models/marts/schema.yml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - unique
          - not_null
      - name: status
        tests:
          - accepted_values:
              values: ['placed', 'shipped', 'delivered', 'cancelled']
      - name: customer_id
        tests:
          - relationships:
              to: ref('dim_customers')
              field: customer_id
```

- **`unique`** — no duplicate values in the column
- **`not_null`** — no `NULL` values
- **`accepted_values`** — every value is in a predefined list
- **`relationships`** — referential integrity: every value exists in the target model's column

**Singular tests** are one-off SQL files in the `tests/` directory. The test passes when the query returns zero rows — any returned row represents a failure:

```sql
-- tests/assert_orders_total_is_positive.sql
select
    order_id,
    order_total
from {{ ref('fct_orders') }}
where order_total <= 0
```

Use singular tests for complex assertions that can't be expressed as a simple column-level generic test, like multi-column invariants or cross-model business rules.

### Configuring Test Severity and Thresholds

Not every failure should halt the pipeline. Configure severity and error thresholds in YAML:

```yaml
- name: order_id
  tests:
    - unique:
        config:
          severity: warn          # 'error' (default) or 'warn'
    - not_null:
        config:
          severity: error
          error_if: ">100"        # fail only if more than 100 rows fail
          warn_if: ">10"          # warn if more than 10 rows fail
```

This is useful for data sources you don't fully control — warn on small anomalies, only hard-fail when the volume of bad data is significant.

### Custom Generic Tests

You can define reusable generic tests in `tests/generic/` using Jinja macros:

```sql
-- tests/generic/is_valid_email.sql
{% test is_valid_email(model, column_name) %}

select {{ column_name }}
from {{ model }}
where {{ column_name }} is not null
  and {{ column_name }} not like '%@%.%'

{% endtest %}
```

Apply it just like a built-in test:

```yaml
- name: email
  tests:
    - is_valid_email
```

### Running Tests Selectively

```bash
# Run all tests
dbt test

# Test a specific model and its dependencies
dbt test --select fct_orders+

# Test only a specific type of test (using tags)
dbt test --select tag:my_tag

# Run tests from dbt-utils package
dbt test --select package:dbt_utils
```

### dbt-utils for Extra Generics

The `dbt-utils` package extends the built-in set with tests like:

```yaml
- name: order_id
  tests:
    - dbt_utils.expression_is_true:
        expression: ">= 1000"           # order IDs start from 1000
- name: created_at
  tests:
    - dbt_utils.not_constant             # column shouldn't have the same value for every row
    - dbt_utils.recency:
        datepart: hour
        interval: 2                      # data should be no older than 2 hours
```

### Best Practices

- **Test every primary key** with `unique` + `not_null` — these two are non-negotiable on any model that other models join to.
- **Test foreign keys** with `relationships` on fact tables — a broken join silently produces NULLs in downstream metrics, which is harder to catch than a test failure.
- **Use `accepted_values` on status/type columns** — the list serves as documentation AND a guard against new enum values silently flowing through untested.
- **Separate source tests from model tests** — run `dbt test --select source:*` as an early step in your pipeline to catch upstream data quality issues before wasting compute on transformations.
- **Set `store_failures: true`** in `dbt_project.yml` to write failing rows to a table in your warehouse for post-hoc debugging:

```yaml
# dbt_project.yml
tests:
  +store_failures: true
  +schema: test_failures
```

A disciplined test suite turns dbt from a transformation tool into a data contract enforcer — one that tells you the moment upstream data or business logic breaks an assumption your models rely on.

## 19:03 - dbt Slim CI: Only Test What Changed

**Slim CI** is the single highest-leverage dbt CI optimization. Instead of running `dbt build` against your entire project on every pull request (which can take 30+ minutes for large projects), Slim CI uses dbt's `state:` selector to run only the models that actually changed — plus their downstream dependents.

### The Core Idea

dbt can compare the compiled state of your current branch against the last successful production run (a saved `manifest.json`). Models that are identical at the SQL level are skipped entirely.

```bash
# Full CI (slow, expensive)
dbt build

# Slim CI (fast — only changed models + children)
dbt build --select state:modified+ --defer --state ./prod-state
```

- `state:modified` — selects models whose compiled SQL differs from the reference state
- `+` suffix — also selects all downstream dependents of those models
- `--defer` — for models NOT selected (the unchanged upstream ones), dbt reads from production instead of rebuilding them in a dev schema
- `--state ./prod-state` — directory containing the reference `manifest.json`

### How to Get the Production Manifest

In **dbt Cloud**, the production job's `manifest.json` is accessible via the Metadata API or directly via the "Artifacts" tab. Download it in your CI job before running:

```bash
# Example: fetch latest prod manifest using dbt Cloud API
curl -H "Authorization: Token $DBT_CLOUD_TOKEN" \
  "https://cloud.getdbt.com/api/v2/accounts/$ACCOUNT_ID/jobs/$JOB_ID/artifacts/manifest.json" \
  -o prod-state/manifest.json
```

In **dbt Core** with a custom orchestrator (Airflow, Prefect, etc.), upload the manifest artifact after each successful production run to a known location (S3, GCS) and download it in your CI step:

```bash
# Upload after prod run
aws s3 cp target/manifest.json s3://my-bucket/dbt/prod/manifest.json

# Download in CI before slim CI run
mkdir prod-state
aws s3 cp s3://my-bucket/dbt/prod/manifest.json prod-state/manifest.json
```

### State Selector Variants

| Selector | What It Matches |
|---|---|
| `state:modified` | Models with changed compiled SQL |
| `state:modified.body` | Only SQL body changes (ignores config/description changes) |
| `state:modified.configs` | Only config-level changes |
| `state:new` | Models that didn't exist in the reference state |
| `state:modified+` | Changed models AND all downstream dependents |
| `state:modified+ state:new` | Both changed and brand-new models |

A practical CI selector that catches everything worth testing:

```bash
dbt build \
  --select "state:modified+ state:new" \
  --defer \
  --state ./prod-state
```

### The `--defer` Flag in Practice

Without `--defer`, your CI run tries to build upstream models in the CI schema even if you didn't change them. With `--defer`, dbt substitutes unselected upstream nodes with their production counterparts — so your changed model can still join against `prod.fct_orders` without you having to rebuild it.

```sql
-- Your changed model references an unchanged upstream:
select *
from {{ ref('fct_orders') }}   -- dbt resolves this to prod.fct_orders when deferred
where {{ ref('dim_customers') }} -- also resolves to prod.dim_customers
```

This means CI tests run against real production data, not an empty dev schema — catching real data-quality failures.

### Full GitHub Actions Example

```yaml
# .github/workflows/dbt-ci.yml
name: dbt Slim CI

on:
  pull_request:
    paths:
      - 'models/**'
      - 'tests/**'
      - 'macros/**'
      - 'dbt_project.yml'

jobs:
  slim-ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install dbt
        run: pip install dbt-snowflake

      - name: Download prod manifest
        run: |
          mkdir prod-state
          aws s3 cp s3://my-bucket/dbt/prod/manifest.json prod-state/manifest.json
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}

      - name: dbt deps
        run: dbt deps

      - name: dbt Slim CI build
        run: |
          dbt build \
            --select "state:modified+ state:new" \
            --defer \
            --state prod-state \
            --target ci
        env:
          DBT_SNOWFLAKE_ACCOUNT: ${{ secrets.SNOWFLAKE_ACCOUNT }}
          DBT_SNOWFLAKE_USER: ${{ secrets.SNOWFLAKE_USER }}
          DBT_SNOWFLAKE_PASSWORD: ${{ secrets.SNOWFLAKE_PASSWORD }}
```

### Best Practices

- **Run `dbt build` not `dbt run && dbt test`** — `build` runs models and their associated tests together, catching test failures at the model level rather than after all models finish.
- **Include `state:new` alongside `state:modified+`** — brand-new models have no reference state and won't be matched by `state:modified` alone.
- **Use `--target ci`** with a separate CI target in `profiles.yml` that writes to a dedicated CI schema (e.g., `dbt_ci_<pr_number>`), keeping CI runs isolated from both dev and prod.
- **Gate on path changes** in your CI trigger — no need to run dbt CI if only a README changed.
- **Clean up CI schemas** after merging — a post-merge job that drops `dbt_ci_*` schemas prevents schema sprawl in your warehouse.

Slim CI is a force multiplier: a project that takes 40 minutes to build fully can have a 2-minute CI loop for small PRs, which dramatically improves the feedback cycle for every contributor.
