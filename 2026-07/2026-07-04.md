## 00:02 - dbt Unit Tests: Validating Model Logic with Mocked Inputs

Introduced in **dbt 1.8**, unit tests let you verify the SQL logic of a model in isolation by supplying fixture data as inputs and asserting the expected output — without touching your warehouse at all.

### Why Unit Tests Matter

Traditional dbt tests (`not_null`, `unique`, custom singular tests) run against *real data* after a model builds. But what if your source data doesn't contain the edge case you're worried about? Unit tests let you construct synthetic inputs to prove your transformation logic handles every scenario correctly.

### Anatomy of a Unit Test

Unit tests are defined in a `.yml` file alongside your models:

```yaml
# models/finance/_finance__models.yml
unit_tests:
  - name: test_order_revenue_excludes_refunds
    model: fct_order_revenue
    given:
      - input: ref('stg_orders')
        rows:
          - {order_id: 1, amount: 100.00, status: 'completed'}
          - {order_id: 2, amount: 50.00,  status: 'refunded'}
          - {order_id: 3, amount: 200.00, status: 'completed'}
      - input: ref('stg_refunds')
        rows:
          - {order_id: 2, refund_amount: 50.00}
    expect:
      rows:
        - {order_id: 1, revenue: 100.00}
        - {order_id: 3, revenue: 200.00}
```

The model under test (`fct_order_revenue`) might look like:

```sql
-- models/finance/fct_order_revenue.sql
select
    o.order_id,
    o.amount as revenue
from {{ ref('stg_orders') }} o
left join {{ ref('stg_refunds') }} r using (order_id)
where o.status = 'completed'
  and r.order_id is null  -- exclude orders that were refunded
```

Running `dbt test --select fct_order_revenue,test_type:unit` executes only unit tests for that model.

### Key Patterns and Tips

**Partial fixture rows** — You only need to specify the columns relevant to your test. dbt auto-fills nulls for any omitted columns, keeping fixtures concise.

**Mocking `source()` inputs** — You can mock upstream sources just like `ref()`:

```yaml
given:
  - input: source('raw', 'events')
    rows:
      - {event_id: 'abc', event_type: 'click', ts: '2026-01-01'}
```

**`overrides` for `var()` and `env_var()`** — If your model uses variables, override them per test:

```yaml
overrides:
  vars:
    start_date: '2026-01-01'
```

**Combine with incremental logic** — Set `overrides.is_incremental: true` to test the incremental branch of your model without actually running it incrementally:

```yaml
overrides:
  is_incremental: true
```

### When to Write Unit Tests

- Complex `case when` logic or multi-step transformations where a single wrong condition silently drops rows.
- Financial or regulatory calculations where correctness is non-negotiable.
- Macros that generate SQL — test the model that *uses* the macro.
- Whenever a bug is fixed — add a unit test to prevent regression.

Unit tests are a game-changer for data teams that want the same confidence in their SQL transformations that software engineers expect from application code.
