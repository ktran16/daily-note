## 00:02 - dbt Unit Tests: Validating Model Logic with Mocked Inputs

Introduced in **dbt 1.8**, unit tests let you verify the SQL logic of a model in isolation by supplying fixture data as inputs and asserting the expected output — without touching your warehouse at all.

### Why Unit Tests Matter

Traditional dbt tests (`not_null`, `unique`, custom singular tests) run against *real data* after a model builds. But what if your source data doesn't contain the edge case you're worried about? Unit tests let you construct synthetic inputs to prove your transformation logic handles every scenario correctly.

### Anatomy of a Unit Test

Unit tests are defined in a `.yml` file alongside your models:

```yaml
# models/finance/_finance__models.yml
unit_tests:
  - name: test_order_revenue_excludes_refunds
    model: fct_order_revenue
    given:
      - input: ref('stg_orders')
        rows:
          - {order_id: 1, amount: 100.00, status: 'completed'}
          - {order_id: 2, amount: 50.00,  status: 'refunded'}
          - {order_id: 3, amount: 200.00, status: 'completed'}
      - input: ref('stg_refunds')
        rows:
          - {order_id: 2, refund_amount: 50.00}
    expect:
      rows:
        - {order_id: 1, revenue: 100.00}
        - {order_id: 3, revenue: 200.00}
```

The model under test (`fct_order_revenue`) might look like:

```sql
-- models/finance/fct_order_revenue.sql
select
    o.order_id,
    o.amount as revenue
from {{ ref('stg_orders') }} o
left join {{ ref('stg_refunds') }} r using (order_id)
where o.status = 'completed'
  and r.order_id is null  -- exclude orders that were refunded
```

Running `dbt test --select fct_order_revenue,test_type:unit` executes only unit tests for that model.

### Key Patterns and Tips

**Partial fixture rows** — You only need to specify the columns relevant to your test. dbt auto-fills nulls for any omitted columns, keeping fixtures concise.

**Mocking `source()` inputs** — You can mock upstream sources just like `ref()`:

```yaml
given:
  - input: source('raw', 'events')
    rows:
      - {event_id: 'abc', event_type: 'click', ts: '2026-01-01'}
```

**`overrides` for `var()` and `env_var()`** — If your model uses variables, override them per test:

```yaml
overrides:
  vars:
    start_date: '2026-01-01'
```

**Combine with incremental logic** — Set `overrides.is_incremental: true` to test the incremental branch of your model without actually running it incrementally:

```yaml
overrides:
  is_incremental: true
```

### When to Write Unit Tests

- Complex `case when` logic or multi-step transformations where a single wrong condition silently drops rows.
- Financial or regulatory calculations where correctness is non-negotiable.
- Macros that generate SQL — test the model that *uses* the macro.
- Whenever a bug is fixed — add a unit test to prevent regression.

Unit tests are a game-changer for data teams that want the same confidence in their SQL transformations that software engineers expect from application code.

## 05:02 - dbt Hooks: Automating DDL Side Effects with pre-hook, post-hook, and on-run-end

**Hooks** let you execute arbitrary SQL before or after a model runs, or at key points in the full dbt invocation. They're the escape hatch for anything dbt can't manage natively: granting permissions, logging run metadata, managing row-level security policies, or notifying downstream consumers.

### The Four Hook Types

| Hook | When It Fires |
|---|---|
| `pre-hook` | Before each model's SQL executes |
| `post-hook` | After each model's SQL executes (and the model table/view exists) |
| `on-run-start` | Once at the very start of `dbt run` / `dbt build` |
| `on-run-end` | Once at the very end of `dbt run` / `dbt build` |

### Common Use Case: Granting Permissions After Build

The most universal pattern is granting `SELECT` to a reporting role after every model materializes:

```yaml
# dbt_project.yml
models:
  my_project:
    +post-hook:
      - "grant select on {{ this }} to role reporter"
```

`{{ this }}` is a special dbt variable that resolves to the fully-qualified relation name of the current model (e.g. `analytics.marts.fct_orders`), so you never hard-code schema names.

### Scoping Hooks to Specific Subfolders

Hooks compose with the config hierarchy. Apply them only where needed:

```yaml
models:
  my_project:
    marts:
      +post-hook:
        - "grant select on {{ this }} to role bi_tool_user"
    staging:
      +post-hook: []  # no grants needed on raw staging models
```

### Run-Level Hooks for Auditing

`on-run-end` has access to `{{ results }}` — a list of every model result from the run — making it perfect for writing a run audit log:

```yaml
# dbt_project.yml
on-run-end:
  - "{{ log_run_results(results) }}"
```

```sql
-- macros/log_run_results.sql
{% macro log_run_results(results) %}
  {% set insert_statements = [] %}
  {% for res in results %}
    {% set _ = insert_statements.append(
      "insert into audit.dbt_run_log values ('"
      ~ res.node.name ~ "', '"
      ~ res.status ~ "', "
      ~ res.execution_time | round(2)
      ~ ", current_timestamp)"
    ) %}
  {% endfor %}
  {% for stmt in insert_statements %}
    {{ stmt }};
  {% endfor %}
{% endmacro %}
```

### Model-Level Hook Override

Hooks can also be set inline in a model's config block for one-off cases:

```sql
-- models/marts/fct_orders.sql
{{ config(
    materialized='table',
    post_hook=[
        "grant select on {{ this }} to role reporter",
        "call analytics.refresh_search_index('{{ this.schema }}', '{{ this.name }}')"
    ]
) }}

select ...
```

### Key Gotchas

- **Hooks run inside the model's transaction** on databases that wrap DDL in transactions (Postgres, Redshift). On Snowflake and BigQuery, DDL is auto-committed, so hooks run as separate statements.
- **Hook failures don't roll back the model** — a failed `post-hook` leaves the model built but the hook unexecuted. Monitor hook results in the run logs.
- **`on-run-end` skips if the run errors out entirely** — structure critical cleanup as `post-hook` on individual models rather than relying on `on-run-end` for guaranteed execution.

Hooks are the right tool any time you need side effects that live outside the model's SELECT logic — keep the model SQL pure and push operational concerns into hooks.
