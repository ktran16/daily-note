## 00:03 - dbt Exposures: Documenting Downstream Consumers of Your Data

**Exposures** are dbt's way of cataloging what consumes your models — dashboards, reports, ML models, applications, or any external system that depends on the data you build. They don't change how data is processed; they complete the **data lineage picture** by making downstream dependencies visible to your team and in dbt docs.

### Why Exposures Matter

Without exposures, your lineage graph ends at the last dbt model. You know how raw data flows into `fct_orders`, but you have no record that the CFO's revenue dashboard, the pricing ML model, and the customer success Salesforce sync all depend on it. Exposures bridge that gap.

**Practical benefits:**
- `dbt docs generate` renders a full lineage graph from source tables all the way to Looker dashboards or Airflow DAGs.
- `dbt ls --select +exposure:cfo_revenue_dashboard` shows every model the dashboard depends on — useful for impact analysis before a refactor.
- Teams know who to notify when a model changes.

### Defining an Exposure

Exposures live in any `.yml` file (commonly `models/exposures.yml` or co-located with the relevant model's `.yml`):

```yaml
# models/exposures.yml
exposures:
  - name: cfo_revenue_dashboard
    label: "CFO Revenue Dashboard"
    type: dashboard
    maturity: high           # high | medium | low
    url: https://looker.company.com/dashboards/42
    description: >
      Executive revenue dashboard tracking MRR, ARR, and churn.
      Refreshed nightly. Primary source of truth for board reporting.

    depends_on:
      - ref('fct_orders')
      - ref('dim_customers')
      - ref('fct_subscriptions')

    owner:
      name: Finance Analytics Team
      email: finance-analytics@company.com

  - name: churn_prediction_model
    label: "Churn Prediction ML Model"
    type: ml               # dashboard | notebook | ml | application | analysis | source
    maturity: medium
    url: https://mlflow.internal/models/churn-v3
    description: >
      XGBoost churn model trained weekly on 90-day customer behavior features.
      Scores are written back to Snowflake and surfaced in Salesforce.

    depends_on:
      - ref('fct_customer_activity')
      - ref('dim_customers')
      - ref('int_feature_store')

    owner:
      name: Data Science
      email: data-science@company.com
```

### Exposure Types

| `type` | When to use |
|---|---|
| `dashboard` | BI tool dashboards (Looker, Tableau, Metabase, Mode) |
| `notebook` | Jupyter / Databricks notebooks used for analysis |
| `ml` | Machine learning models trained on your data |
| `application` | Operational apps reading from your warehouse |
| `analysis` | Ad-hoc or recurring SQL analyses |
| `source` | Reverse-ETL / sync back to operational systems |

### Querying Exposures with dbt CLI

```bash
# List all exposures in the project
dbt ls --resource-type exposure

# List all models that feed a specific exposure (full upstream graph)
dbt ls --select +exposure:cfo_revenue_dashboard

# Build every model the dashboard depends on (and their ancestors)
dbt build --select +exposure:cfo_revenue_dashboard

# Run tests only on models feeding a given exposure
dbt test --select +exposure:churn_prediction_model
```

The `+exposure:<name>` selector is especially powerful in CI: you can run a targeted build for only the subset of models that matter to a specific downstream consumer.

### Maturity as a Contract Signal

The `maturity` field (`high` / `medium` / `low`) is a soft indicator of how stable and reliable an exposure is:

- **high** — production-grade, board-level reporting; changes need change management.
- **medium** — established but still evolving; stakeholders are informed of changes.
- **low** — experimental or prototype; consumers accept that the schema may shift.

Pair `maturity: high` exposures with **model contracts** (`contract: {enforced: true}`) on their upstream models so that schema-breaking changes fail CI before reaching production.

### Viewing Exposures in dbt Docs

```bash
dbt docs generate && dbt docs serve
```

Navigate to any exposure in the docs UI to see:
- A description and owner contact.
- A full lineage graph from raw sources through every intermediate model.
- The `maturity` badge and a clickable URL to the dashboard or notebook.

### Key Takeaways

- Exposures are **metadata-only** — they don't run SQL or affect builds, but they make lineage and ownership explicit.
- Use `+exposure:<name>` selectors in CI to build and test only the models that a specific consumer needs.
- Set `owner.email` on every exposure so dbt docs serves as a living on-call directory for downstream consumers.
- Combine high-maturity exposures with model contracts and model versions for a robust, breaking-change-safe data API.
- Even two or three well-defined exposures dramatically improve discoverability for new team members browsing the lineage graph.

## 05:03 - dbt Hooks: Running SQL Before and After Your Models Build

**Hooks** let you execute arbitrary SQL at specific points in the dbt build lifecycle. They're the escape hatch for things dbt doesn't natively support — granting permissions, logging audit rows, calling stored procedures, or manipulating warehouse session settings — without wrapping your logic in a custom macro.

### The Four Hook Types

| Hook | Runs... | Configured in... |
|---|---|---|
| `pre-hook` | Before a model/seed/snapshot materializes | model config, `dbt_project.yml` |
| `post-hook` | After a model/seed/snapshot materializes | model config, `dbt_project.yml` |
| `on-run-start` | Once at the very start of a `dbt run` | `dbt_project.yml` only |
| `on-run-end` | Once at the very end of a `dbt run` | `dbt_project.yml` only |

### post-hook: Granting Permissions After Each Build

The most common use case — granting SELECT to a BI role after a table is rebuilt:

```sql
-- models/marts/fct_orders.sql
{{ config(
    materialized='table',
    post_hook=[
        "grant select on {{ this }} to role REPORTER",
        "grant select on {{ this }} to role BI_TOOL"
    ]
) }}

select ...
```

`{{ this }}` is a dbt built-in that resolves to the fully-qualified relation name (`database.schema.model_name`) at runtime — you never hard-code it.

### Project-Wide Hooks in dbt_project.yml

Apply a hook to every model in a folder without touching individual files:

```yaml
# dbt_project.yml
models:
  my_project:
    marts:
      +post_hook:
        - "grant select on {{ this }} to role REPORTER"

on-run-start:
  - "call my_db.admin.start_audit_log('{{ invocation_id }}')"

on-run-end:
  - "call my_db.admin.end_audit_log('{{ invocation_id }}', {{ results | tojson }})"
```

`{{ invocation_id }}` is a unique UUID dbt generates per run — ideal for correlating audit log entries across models in a single execution.

### pre-hook: Session Configuration

Some warehouses benefit from session-level settings applied before expensive builds:

```sql
-- models/marts/fct_events.sql (Snowflake example)
{{ config(
    materialized='incremental',
    pre_hook="alter session set query_tag = 'dbt_{{ model.name }}_{{ run_started_at }}'",
    post_hook="alter session unset query_tag"
) }}

select ...
```

This tags every Snowflake query from this model's build with a structured label, making cost attribution and query history filtering trivial.

### Calling a Macro from a Hook

Hooks accept Jinja, so you can call a macro for reusable logic:

```sql
-- macros/grant_select.sql
{% macro grant_select(role='REPORTER') %}
    grant select on {{ this }} to role {{ role }}
{% endmacro %}
```

```yaml
# dbt_project.yml
models:
  my_project:
    +post_hook: "{{ grant_select('REPORTER') }}"
```

### Hook Execution Order and Error Behavior

- Multiple hooks on the same model run **in the order defined** (list order).
- A hook failure **fails the model** — the run stops for that node. Wrap non-critical hooks in a `try/except`-style stored procedure if needed.
- `on-run-end` hooks receive a `results` context variable — a list of result objects for every node — so you can conditionally notify or log based on pass/fail counts.

```yaml
# Only send a Slack alert if any model failed
on-run-end:
  - "{{ alert_on_failure(results) if results | selectattr('status', 'equalto', 'error') | list }}"
```

### Key Takeaways

- Use `post-hook` for permission grants — it's the canonical pattern in every dbt project deployed to a shared warehouse.
- Prefer project-level hooks in `dbt_project.yml` over per-model configs to keep individual model files clean.
- `{{ this }}` and `{{ invocation_id }}` are your most useful hook context variables.
- `on-run-end` with the `results` context is powerful for build-level alerting and audit logging without external orchestration.
- Keep hook SQL idempotent: hooks re-run on every build, so `grant` (not `create`) and upsert-style audit inserts are safer than append-only writes.

## 10:02 - dbt Unit Tests: Testing SQL Logic Without Real Data

**Unit tests** (introduced in dbt 1.8) let you assert that a model produces the correct output for a given set of mock inputs — entirely in YAML, without touching real warehouse data. They're different from dbt's generic/singular data tests, which validate the *state* of data already in the warehouse. Unit tests validate the *SQL logic* of a model by feeding it fake rows and checking what comes out.

### Why Unit Tests?

Generic tests like `not_null` or `unique` tell you data quality is intact. They don't tell you whether your `case when` logic, date truncation, or window function behaves correctly for edge cases. Unit tests fill that gap:

- Catch regressions in business logic before a PR merges.
- Test edge cases (null inputs, boundary dates, zero-division guards) that may never appear in dev data.
- Run without needing the warehouse populated — fast feedback in CI.

### Anatomy of a Unit Test

```yaml
# models/marts/fct_orders.yml
unit_tests:
  - name: test_discount_applied_correctly
    model: fct_orders
    given:
      - input: ref('stg_orders')
        rows:
          - {order_id: 1, subtotal: 100.00, coupon_pct: 10}
          - {order_id: 2, subtotal: 200.00, coupon_pct: 0}
          - {order_id: 3, subtotal: 50.00,  coupon_pct: null}
      - input: ref('dim_coupons')
        rows:
          - {coupon_pct: 10, is_active: true}
    expect:
      rows:
        - {order_id: 1, final_amount: 90.00}
        - {order_id: 2, final_amount: 200.00}
        - {order_id: 3, final_amount: 50.00}
```

Each `given` block mocks one upstream `ref()` or `source()` with literal rows. The `expect` block defines the output rows you assert against. Only the columns you list in `expect` are compared — extra columns in the model output are ignored.

### Running Unit Tests

```bash
# Run all unit tests
dbt test --select test_type:unit

# Run unit tests for a specific model
dbt test --select fct_orders --select test_type:unit

# Run unit tests alongside data tests (default behavior of dbt build)
dbt build --select fct_orders
```

`dbt build` runs unit tests, data tests, and materializes models in dependency order — so unit tests act as a pre-materialization gate on your logic.

### Mocking Sources and Seeds

You can mock `source()` references the same way as `ref()`:

```yaml
given:
  - input: source('stripe', 'payments')
    rows:
      - {payment_id: 'py_001', amount_cents: 9900, status: 'succeeded'}
      - {payment_id: 'py_002', amount_cents: 4900, status: 'failed'}
```

Seeds are referenced by `ref()` just like models, so they mock the same way.

### Overriding dbt Variables and Macros

Unit tests support overriding `var()` values and `is_incremental()` for models that branch on run-time context:

```yaml
unit_tests:
  - name: test_incremental_filter_logic
    model: fct_events
    overrides:
      macros:
        is_incremental: true     # simulate an incremental run
      vars:
        event_lookback_days: 7
    given:
      - input: ref('stg_events')
        rows:
          - {event_id: 1, created_at: '2026-07-20', event_type: 'click'}
          - {event_id: 2, created_at: '2026-07-10', event_type: 'view'}
    expect:
      rows:
        - {event_id: 1}   # only the recent row should survive the incremental filter
```

### Key Takeaways

- Unit tests validate **SQL logic**; data tests validate **data quality** — you need both.
- Define unit tests in the same `.yml` file as your model's schema or in a dedicated `unit_tests.yml` in the same directory.
- Use `overrides.macros.is_incremental: true` to test the incremental branch of a model without actually running incrementally.
- Only columns explicitly listed in `expect.rows` are asserted — omitting a column means "I don't care what value is here."
- Unit tests run in the warehouse (they compile to SQL `select` statements with inline CTEs), so adapter-specific SQL functions still behave correctly.
- Add `dbt test --select test_type:unit` as a fast CI step before building anything — it catches logic bugs cheaply before materialization costs are incurred.

## 15:02 - dbt Snapshots: Tracking Historical Changes with Type 2 SCDs

**Snapshots** are dbt's mechanism for capturing row-level history in source tables that change over time. They implement a **Type 2 Slowly Changing Dimension (SCD)** pattern: instead of overwriting a row when it changes, dbt inserts a new row and closes the old one with an `dbt_valid_to` timestamp, preserving the full history of every record.

### When to Use Snapshots

Use snapshots when you need to answer questions like:
- "What was this customer's plan tier on a specific date?"
- "What was the order status when the invoice was generated?"
- "Which sales rep owned this account six months ago?"

These are impossible to answer if your pipeline just overwrites the current state.

### Defining a Snapshot

Snapshots live in the `snapshots/` directory and use their own `{% snapshot %}` block syntax:

```sql
-- snapshots/snap_customers.sql
{% snapshot snap_customers %}

{{
    config(
        target_schema='snapshots',
        unique_key='customer_id',
        strategy='timestamp',
        updated_at='updated_at',
        invalidate_hard_deletes=True
    )
}}

select
    customer_id,
    email,
    plan,
    status,
    updated_at
from {{ source('crm', 'customers') }}

{% endsnapshot %}
```

Run with:
```bash
dbt snapshot
```

### Snapshot Strategies

dbt ships two built-in strategies:

| Strategy | How it detects changes | Use when... |
|---|---|---|
| `timestamp` | Compares `updated_at` column to its previous value | Source table has a reliable `updated_at` |
| `check` | Hashes a list of columns you specify | Source has no timestamp (e.g., a vendor flat file) |

**`check` strategy example:**

```sql
{{
    config(
        strategy='check',
        check_cols=['email', 'plan', 'status'],   -- only these columns trigger a new snapshot row
        unique_key='customer_id',
        target_schema='snapshots'
    )
}}
```

### What dbt Adds to the Snapshot Table

dbt appends four metadata columns automatically:

| Column | Description |
|---|---|
| `dbt_scd_id` | Unique hash identifying this specific version of the row |
| `dbt_updated_at` | Timestamp when dbt last touched this row |
| `dbt_valid_from` | When this row version became active |
| `dbt_valid_to` | When this row version was superseded (`null` = still current) |

### Querying Point-in-Time State

The snapshot table lets you reconstruct the state of any record at any past moment:

```sql
-- What plan was customer 42 on during the November 2025 billing cycle?
select *
from {{ ref('snap_customers') }}
where
    customer_id = 42
    and dbt_valid_from <= '2025-11-01'
    and (dbt_valid_to > '2025-11-01' or dbt_valid_to is null)
```

A common pattern is to join fact tables to snapshots using this bracket logic, keeping historical accuracy even as dimension values drift.

### Hard Deletes

By default, rows deleted from the source are never invalidated in the snapshot — they keep `dbt_valid_to = null` forever, appearing "still current." Set `invalidate_hard_deletes=True` in the config and dbt will close those rows on the next snapshot run.

### Key Takeaways

- Run `dbt snapshot` on a separate, more frequent schedule than `dbt run` if your source tables change throughout the day.
- Snapshot tables accumulate rows forever — partition by `dbt_valid_from` or cluster on `unique_key` to keep queries fast as history grows.
- Use the `timestamp` strategy when available; it's cheaper than `check` because dbt doesn't need to hash every monitored column on every row.
- Keep snapshot logic thin — select only the columns needed for downstream joins. Heavy transformations belong in downstream models that `ref()` the snapshot.
- Combine snapshots with a date spine to produce a "state per day" dimension table that's easy to join in BI tools without bracket-style logic.
