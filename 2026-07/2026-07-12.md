## 00:02 - dbt Python Models: Running Python Logic Inside Your dbt DAG

dbt Python models (`.py` files) let you run Python code as a first-class node in your dbt DAG — useful when SQL falls short: ML inference, complex string parsing, pandas-style reshaping, or calling external APIs during transformation.

### When to Use Python Models

- Applying a trained ML model to a table of features
- Deduplication or fuzzy matching with libraries like `rapidfuzz`
- Multi-step dataframe transformations that would be awkward in SQL window functions
- Calling external REST APIs and persisting the results

They are currently supported on **Snowflake** (Snowpark), **Databricks** (PySpark / pandas-on-Spark), **BigQuery** (Dataproc), and **Amazon Athena** (Spark).

### Anatomy of a Python Model

```python
# models/ml/scored_customers.py

def model(dbt, session):
    # Reference upstream dbt models or sources
    customers = dbt.ref("stg_customers")
    orders    = dbt.ref("fct_orders")

    # Use native DataFrame API (Snowpark / PySpark)
    df = customers.join(orders, on="customer_id", how="left")
    df = df.with_column("clv_score", df["lifetime_value"] * 0.85)

    return df
```

The `model(dbt, session)` signature is required. `dbt.ref()` and `dbt.source()` return native DataFrame objects for the active platform.

### Config Stays the Same

```python
def model(dbt, session):
    dbt.config(
        materialized="table",
        packages=["scikit-learn==1.4.0"],  # Snowflake: install in the session
    )
    ...
```

All standard dbt configs — `materialized`, `tags`, `meta`, `grants` — work identically to SQL models.

### Mixing SQL and Python in One DAG

Python models are full DAG citizens. A downstream SQL model can `ref()` a Python model and vice-versa:

```sql
-- models/marts/enriched_customers.sql
select *
from {{ ref('scored_customers') }}   -- Python model upstream
where clv_score > 0.7
```

### Key Limitations

| Constraint | Detail |
|---|---|
| Materialization | Only `table` and `incremental` (no `view` — Python runs in a session, not a query) |
| Incremental logic | Use `dbt.is_incremental()` and filter the input DataFrame manually |
| Testing | Same generic/singular tests apply to the output table |
| IDE linting | SQL linters won't help; use `mypy` or `ruff` in a separate pre-commit step |

### Incremental Python Model Pattern

```python
def model(dbt, session):
    dbt.config(materialized="incremental", unique_key="event_id")

    df = dbt.ref("raw_events")

    if dbt.is_incremental():
        # Only process events newer than the latest already loaded
        max_ts = session.sql(
            f"select max(event_ts) from {dbt.this}"
        ).collect()[0][0]
        df = df.filter(df["event_ts"] > max_ts)

    return df
```

### Best Practices

- Keep Python models **thin** — do heavy lifting in SQL upstream, use Python only for what SQL can't do.
- Pin library versions in `dbt.config(packages=[...])` to avoid silent breakage across runs.
- Store shared Python helpers in `macros/` as `.py` files (dbt 1.8+) or in a separate `utils` package.
- Python models are slower to run than SQL models (session spin-up overhead), so avoid using them for simple transformations.

## 05:02 - dbt Model Contracts: Enforcing Schema Integrity Across Model Boundaries

dbt Model Contracts (stable since dbt 1.5) let you declare a model's expected columns, data types, and constraints as part of the model definition itself — turning your schema into a **compile-time and run-time guarantee** rather than a hope.

### Why Contracts Matter

Without contracts, a model's schema is purely implicit. A refactor that renames a column silently breaks every downstream consumer. Contracts make that break **explicit and immediate**, catching it before bad data reaches production.

### Enabling a Contract

Add `contract: {enforced: true}` to your model config and declare each column with its data type:

```yaml
# models/marts/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_total
        data_type: numeric
      - name: ordered_at
        data_type: timestamp
```

When `enforced: true`, dbt will **fail at compile time** if the model's SELECT list doesn't return all declared columns, and some adapters (Snowflake, BigQuery, Redshift, Postgres) enforce the `not_null` and `primary_key` constraints at the database level.

### Supported Constraint Types

| Constraint | Adapter enforcement | dbt-level check |
|---|---|---|
| `not_null` | Yes (most adapters) | Yes |
| `primary_key` | Varies (some adapters create PK) | Yes (uniqueness not guaranteed) |
| `unique` | Varies | No — use generic `unique` test |
| `foreign_key` | Varies | No — use relationships test |
| `check` | Varies | No |
| `custom` | Adapter-defined | No |

### Contracts + Versioned Models

Contracts shine brightest with **model versioning** (`v1`, `v2`, …). A version bump signals to consumers that a breaking change happened; the contract documents exactly what changed:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1
        config:
          contract: {enforced: true}
      - v: 2
        config:
          contract: {enforced: true}
        columns:
          - name: order_id
            data_type: bigint
          - name: order_total_usd   # renamed from order_total
            data_type: numeric
```

Downstream models using `ref('fct_orders', v=1)` keep working; only those that explicitly upgrade to `v=2` need to handle the rename.

### Best Practices

- Enforce contracts on **public models** (those consumed by other teams or external tools) — not on every staging model.
- Always pair a `primary_key` constraint with a `unique` + `not_null` dbt test; the constraint documents intent while the test actually validates it.
- Use contracts as the source of truth for documentation: tools like dbt Docs and the semantic layer can read declared column types directly.
- CI should run `dbt compile` on PRs to catch contract violations before merge — no data needs to move, just the DAG needs to parse.

## 10:03 - dbt Snapshots: Tracking Slowly Changing Dimensions Over Time

dbt Snapshots implement **Type 2 Slowly Changing Dimensions (SCD2)** — automatically capturing row-level history as source data changes over time. Instead of every query seeing only the current state of a record, snapshots let you ask "what did this record look like on a given date?"

### How Snapshots Work

dbt adds four metadata columns to the snapshot table on each `dbt snapshot` run:

| Column | Meaning |
|---|---|
| `dbt_scd_id` | Unique identifier for this row version (hash) |
| `dbt_updated_at` | Timestamp of when this version was captured |
| `dbt_valid_from` | When this version became active |
| `dbt_valid_to` | When it was superseded (`NULL` = current row) |

### Defining a Snapshot

Snapshots live in the `snapshots/` directory and use a `{% snapshot %}` block:

```sql
-- snapshots/snap_customers.sql

{% snapshot snap_customers %}

  {{
    config(
      target_schema='snapshots',
      unique_key='customer_id',
      strategy='timestamp',
      updated_at='updated_at',
    )
  }}

  select * from {{ source('app_db', 'customers') }}

{% endsnapshot %}
```

Run with:

```bash
dbt snapshot
```

### Two Snapshot Strategies

**`timestamp`** — compares an `updated_at` column to detect changes. Fast and reliable when the source has a reliable update timestamp.

```yaml
strategy: timestamp
updated_at: updated_at
```

**`check`** — hashes a list of columns and flags a change when any column value differs. Useful when there is no `updated_at` column in the source.

```yaml
strategy: check
check_cols:
  - email
  - phone
  - address_line_1
  - plan_tier
# or: check_cols: all
```

### Querying Snapshot History

```sql
-- Current state only
select * from {{ ref('snap_customers') }}
where dbt_valid_to is null;

-- Point-in-time: what was each customer's plan on 2025-01-01?
select *
from {{ ref('snap_customers') }}
where '2025-01-01'
    between dbt_valid_from and coalesce(dbt_valid_to, current_timestamp);

-- Full history for a specific customer
select *
from {{ ref('snap_customers') }}
where customer_id = 42
order by dbt_valid_from;
```

### Downstream Models Using Snapshot History

A fact table can join to the snapshot using the point-in-time pattern to capture the attribute value **at the time of the event**, not today's value:

```sql
-- models/marts/fct_orders_with_customer_plan.sql
select
    o.order_id,
    o.ordered_at,
    c.plan_tier as plan_tier_at_order_time
from {{ ref('fct_orders') }} o
join {{ ref('snap_customers') }} c
  on  o.customer_id = c.customer_id
  and o.ordered_at between c.dbt_valid_from
                       and coalesce(c.dbt_valid_to, current_timestamp)
```

### Best Practices

- Run snapshots **before** your standard `dbt run` in your pipeline so models always see the freshest history.
- Snapshots are **append-only** — never truncate or rebuild the target table; you will lose historical versions.
- Keep snapshot queries simple (usually `select * from source`). Apply transformations in downstream models that `ref()` the snapshot.
- Set `target_schema` to a separate schema (e.g., `snapshots`) to keep them visually distinct from regular models.
- Add a `dbt_valid_to is null` filter as a default in your BI tool's table definition so analysts always start from the current view.
