## 00:02 - dbt Hooks: Running SQL Before and After Your Models

dbt **hooks** let you execute arbitrary SQL at key points in a model's lifecycle — before it runs (`pre-hook`) or after it finishes (`post-hook`). They're the escape hatch for anything dbt doesn't handle natively: granting permissions, managing table statistics, logging audit trails, or toggling warehouse settings.

### Where to Define Hooks

Hooks can be set in three places (narrower scope wins):

**`dbt_project.yml`** — applies to all models in a path:
```yaml
models:
  my_project:
    +post-hook:
      - "grant select on {{ this }} to role reporter"
    marts:
      +pre-hook: "call sp_lock_table('{{ this }}')"
```

**`schema.yml`** — applies to one model:
```yaml
models:
  - name: orders
    config:
      post-hook: "analyze table {{ this }}"
```

**In-model config block** — highest precedence:
```sql
{{ config(
    materialized='table',
    post-hook=[
      "grant select on {{ this }} to role analyst",
      "insert into audit_log values ('{{ this }}', current_timestamp())"
    ]
) }}

select * from {{ ref('stg_orders') }}
```

### The `{{ this }}` Variable

`{{ this }}` resolves to the fully-qualified relation that dbt just built — `database.schema.model_name`. Always use it instead of hardcoding names so hooks work across environments (dev, prod, CI).

### Common Patterns

**Grant permissions after every table/incremental build:**
```sql
{{ config(
    materialized='incremental',
    post-hook="grant select on {{ this }} to role bi_reader"
) }}
```

**Run `ANALYZE` to update table statistics (Redshift, Postgres):**
```yaml
+post-hook: "analyze {{ this }}"
```

**Insert a row into an audit table:**
```sql
post-hook: >
  insert into dbt_audit.model_runs (model, run_at)
  values ('{{ this }}', current_timestamp())
```

**Cluster a BigQuery table after load:**
```sql
{{ config(
    materialized='table',
    post-hook="alter table {{ this }} cluster by event_date, user_id"
) }}
```

### Hooks vs. Operations

For warehouse-wide work not tied to a specific model (e.g., running a stored procedure once per pipeline), use `dbt run-operation` with a macro instead of a hook. Hooks are model-scoped; operations are project-scoped.

### Key Gotchas

- **Multiple hooks** run in order — pass a YAML list, not a single string, when you need sequencing.
- Hooks run **inside the same transaction** as the model by default on databases that support DDL transactions; on BigQuery/Snowflake they run in separate statements.
- A **failing hook fails the model** — dbt marks it as an error, so keep hook SQL simple and idempotent.
- Use `{{ target.schema }}` or `{{ env_var() }}` in hook SQL to avoid hardcoding environment-specific values.

## 05:02 - dbt `store_failures`: Persisting Test Failures for Easy Debugging

When a dbt test fails, the default behavior is to print a count — "3 rows failed" — and move on. That number alone rarely tells you *which* rows are bad or *why*. The `store_failures` config fixes this by materializing the failing rows into a table in your warehouse so you can query them directly.

### Enabling `store_failures`

Turn it on at any scope:

**Project-wide** (`dbt_project.yml`):
```yaml
tests:
  +store_failures: true
  +store_failures_as: table   # 'table' (default) or 'view'
  +schema: dbt_test_failures  # custom schema to keep failures separate
```

**Single test in `schema.yml`**:
```yaml
models:
  - name: orders
    columns:
      - name: customer_id
        tests:
          - not_null:
              config:
                store_failures: true
          - relationships:
              to: ref('customers')
              field: id
              config:
                store_failures: true
                store_failures_as: view
```

**Inline on a singular test** (`tests/assert_positive_revenue.sql`):
```sql
{{ config(store_failures=true, schema='dbt_test_failures') }}

select order_id, revenue
from {{ ref('fct_orders') }}
where revenue < 0
```

### What Gets Stored

dbt materializes the *query that backs the test* — i.e., the rows that caused the failure. For a `not_null` test on `customer_id`, you get a table of every row where `customer_id IS NULL`. The table is named after the test: `not_null_orders_customer_id`.

```sql
-- After a test run, query the failures directly:
select * from dbt_test_failures.not_null_orders_customer_id limit 50;
```

### `store_failures_as: view` vs `table`

| Setting | Behavior |
|---|---|
| `table` | Rows written at test time; fast to query later |
| `view` | Re-runs the failing query on demand; always fresh |

Use `view` in development (always current), `table` in CI/production (snapshot failures at run time for audit purposes).

### Combining with `warn_if` / `error_if`

You can soften a test to a warning while still persisting failures for investigation:

```yaml
- name: amount
  tests:
    - not_null:
        config:
          store_failures: true
          warn_if: ">0"     # warn instead of error
          error_if: ">100"  # only hard-fail above threshold
```

### Key Tips

- Set a dedicated `schema` (e.g., `dbt_test_failures`) so failure tables don't clutter your main schema.
- Failure tables persist across runs — each run **replaces** the previous failures for that test, so they always reflect the most recent state.
- Use `dbt test --store-failures` as a CLI flag to enable it ad-hoc without changing `schema.yml`.
- In CI, enable `store_failures` on critical tests and export the schema to an artifact or alerting system for faster triage.
