## 00:02 - dbt Hooks: Running SQL Before and After Your Models

dbt **hooks** let you execute arbitrary SQL at key points in a model's lifecycle — before it runs (`pre-hook`) or after it finishes (`post-hook`). They're the escape hatch for anything dbt doesn't handle natively: granting permissions, managing table statistics, logging audit trails, or toggling warehouse settings.

### Where to Define Hooks

Hooks can be set in three places (narrower scope wins):

**`dbt_project.yml`** — applies to all models in a path:
```yaml
models:
  my_project:
    +post-hook:
      - "grant select on {{ this }} to role reporter"
    marts:
      +pre-hook: "call sp_lock_table('{{ this }}')"
```

**`schema.yml`** — applies to one model:
```yaml
models:
  - name: orders
    config:
      post-hook: "analyze table {{ this }}"
```

**In-model config block** — highest precedence:
```sql
{{ config(
    materialized='table',
    post-hook=[
      "grant select on {{ this }} to role analyst",
      "insert into audit_log values ('{{ this }}', current_timestamp())"
    ]
) }}

select * from {{ ref('stg_orders') }}
```

### The `{{ this }}` Variable

`{{ this }}` resolves to the fully-qualified relation that dbt just built — `database.schema.model_name`. Always use it instead of hardcoding names so hooks work across environments (dev, prod, CI).

### Common Patterns

**Grant permissions after every table/incremental build:**
```sql
{{ config(
    materialized='incremental',
    post-hook="grant select on {{ this }} to role bi_reader"
) }}
```

**Run `ANALYZE` to update table statistics (Redshift, Postgres):**
```yaml
+post-hook: "analyze {{ this }}"
```

**Insert a row into an audit table:**
```sql
post-hook: >
  insert into dbt_audit.model_runs (model, run_at)
  values ('{{ this }}', current_timestamp())
```

**Cluster a BigQuery table after load:**
```sql
{{ config(
    materialized='table',
    post-hook="alter table {{ this }} cluster by event_date, user_id"
) }}
```

### Hooks vs. Operations

For warehouse-wide work not tied to a specific model (e.g., running a stored procedure once per pipeline), use `dbt run-operation` with a macro instead of a hook. Hooks are model-scoped; operations are project-scoped.

### Key Gotchas

- **Multiple hooks** run in order — pass a YAML list, not a single string, when you need sequencing.
- Hooks run **inside the same transaction** as the model by default on databases that support DDL transactions; on BigQuery/Snowflake they run in separate statements.
- A **failing hook fails the model** — dbt marks it as an error, so keep hook SQL simple and idempotent.
- Use `{{ target.schema }}` or `{{ env_var() }}` in hook SQL to avoid hardcoding environment-specific values.

## 05:02 - dbt `store_failures`: Persisting Test Failures for Easy Debugging

When a dbt test fails, the default behavior is to print a count — "3 rows failed" — and move on. That number alone rarely tells you *which* rows are bad or *why*. The `store_failures` config fixes this by materializing the failing rows into a table in your warehouse so you can query them directly.

### Enabling `store_failures`

Turn it on at any scope:

**Project-wide** (`dbt_project.yml`):
```yaml
tests:
  +store_failures: true
  +store_failures_as: table   # 'table' (default) or 'view'
  +schema: dbt_test_failures  # custom schema to keep failures separate
```

**Single test in `schema.yml`**:
```yaml
models:
  - name: orders
    columns:
      - name: customer_id
        tests:
          - not_null:
              config:
                store_failures: true
          - relationships:
              to: ref('customers')
              field: id
              config:
                store_failures: true
                store_failures_as: view
```

**Inline on a singular test** (`tests/assert_positive_revenue.sql`):
```sql
{{ config(store_failures=true, schema='dbt_test_failures') }}

select order_id, revenue
from {{ ref('fct_orders') }}
where revenue < 0
```

### What Gets Stored

dbt materializes the *query that backs the test* — i.e., the rows that caused the failure. For a `not_null` test on `customer_id`, you get a table of every row where `customer_id IS NULL`. The table is named after the test: `not_null_orders_customer_id`.

```sql
-- After a test run, query the failures directly:
select * from dbt_test_failures.not_null_orders_customer_id limit 50;
```

### `store_failures_as: view` vs `table`

| Setting | Behavior |
|---|---|
| `table` | Rows written at test time; fast to query later |
| `view` | Re-runs the failing query on demand; always fresh |

Use `view` in development (always current), `table` in CI/production (snapshot failures at run time for audit purposes).

### Combining with `warn_if` / `error_if`

You can soften a test to a warning while still persisting failures for investigation:

```yaml
- name: amount
  tests:
    - not_null:
        config:
          store_failures: true
          warn_if: ">0"     # warn instead of error
          error_if: ">100"  # only hard-fail above threshold
```

### Key Tips

- Set a dedicated `schema` (e.g., `dbt_test_failures`) so failure tables don't clutter your main schema.
- Failure tables persist across runs — each run **replaces** the previous failures for that test, so they always reflect the most recent state.
- Use `dbt test --store-failures` as a CLI flag to enable it ad-hoc without changing `schema.yml`.
- In CI, enable `store_failures` on critical tests and export the schema to an artifact or alerting system for faster triage.

## 10:02 - dbt Cross-Database Macros with `adapter.dispatch()`

Most SQL dialects differ just enough to be painful — `DATEADD` on Snowflake, `DATE_ADD` on BigQuery, `DATEADD` again on Redshift but with different argument order. The `adapter.dispatch()` pattern lets you write a single macro name that resolves to the right implementation at runtime based on the active adapter.

### How It Works

`adapter.dispatch('macro_name', 'package_name')` returns a macro reference. dbt searches for implementations in this priority order:

1. `<adapter>__<macro_name>` in the current project
2. `<adapter>__<macro_name>` in the specified package
3. `default__<macro_name>` as the fallback

### Writing a Cross-Database Macro

```jinja
{# macros/safe_divide.sql #}

{% macro safe_divide(numerator, denominator) %}
    {{ return(adapter.dispatch('safe_divide', 'my_project')(numerator, denominator)) }}
{% endmacro %}

{# Default fallback (ANSI SQL) #}
{% macro default__safe_divide(numerator, denominator) %}
    case when {{ denominator }} = 0 then null
         else {{ numerator }} / nullif({{ denominator }}, 0)
    end
{% endmacro %}

{# Snowflake-specific override #}
{% macro snowflake__safe_divide(numerator, denominator) %}
    div0({{ numerator }}, {{ denominator }})
{% endmacro %}

{# BigQuery-specific override #}
{% macro bigquery__safe_divide(numerator, denominator) %}
    ieee_divide({{ numerator }}, {{ denominator }})
{% endmacro %}
```

Call it like any other macro:

```sql
select
    order_id,
    {{ safe_divide('revenue', 'units') }} as revenue_per_unit
from {{ ref('fct_orders') }}
```

dbt picks the right implementation automatically — `div0()` on Snowflake, `ieee_divide()` on BigQuery, the ANSI fallback everywhere else.

### The `dispatch` Search Path

Control which projects dbt searches for overrides via `dispatch` in `dbt_project.yml`:

```yaml
dispatch:
  - macro_namespace: dbt_utils
    search_order: ['my_project', 'dbt_utils']
```

This means your project's `snowflake__<macro>` overrides will win over `dbt_utils` implementations — useful for patching a package macro without forking it.

### When to Use This Pattern

- **Writing reusable packages** that need to run on multiple adapters.
- **Overriding dbt-utils macros** when the default implementation doesn't fit your warehouse's quirks.
- **Abstracting dialect differences** (date functions, string aggregation, JSON parsing) behind a stable macro API so models stay clean.

### Adapter Names for Dispatch

| Warehouse | Adapter prefix |
|---|---|
| Snowflake | `snowflake` |
| BigQuery | `bigquery` |
| Redshift | `redshift` |
| Postgres | `postgres` |
| DuckDB | `duckdb` |
| Databricks | `databricks` |
| Trino | `trino` |

### Key Takeaway

`adapter.dispatch()` is the mechanism behind every cross-database macro in `dbt_utils` and `dbt-core` itself. Learning it unlocks writing reusable, adapter-agnostic logic — and lets you surgically override any package macro without patching the upstream source.

## 15:02 - dbt Model Contracts: Enforcing Schemas at Build Time

dbt **model contracts** (introduced in dbt 1.5) let you declare the exact shape a model must produce — column names, data types, and constraints — and have dbt enforce it before or during the build. Instead of discovering a broken schema downstream in BI tools or failing pipelines, contracts catch mismatches at compile or run time.

### Enabling a Contract

Add `contract: {enforced: true}` to a model's config, then declare every column with its `data_type`:

```yaml
# models/schema.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric
      - name: status
        data_type: varchar
```

When `enforced: true`, dbt will **fail the run** if the model's SELECT produces columns with wrong names, wrong types, or in the wrong order (for adapters that care about order).

### What Gets Enforced

| Check | Enforced? |
|---|---|
| Column names present | Yes |
| Data types match | Yes (adapter-dependent casting) |
| Column order | Yes on some adapters (Snowflake, Redshift) |
| `not_null` constraint | Yes — added as a DDL constraint where supported |
| `primary_key` / `unique` constraint | Adapter-dependent (Snowflake enforces; BigQuery is advisory) |

### Constraints in SQL DDL

On supported adapters, dbt translates constraints directly into `CREATE TABLE` DDL:

```sql
-- Generated DDL on Snowflake:
create or replace table analytics.fct_orders (
    order_id    bigint     not null primary key,
    customer_id bigint     not null,
    order_date  date,
    total_amount numeric,
    status      varchar
) as (
    select ...
);
```

On adapters that don't support DDL constraints (e.g., BigQuery), dbt still validates column names and types but skips constraint clauses.

### Model-Level Constraints

You can also add constraints at the model level (not column-level):

```yaml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    constraints:
      - type: primary_key
        columns: [order_id]
      - type: unique
        columns: [order_id, order_date]
    columns:
      ...
```

### Contracts + Versioned Models

Contracts pair perfectly with **model versioning** (`versions:` in schema.yml). When you bump a model version, the contract documents exactly what changed in the interface — breaking vs. non-breaking — making the diff reviewable and safe for downstream consumers.

### Key Takeaways

- Contracts shift schema validation **left** — catching errors at build time instead of query time.
- They are the foundation for treating dbt models as stable, documented APIs consumed by other teams or tools.
- On warehouses without DDL constraint support, contracts still provide documentation and name/type validation — they're not a no-op.
- Use `dbt compile` to see the generated DDL and verify contracts before running.

## 19:02 - dbt Snapshots: Capturing Slowly Changing Dimensions

dbt **snapshots** solve a classic data warehousing problem: your source system only stores current state, but you need to know what a record *looked like* in the past. Snapshots implement SCD Type 2 — appending rows with effective date ranges whenever a record changes — without writing custom merge logic.

### How Snapshots Work

dbt compares the current source data against the last snapshot using a chosen strategy. When a row changes, it:

1. Closes the old row by setting `dbt_valid_to` to the current timestamp
2. Inserts a new row with `dbt_valid_from = now()` and `dbt_valid_to = NULL`

Unchanged rows are left alone, so the operation is efficient even on large sources.

### Defining a Snapshot

Snapshots live in the `snapshots/` directory (configurable) and use a `{% snapshot %}` block:

```sql
-- snapshots/customers_snapshot.sql
{% snapshot customers_snapshot %}

{{
    config(
        target_schema='snapshots',
        unique_key='customer_id',
        strategy='timestamp',
        updated_at='updated_at',
        invalidate_hard_deletes=True
    )
}}

select
    customer_id,
    email,
    plan_tier,
    status,
    updated_at
from {{ source('crm', 'customers') }}

{% endsnapshot %}
```

Run snapshots with:
```bash
dbt snapshot
# or target a single snapshot:
dbt snapshot --select customers_snapshot
```

### The Two Strategies

**`timestamp`** — compares an `updated_at` column. Use this when your source reliably updates a timestamp on every change. Fast and precise.

```yaml
strategy: timestamp
updated_at: updated_at
```

**`check`** — hashes a list of columns and flags a change if any column value differs. Use when there is no reliable `updated_at`.

```yaml
strategy: check
check_cols: ['email', 'plan_tier', 'status']
# or check all columns:
check_cols: all
```

### Snapshot Metadata Columns

dbt automatically adds four columns to every snapshot table:

| Column | Description |
|---|---|
| `dbt_scd_id` | Unique surrogate key for each snapshot row |
| `dbt_updated_at` | When dbt last touched this row |
| `dbt_valid_from` | When this version of the record became active |
| `dbt_valid_to` | When this version was superseded (`NULL` = current) |

### Querying Snapshot Data

**Current state only:**
```sql
select * from {{ ref('customers_snapshot') }}
where dbt_valid_to is null;
```

**Point-in-time lookup (what did the record look like on a given date?):**
```sql
select * from {{ ref('customers_snapshot') }}
where customer_id = 42
  and '2025-03-01'::date between dbt_valid_from and coalesce(dbt_valid_to, current_date);
```

**Full history for a customer:**
```sql
select customer_id, plan_tier, dbt_valid_from, dbt_valid_to
from {{ ref('customers_snapshot') }}
where customer_id = 42
order by dbt_valid_from;
```

### `invalidate_hard_deletes`

By default, if a record disappears from the source, its snapshot row stays open (`dbt_valid_to = NULL`) — dbt can't distinguish "deleted" from "not yet loaded." Setting `invalidate_hard_deletes: True` closes out rows for records no longer present in the source query, giving you accurate `dbt_valid_to` dates for deletes.

### Key Best Practices

- **Run snapshots frequently** — the granularity of your history depends on how often `dbt snapshot` runs. A daily job means you can only reconstruct daily state, not intra-day changes.
- **Snapshot at the source layer** — snapshot raw/staging tables, not transformed ones. You want to capture changes before any business logic is applied.
- **Use dedicated schemas** — put snapshots in a separate schema (e.g., `snapshots`) to keep them clearly separated from marts and staging models.
- **Don't add `where` clauses** to the snapshot query — if you filter out rows, dbt treats the missing rows as deleted (when `invalidate_hard_deletes` is on), which may not be what you want.
- **Test snapshot outputs** like any other model — `not_null`, `unique` on `dbt_scd_id`, and `relationships` tests to verify referential integrity downstream.
