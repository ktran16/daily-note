## 00:02 - dbt Packages: Extending dbt with dbt-utils and the Package Hub

dbt packages let you import pre-built macros, models, and tests directly into your project — think of them as npm or pip packages but for dbt. The primary registry is [dbt Hub](https://hub.getdbt.com/), and the community package `dbt-utils` is the one nearly every dbt project installs.

### Installing Packages

Declare packages in `packages.yml` at the root of your project:

```yaml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0
  - package: calogica/dbt_expectations
    version: 0.10.3
  - git: "https://github.com/your-org/internal-dbt-macros.git"
    revision: v1.2.0
```

Then run:

```bash
dbt deps
```

This downloads packages into the `dbt_packages/` directory (add it to `.gitignore`).

### Key dbt-utils Macros

**`generate_surrogate_key`** — creates a hashed surrogate key from one or more columns, avoiding warehouse-specific `MD5` / `SHA256` calls:

```sql
select
    {{ dbt_utils.generate_surrogate_key(['order_id', 'line_item_id']) }} as sk_order_line,
    order_id,
    line_item_id,
    product_name
from {{ ref('stg_order_lines') }}
```

**`date_spine`** — generates a contiguous sequence of dates, invaluable for filling gaps in time-series data:

```sql
with date_series as (
    {{ dbt_utils.date_spine(
        datepart="day",
        start_date="cast('2024-01-01' as date)",
        end_date="current_date"
    ) }}
)
select
    date_day,
    coalesce(revenue, 0) as revenue
from date_series
left join {{ ref('fct_daily_revenue') }} using (date_day)
```

**`pivot`** — turns row values into columns without hard-coding every case:

```sql
select
    store_id,
    {{ dbt_utils.pivot(
        column='status',
        values=['pending', 'shipped', 'delivered', 'returned'],
        agg='count',
        then_value='order_id'
    ) }}
from {{ ref('stg_orders') }}
group by 1
```

**`get_column_values`** — dynamically fetches distinct values from a column at compile time, powering the `pivot` macro above and enabling data-driven logic without hardcoding lists.

### dbt-utils Generic Tests

dbt-utils ships extra schema tests you can apply in `schema.yml`:

```yaml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - dbt_utils.not_empty_string
          - dbt_utils.expression_is_true:
              expression: "total_amount >= 0"
      - name: customer_id
        tests:
          - dbt_utils.relationships_where:
              to: ref('dim_customers')
              field: customer_id
              from_condition: "status != 'test'"
```

### Best Practices

- **Pin exact versions** in `packages.yml` — `">=1.0.0,<2.0.0"` ranges can break your project when a major update ships.
- **Run `dbt deps` in CI** before `dbt build` so every pipeline has fresh, verified packages.
- **Namespace your own macros** with a project prefix (e.g. `myco_generate_sk`) to avoid conflicts with package macros of the same name.
- **Internal packages**: split shared macros into a private Git-hosted package so multiple dbt projects consume the same tested logic rather than copy-pasting.

Packages dramatically reduce boilerplate; leaning on `dbt-utils` for surrogate keys, date spines, and cross-database SQL abstractions is one of the highest-leverage moves in a dbt project.

## 05:03 - dbt Model Contracts: Enforcing Data Schemas at Build Time

dbt model contracts (introduced in dbt-core 1.5) let you declare a **binding schema** for a model — column names, data types, and constraints — and have dbt verify that the compiled SQL actually produces those columns before the model is materialized. Think of it as an interface contract between model authors and downstream consumers.

### Enabling a Contract

Add `contract.enforced: true` under the model's `config` block in `schema.yml`:

```yaml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: varchar
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_total
        data_type: numeric
      - name: created_at
        data_type: timestamp
```

When `enforced: true`, dbt will **fail at compile/run time** if the model's SELECT list does not include every declared column with the correct data type. This catches schema drift before bad data reaches your warehouse.

### Constraints

dbt supports several constraint types (warehouse support varies):

| Constraint | Description |
|---|---|
| `not_null` | Column cannot contain NULLs |
| `primary_key` | Uniqueness + not null (DDL-level on supported warehouses) |
| `unique` | Unique values within the column |
| `foreign_key` | Referential integrity to another table |
| `check` | Custom SQL expression (e.g. `order_total >= 0`) |

```yaml
- name: order_total
  data_type: numeric
  constraints:
    - type: check
      expression: "order_total >= 0"
```

### Model-Level vs Column-Level Constraints

Constraints can also be declared at the model level for multi-column keys:

```yaml
models:
  - name: fct_order_lines
    config:
      contract:
        enforced: true
    constraints:
      - type: primary_key
        columns: [order_id, line_item_id]
    columns:
      - name: order_id
        data_type: bigint
      - name: line_item_id
        data_type: integer
```

### Contracts and Versioned Models

Contracts are most powerful when combined with **model versioning**. When you bump a model to v2, the contract for v1 remains frozen, giving downstream consumers time to migrate without breaking:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1
        config:
          contract:
            enforced: true
      - v: 2
        config:
          contract:
            enforced: true
```

Downstream models reference a specific version with `ref('fct_orders', v=1)` — they won't silently receive breaking schema changes.

### Best Practices

- **Apply contracts to public models first** — those consumed by other teams or downstream tools (BI, ML) benefit most from a guaranteed schema.
- **Keep data types warehouse-agnostic** where possible (`varchar`, `numeric`, `boolean`, `timestamp`) for portability; use adapter-specific types only when you need precision (e.g. `varchar(255)` on Redshift).
- **Combine with dbt tests** — contracts enforce structure; tests enforce data quality. Both layers are needed.
- **Start enforcing in staging, not production** — surface contract violations in CI (`dbt build --select fct_orders`) before they can affect live dashboards.

Model contracts shift schema governance left, catching breaking changes at authoring time rather than discovering them when a downstream dashboard breaks at 9 AM.

## 10:02 - dbt Snapshots: Tracking Slowly Changing Dimensions

dbt **snapshots** give you a built-in mechanism for capturing historical change in mutable source tables — the classic Type 2 Slowly Changing Dimension (SCD2) pattern — without writing any custom ETL infrastructure. Every time you run `dbt snapshot`, dbt compares the source to the existing snapshot table and inserts new rows for records that changed, recording when each version was valid.

### Defining a Snapshot

Create a `.sql` file inside your `snapshots/` directory and wrap it in a `{% snapshot %}` block:

```sql
{% snapshot snap_customers %}

{{
    config(
        target_schema='snapshots',
        strategy='timestamp',
        unique_key='customer_id',
        updated_at='updated_at',
        invalidate_hard_deletes=True
    )
}}

select
    customer_id,
    email,
    plan_tier,
    status,
    updated_at
from {{ source('app_db', 'customers') }}

{% endsnapshot %}
```

Run with:

```bash
dbt snapshot
```

### Snapshot Strategies

| Strategy | When to use | Required config |
|---|---|---|
| `timestamp` | Source has a reliable `updated_at` column | `unique_key`, `updated_at` |
| `check` | No reliable timestamp; compare specific columns | `unique_key`, `check_cols` |

**Check strategy example** — tracks changes across selected columns even without an `updated_at`:

```sql
{{
    config(
        strategy='check',
        unique_key='product_id',
        check_cols=['price', 'category', 'is_active']
    )
}}
```

### Output Columns

dbt automatically adds four metadata columns to every snapshot table:

| Column | Meaning |
|---|---|
| `dbt_scd_id` | Unique surrogate key for each snapshot row |
| `dbt_updated_at` | When this row was last evaluated |
| `dbt_valid_from` | Timestamp this version became active |
| `dbt_valid_to` | Timestamp this version was superseded (NULL = current) |

### Querying Snapshots

To get the **current record** for each customer:

```sql
select *
from {{ ref('snap_customers') }}
where dbt_valid_to is null
```

To **reconstruct state at a point in time** (e.g. what plan each customer had on 2025-01-01):

```sql
select *
from {{ ref('snap_customers') }}
where dbt_valid_from <= '2025-01-01'
  and (dbt_valid_to > '2025-01-01' or dbt_valid_to is null)
```

### Hard Deletes

With `invalidate_hard_deletes=True`, rows that disappear from the source get their `dbt_valid_to` stamped with the current snapshot time rather than being silently orphaned — critical for regulatory audit trails and churn analysis.

### Best Practices

- **Run snapshots before models** in your orchestration pipeline so downstream models always see the latest history (e.g. in Airflow: `dbt snapshot → dbt run`).
- **Snapshot as close to the source as possible** — snapshot raw/staged data, not already-transformed models, to minimize data loss from upstream filters.
- **Don't snapshot unnecessarily** — snapshots grow indefinitely; only apply SCD2 to entities where historical state genuinely matters (customers, accounts, pricing).
- **Index `dbt_valid_to`** on large snapshot tables in your warehouse for fast "current record" queries.

Snapshots are one of dbt's most underused features outside of analytics engineering teams — they eliminate a whole class of custom CDC pipelines when your source system doesn't expose change history natively.

## 15:03 - dbt Artifacts: Understanding manifest.json, run_results.json, and catalog.json

Every `dbt` invocation writes **artifact files** into the `target/` directory. These JSON files are the backbone of state-based CI, documentation, observability integrations, and the dbt Cloud UI — understanding them unlocks a lot of advanced dbt tooling.

### The Three Core Artifacts

**`manifest.json`** — the complete compiled graph of your project. Generated by `dbt compile`, `dbt run`, `dbt build`, etc.

Key contents:
- Every node (model, test, snapshot, seed, macro) with its compiled SQL, config, and metadata
- The full DAG expressed as `depends_on.nodes` relationships
- Column-level descriptions from `schema.yml`

```bash
# Inspect a single model's compiled SQL from the manifest
cat target/manifest.json | jq '.nodes["model.my_project.fct_orders"].compiled_code'
```

**`run_results.json`** — execution results from the most recent `dbt run/test/build`.

```bash
# Find all failed nodes from the last run
cat target/run_results.json | jq '[.results[] | select(.status == "error") | {node: .unique_id, message: .message}]'
```

**`catalog.json`** — warehouse introspection output from `dbt docs generate`. Contains the actual columns, types, and row counts as seen in the warehouse — not just what's declared in `schema.yml`.

```bash
dbt docs generate   # writes catalog.json
dbt docs serve      # serves the docs UI locally on port 8080
```

### State-Based Builds with manifest.json

The most powerful use of artifacts is **deferred state** in CI: compare the current branch's manifest against the production manifest to run only changed models and their downstream dependencies.

```bash
# In CI: fetch production manifest, then run only what changed
dbt run \
  --select "state:modified+" \
  --defer \
  --state ./prod-manifest/
```

`--defer` tells dbt to resolve `ref()` calls for *unselected* nodes against the production environment rather than rebuilding them — so a PR that only touches `fct_orders` doesn't need to rebuild all its upstream staging models.

### Storing Artifacts for State Comparison

In practice you need to persist the production manifest somewhere CI can fetch it:

```yaml
# Example: GitHub Actions step storing the artifact
- name: Download prod manifest
  run: |
    aws s3 cp s3://my-dbt-artifacts/prod/manifest.json ./prod-manifest/manifest.json

- name: Run slim CI
  run: |
    dbt build --select "state:modified+" --defer --state ./prod-manifest/
```

dbt Cloud handles this automatically via its **"Defer to production"** setting — it stores and retrieves artifacts between runs without any extra configuration.

### Parsing Artifacts in Python

The `dbt-artifacts` and `elementary` packages expose parsed artifact models directly in your warehouse. For raw access, the schemas are stable and documented:

```python
import json

with open("target/run_results.json") as f:
    results = json.load(f)

failures = [
    r for r in results["results"]
    if r["status"] in ("error", "fail")
]
for f in failures:
    print(f["unique_id"], f["message"])
```

### Best Practices

- **Always archive `manifest.json` from your production job** — this single file unlocks slim CI, drift detection, and cross-environment comparison.
- **Use `catalog.json` to catch schema drift** — diff it between releases to spot unexpected column additions, type changes, or dropped columns.
- **Don't commit `target/`** — artifacts are generated outputs; add `target/` to `.gitignore` and store artifacts externally (S3, GCS, dbt Cloud).
- **Elementary and re_data** both consume `run_results.json` to power data observability dashboards with zero extra instrumentation.

Artifacts make dbt a first-class citizen in DataOps pipelines — they're the API surface that everything from Slim CI to lineage visualization to alerting systems builds on.

## 19:02 - dbt Access & Model Groups: Enforcing Visibility Boundaries Across Teams

dbt **model groups** and the `access` property (introduced in dbt-core 1.5) give large data teams a way to enforce ownership and encapsulation directly in the project. They answer the question: *which models are internal implementation details vs. stable interfaces that other teams can depend on?*

### Model Access Levels

Every model has an `access` property with one of three values:

| Level | Who can `ref()` it | Default |
|---|---|---|
| `private` | Only models in the same group | No |
| `protected` | Any model in the same project | Yes (the current default) |
| `public` | Any model in any project (cross-project refs) | No |

### Defining Groups and Access

Declare groups in `schema.yml`. Each group has an `owner` (team or person) and models that belong to it:

```yaml
# models/finance/schema.yml
groups:
  - name: finance
    owner:
      name: Finance Analytics Team
      email: analytics@finance.example.com

models:
  - name: stg_invoices
    group: finance
    config:
      access: private       # internal staging — other teams can't ref() this

  - name: int_invoice_aggregates
    group: finance
    config:
      access: private       # intermediate logic, also hidden

  - name: fct_revenue
    group: finance
    config:
      access: public        # stable interface — other teams can depend on this
```

### Enforcing Access at Runtime

If a model outside the `finance` group tries to `ref('stg_invoices')` — a `private` model — dbt raises an error at parse time:

```
Error: Model 'marketing.fct_campaigns' attempted to reference 'finance.stg_invoices',
which is private to the 'finance' group.
```

This prevents teams from reaching into another team's internal models and building silent dependencies that break when implementation changes.

### Cross-Project References (dbt Mesh)

`public` models are the foundation of **dbt Mesh** — dbt's multi-project architecture. A downstream project references a public model from an upstream project using a special `ref()` syntax:

```sql
-- In the marketing dbt project, referencing finance's public model
select *
from {{ ref('finance', 'fct_revenue') }}
```

The upstream project must declare the model as `public` and typically enforce a contract on it, since downstream teams are taking a dependency on its schema:

```yaml
models:
  - name: fct_revenue
    group: finance
    config:
      access: public
      contract:
        enforced: true
    columns:
      - name: revenue_date
        data_type: date
      - name: total_revenue_usd
        data_type: numeric
```

### Listing Public Models

You can query which models a project exposes to others:

```bash
dbt ls --select "config.access:public"
```

### Best Practices

- **Default to `private` for staging models** — `stg_*` models are transformation internals; downstream teams should consume marts, not raw staging logic.
- **Apply contracts to every `public` model** — an uncontracted public model is a promise with no guarantee; consumers can't safely build on it.
- **Use groups to mirror your org structure** — one group per team or domain (finance, marketing, supply-chain) makes ownership clear and searchable in `dbt docs`.
- **Treat `public` promotion like an API release** — once other projects depend on a public model, changing its schema is a breaking change. Version the model (`v1`, `v2`) rather than mutating in place.

Model groups and access levels bring software-engineering encapsulation to dbt projects, making it safe for multiple teams to share a warehouse without stepping on each other's internal models.
