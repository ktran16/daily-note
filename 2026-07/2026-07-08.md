## 00:02 - dbt Packages: Extending dbt with dbt-utils and the Package Hub

dbt packages let you import pre-built macros, models, and tests directly into your project â€” think of them as npm or pip packages but for dbt. The primary registry is [dbt Hub](https://hub.getdbt.com/), and the community package `dbt-utils` is the one nearly every dbt project installs.

### Installing Packages

Declare packages in `packages.yml` at the root of your project:

```yaml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0
  - package: calogica/dbt_expectations
    version: 0.10.3
  - git: "https://github.com/your-org/internal-dbt-macros.git"
    revision: v1.2.0
```

Then run:

```bash
dbt deps
```

This downloads packages into the `dbt_packages/` directory (add it to `.gitignore`).

### Key dbt-utils Macros

**`generate_surrogate_key`** â€” creates a hashed surrogate key from one or more columns, avoiding warehouse-specific `MD5` / `SHA256` calls:

```sql
select
    {{ dbt_utils.generate_surrogate_key(['order_id', 'line_item_id']) }} as sk_order_line,
    order_id,
    line_item_id,
    product_name
from {{ ref('stg_order_lines') }}
```

**`date_spine`** â€” generates a contiguous sequence of dates, invaluable for filling gaps in time-series data:

```sql
with date_series as (
    {{ dbt_utils.date_spine(
        datepart="day",
        start_date="cast('2024-01-01' as date)",
        end_date="current_date"
    ) }}
)
select
    date_day,
    coalesce(revenue, 0) as revenue
from date_series
left join {{ ref('fct_daily_revenue') }} using (date_day)
```

**`pivot`** â€” turns row values into columns without hard-coding every case:

```sql
select
    store_id,
    {{ dbt_utils.pivot(
        column='status',
        values=['pending', 'shipped', 'delivered', 'returned'],
        agg='count',
        then_value='order_id'
    ) }}
from {{ ref('stg_orders') }}
group by 1
```

**`get_column_values`** â€” dynamically fetches distinct values from a column at compile time, powering the `pivot` macro above and enabling data-driven logic without hardcoding lists.

### dbt-utils Generic Tests

dbt-utils ships extra schema tests you can apply in `schema.yml`:

```yaml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - dbt_utils.not_empty_string
          - dbt_utils.expression_is_true:
              expression: "total_amount >= 0"
      - name: customer_id
        tests:
          - dbt_utils.relationships_where:
              to: ref('dim_customers')
              field: customer_id
              from_condition: "status != 'test'"
```

### Best Practices

- **Pin exact versions** in `packages.yml` â€” `">=1.0.0,<2.0.0"` ranges can break your project when a major update ships.
- **Run `dbt deps` in CI** before `dbt build` so every pipeline has fresh, verified packages.
- **Namespace your own macros** with a project prefix (e.g. `myco_generate_sk`) to avoid conflicts with package macros of the same name.
- **Internal packages**: split shared macros into a private Git-hosted package so multiple dbt projects consume the same tested logic rather than copy-pasting.

Packages dramatically reduce boilerplate; leaning on `dbt-utils` for surrogate keys, date spines, and cross-database SQL abstractions is one of the highest-leverage moves in a dbt project.
