## 00:02 - dbt Exposures: Documenting Downstream Consumers and Closing the Lineage Loop

### What Are Exposures?

Most dbt projects carefully document models, sources, and tests — but the lineage graph often stops at the warehouse. **Exposures** let you declare what consumes your dbt models downstream: dashboards, reports, ML pipelines, APIs, or any external artifact. They close the lineage loop so your team knows who depends on what before making a breaking change.

Exposures live in `.yml` files (typically `models/exposures.yml`) and are first-class citizens in dbt docs.

---

### Defining an Exposure

```yaml
# models/exposures.yml
exposures:
  - name: weekly_revenue_dashboard
    type: dashboard
    maturity: high
    url: https://bi.company.com/dashboards/revenue
    description: >
      Weekly revenue dashboard used by the Finance team.
      Pulls from the fct_orders and dim_customers models.
    depends_on:
      - ref('fct_orders')
      - ref('dim_customers')
    owner:
      name: Sarah Chen
      email: sarah.chen@company.com
    tags:
      - finance
      - revenue
```

**Key fields:**
- `type` — `dashboard`, `notebook`, `analysis`, `ml`, `application`
- `maturity` — `low`, `medium`, `high` (signals how stable the exposure is)
- `depends_on` — accepts `ref()` and `source()` — this is what builds the lineage edge
- `owner` — who is responsible if something breaks

---

### Why Exposures Matter

**1. Impact analysis before refactoring.** When you run `dbt ls --select fct_orders+`, exposures appear in the output, so you know a Finance dashboard will break before you rename a column.

**2. Richer DAG in dbt docs.** Exposures render as terminal nodes in `dbt docs generate` — your stakeholders can trace data all the way from raw source to the Tableau workbook they use every day.

**3. Freshness SLAs.** Combine exposures with source freshness checks to build end-to-end SLA reporting: if an upstream source is stale, you know exactly which dashboards are at risk.

---

### Selecting Exposures in dbt Commands

```bash
# Build all models that feed a specific exposure
dbt build --select +exposure:weekly_revenue_dashboard

# List everything an exposure depends on
dbt ls --select +exposure:weekly_revenue_dashboard --resource-type model
```

The `+` graph operator traverses all parents, so you can run only the models that matter for a given downstream consumer — useful for targeted refreshes.

---

### Best Practice: One Exposure Per Consumer

Resist the urge to create a single "all dashboards" exposure. A 1:1 mapping between exposures and actual consumers gives you granular impact analysis and makes ownership unambiguous. If a dashboard is decommissioned, delete its exposure — it's a lightweight, high-value record of intent.

Exposures cost almost nothing to maintain and pay off the moment someone asks "what breaks if I change this model?"

## 05:02 - dbt Semantic Layer & MetricFlow: One Metric Definition, Every Tool

### The Problem: Metric Sprawl

In most data teams, the same metric gets defined in five different places — a Looker LookML file, a Tableau calculated field, a Jupyter notebook, a dbt model, and a Slack reply. They all disagree on edge cases: do we include refunded orders? Which date field drives "revenue this month"? The dbt Semantic Layer, powered by **MetricFlow**, solves this by making metrics a first-class dbt artifact: define once, query everywhere.

---

### How It Works

MetricFlow introduces two YAML abstractions on top of your existing dbt models:

**1. Semantic Models** — declare the "grain" of a model and its dimensions/measures.

```yaml
# models/semantics/orders.yml
semantic_models:
  - name: orders
    model: ref('fct_orders')
    entities:
      - name: order_id
        type: primary
      - name: customer_id
        type: foreign
    dimensions:
      - name: order_date
        type: time
        type_params:
          time_granularity: day
      - name: status
        type: categorical
    measures:
      - name: revenue
        agg: sum
        expr: amount_usd
      - name: order_count
        agg: count_distinct
        expr: order_id
```

**2. Metrics** — build on measures with optional filters, ratios, and derived expressions.

```yaml
metrics:
  - name: monthly_revenue
    type: simple
    label: Monthly Revenue (USD)
    type_params:
      measure: revenue

  - name: refund_rate
    type: ratio
    label: Refund Rate
    type_params:
      numerator: refund_count
      denominator: order_count
    filter: |
      {{ Dimension('orders__status') }} != 'cancelled'

  - name: revenue_7d_avg
    type: cumulative
    label: 7-Day Rolling Revenue
    type_params:
      measure: revenue
      window: 7 days
```

---

### Querying Metrics with `mf query`

Once defined, MetricFlow's CLI lets you query metrics with arbitrary dimension slices — no raw SQL needed:

```bash
# Revenue by month and status
mf query \
  --metrics monthly_revenue \
  --group-by metric_time__month,orders__status

# Refund rate for the last 30 days
mf query \
  --metrics refund_rate \
  --start-time 2026-06-03 \
  --end-time 2026-07-03
```

MetricFlow generates and executes the correct SQL against your warehouse, handling fan-out joins, time-spine alignment, and dimension expansion automatically.

---

### dbt Cloud Semantic Layer: BI Tool Integration

In dbt Cloud (Team/Enterprise), the Semantic Layer exposes a **JDBC/GraphQL API** that BI tools query directly:

- **Tableau**, **Looker**, **Hex**, **Mode**, **Excel** connect via the dbt Semantic Layer integration and pull live metric results without any SQL.
- Metric definitions update in dbt → every connected tool sees the change instantly.
- No more "which revenue definition is correct?" — there's only one.

---

### When to Adopt

The Semantic Layer pays off most when:
1. Multiple BI tools or notebooks consume the same metrics
2. You have ratio or cumulative metric types that are tedious to replicate in SQL
3. Metric definitions cause disagreements across teams

For small teams with a single BI tool, well-named `fct_` models with a metrics mart layer are often sufficient. But as your org scales, the Semantic Layer eliminates the #1 data trust problem: inconsistent numbers.

## 10:02 - dbt Slim CI: State-Based Builds for Faster CI Pipelines

### The Problem with Full Refreshes in CI

Running `dbt build` on every pull request works fine for small projects, but once your DAG has hundreds of models it becomes painfully slow and expensive. A PR that touches two models should not rebuild the entire warehouse. **Slim CI** is dbt's answer: build only what changed, relative to a known-good production state.

---

### How Slim CI Works

dbt tracks the compiled state of your project in an **artifacts** file (`manifest.json`). Slim CI compares the current PR's manifest against production's manifest and uses the `state:modified` selector to limit what gets built.

**Step 1 — Fetch the production manifest in CI.**

In dbt Cloud, the "Defer to production" feature does this automatically. In dbt Core + any CI runner, you download the latest production `manifest.json` before running:

```bash
# Example: fetch from an S3 artifact store
aws s3 cp s3://my-dbt-artifacts/prod/manifest.json ./prod/manifest.json
```

**Step 2 — Run only modified models and their downstream dependents.**

```bash
dbt build \
  --select state:modified+ \
  --defer \
  --state ./prod/
```

- `state:modified` — selects every model whose SQL, config, or upstream ref changed
- `+` — also selects all downstream dependents (so broken children are caught)
- `--defer` — for nodes NOT selected, resolve `ref()` calls against the production schema instead of rebuilding them

---

### What Gets Caught as "Modified"

| Change type | Selected? |
|---|---|
| SQL body changed | Yes |
| `config()` block changed (e.g., materialization) | Yes |
| Upstream model changed | Yes (via `+`) |
| Only a test changed | Yes (tests are state-aware too) |
| Unrelated model | No |

---

### GitHub Actions Example

```yaml
# .github/workflows/dbt-ci.yml
jobs:
  slim-ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Download production manifest
        run: |
          aws s3 cp s3://my-dbt-artifacts/prod/manifest.json prod/manifest.json

      - name: dbt build (modified only)
        run: |
          dbt deps
          dbt build \
            --select state:modified+ \
            --defer \
            --state prod/ \
            --target ci
        env:
          DBT_PROFILES_DIR: .
```

The `--target ci` flag points to a separate CI schema (e.g., `dbt_ci_pr_123`) so PR runs never pollute production tables.

---

### dbt Cloud: Slim CI Is Built In

In dbt Cloud, enable **"Defer to a previous run state"** on your CI job and point it at your production job. dbt Cloud fetches the manifest automatically — no S3 bucket or custom scripting needed.

---

### Key Benefit: Cost and Speed

A project with 300 models where a PR changes 4 models might run in 2 minutes instead of 40. Over hundreds of PRs per month, Slim CI pays for itself in compute costs and developer feedback speed alone. It's one of the highest-leverage practices a growing dbt team can adopt.

## 15:02 - dbt Model Contracts: Enforcing Schema Stability at the Warehouse Level

### Why Contracts Exist

dbt models are flexible by design — you can rename a column or drop a field and the project compiles just fine. But downstream consumers (BI tools, APIs, other dbt projects) break silently. **Model contracts** (introduced in dbt Core 1.5) let you declare a model's schema as a binding promise: if the compiled output doesn't match the contract, the build fails before any data is written.

---

### Defining a Contract

Add `contract: {enforced: true}` inside the model's `config` block in YAML, then list every column with its data type:

```yaml
# models/marts/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: amount_usd
        data_type: numeric
      - name: status
        data_type: varchar
```

At run time, dbt compares the contract against the model's actual compiled columns. Any mismatch — wrong type, missing column, extra undeclared column — raises a hard error before the `CREATE TABLE` or `INSERT` runs.

---

### Supported Constraints

| Constraint type | Enforced in warehouse? | Notes |
|---|---|---|
| `not_null` | Yes (most adapters) | Column-level |
| `primary_key` | Yes (most adapters) | Column or model level |
| `unique` | Yes (some adapters) | Column-level |
| `foreign_key` | Partial | Snowflake, BigQuery enforce at query time only |
| `check` | Partial | Adapter-dependent |
| `custom` | Yes | Raw SQL expression via `expression:` |

Not every database enforces constraints the same way — Snowflake accepts `NOT NULL` DDL but doesn't check it on insert by default. Pair warehouse constraints with dbt tests (`not_null`, `unique`) for full coverage across all adapters.

---

### Constraints at the Model Level

Some constraints span multiple columns and must be declared at the model level:

```yaml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    constraints:
      - type: primary_key
        columns: [order_id, order_date]   # composite PK
      - type: check
        expression: "amount_usd >= 0"
        name: chk_non_negative_amount
    columns:
      - name: order_id
        data_type: bigint
      - name: order_date
        data_type: date
      - name: amount_usd
        data_type: numeric
```

---

### Contracts + Versioning: A Powerful Combo

Model contracts pair naturally with **model versioning** (also dbt 1.5+). When you need to change a contracted model's schema, instead of breaking downstream consumers you:

1. Create `fct_orders_v2` with the new schema.
2. Bump `latest_version: 2` in the config.
3. Give consumers time to migrate from `ref('fct_orders', v=1)` to `ref('fct_orders', v=2)`.
4. Deprecate v1 once all consumers are migrated.

This lets an API or BI tool pin to a stable schema version while the data team iterates safely.

---

### When to Enforce Contracts

Contracts add overhead — every column must be declared and typed. Apply them to:

- **Public models** consumed by other dbt projects or external tools.
- **Models backing production dashboards or SLA-bound APIs.**
- Any model where an undeclared schema change has caused an incident before.

Avoid contracts on exploratory or rapidly changing models in `staging/`; enforce them at the `marts/` boundary where stability is expected. The moment a model graduates from "internal" to "public interface," a contract is the right tool to signal that commitment.
