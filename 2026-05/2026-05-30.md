## 14:18 - Feature Stores: The Missing Infrastructure Layer in ML Systems

A **feature store** is a centralized repository for storing, sharing, and serving machine learning features — and it solves one of the most underappreciated problems in production ML: feature inconsistency between training and inference.

### The Training-Serving Skew Problem

Without a feature store, teams often compute features twice: once in Python/Spark for model training and again in a backend language like Java or Go for real-time serving. These two implementations inevitably drift apart, causing model degradation that's notoriously hard to debug. A feature store eliminates this by defining features once and materializing them for both offline and online access.

### How Feature Stores Work

A feature store has two main serving layers:

- **Offline store** (e.g., S3, BigQuery, Hive): Stores historical feature values for training and batch inference. Optimized for high-throughput reads of large datasets.
- **Online store** (e.g., Redis, DynamoDB, Cassandra): Stores the latest feature values for low-latency real-time inference. Optimized for point lookups by entity key (e.g., `user_id`, `item_id`).

A **materialization job** continuously moves data from the offline store to the online store, ensuring freshness.

### Point-in-Time Correctness

One of the most valuable guarantees a feature store provides is **point-in-time correct** feature retrieval for training. When constructing a training dataset, it returns the feature value as it existed *at the time of the training event* — not the current value. This prevents **data leakage**, where future information bleeds into training labels.

### When to Adopt One

Feature stores pay off when:
1. Multiple models share the same features (reusability).
2. Features are expensive to compute and should be cached.
3. You need sub-10ms feature lookup latency in production.
4. Your team needs to audit what feature values a model saw for a given prediction.

Popular open-source options include **Feast** and **Hopsworks**. Cloud-managed options include AWS SageMaker Feature Store, GCP Vertex AI Feature Store, and Databricks Feature Store.

The key insight: a feature store is not just a performance optimization — it's a correctness and governance tool that makes ML systems reproducible and auditable.

## 15:03 - Apache Iceberg: Why the Table Format Revolution Matters

**Apache Iceberg** is an open table format for large analytic datasets that is rapidly replacing the traditional Hive table format in modern data lakehouses. Understanding what it solves explains why virtually every major data platform — Snowflake, Databricks, AWS, Google Cloud — has adopted it.

### The Problem with Traditional Data Lakes

Classic Parquet-on-S3 data lakes had three crippling weaknesses:

1. **No ACID transactions**: Two simultaneous writes could corrupt a table, so teams serialized all writes or used fragile locking mechanisms.
2. **Expensive planning**: Query engines had to list every file in a partition to plan a query — on large tables, this S3 `LIST` operation alone could take minutes.
3. **No time travel**: Once data was overwritten or deleted, it was gone. Auditing historical states required maintaining separate snapshots manually.

### What Iceberg Adds

Iceberg sits as a **metadata layer** on top of your existing object storage. It maintains a tree of metadata files (snapshot → manifest list → manifest files → data files) that lets the query engine know exactly which Parquet files belong to the current table state without scanning the entire storage prefix.

Key capabilities this unlocks:

- **ACID commits via optimistic concurrency**: Each write creates a new snapshot atomically. Readers always see a consistent snapshot; writers fail fast on conflicts rather than silently corrupting data.
- **Hidden partitioning**: Iceberg tracks partition evolution in metadata, so you can repartition a table without rewriting data or breaking existing queries.
- **Time travel & rollback**: Query any historical snapshot with `AS OF TIMESTAMP` or `AS OF VERSION`. Roll back a bad write in seconds.
- **Incremental reads**: Downstream pipelines can ask "what changed since snapshot X?" — critical for building efficient CDC pipelines without full table scans.

### The Interoperability Dividend

Because Iceberg is an open standard, you can write from Spark, read from Trino, and query directly in Snowflake — all against the same physical files. This **multi-engine interoperability** is the core reason it has become the default choice for new data lakehouse builds in 2025-2026.

The practical takeaway: if you're building a new data platform on object storage today, defaulting to Iceberg instead of raw Parquet gives you ACID correctness, faster query planning, and zero-cost time travel with no additional infrastructure.

## 19:02 - RAG Beyond Naive Chunking: Advanced Retrieval Patterns That Actually Work

Retrieval-Augmented Generation (RAG) is now mainstream, but most production deployments suffer from the same problem: naive chunking produces retrievals that are semantically incomplete, causing LLMs to hallucinate or hedge even when the answer exists in the knowledge base. Moving beyond fixed-size chunking is the single highest-leverage improvement in most RAG systems.

### Why Naive Chunking Fails

Splitting documents into 512-token windows destroys context. A sentence like "It increased by 23% year-over-year" is meaningless without knowing what "it" refers to — information that may live in a prior chunk that was never retrieved. The embedding model then produces a vector for decontextualized text, leading to poor recall.

### Pattern 1: Hierarchical (Parent-Child) Retrieval

Store two granularities: small **child chunks** (128 tokens) for precise embedding-based retrieval, and larger **parent chunks** (512–1024 tokens) that preserve surrounding context. At query time, retrieve child chunks by similarity, then return their parent chunks to the LLM. This gives you precision at retrieval and context at generation — the best of both worlds.

### Pattern 2: Contextual Chunk Headers

Before embedding each chunk, prepend a generated header summarizing the document and section it came from: `"Document: Q3 2025 Earnings Report. Section: Revenue Breakdown. {chunk text}"`. This forces the embedding to carry document-level context, dramatically improving recall for generic queries.

### Pattern 3: Hybrid Search (Dense + Sparse)

Pure vector search struggles with exact-match queries — product codes, names, IDs. Combine dense retrieval (embeddings via cosine similarity) with sparse retrieval (BM25 keyword search) using **Reciprocal Rank Fusion (RRF)** to merge the result lists. Most modern vector DBs (Weaviate, Elasticsearch, Azure AI Search) support hybrid search natively.

### Pattern 4: Reranking as a Second Pass

After retrieving the top-K candidates, run a **cross-encoder reranker** (e.g., Cohere Rerank, BGE-Reranker) that scores each document against the query jointly rather than independently. Cross-encoders are too slow to rank all documents but are far more accurate than bi-encoders for the top-20 candidates. This typically improves answer quality by 10–20% with minimal latency overhead.

### The Practical Stack

For a production RAG pipeline: **hybrid search → parent-child retrieval → reranking → LLM**. Each layer is independently testable. Measure retrieval quality separately from generation quality using metrics like NDCG@10 and context recall — don't wait for end-to-end answer quality to diagnose retrieval failures.
