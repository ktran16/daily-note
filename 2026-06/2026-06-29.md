## 00:06 - dbt Incremental Model Strategies: append, merge, delete+insert, and microbatch

Incremental models let dbt process only new or changed rows instead of rebuilding an entire table from scratch. The `incremental_strategy` config controls *how* those rows are written, and choosing the right strategy has a big impact on performance, correctness, and cost.

### The Four Core Strategies

**1. `append` (default on some adapters)**
Simply inserts new rows. No deduplication, no updates. Use it when your source data is immutable event logs and you're certain there are no late-arriving or corrected rows.

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='append'
) }}

select * from {{ ref('stg_events') }}
{% if is_incremental() %}
  where event_timestamp > (select max(event_timestamp) from {{ this }})
{% endif %}
```

**2. `merge` (default on BigQuery, Snowflake, Databricks)**
Issues a `MERGE` statement (upsert) — matching on a `unique_key`, updating existing rows, and inserting new ones. Best for mutable records like user profiles or order statuses.

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='merge',
    unique_key='order_id'
) }}

select order_id, customer_id, status, updated_at
from {{ ref('stg_orders') }}
{% if is_incremental() %}
  where updated_at > (select max(updated_at) from {{ this }})
{% endif %}
```

**3. `delete+insert` (Snowflake, Spark, Redshift)**
Deletes matching rows first, then inserts the full replacement set. More predictable than `merge` for adapters where `MERGE` has edge cases, and useful when you want to re-process entire partitions cleanly.

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='delete+insert',
    unique_key='date_day'
) }}
```

**4. `microbatch` (dbt 1.9+, event-oriented models)**
A newer strategy designed for high-volume event streams. dbt automatically batches the incremental load into fixed time windows (e.g., hourly) and retries each batch independently. Requires an `event_time` column to be declared.

```yaml
models:
  - name: fct_page_views
    config:
      materialized: incremental
      incremental_strategy: microbatch
      event_time: viewed_at
      batch_size: hour
      lookback: 3
```

### Key Config Options

| Config | Purpose |
|--------|---------|
| `unique_key` | Column(s) used for matching in `merge` / `delete+insert` |
| `on_schema_change` | `ignore`, `fail`, `append_new_columns`, or `sync_all_columns` |
| `partition_by` | Limit the merge/delete scan to relevant partitions (BigQuery especially) |

### Choosing a Strategy

- **Immutable events, append-only source** → `append`
- **Mutable records with a natural key** → `merge`
- **Partitioned tables, replace full partitions** → `delete+insert`
- **High-throughput event tables needing reliable backfill** → `microbatch`

Always pair your incremental filter with the right `unique_key` — a missing or wrong key is the most common source of duplicates in incremental models.

## 05:04 - dbt Exposures: Documenting Downstream Consumers of Your Models

Exposures let you declare *what consumes* your dbt models — dashboards, ML pipelines, APIs, reverse ETL jobs — directly inside your dbt project. They close the lineage gap: without them, the DAG ends at your final mart models, leaving stakeholders blind to which business reports or applications depend on which data.

### Why Exposures Matter

- **End-to-end lineage**: dbt docs shows a full graph from raw sources → staging → marts → BI dashboard.
- **Impact analysis**: run `dbt ls --select +exposure:weekly_revenue_report` to find every model that feeds a given exposure before making a breaking change.
- **Ownership clarity**: each exposure records an owner and type, making it obvious who to notify when upstream models change.

### Defining an Exposure

Exposures live in `.yml` files (typically `models/exposures.yml`):

```yaml
exposures:
  - name: weekly_revenue_report
    label: Weekly Revenue Report
    type: dashboard          # dashboard | notebook | analysis | ml | application
    maturity: high           # low | medium | high
    url: https://looker.example.com/dashboards/42
    description: >
      Executive dashboard tracking weekly revenue by region and product line.
      Refreshes every Monday at 06:00 UTC.

    owner:
      name: Data Analytics Team
      email: analytics@example.com

    depends_on:
      - ref('fct_orders')
      - ref('dim_customers')
      - ref('dim_products')
```

### Selecting and Testing via Exposures

```bash
# Build everything that feeds this dashboard
dbt build --select +exposure:weekly_revenue_report

# Run tests only on models feeding the exposure
dbt test --select +exposure:weekly_revenue_report

# List all exposures in the project
dbt ls --resource-type exposure
```

### Best Practices

- **Register all production consumers**: even if a notebook is exploratory, promoting it to an exposure signals that upstream changes need coordination.
- **Set `maturity: high`** for any exposure used in executive reporting or SLA-bound pipelines — this appears prominently in dbt docs.
- **Include the URL**: a direct link to the Looker dashboard, Hex notebook, or Airflow DAG makes impact analysis instantaneous.
- **Pair with model contracts**: if an exposure depends on `fct_orders`, add a contract to that model so column renames fail at CI time rather than silently breaking the dashboard.

Exposures are low-effort, high-value documentation — they turn your dbt DAG into a complete map of your organisation's data dependencies.

## 10:04 - dbt Seeds: Loading Static Reference Data into Your Warehouse

Seeds are CSV files that dbt compiles into tables in your warehouse. They are the right tool for small, slowly-changing reference datasets — country codes, product category mappings, marketing channel labels, cost centres — that live better in version control than in a one-off `INSERT` script.

### Project Structure

Drop CSV files into the `seeds/` directory (configurable via `seed-paths` in `dbt_project.yml`):

```
seeds/
  country_codes.csv
  marketing_channels.csv
  finance/
    cost_centres.csv
```

Run them with:

```bash
dbt seed                         # load all seeds
dbt seed --select country_codes  # load a single seed
dbt seed --full-refresh          # drop and recreate (overwrite existing data)
```

### Configuring Seeds

Seeds follow the same config block pattern as models, set in `dbt_project.yml` or a `schema.yml` alongside the CSV:

```yaml
# dbt_project.yml
seeds:
  my_project:
    +schema: reference          # load into a dedicated schema
    +quote_columns: false
    finance:
      cost_centres:
        +column_types:
          cost_centre_id: varchar(10)
          budget_usd: numeric(18, 2)
```

For documentation and testing, treat seeds like models in `schema.yml`:

```yaml
seeds:
  - name: country_codes
    description: "ISO 3166-1 alpha-2 country codes and display names."
    columns:
      - name: country_code
        data_tests:
          - unique
          - not_null
      - name: country_name
        data_tests:
          - not_null
```

### Referencing Seeds in Models

Use `{{ ref() }}` exactly as you would for any model — dbt adds the seed to the DAG automatically:

```sql
-- models/marts/fct_orders_enriched.sql
select
    o.order_id,
    o.country_code,
    c.country_name,
    m.channel_label
from {{ ref('fct_orders') }} o
left join {{ ref('country_codes') }} c using (country_code)
left join {{ ref('marketing_channels') }} m using (channel_id)
```

### When to Use Seeds (and When Not To)

**Good fit:**
- Static lookup tables under ~1 MB that change infrequently
- Reference data that should be reviewed via code review (e.g., mapping tables owned by analysts)
- Data needed in CI that isn't available in staging environments

**Poor fit:**
- Tables over a few thousand rows — use a source + staging model instead
- Data that changes daily — seeds require a `dbt seed` run to update; they aren't pulled automatically
- Sensitive PII — CSV files in source control are not the right place for that data

### Pro Tip: Guarding Against Accidental Overwrites

By default, `dbt seed` will not truncate an existing table unless you pass `--full-refresh`. This is intentional — it prevents accidental data loss in production. Gate `--full-refresh` behind a CI flag or a manual approval step in your deployment pipeline.
