## 00:06 - dbt Incremental Model Strategies: append, merge, delete+insert, and microbatch

Incremental models let dbt process only new or changed rows instead of rebuilding an entire table from scratch. The `incremental_strategy` config controls *how* those rows are written, and choosing the right strategy has a big impact on performance, correctness, and cost.

### The Four Core Strategies

**1. `append` (default on some adapters)**
Simply inserts new rows. No deduplication, no updates. Use it when your source data is immutable event logs and you're certain there are no late-arriving or corrected rows.

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='append'
) }}

select * from {{ ref('stg_events') }}
{% if is_incremental() %}
  where event_timestamp > (select max(event_timestamp) from {{ this }})
{% endif %}
```

**2. `merge` (default on BigQuery, Snowflake, Databricks)**
Issues a `MERGE` statement (upsert) ā€” matching on a `unique_key`, updating existing rows, and inserting new ones. Best for mutable records like user profiles or order statuses.

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='merge',
    unique_key='order_id'
) }}

select order_id, customer_id, status, updated_at
from {{ ref('stg_orders') }}
{% if is_incremental() %}
  where updated_at > (select max(updated_at) from {{ this }})
{% endif %}
```

**3. `delete+insert` (Snowflake, Spark, Redshift)**
Deletes matching rows first, then inserts the full replacement set. More predictable than `merge` for adapters where `MERGE` has edge cases, and useful when you want to re-process entire partitions cleanly.

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='delete+insert',
    unique_key='date_day'
) }}
```

**4. `microbatch` (dbt 1.9+, event-oriented models)**
A newer strategy designed for high-volume event streams. dbt automatically batches the incremental load into fixed time windows (e.g., hourly) and retries each batch independently. Requires an `event_time` column to be declared.

```yaml
models:
  - name: fct_page_views
    config:
      materialized: incremental
      incremental_strategy: microbatch
      event_time: viewed_at
      batch_size: hour
      lookback: 3
```

### Key Config Options

| Config | Purpose |
|--------|---------|
| `unique_key` | Column(s) used for matching in `merge` / `delete+insert` |
| `on_schema_change` | `ignore`, `fail`, `append_new_columns`, or `sync_all_columns` |
| `partition_by` | Limit the merge/delete scan to relevant partitions (BigQuery especially) |

### Choosing a Strategy

- **Immutable events, append-only source** ā†’ `append`
- **Mutable records with a natural key** ā†’ `merge`
- **Partitioned tables, replace full partitions** ā†’ `delete+insert`
- **High-throughput event tables needing reliable backfill** ā†’ `microbatch`

Always pair your incremental filter with the right `unique_key` ā€” a missing or wrong key is the most common source of duplicates in incremental models.

## 05:04 - dbt Exposures: Documenting Downstream Consumers of Your Models

Exposures let you declare *what consumes* your dbt models ā€” dashboards, ML pipelines, APIs, reverse ETL jobs ā€” directly inside your dbt project. They close the lineage gap: without them, the DAG ends at your final mart models, leaving stakeholders blind to which business reports or applications depend on which data.

### Why Exposures Matter

- **End-to-end lineage**: dbt docs shows a full graph from raw sources ā†’ staging ā†’ marts ā†’ BI dashboard.
- **Impact analysis**: run `dbt ls --select +exposure:weekly_revenue_report` to find every model that feeds a given exposure before making a breaking change.
- **Ownership clarity**: each exposure records an owner and type, making it obvious who to notify when upstream models change.

### Defining an Exposure

Exposures live in `.yml` files (typically `models/exposures.yml`):

```yaml
exposures:
  - name: weekly_revenue_report
    label: Weekly Revenue Report
    type: dashboard          # dashboard | notebook | analysis | ml | application
    maturity: high           # low | medium | high
    url: https://looker.example.com/dashboards/42
    description: >
      Executive dashboard tracking weekly revenue by region and product line.
      Refreshes every Monday at 06:00 UTC.

    owner:
      name: Data Analytics Team
      email: analytics@example.com

    depends_on:
      - ref('fct_orders')
      - ref('dim_customers')
      - ref('dim_products')
```

### Selecting and Testing via Exposures

```bash
# Build everything that feeds this dashboard
dbt build --select +exposure:weekly_revenue_report

# Run tests only on models feeding the exposure
dbt test --select +exposure:weekly_revenue_report

# List all exposures in the project
dbt ls --resource-type exposure
```

### Best Practices

- **Register all production consumers**: even if a notebook is exploratory, promoting it to an exposure signals that upstream changes need coordination.
- **Set `maturity: high`** for any exposure used in executive reporting or SLA-bound pipelines ā€” this appears prominently in dbt docs.
- **Include the URL**: a direct link to the Looker dashboard, Hex notebook, or Airflow DAG makes impact analysis instantaneous.
- **Pair with model contracts**: if an exposure depends on `fct_orders`, add a contract to that model so column renames fail at CI time rather than silently breaking the dashboard.

Exposures are low-effort, high-value documentation ā€” they turn your dbt DAG into a complete map of your organisation's data dependencies.

## 10:04 - dbt Seeds: Loading Static Reference Data into Your Warehouse

Seeds are CSV files that dbt compiles into tables in your warehouse. They are the right tool for small, slowly-changing reference datasets ā€” country codes, product category mappings, marketing channel labels, cost centres ā€” that live better in version control than in a one-off `INSERT` script.

### Project Structure

Drop CSV files into the `seeds/` directory (configurable via `seed-paths` in `dbt_project.yml`):

```
seeds/
  country_codes.csv
  marketing_channels.csv
  finance/
    cost_centres.csv
```

Run them with:

```bash
dbt seed                         # load all seeds
dbt seed --select country_codes  # load a single seed
dbt seed --full-refresh          # drop and recreate (overwrite existing data)
```

### Configuring Seeds

Seeds follow the same config block pattern as models, set in `dbt_project.yml` or a `schema.yml` alongside the CSV:

```yaml
# dbt_project.yml
seeds:
  my_project:
    +schema: reference          # load into a dedicated schema
    +quote_columns: false
    finance:
      cost_centres:
        +column_types:
          cost_centre_id: varchar(10)
          budget_usd: numeric(18, 2)
```

For documentation and testing, treat seeds like models in `schema.yml`:

```yaml
seeds:
  - name: country_codes
    description: "ISO 3166-1 alpha-2 country codes and display names."
    columns:
      - name: country_code
        data_tests:
          - unique
          - not_null
      - name: country_name
        data_tests:
          - not_null
```

### Referencing Seeds in Models

Use `{{ ref() }}` exactly as you would for any model ā€” dbt adds the seed to the DAG automatically:

```sql
-- models/marts/fct_orders_enriched.sql
select
    o.order_id,
    o.country_code,
    c.country_name,
    m.channel_label
from {{ ref('fct_orders') }} o
left join {{ ref('country_codes') }} c using (country_code)
left join {{ ref('marketing_channels') }} m using (channel_id)
```

### When to Use Seeds (and When Not To)

**Good fit:**
- Static lookup tables under ~1 MB that change infrequently
- Reference data that should be reviewed via code review (e.g., mapping tables owned by analysts)
- Data needed in CI that isn't available in staging environments

**Poor fit:**
- Tables over a few thousand rows ā€” use a source + staging model instead
- Data that changes daily ā€” seeds require a `dbt seed` run to update; they aren't pulled automatically
- Sensitive PII ā€” CSV files in source control are not the right place for that data

### Pro Tip: Guarding Against Accidental Overwrites

By default, `dbt seed` will not truncate an existing table unless you pass `--full-refresh`. This is intentional ā€” it prevents accidental data loss in production. Gate `--full-refresh` behind a CI flag or a manual approval step in your deployment pipeline.

## 15:04 - dbt Model Contracts: Enforcing Schema Stability Across Teams

Model contracts (introduced in dbt 1.5) let you declare a model's expected column names and data types as a **binding promise** rather than documentation. When a contract is active, dbt validates the contract against the compiled SQL at runtime and raises an error before any rows touch the warehouse if the shape doesn't match.

### Why Contracts Matter

Without contracts, renaming a column in `fct_orders` silently breaks every downstream BI dashboard, ML feature store, or reverse ETL job that reads it. Contracts shift that failure left ā€” from "the dashboard is broken in production" to "the CI pipeline failed before the PR merged."

### Enabling a Contract

Add a `contract` block to the model's `schema.yml` entry and list every column with its data type:

```yaml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true       # activates the contract
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key   # informational on most adapters
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount_usd
        data_type: numeric
      - name: status
        data_type: varchar
```

dbt will now compare this declaration to the model's SELECT output. A column that is missing, has the wrong type, or appears in the wrong order (on adapters where order matters, like Redshift) will cause `dbt run` to fail immediately.

### Constraints vs. Tests

| Feature | Constraints | Data Tests |
|---|---|---|
| When checked | Compile / run time (schema) | After the model runs (row data) |
| What they check | Column existence & type | Row-level values (uniqueness, not_null, etc.) |
| Adapter support | Varies ā€” some are informational only | Universal |

Constraints like `not_null` and `primary_key` are schema-level hints. For actual data quality guarantees, still add `data_tests` in addition to constraints.

### Contracts and Versioned Models

Contracts are especially powerful when combined with **model versions** (`dbt 1.6+`). You can publish `fct_orders.v1` with a contract, then introduce breaking changes as `fct_orders.v2` ā€” downstream consumers pin to `v1` via `ref('fct_orders', version=1)` and migrate on their own schedule:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1
        config:
          contract:
            enforced: true
      - v: 2
        config:
          contract:
            enforced: true
```

### Best Practices

- **Enforce contracts on public interface models** ā€” marts and any model referenced by external tools. Leave staging models contract-free for flexibility.
- **Start with `enforced: false`** to generate a contract skeleton from your existing model, then enable enforcement once you've reviewed the output of `dbt run --select fct_orders`.
- **Pair with CI**: contracts only protect you if `dbt run` (or `dbt build`) runs in your pull request pipeline. A contract that's never checked at PR time gives a false sense of safety.
- **Document `data_type` using warehouse-native types** (e.g., `varchar` not `string` on Snowflake) ā€” dbt does light normalisation but exact type strings vary by adapter.
