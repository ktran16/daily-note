## 00:04 - dbt Incremental Models: Efficient Processing of Large Datasets

Incremental models are one of dbt's most powerful materializations. Instead of rebuilding an entire table on every run, dbt only processes new or changed rows ŌĆö dramatically reducing query cost and runtime on large datasets.

### How It Works

Mark a model as incremental in its config block:

```sql
{{ config(
    materialized='incremental',
    unique_key='event_id'
) }}

SELECT
    event_id,
    user_id,
    event_type,
    created_at
FROM {{ source('raw', 'events') }}

{% if is_incremental() %}
    WHERE created_at > (SELECT MAX(created_at) FROM {{ this }})
{% endif %}
```

The `is_incremental()` macro returns `true` on every run except the very first (full refresh). `{{ this }}` references the existing table in your warehouse, letting you filter only for rows that are newer than what's already been loaded.

### The `unique_key` Parameter

When you specify a `unique_key`, dbt performs an **upsert** instead of a plain append:

- **Without `unique_key`**: new rows are appended. Fine for immutable event logs.
- **With `unique_key`**: existing rows matching the key are updated, new rows are inserted. Essential for mutable records like user profiles or order statuses.

### Incremental Strategies

Different warehouses support different strategies via `incremental_strategy`:

| Strategy | Best For | Warehouses |
|---|---|---|
| `append` | Immutable, append-only logs | All |
| `merge` | Upserts on a unique key | Snowflake, BigQuery, Databricks |
| `delete+insert` | Replacing partitions | BigQuery, Redshift |
| `insert_overwrite` | Partition-level replacement | Spark, Databricks |

```sql
{{ config(
    materialized='incremental',
    unique_key='order_id',
    incremental_strategy='merge',
    merge_update_columns=['status', 'updated_at']
) }}
```

`merge_update_columns` is a useful Snowflake/BigQuery option that restricts which columns get updated on a match ŌĆö protecting columns like `created_at` from being overwritten.

### When to Use `--full-refresh`

Run `dbt run --full-refresh` to rebuild the table from scratch when:
- The model's schema changes (added/removed columns)
- The filter logic changes and historical data needs reprocessing
- Data has been backfilled upstream

### Common Pitfall: Late-Arriving Data

A simple `WHERE created_at > MAX(created_at)` filter breaks when upstream data arrives late. A common fix is to look back a safe window:

```sql
{% if is_incremental() %}
    WHERE created_at > (
        SELECT DATEADD('day', -3, MAX(created_at)) FROM {{ this }}
    )
{% endif %}
```

This reprocesses the last 3 days on every run, trading a small amount of extra compute for correctness on late data.

### Key Takeaways

- Use `is_incremental()` to gate the filter; never hardcode dates.
- Choose `unique_key` + `merge` for mutable records; omit it for append-only event streams.
- Build in a lookback window when upstream data can arrive late.
- Always test incremental logic with a `--full-refresh` to ensure results match the full dataset.

## 05:03 - dbt Model Contracts: Enforcing Schema Guarantees

dbt model contracts (introduced in dbt Core 1.5) let you declare the expected schema of a model as part of the project ŌĆö and have dbt enforce it at build time. Think of them as compile-time type safety for your data warehouse.

### Why Contracts Matter

Without contracts, a model's column list is purely documentation. Any upstream rename or accidental column drop silently makes it downstream. Contracts turn that silent breakage into a loud build failure *before* bad data reaches consumers.

### Enabling a Contract

Add `contract: {enforced: true}` to the model's config and declare every column with its data type:

```yaml
# models/marts/orders.yml
models:
  - name: orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: status
        data_type: varchar
      - name: total_amount
        data_type: numeric
      - name: created_at
        data_type: timestamp
```

When `enforced: true`, dbt will fail the run if:
- The model's SELECT produces a column not listed in the YAML
- A declared column is missing from the SELECT
- The inferred data type doesn't match the declared `data_type`

### Supported Constraints

| Constraint | Effect | Warehouse Support |
|---|---|---|
| `not_null` | Column must not contain NULLs | All (enforced or warned) |
| `primary_key` | Unique + not null | Snowflake, BigQuery, Postgres |
| `unique` | No duplicate values | Snowflake, BigQuery |
| `foreign_key` | Referential integrity | Snowflake, BigQuery |
| `check` | Custom SQL expression | Postgres, Snowflake |

Note: some warehouses (BigQuery, Snowflake) support constraints at the DDL level but don't physically enforce them ŌĆö dbt will still validate during the build using its own checks.

### Contracts and Model Versions

Contracts pair naturally with **model versioning**. When you bump a model to `v2`, the contract on `v1` stays frozen, guaranteeing downstream consumers can rely on it while you iterate on the new version:

```yaml
models:
  - name: orders
    latest_version: 2
    versions:
      - v: 1
        config:
          contract:
            enforced: true
      - v: 2
        config:
          contract:
            enforced: true
```

### Practical Tips

- Start contracts on **public models** (those consumed by other teams or BI tools) first ŌĆö that's where schema drift causes the most pain.
- Use `data_type` values that match your warehouse's native type names (e.g., `varchar` for Snowflake, `STRING` for BigQuery).
- Combine contracts with **exposures** to map which dashboards depend on which contracted models, giving you a full impact-analysis chain from source to report.
- Contracts do NOT replace `dbt test` ŌĆö they catch structural issues at build time, while tests catch data quality issues at test time. Use both.

## 10:04 - dbt Snapshots: Tracking Slowly Changing Dimensions (SCD Type 2)

dbt snapshots capture the state of a mutable source table over time, automatically maintaining a history of every row change. This is the standard way to implement **SCD Type 2** (slowly changing dimensions) without any custom orchestration.

### When to Use Snapshots

Snapshots are ideal for source data that can change in place ŌĆö like a CRM contacts table, an orders status table, or a product catalog ŌĆö where you need to know not just the *current* state, but *what it looked like* at any past point in time.

### Defining a Snapshot

Snapshots live in the `snapshots/` directory and use a dedicated `snapshot` block:

```sql
-- snapshots/orders_snapshot.sql
{% snapshot orders_snapshot %}

{{
    config(
        target_schema='snapshots',
        unique_key='order_id',
        strategy='timestamp',
        updated_at='updated_at'
    )
}}

SELECT
    order_id,
    customer_id,
    status,
    total_amount,
    updated_at
FROM {{ source('raw', 'orders') }}

{% endsnapshot %}
```

Run it with `dbt snapshot`. dbt appends four metadata columns to the snapshot table:

| Column | Meaning |
|---|---|
| `dbt_scd_id` | Unique key for each snapshot row (hash of `unique_key` + `updated_at`) |
| `dbt_updated_at` | When this snapshot record was last written |
| `dbt_valid_from` | When this version of the row became active |
| `dbt_valid_to` | When this version was superseded (`NULL` = current row) |

### Two Snapshot Strategies

**`timestamp`** ŌĆö Recommended. Uses a column (e.g., `updated_at`) to detect changes. Reliable and cheap.

```sql
config(
    strategy='timestamp',
    updated_at='updated_at'
)
```

**`check`** ŌĆö Compares a list of columns row-by-row. Use when the source has no reliable `updated_at`:

```sql
config(
    strategy='check',
    check_cols=['status', 'total_amount']
    -- or check_cols='all' to compare every column
)
```

`check` is more expensive since dbt must compare columns on every run.

### Querying Snapshot History

To get the full history of an order:

```sql
SELECT *
FROM snapshots.orders_snapshot
WHERE order_id = 12345
ORDER BY dbt_valid_from;
```

To reconstruct the state at a specific point in time (point-in-time query):

```sql
SELECT *
FROM snapshots.orders_snapshot
WHERE '2025-11-01 00:00:00' BETWEEN dbt_valid_from AND COALESCE(dbt_valid_to, '9999-12-31')
```

### Best Practices

- **Run snapshots before other models** ŌĆö snapshots should capture source state before transformations run on that data.
- **Never modify snapshot SQL logic** once in production without a full rebuild; changing `unique_key` or `strategy` corrupts the history.
- **Add a `target_database`** config in multi-database setups to keep snapshot tables isolated from mart tables.
- **Keep snapshot models thin** ŌĆö select only the columns you need for history; join in enrichments downstream in a separate model.
- Snapshots are a great companion to the `{{ ref() }}` system: downstream models can reference `ref('orders_snapshot')` and filter on `dbt_valid_to IS NULL` to get current rows only.
