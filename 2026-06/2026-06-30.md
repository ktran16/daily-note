## 00:04 - dbt Slim CI: Run Only What Changed in Pull Requests

Slim CI is dbt's approach to cutting CI costs by running and testing only the models that changed in a PR — rather than rebuilding the full project. It relies on **state comparison**: dbt compares your branch's compiled artifacts against a known baseline (typically production) and selects only modified nodes.

### How It Works

1. **Fetch production artifacts** (`manifest.json`) into a `./prod-artifacts/` folder before running CI.
2. **Use `--state` and `state:modified+`** to select changed models and their downstream dependents.

```bash
# Download production manifest (from dbt Cloud artifact storage or S3)
dbt ls \
  --state ./prod-artifacts \
  --select state:modified+

# Build only what changed, deferring unmodified refs to production
dbt build \
  --state ./prod-artifacts \
  --select state:modified+ \
  --defer
```

The `--defer` flag tells dbt to resolve `{{ ref() }}` calls for *unselected* upstream models against production, so your tests can run without rebuilding the whole DAG.

### State Modifiers

| Selector | What it catches |
|---|---|
| `state:modified` | Models whose SQL or config changed |
| `state:modified.body` | SQL body changes only |
| `state:modified.configs` | Config changes only (e.g. materialization) |
| `state:modified+` | Modified nodes **plus** all downstream dependents |
| `state:new` | Brand-new models with no prior state |

Combining `state:new,state:modified+` is the most common CI pattern — it catches both new additions and ripple effects.

### Example GitHub Actions Step

```yaml
- name: dbt Slim CI
  run: |
    dbt deps
    dbt build \
      --profiles-dir . \
      --state ./prod-artifacts \
      --select state:new,state:modified+ \
      --defer
  env:
    DBT_TARGET: ci
```

### Where to Get the Baseline Manifest

- **dbt Cloud**: available via the API at `/api/v2/accounts/{id}/jobs/{job_id}/artifacts/manifest.json`
- **Self-hosted**: upload `target/manifest.json` after each production run to S3/GCS and download it in CI
- **dbt Core + GitHub Actions**: use a dedicated "fetch artifacts" job that runs before the CI build job

### Key Benefits

- CI runs in minutes instead of hours on large projects
- Warehouse compute costs drop dramatically
- Faster feedback loop for PR authors
- Tests still cover downstream models that depend on changed code

Slim CI is one of the highest-leverage dbt practices for teams with more than a few dozen models — the setup cost is low and the time savings compound across every PR.

## 05:02 - dbt Custom Generic Tests: Building Reusable Test Macros

dbt ships with four built-in generic tests (`not_null`, `unique`, `accepted_values`, `relationships`), but real projects inevitably need more. **Custom generic tests** let you package any assertion as a reusable macro that can be applied to any column or model via `schema.yml` — exactly like the built-ins.

### Anatomy of a Custom Generic Test

A generic test is a Jinja macro that returns a SQL `SELECT` statement. dbt runs it and **expects zero rows** — any row returned is a test failure.

Place the file in `tests/generic/` (or `macros/` — dbt scans both):

```sql
-- tests/generic/assert_column_is_positive.sql
{% test assert_column_is_positive(model, column_name) %}

select
    {{ column_name }} as failing_value
from {{ model }}
where {{ column_name }} <= 0

{% endtest %}
```

Apply it in `schema.yml` just like a built-in:

```yaml
models:
  - name: orders
    columns:
      - name: order_amount
        tests:
          - assert_column_is_positive
```

### Accepting Additional Arguments

Tests can take extra parameters beyond the required `model` and `column_name`:

```sql
-- tests/generic/assert_column_between.sql
{% test assert_column_between(model, column_name, min_value, max_value) %}

select
    {{ column_name }} as failing_value
from {{ model }}
where {{ column_name }} < {{ min_value }}
   or {{ column_name }} > {{ max_value }}

{% endtest %}
```

```yaml
columns:
  - name: discount_pct
    tests:
      - assert_column_between:
          min_value: 0
          max_value: 100
```

### Model-Level Custom Tests

Not all tests target a single column. Use `model` alone for row-level assertions:

```sql
-- tests/generic/assert_no_duplicate_keys.sql
{% test assert_no_duplicate_keys(model, column_name) %}

select
    {{ column_name }},
    count(*) as row_count
from {{ model }}
group by {{ column_name }}
having count(*) > 1

{% endtest %}
```

### Using dbt-utils Instead of Writing From Scratch

The `dbt-utils` package ships dozens of ready-made generic tests — check if what you need already exists before writing your own:

```yaml
# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: [">=1.0.0", "<2.0.0"]
```

Common `dbt-utils` tests: `expression_is_true`, `recency`, `equal_rowcount`, `not_constant`, `not_empty_string`, `at_least_one`.

### Best Practices

- **Name tests with intent**: `assert_*` or `test_*` prefixes make the purpose clear in the DAG.
- **Return meaningful columns**: include `failing_value` or `row_id` so failures are easy to diagnose with `store_failures: true`.
- **Keep tests deterministic**: avoid non-deterministic SQL (`RAND()`, `NOW()`) so results are reproducible.
- **Namespace in packages**: if sharing via a dbt package, prefix with your package name (e.g. `myteam_assert_positive`) to avoid collisions.

Custom generic tests turn one-off SQL checks into first-class, versioned, documented data quality rules that scale across your entire project.

## 10:03 - dbt Model Contracts: Enforcing Schemas and Breaking Change Detection

dbt **model contracts** (introduced in dbt Core 1.5) let you declare the exact shape a model must produce — column names, data types, and constraints — and have dbt validate that shape at build time. This bridges the gap between analytics engineering and downstream consumers who depend on stable interfaces.

### Why Contracts Matter

Without contracts, a model's schema is implicit: consumers learn about changes only when something breaks. Contracts make the schema **explicit and machine-enforced**, enabling:

- Breaking-change detection before code hits production
- Safe cross-team or cross-project dependencies (especially with dbt Mesh)
- A clear API surface for BI tools, ML pipelines, and reverse-ETL

### Defining a Contract

Add `contract: {enforced: true}` to a model's config, then enumerate every column with its `data_type`:

```yaml
# models/schema.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_amount
        data_type: numeric
      - name: ordered_at
        data_type: timestamp
```

When `enforced: true`, dbt will:
1. Compile the model's SQL
2. Compare the resulting column list and types against the declared contract
3. **Fail the run** if any column is missing, extra, or has the wrong type

### Model-Level Constraints

Beyond column-level constraints, you can declare model-level ones:

```yaml
config:
  contract:
    enforced: true
constraints:
  - type: primary_key
    columns: [order_id]
  - type: check
    expression: "order_amount >= 0"
    name: chk_order_amount_positive
```

Not all constraint types are supported by every warehouse — dbt translates what it can and warns on unsupported ones.

### Contracts and `access` Levels

Contracts pair naturally with dbt's **access** property. Mark a model `public` so other projects can reference it via dbt Mesh, and enforce a contract so those references never silently break:

```yaml
models:
  - name: fct_orders
    access: public
    config:
      contract:
        enforced: true
```

A `public` model without a contract is risky — anyone can reference it, but schema changes are unchecked. `public` + `enforced: true` is the recommended combination for cross-project models.

### Versioning Contracts

When you need to make a breaking change (drop a column, rename one), use **model versioning** instead of editing in place:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1
        defined_in: fct_orders_v1   # keep old model for consumers
      - v: 2
        defined_in: fct_orders      # new shape with contract
```

Consumers pin to `ref('fct_orders', v=1)` until they migrate, giving you a deprecation runway.

### Practical Tips

- Start with your most-referenced `public` models — that's where contract breakage hurts most
- Use `dbt compile` in CI to catch contract violations before `dbt build`
- `data_type` values must match your warehouse's native type names (e.g., `STRING` on BigQuery, `VARCHAR` on Snowflake/Redshift)
- Contracts work with all materializations except `ephemeral` (ephemeral models are inlined and never produce a physical schema to validate against)

Model contracts shift schema governance left: instead of discovering interface breaks in production, teams catch them in the PR — exactly where they're cheapest to fix.

## 15:03 - dbt Snapshots: Tracking Slowly Changing Dimensions (SCD Type 2)

Most source tables overwrite records when data changes — a customer updates their email address and the old one is gone forever. dbt **snapshots** solve this by capturing row-level history over time, automatically implementing SCD Type 2 without custom ETL code.

### How Snapshots Work

A snapshot runs `dbt snapshot` and queries a source table. On each run dbt:
1. Compares current rows against the previously captured snapshot
2. **Closes** changed rows by setting `dbt_valid_to` to the current timestamp
3. **Inserts** new rows for the updated values with `dbt_valid_to = NULL` (open/current record)
4. Passes through unchanged rows untouched

### Defining a Snapshot

Snapshots live in the `snapshots/` directory:

```sql
-- snapshots/customers_snapshot.sql
{% snapshot customers_snapshot %}

{{
  config(
    target_schema='snapshots',
    unique_key='customer_id',
    strategy='timestamp',
    updated_at='updated_at'
  )
}}

select * from {{ source('app_db', 'customers') }}

{% endsnapshot %}
```

Run it with:

```bash
dbt snapshot
# or target a specific snapshot
dbt snapshot --select customers_snapshot
```

### The Two Snapshot Strategies

**`timestamp`** — recommended when the source has a reliable `updated_at` column:

```sql
config(
  strategy='timestamp',
  unique_key='customer_id',
  updated_at='updated_at'
)
```

**`check`** — for sources without a reliable updated timestamp; compares a list of columns for any change:

```sql
config(
  strategy='check',
  unique_key='customer_id',
  check_cols=['email', 'plan_tier', 'status']
  -- or check_cols='all' to compare every column
)
```

`check` is slower (full column comparison on every run) and misses changes that revert before the next snapshot run. Prefer `timestamp` whenever possible.

### dbt-Generated Snapshot Columns

dbt automatically adds four metadata columns to every snapshot table:

| Column | Description |
|---|---|
| `dbt_scd_id` | Unique surrogate key per snapshot row (hash of PK + `dbt_updated_at`) |
| `dbt_updated_at` | Timestamp when dbt last touched this row |
| `dbt_valid_from` | When this version of the record became active |
| `dbt_valid_to` | When it was superseded (`NULL` = current record) |

### Querying Snapshot History

```sql
-- Current state of all customers
select * from {{ ref('customers_snapshot') }}
where dbt_valid_to is null;

-- What did a customer's record look like on a given date?
select * from {{ ref('customers_snapshot') }}
where customer_id = 42
  and dbt_valid_from <= '2025-03-01'
  and (dbt_valid_to > '2025-03-01' or dbt_valid_to is null);

-- Full history for a customer, ordered chronologically
select customer_id, email, plan_tier, dbt_valid_from, dbt_valid_to
from {{ ref('customers_snapshot') }}
where customer_id = 42
order by dbt_valid_from;
```

### Snapshot Best Practices

- **Run snapshots before your models**: in orchestration, snapshot jobs should precede the main `dbt build` so downstream models always see the latest history.
- **Use a dedicated schema**: set `target_schema='snapshots'` to keep snapshot tables separate from marts and staging.
- **Never delete from source tables before snapshotting**: hard deletes are invisible to snapshots — use soft deletes (`is_deleted` flags) or handle them explicitly with a `where` filter or hard-delete detection macro.
- **Snapshot cadence matters**: a snapshot run once a day misses changes that happen and revert within 24 hours. Match cadence to how frequently source data changes.
- **Don't `ref()` a snapshot from another snapshot**: snapshots should always point at sources, not at each other.

Snapshots are one of dbt's most underused features outside of data warehousing teams — once set up, they provide a complete, auditable history of any mutable source table with virtually no ongoing maintenance.

## 19:02 - dbt Performance Tuning: Making Your Models Fast and Cost-Efficient

Slow dbt builds and expensive warehouse queries are usually symptoms of a handful of fixable problems: wrong materializations, missing clustering, full-table scans on incremental models, or unbounded `SELECT *` chains. Here is a practical playbook for tuning dbt performance at the model, build, and warehouse level.

### 1. Choose the Right Materialization

Materialization choice is the highest-leverage performance decision:

| Materialization | Cost to build | Cost to query | When to use |
|---|---|---|---|
| `view` | None | High (reruns query each time) | Cheap intermediate transforms, rarely queried |
| `table` | High | Low | Frequently queried models, heavy aggregations |
| `incremental` | Low (only new rows) | Low | Append-only or merge-friendly large fact tables |
| `ephemeral` | None | Medium (inlined as CTE) | Lightweight reusable CTEs, never queried directly |

If a view is slow to query, materialize it as a table. If a full table refresh is too slow, convert to incremental.

### 2. Tune Incremental Models to Avoid Full Scans

The most common incremental performance bug is a predicate that doesn't filter the source table:

```sql
-- BAD: scans the entire source on every run
{{ config(materialized='incremental', unique_key='event_id') }}

select * from {{ source('events', 'raw_events') }}
{% if is_incremental() %}
  where event_ts > (select max(event_ts) from {{ this }})
{% endif %}
```

The `WHERE` on `{{ this }}` is correct, but the source table also needs a partition filter or the warehouse still scans everything:

```sql
-- GOOD: filter the source too
{% if is_incremental() %}
  where event_ts > (select max(event_ts) from {{ this }})
    and event_ts >= dateadd('day', -3, current_date)  -- Snowflake partition pruning
{% endif %}
```

Adding a small lookback window (`-3 days`) guards against late-arriving data while keeping the source scan narrow.

### 3. Use Warehouse-Native Clustering and Partitioning

Push clustering/partitioning config into the model so the warehouse can prune partitions at query time:

```yaml
# Snowflake
{{ config(
  materialized='table',
  cluster_by=['customer_id', 'event_date']
) }}

# BigQuery
{{ config(
  materialized='table',
  partition_by={"field": "event_date", "data_type": "date"},
  cluster_by=["customer_id", "event_type"]
) }}

# Redshift
{{ config(
  materialized='table',
  sort=['event_date', 'customer_id'],
  dist='customer_id'
) }}
```

Match cluster/sort keys to the `WHERE` and `JOIN` columns most commonly used by downstream queries.

### 4. Avoid `SELECT *` in Intermediate Models

Wide `SELECT *` chains propagate columns that nothing downstream uses, multiplying scan width at every step:

```sql
-- BAD: carries 60 columns through 3 transform layers
select * from {{ ref('stg_orders') }}

-- GOOD: project only what downstream models need
select
    order_id,
    customer_id,
    order_amount,
    ordered_at
from {{ ref('stg_orders') }}
```

This is especially impactful on columnar warehouses (BigQuery, Redshift, Snowflake) where scan cost is proportional to columns selected.

### 5. Speed Up dbt Builds with Parallelism and Slim CI

dbt runs models sequentially by default within a thread. Increase threads in `profiles.yml`:

```yaml
# profiles.yml
my_project:
  target: prod
  outputs:
    prod:
      threads: 8   # run up to 8 models in parallel (respects DAG dependencies)
```

Combine with **Slim CI** (`--select state:modified+ --defer --state ./prod-artifacts`) so PRs only rebuild changed models and their dependents — this often cuts CI time by 80-90% on mature projects.

### 6. Profile Slow Models with `query_comment`

Add a query comment so slow queries are easy to find in your warehouse's query history:

```yaml
# dbt_project.yml
query-comment:
  comment: "dbt={{ dbt_version }} | project={{ project_name }} | model={{ model.name }} | profile={{ target.name }}"
  append: true
```

Now every dbt-generated query in Snowflake's `QUERY_HISTORY` or BigQuery's `INFORMATION_SCHEMA.JOBS` is tagged, making it trivial to identify which model is responsible for a slow or expensive query.

### Quick Wins Checklist

- [ ] Replace `view` with `table` for models queried >10×/day
- [ ] Add source-side date filters to incremental models
- [ ] Set `cluster_by` / `partition_by` on large fact tables
- [ ] Raise `threads` to 4-8 in production profiles
- [ ] Remove `SELECT *` in staging and intermediate layers
- [ ] Enable `store_failures: true` on tests in dev to debug without re-running
- [ ] Use `dbt ls --select state:modified+` before builds to preview scope

Small configuration changes at the model level compound across hundreds of runs — performance tuning is one of the highest-ROI investments in a mature dbt project.
