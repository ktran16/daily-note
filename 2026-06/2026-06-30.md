# Daily Notes - 2026-06-30

## 00:04 - dbt Slim CI: Run Only What Changed in Pull Requests

Slim CI is dbt's approach to cutting CI costs by running and testing only the models that changed in a PR — rather than rebuilding the full project. It relies on **state comparison**: dbt compares your branch's compiled artifacts against a known baseline (typically production) and selects only modified nodes.

### How It Works

1. **Fetch production artifacts** (`manifest.json`) into a `./prod-artifacts/` folder before running CI.
2. **Use `--state` and `state:modified+`** to select changed models and their downstream dependents.

```bash
# Download production manifest (from dbt Cloud artifact storage or S3)
dbt ls \
  --state ./prod-artifacts \
  --select state:modified+

# Build only what changed, deferring unmodified refs to production
dbt build \
  --state ./prod-artifacts \
  --select state:modified+ \
  --defer
```

The `--defer` flag tells dbt to resolve `{{ ref() }}` calls for *unselected* upstream models against production, so your tests can run without rebuilding the whole DAG.

### State Modifiers

| Selector | What it catches |
|---|---|
| `state:modified` | Models whose SQL or config changed |
| `state:modified.body` | SQL body changes only |
| `state:modified.configs` | Config changes only (e.g. materialization) |
| `state:modified+` | Modified nodes **plus** all downstream dependents |
| `state:new` | Brand-new models with no prior state |

Combining `state:new,state:modified+` is the most common CI pattern — it catches both new additions and ripple effects.

### Example GitHub Actions Step

```yaml
- name: dbt Slim CI
  run: |
    dbt deps
    dbt build \
      --profiles-dir . \
      --state ./prod-artifacts \
      --select state:new,state:modified+ \
      --defer
  env:
    DBT_TARGET: ci
```

### Where to Get the Baseline Manifest

- **dbt Cloud**: available via the API at `/api/v2/accounts/{id}/jobs/{job_id}/artifacts/manifest.json`
- **Self-hosted**: upload `target/manifest.json` after each production run to S3/GCS and download it in CI
- **dbt Core + GitHub Actions**: use a dedicated "fetch artifacts" job that runs before the CI build job

### Key Benefits

- CI runs in minutes instead of hours on large projects
- Warehouse compute costs drop dramatically
- Faster feedback loop for PR authors
- Tests still cover downstream models that depend on changed code

Slim CI is one of the highest-leverage dbt practices for teams with more than a few dozen models — the setup cost is low and the time savings compound across every PR.
