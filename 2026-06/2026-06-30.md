## 00:04 - dbt Slim CI: Run Only What Changed in Pull Requests

Slim CI is dbt's approach to cutting CI costs by running and testing only the models that changed in a PR — rather than rebuilding the full project. It relies on **state comparison**: dbt compares your branch's compiled artifacts against a known baseline (typically production) and selects only modified nodes.

### How It Works

1. **Fetch production artifacts** (`manifest.json`) into a `./prod-artifacts/` folder before running CI.
2. **Use `--state` and `state:modified+`** to select changed models and their downstream dependents.

```bash
# Download production manifest (from dbt Cloud artifact storage or S3)
dbt ls \
  --state ./prod-artifacts \
  --select state:modified+

# Build only what changed, deferring unmodified refs to production
dbt build \
  --state ./prod-artifacts \
  --select state:modified+ \
  --defer
```

The `--defer` flag tells dbt to resolve `{{ ref() }}` calls for *unselected* upstream models against production, so your tests can run without rebuilding the whole DAG.

### State Modifiers

| Selector | What it catches |
|---|---|
| `state:modified` | Models whose SQL or config changed |
| `state:modified.body` | SQL body changes only |
| `state:modified.configs` | Config changes only (e.g. materialization) |
| `state:modified+` | Modified nodes **plus** all downstream dependents |
| `state:new` | Brand-new models with no prior state |

Combining `state:new,state:modified+` is the most common CI pattern — it catches both new additions and ripple effects.

### Example GitHub Actions Step

```yaml
- name: dbt Slim CI
  run: |
    dbt deps
    dbt build \
      --profiles-dir . \
      --state ./prod-artifacts \
      --select state:new,state:modified+ \
      --defer
  env:
    DBT_TARGET: ci
```

### Where to Get the Baseline Manifest

- **dbt Cloud**: available via the API at `/api/v2/accounts/{id}/jobs/{job_id}/artifacts/manifest.json`
- **Self-hosted**: upload `target/manifest.json` after each production run to S3/GCS and download it in CI
- **dbt Core + GitHub Actions**: use a dedicated "fetch artifacts" job that runs before the CI build job

### Key Benefits

- CI runs in minutes instead of hours on large projects
- Warehouse compute costs drop dramatically
- Faster feedback loop for PR authors
- Tests still cover downstream models that depend on changed code

Slim CI is one of the highest-leverage dbt practices for teams with more than a few dozen models — the setup cost is low and the time savings compound across every PR.

## 05:02 - dbt Custom Generic Tests: Building Reusable Test Macros

dbt ships with four built-in generic tests (`not_null`, `unique`, `accepted_values`, `relationships`), but real projects inevitably need more. **Custom generic tests** let you package any assertion as a reusable macro that can be applied to any column or model via `schema.yml` — exactly like the built-ins.

### Anatomy of a Custom Generic Test

A generic test is a Jinja macro that returns a SQL `SELECT` statement. dbt runs it and **expects zero rows** — any row returned is a test failure.

Place the file in `tests/generic/` (or `macros/` — dbt scans both):

```sql
-- tests/generic/assert_column_is_positive.sql
{% test assert_column_is_positive(model, column_name) %}

select
    {{ column_name }} as failing_value
from {{ model }}
where {{ column_name }} <= 0

{% endtest %}
```

Apply it in `schema.yml` just like a built-in:

```yaml
models:
  - name: orders
    columns:
      - name: order_amount
        tests:
          - assert_column_is_positive
```

### Accepting Additional Arguments

Tests can take extra parameters beyond the required `model` and `column_name`:

```sql
-- tests/generic/assert_column_between.sql
{% test assert_column_between(model, column_name, min_value, max_value) %}

select
    {{ column_name }} as failing_value
from {{ model }}
where {{ column_name }} < {{ min_value }}
   or {{ column_name }} > {{ max_value }}

{% endtest %}
```

```yaml
columns:
  - name: discount_pct
    tests:
      - assert_column_between:
          min_value: 0
          max_value: 100
```

### Model-Level Custom Tests

Not all tests target a single column. Use `model` alone for row-level assertions:

```sql
-- tests/generic/assert_no_duplicate_keys.sql
{% test assert_no_duplicate_keys(model, column_name) %}

select
    {{ column_name }},
    count(*) as row_count
from {{ model }}
group by {{ column_name }}
having count(*) > 1

{% endtest %}
```

### Using dbt-utils Instead of Writing From Scratch

The `dbt-utils` package ships dozens of ready-made generic tests — check if what you need already exists before writing your own:

```yaml
# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: [">=1.0.0", "<2.0.0"]
```

Common `dbt-utils` tests: `expression_is_true`, `recency`, `equal_rowcount`, `not_constant`, `not_empty_string`, `at_least_one`.

### Best Practices

- **Name tests with intent**: `assert_*` or `test_*` prefixes make the purpose clear in the DAG.
- **Return meaningful columns**: include `failing_value` or `row_id` so failures are easy to diagnose with `store_failures: true`.
- **Keep tests deterministic**: avoid non-deterministic SQL (`RAND()`, `NOW()`) so results are reproducible.
- **Namespace in packages**: if sharing via a dbt package, prefix with your package name (e.g. `myteam_assert_positive`) to avoid collisions.

Custom generic tests turn one-off SQL checks into first-class, versioned, documented data quality rules that scale across your entire project.

## 10:03 - dbt Model Contracts: Enforcing Schemas and Breaking Change Detection

dbt **model contracts** (introduced in dbt Core 1.5) let you declare the exact shape a model must produce — column names, data types, and constraints — and have dbt validate that shape at build time. This bridges the gap between analytics engineering and downstream consumers who depend on stable interfaces.

### Why Contracts Matter

Without contracts, a model's schema is implicit: consumers learn about changes only when something breaks. Contracts make the schema **explicit and machine-enforced**, enabling:

- Breaking-change detection before code hits production
- Safe cross-team or cross-project dependencies (especially with dbt Mesh)
- A clear API surface for BI tools, ML pipelines, and reverse-ETL

### Defining a Contract

Add `contract: {enforced: true}` to a model's config, then enumerate every column with its `data_type`:

```yaml
# models/schema.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_amount
        data_type: numeric
      - name: ordered_at
        data_type: timestamp
```

When `enforced: true`, dbt will:
1. Compile the model's SQL
2. Compare the resulting column list and types against the declared contract
3. **Fail the run** if any column is missing, extra, or has the wrong type

### Model-Level Constraints

Beyond column-level constraints, you can declare model-level ones:

```yaml
config:
  contract:
    enforced: true
constraints:
  - type: primary_key
    columns: [order_id]
  - type: check
    expression: "order_amount >= 0"
    name: chk_order_amount_positive
```

Not all constraint types are supported by every warehouse — dbt translates what it can and warns on unsupported ones.

### Contracts and `access` Levels

Contracts pair naturally with dbt's **access** property. Mark a model `public` so other projects can reference it via dbt Mesh, and enforce a contract so those references never silently break:

```yaml
models:
  - name: fct_orders
    access: public
    config:
      contract:
        enforced: true
```

A `public` model without a contract is risky — anyone can reference it, but schema changes are unchecked. `public` + `enforced: true` is the recommended combination for cross-project models.

### Versioning Contracts

When you need to make a breaking change (drop a column, rename one), use **model versioning** instead of editing in place:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1
        defined_in: fct_orders_v1   # keep old model for consumers
      - v: 2
        defined_in: fct_orders      # new shape with contract
```

Consumers pin to `ref('fct_orders', v=1)` until they migrate, giving you a deprecation runway.

### Practical Tips

- Start with your most-referenced `public` models — that's where contract breakage hurts most
- Use `dbt compile` in CI to catch contract violations before `dbt build`
- `data_type` values must match your warehouse's native type names (e.g., `STRING` on BigQuery, `VARCHAR` on Snowflake/Redshift)
- Contracts work with all materializations except `ephemeral` (ephemeral models are inlined and never produce a physical schema to validate against)

Model contracts shift schema governance left: instead of discovering interface breaks in production, teams catch them in the PR — exactly where they're cheapest to fix.
