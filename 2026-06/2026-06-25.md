## 00:04 - dbt Slim CI: Run Only What Changed with State-Based Selection

Running every model on every pull request is slow and expensive. dbt's **Slim CI** pattern uses state-based selection to run only the models (and their dependents) that changed since the last successful production run.

### How It Works

dbt can compare the current project state against a previous **manifest** (`manifest.json`). The manifest is an artifact produced by every `dbt run` or `dbt build` — it captures compiled SQL, node graph, and checksums for every node.

```bash
# In CI, fetch the production manifest first
dbt ls --target prod  # produces manifest.json in ./target/

# Then run only models modified vs that manifest
dbt build \
  --select state:modified+ \
  --defer \
  --state ./prod-manifest/
```

- `state:modified` — nodes whose SQL or config changed
- `state:modified+` — those nodes **plus all downstream dependents**
- `--defer` — for unselected models, resolve `{{ ref() }}` against prod tables instead of rebuilding them

### Typical CI Pipeline Pattern

```yaml
# .github/workflows/ci.yml
steps:
  - name: Download production manifest
    run: |
      dbt-cloud artifact download \
        --run-id $PROD_RUN_ID \
        --path target/manifest.json \
        --output-dir ./prod-manifest/

  - name: Run dbt Slim CI
    run: |
      dbt build \
        --select state:modified+ \
        --defer \
        --state ./prod-manifest/ \
        --target ci
```

### Other Useful State Selectors

| Selector | What it matches |
|---|---|
| `state:new` | Nodes that don't exist in the prior manifest |
| `state:modified.body` | SQL body changed |
| `state:modified.configs` | Config (materialization, tags, etc.) changed |
| `state:modified.persisted_descriptions` | Descriptions changed on persisted docs |
| `result:error` | Nodes that errored in a prior run (great for retries) |

### Why `--defer` Matters

Without `--defer`, a model selecting `orders` that depends on `customers` would fail because `customers` wasn't rebuilt. With `--defer`, dbt resolves the missing upstream ref to the production relation — so tests and downstream models can still run without a full rebuild.

### Key Takeaway

Store and version your production `manifest.json` (dbt Cloud does this automatically; in dbt Core use artifact storage like S3 or GCS). This single file is what makes Slim CI possible — without it, you're always running full builds.

## 05:03 - dbt Grants: Declarative Access Control for Your Models

Managing database permissions manually is fragile — grants get forgotten, revoked by accident, or differ between environments. dbt's **grants** config lets you declare access control alongside your models, so permissions are applied automatically every time the model materializes.

### Basic Setup

Define grants in `dbt_project.yml` or in a model's config block:

```yaml
# dbt_project.yml
models:
  my_project:
    marts:
      +grants:
        select: ['reporter', 'bi_user']
    staging:
      +grants:
        select: ['transformer']
```

Or in a model file directly:

```sql
-- models/marts/fct_orders.sql
{{ config(
    materialized='table',
    grants={
        'select': ['reporter', 'bi_user', 'analyst_role']
    }
) }}

select * from {{ ref('int_orders_enriched') }}
```

### How Grants Are Applied

After a model materializes (or increments), dbt runs the appropriate `GRANT` statement for the adapter:

```sql
-- Snowflake / BigQuery / Redshift equivalent
GRANT SELECT ON TABLE analytics.marts.fct_orders TO ROLE reporter;
GRANT SELECT ON TABLE analytics.marts.fct_orders TO ROLE bi_user;
```

For **views**, the grant is applied each run. For **tables** and **incremental** models, dbt applies grants after every full refresh, and re-applies on incremental runs too.

### `copy_grants` for Views (Snowflake)

On Snowflake, recreating a view drops existing grants by default. Use `copy_grants` to preserve them:

```sql
{{ config(
    materialized='view',
    copy_grants=true
) }}
```

### Revoking Stale Grants

By default, dbt only **adds** the grants you declare — it doesn't revoke grants that already exist. To enforce the declared set as the complete set (revoking any extras), set:

```yaml
# dbt_project.yml
models:
  +grant_access_to: []  # not this — use the grants block
```

For strict revocation, check your adapter docs; some adapters support `on-run-end` hooks using `revoke` for cleanup, but native revoke support varies.

### Environment-Specific Grants

Use `target.name` in Jinja to apply different grants per environment:

```yaml
models:
  my_project:
    marts:
      +grants: "{{ {'select': ['reporter']} if target.name == 'prod' else {} }}"
```

Or keep it readable with a macro:

```sql
{{ config(grants=var('mart_grants', {})) }}
```

### Key Takeaway

Put grants in version control alongside your models — not in a runbook or someone's memory. When a model is rebuilt or a new engineer recreates your schema, the correct permissions follow automatically.
