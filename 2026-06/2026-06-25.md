## 00:04 - dbt Slim CI: Run Only What Changed with State-Based Selection

Running every model on every pull request is slow and expensive. dbt's **Slim CI** pattern uses state-based selection to run only the models (and their dependents) that changed since the last successful production run.

### How It Works

dbt can compare the current project state against a previous **manifest** (`manifest.json`). The manifest is an artifact produced by every `dbt run` or `dbt build` — it captures compiled SQL, node graph, and checksums for every node.

```bash
# In CI, fetch the production manifest first
dbt ls --target prod  # produces manifest.json in ./target/

# Then run only models modified vs that manifest
dbt build \
  --select state:modified+ \
  --defer \
  --state ./prod-manifest/
```

- `state:modified` — nodes whose SQL or config changed
- `state:modified+` — those nodes **plus all downstream dependents**
- `--defer` — for unselected models, resolve `{{ ref() }}` against prod tables instead of rebuilding them

### Typical CI Pipeline Pattern

```yaml
# .github/workflows/ci.yml
steps:
  - name: Download production manifest
    run: |
      dbt-cloud artifact download \
        --run-id $PROD_RUN_ID \
        --path target/manifest.json \
        --output-dir ./prod-manifest/

  - name: Run dbt Slim CI
    run: |
      dbt build \
        --select state:modified+ \
        --defer \
        --state ./prod-manifest/ \
        --target ci
```

### Other Useful State Selectors

| Selector | What it matches |
|---|---|
| `state:new` | Nodes that don't exist in the prior manifest |
| `state:modified.body` | SQL body changed |
| `state:modified.configs` | Config (materialization, tags, etc.) changed |
| `state:modified.persisted_descriptions` | Descriptions changed on persisted docs |
| `result:error` | Nodes that errored in a prior run (great for retries) |

### Why `--defer` Matters

Without `--defer`, a model selecting `orders` that depends on `customers` would fail because `customers` wasn't rebuilt. With `--defer`, dbt resolves the missing upstream ref to the production relation — so tests and downstream models can still run without a full rebuild.

### Key Takeaway

Store and version your production `manifest.json` (dbt Cloud does this automatically; in dbt Core use artifact storage like S3 or GCS). This single file is what makes Slim CI possible — without it, you're always running full builds.

## 05:03 - dbt Grants: Declarative Access Control for Your Models

Managing database permissions manually is fragile — grants get forgotten, revoked by accident, or differ between environments. dbt's **grants** config lets you declare access control alongside your models, so permissions are applied automatically every time the model materializes.

### Basic Setup

Define grants in `dbt_project.yml` or in a model's config block:

```yaml
# dbt_project.yml
models:
  my_project:
    marts:
      +grants:
        select: ['reporter', 'bi_user']
    staging:
      +grants:
        select: ['transformer']
```

Or in a model file directly:

```sql
-- models/marts/fct_orders.sql
{{ config(
    materialized='table',
    grants={
        'select': ['reporter', 'bi_user', 'analyst_role']
    }
) }}

select * from {{ ref('int_orders_enriched') }}
```

### How Grants Are Applied

After a model materializes (or increments), dbt runs the appropriate `GRANT` statement for the adapter:

```sql
-- Snowflake / BigQuery / Redshift equivalent
GRANT SELECT ON TABLE analytics.marts.fct_orders TO ROLE reporter;
GRANT SELECT ON TABLE analytics.marts.fct_orders TO ROLE bi_user;
```

For **views**, the grant is applied each run. For **tables** and **incremental** models, dbt applies grants after every full refresh, and re-applies on incremental runs too.

### `copy_grants` for Views (Snowflake)

On Snowflake, recreating a view drops existing grants by default. Use `copy_grants` to preserve them:

```sql
{{ config(
    materialized='view',
    copy_grants=true
) }}
```

### Revoking Stale Grants

By default, dbt only **adds** the grants you declare — it doesn't revoke grants that already exist. To enforce the declared set as the complete set (revoking any extras), set:

```yaml
# dbt_project.yml
models:
  +grant_access_to: []  # not this — use the grants block
```

For strict revocation, check your adapter docs; some adapters support `on-run-end` hooks using `revoke` for cleanup, but native revoke support varies.

### Environment-Specific Grants

Use `target.name` in Jinja to apply different grants per environment:

```yaml
models:
  my_project:
    marts:
      +grants: "{{ {'select': ['reporter']} if target.name == 'prod' else {} }}"
```

Or keep it readable with a macro:

```sql
{{ config(grants=var('mart_grants', {})) }}
```

### Key Takeaway

Put grants in version control alongside your models — not in a runbook or someone's memory. When a model is rebuilt or a new engineer recreates your schema, the correct permissions follow automatically.

## 10:03 - dbt store_failures: Persisting Test Failures as Queryable Tables

When a dbt test fails, the default output is a row count — you know *how many* rows failed, but not *which* rows or *why*. The `store_failures` config solves this by writing failing rows into a table in your warehouse, making debugging fast and iterative.

### Enabling store_failures

Set it globally in `dbt_project.yml`, or per-test in `schema.yml`:

```yaml
# dbt_project.yml — enable for all tests
tests:
  +store_failures: true
  +schema: dbt_test__audit   # optional: write to a dedicated schema
```

Or selectively in `schema.yml`:

```yaml
models:
  - name: orders
    columns:
      - name: customer_id
        tests:
          - not_null:
              config:
                store_failures: true
          - relationships:
              to: ref('customers')
              field: id
              config:
                store_failures: true
                store_failures_as: view   # table (default) or view
```

### What Gets Stored

After `dbt test --store-failures`, dbt materializes a relation for each failing test — named after the test itself — containing only the rows that violated the assertion:

```sql
-- Query the failure table directly
SELECT *
FROM analytics.dbt_test__audit.not_null_orders_customer_id
LIMIT 100;

-- For a relationships test, you'd see orders with orphaned customer_ids
SELECT *
FROM analytics.dbt_test__audit.relationships_orders_customer_id__id__ref_customers_
```

### store_failures_as: view vs. table

| Setting | Behavior |
|---|---|
| `table` (default) | Persists rows; re-runs overwrite; survives session |
| `view` | Recalculates on query; no extra storage; always fresh |

Use `view` when the source model is a view itself — the failure set is always current. Use `table` when you want a snapshot of what was wrong at test time.

### Combining with Severity

Pair `store_failures` with `warn_severity` to capture near-misses without blocking your pipeline:

```yaml
- name: order_total
  tests:
    - dbt_utils.accepted_range:
        min_value: 0
        max_value: 1000000
        config:
          store_failures: true
          severity: warn   # pipeline continues, failures are still persisted
```

### Key Takeaway

`store_failures` turns tests from a pass/fail gate into a debugging tool. Enable it on your most critical tests in production — when an alert fires, you already have the bad rows in a queryable table, not just a count.

## 15:04 - dbt Incremental Models: Choosing the Right Strategy

Incremental models avoid full rebuilds by processing only new or changed data. But the performance and correctness guarantees differ dramatically by **incremental strategy** — choosing the wrong one is one of the most common dbt performance mistakes.

### The four strategies

| Strategy | How it works | Full table scan? | Handles late-arriving data? |
|---|---|---|---|
| `append` | INSERT only new rows | No | No |
| `merge` | MERGE (upsert) on a unique key | No | Yes |
| `delete+insert` | DELETE matching rows, then INSERT | Partial | Yes |
| `insert_overwrite` | Replace entire partitions | Partition-scoped | Yes (per partition) |

### `append` — fastest, but risky

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='append'
) }}

select * from {{ source('events', 'raw_clicks') }}
{% if is_incremental() %}
  where event_ts > (select max(event_ts) from {{ this }})
{% endif %}
```

Use when: source data is truly immutable (e.g., append-only event streams). Never use when source rows can be updated — you'll get duplicates.

### `merge` — safest for mutable sources

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='merge',
    unique_key='order_id'
) }}

select
    order_id,
    customer_id,
    status,
    updated_at
from {{ source('app', 'orders') }}
{% if is_incremental() %}
  where updated_at > (select max(updated_at) from {{ this }})
{% endif %}
```

dbt compiles this into a `MERGE` statement that updates existing rows and inserts new ones. Supported on Snowflake, BigQuery, Databricks, and Redshift (dbt 1.6+). Specify `merge_update_columns` to update only specific columns on match instead of the whole row:

```sql
{{ config(
    incremental_strategy='merge',
    unique_key='order_id',
    merge_update_columns=['status', 'updated_at']
) }}
```

### `delete+insert` — partition-safe alternative to merge

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='delete+insert',
    unique_key='order_id'
) }}
```

dbt DELETEs rows from the target where `unique_key` matches, then INSERTs the new batch. Also useful when you want to reprocess a rolling lookback window to heal late-arriving data:

```sql
{% if is_incremental() %}
  where date_trunc('day', event_ts) >= (
    select dateadd(day, -3, max(date_trunc('day', event_ts))) from {{ this }}
  )
{% endif %}
```

This re-processes the last 3 days on every run, catching any rows that arrived late.

### `insert_overwrite` — partition replacement (BigQuery / Spark)

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='insert_overwrite',
    partition_by={
        'field': 'event_date',
        'data_type': 'date'
    }
) }}
```

Replaces entire partitions atomically. Extremely efficient on BigQuery and Databricks for date-partitioned tables — late arrivals within the partition window are healed automatically because the whole partition is replaced.

### Practical decision tree

1. **Append-only source, no late data** → `append`
2. **Source rows can update, warehouse supports MERGE** → `merge`
3. **Source rows can update, MERGE is slow or unsupported** → `delete+insert` with a lookback window
4. **Large date-partitioned table on BigQuery/Spark** → `insert_overwrite`

### Key Takeaway

Always include the `{% if is_incremental() %}` filter clause. Without it, the model reads the full source on every run — giving you the overhead of an incremental model with none of the performance benefit. The first run (or `--full-refresh`) intentionally skips the filter to build the complete history.
