## 00:03 - dbt Packages: Extending dbt with dbt-utils, dbt-expectations, and the Hub

dbt packages are reusable collections of models, macros, tests, and seeds that you install into your project — similar to libraries in traditional software development. They let you avoid reinventing the wheel and bring battle-tested utilities directly into your DAG.

### Installing Packages

Declare dependencies in `packages.yml` at the project root:

```yaml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0
  - package: calogica/dbt_expectations
    version: 0.10.4
  - package: dbt-labs/audit_helper
    version: 0.12.0
```

Then run `dbt deps` to install them into the `dbt_packages/` directory (add this to `.gitignore`).

### dbt-utils: The Swiss Army Knife

`dbt-labs/dbt_utils` is the most widely used package. Key utilities:

```sql
-- Generate a surrogate key from multiple columns
{{ dbt_utils.generate_surrogate_key(['order_id', 'line_item_id']) }}

-- Pivot rows to columns dynamically
{{ dbt_utils.pivot('status', dbt_utils.get_column_values(ref('orders'), 'status')) }}

-- Date spine: generate a row for every date in a range
{{ dbt_utils.date_spine(
    datepart="day",
    start_date="cast('2024-01-01' as date)",
    end_date="cast('2024-12-31' as date)"
) }}
```

Generic tests from dbt-utils are also invaluable:

```yaml
models:
  - name: orders
    columns:
      - name: order_id
        tests:
          - dbt_utils.not_empty_string
          - dbt_utils.expression_is_true:
              expression: "amount > 0"
```

### dbt-expectations: Great Expectations for dbt

Inspired by the Great Expectations library, this package provides rich column-level assertions:

```yaml
- name: unit_price
  tests:
    - dbt_expectations.expect_column_values_to_be_between:
        min_value: 0
        max_value: 10000
    - dbt_expectations.expect_column_values_to_not_be_null
    - dbt_expectations.expect_column_value_lengths_to_be_between:
        min_value: 1
        max_value: 50
```

### audit_helper: Safe Refactors

When rewriting a model, `audit_helper` lets you compare old vs. new results before swapping in production:

```sql
{% set old_relation = ref('orders') %}
{% set new_relation = ref('orders_v2') %}

{{ audit_helper.compare_relations(
    a_relation=old_relation,
    b_relation=new_relation,
    primary_key="order_id"
) }}
```

### Best Practices

- **Pin exact versions** in `packages.yml` — semver ranges can break on `dbt deps` upgrades.
- **Check the dbt Package Hub** (`hub.getdbt.com`) before writing custom macros; the problem is often already solved.
- **Namespace awareness**: macros from packages are called with their package prefix (e.g., `dbt_utils.surrogate_key`), avoiding collisions with your own macros.
- **Audit your deps**: packages run code in your warehouse — review them like any third-party dependency.

Packages dramatically accelerate dbt development by standardizing common patterns across the ecosystem. Start with `dbt-utils`; it alone will save hours of custom SQL.

## 05:02 - dbt Model Contracts: Enforcing Schema Guarantees

Model contracts (introduced in dbt 1.5) let you declare a binding schema promise for a model — column names, data types, and constraints — that dbt enforces at build time. They're the mechanism that turns a dbt model into a stable, versioned interface other teams can depend on.

### Why Contracts Matter

Without contracts, a model's schema is implicit: whoever queries `ref('orders')` hopes the columns they need still exist. Contracts make that promise explicit and machine-checked, preventing silent breaking changes from reaching downstream consumers.

### Enabling a Contract

In the model's `.yml` file, set `contract.enforced: true` and declare every column with its data type:

```yaml
models:
  - name: orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: status
        data_type: varchar
      - name: total_amount
        data_type: numeric
      - name: created_at
        data_type: timestamp
```

When `dbt run` executes, dbt compares the model's compiled SQL column list and types against this declaration. Any mismatch — a missing column, a renamed column, a type change — **fails the build before the table is written**.

### Constraints

Constraints add database-level or dbt-level rules:

```yaml
constraints:
  - type: not_null          # enforced by dbt (pre-run check)
  - type: primary_key       # enforced by the warehouse (if supported)
  - type: unique            # warehouse-enforced
  - type: foreign_key
    to: ref('customers')
    to_columns: [customer_id]
  - type: check
    expression: "total_amount >= 0"
```

Support varies by adapter — BigQuery, Snowflake, and Databricks each expose different constraint types. Use `dbt docs generate` to see what your adapter supports.

### Contracts + Model Versions

Contracts shine alongside **model versioning** (`dbt 1.6+`). You can deprecate `v1` while shipping `v2` with a new contract, giving consumers a migration window:

```yaml
models:
  - name: orders
    latest_version: 2
    versions:
      - v: 1
        defined_in: orders_v1
      - v: 2
        config:
          contract:
            enforced: true
```

Downstream models reference `ref('orders', v=1)` explicitly and upgrade on their own schedule.

### Best Practices

- **Start contracts on public models only** — the ones exposed to other dbt projects or BI tools. Internal staging models don't need them.
- **Use `data_type` values your warehouse actually uses** (e.g., `varchar` not `string` for Snowflake; `string` not `varchar` for BigQuery).
- **Contracts fail loudly at build time**, not silently at query time — treat a contract failure like a compile error, not a test failure.
- Pair contracts with **`access: public`** in multi-project setups so the contract is the cross-project API.

Model contracts shift dbt from a script runner into an interface-driven system — a key step toward treating data models with the same rigor as microservice APIs.

## 10:02 - dbt Snapshots: Tracking Slowly Changing Dimensions Over Time

dbt snapshots let you capture the historical state of a mutable source table over time — solving the classic **Slowly Changing Dimension (SCD Type 2)** problem without custom ETL code. Every time a row changes, dbt records the old value with a validity window so you can reconstruct "what did this record look like on date X?"

### The Problem They Solve

Operational databases overwrite rows. If a customer's `status` changes from `active` to `churned`, the old value is gone. Snapshots intercept this and maintain a versioned history table.

### Defining a Snapshot

Create a `.sql` file inside the `snapshots/` directory:

```sql
{% snapshot orders_snapshot %}

{{
  config(
    target_schema='snapshots',
    unique_key='order_id',
    strategy='timestamp',
    updated_at='updated_at'
  )
}}

select * from {{ source('raw', 'orders') }}

{% endsnapshot %}
```

Run with `dbt snapshot`. dbt will:
1. On first run — copy all rows into a new snapshot table and stamp each with `dbt_valid_from = now()` and `dbt_valid_to = null` (current record).
2. On subsequent runs — detect changed rows (via the strategy), close the old record (`dbt_valid_to = now()`), and insert the new version with a fresh `dbt_valid_from`.

### Two Snapshot Strategies

**`timestamp`** — relies on an `updated_at` column in the source. Fast and reliable when sources expose it:

```sql
config(
  strategy='timestamp',
  updated_at='updated_at'
)
```

**`check`** — compares a list of columns for any change. Use when there's no reliable timestamp:

```sql
config(
  strategy='check',
  check_cols=['status', 'plan_tier', 'email']
  -- or check_cols='all' to diff every column
)
```

### The Snapshot Table's Metadata Columns

| Column | Meaning |
|---|---|
| `dbt_scd_id` | Unique surrogate key per snapshot row |
| `dbt_updated_at` | When dbt detected the change |
| `dbt_valid_from` | When this version became active |
| `dbt_valid_to` | When this version was superseded (`null` = current) |

Querying the current state: `WHERE dbt_valid_to IS NULL`  
Querying as-of a date: `WHERE dbt_valid_from <= '2025-03-01' AND (dbt_valid_to > '2025-03-01' OR dbt_valid_to IS NULL)`

### Referencing Snapshots in Models

Use `ref()` just like any model:

```sql
-- models/marts/customer_history.sql
select
    order_id,
    status,
    dbt_valid_from,
    dbt_valid_to
from {{ ref('orders_snapshot') }}
where dbt_valid_to is null  -- current records only
```

### Best Practices

- **Snapshot source tables, not transformed models** — snapshots should sit as close to raw data as possible so no transformation layer obscures changes.
- **Run snapshots before `dbt run`** in your pipeline (`dbt snapshot && dbt run`) so downstream models see the freshest snapshot state.
- **Use `target_schema` to isolate snapshot tables** from mart/staging tables — they grow unboundedly and need separate retention policies.
- **Don't apply heavy filters in the snapshot query** — if you exclude rows, you'll lose history for them entirely.
- **Test `dbt_valid_to IS NULL` counts** against source row counts as a freshness check.

Snapshots are one of dbt's most powerful primitives for analytics engineering — they turn an append-only history into a first-class SQL table, enabling point-in-time analysis without any bespoke infrastructure.

## 15:03 - dbt CI/CD and Deployment: From Laptop to Production Safely

A dbt project is a codebase, and it deserves the same CI/CD discipline as application code. A well-structured dbt deployment pipeline catches schema regressions, test failures, and broken references before they reach production data consumers.

### The Core Deployment Loop

A healthy dbt pipeline typically has two jobs:

1. **CI job** — runs on every pull request, builds only changed models, and runs all tests.
2. **Production job** — runs on a schedule (hourly, daily), builds everything, and updates the docs site.

### CI: Slim Builds with State

Running a full `dbt build` on every PR is expensive. Use **dbt's state-based comparison** to build only what changed:

```bash
# In CI, fetch the production manifest
dbt ls --select state:modified+ --state ./prod-manifest

# Build only modified models and their downstream dependents
dbt build --select state:modified+ --state ./prod-manifest
```

The `prod-manifest/` directory holds the `manifest.json` from the last successful production run. dbt diffs the current project against it and identifies which nodes changed. The `+` suffix expands selection downstream so you also test models that depend on your changes.

### CI Environment Setup

Keep CI isolated from production by using a separate schema per PR:

```yaml
# profiles.yml (or dbt Cloud environment)
ci:
  target: ci
  outputs:
    ci:
      type: snowflake
      schema: "ci_{{ env_var('PR_NUMBER') }}"  # e.g., ci_1042
```

This gives each PR its own scratch schema, preventing parallel CI runs from colliding and making cleanup trivial — just drop `ci_*` schemas on merge.

### GitHub Actions Example

```yaml
# .github/workflows/dbt-ci.yml
name: dbt CI
on:
  pull_request:
    paths:
      - 'models/**'
      - 'macros/**'
      - 'tests/**'
      - 'dbt_project.yml'

jobs:
  dbt-ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install dbt
        run: pip install dbt-snowflake==1.8.0

      - name: Download production manifest
        run: |
          aws s3 cp s3://my-bucket/dbt/prod/manifest.json ./prod-manifest/manifest.json

      - name: dbt deps
        run: dbt deps

      - name: dbt build (changed models only)
        run: |
          dbt build \
            --select state:modified+ \
            --state ./prod-manifest \
            --target ci
        env:
          DBT_SNOWFLAKE_ACCOUNT: ${{ secrets.SNOWFLAKE_ACCOUNT }}
          PR_NUMBER: ${{ github.event.number }}
```

### Production Job Checklist

A production dbt run should do more than just `dbt run`:

```bash
dbt source freshness          # fail fast if upstream data is stale
dbt build --target prod       # run + test in one command
dbt docs generate             # update the docs site
# Upload manifest.json to artifact storage for next CI diff
aws s3 cp ./target/manifest.json s3://my-bucket/dbt/prod/manifest.json
```

### Deployment Environments

| Environment | Schema prefix | Trigger | Purpose |
|---|---|---|---|
| dev | `dev_<username>` | Manual | Individual development |
| ci | `ci_<pr_number>` | PR open/push | Pre-merge validation |
| staging | `staging` | Merge to main | Integration testing |
| production | (no prefix) | Schedule | Live data for consumers |

### Best Practices

- **Store `manifest.json` as a CI artifact** after every successful production run — it's the baseline for slim CI builds.
- **Use `dbt build` over `dbt run` + `dbt test`** — it interleaves tests immediately after each model so failures surface faster.
- **Set `--fail-fast`** in CI to abort on the first test failure rather than running everything and drowning in errors.
- **Pin dbt and adapter versions** in `requirements.txt` — patch releases can change behavior unexpectedly.
- **Alert on `source freshness` failures separately** from model failures; a stale source is an upstream problem, not a code problem.

A mature dbt CI/CD pipeline turns data pipeline changes into a safe, reviewable, reproducible process — the same confidence software teams have when shipping application code.
