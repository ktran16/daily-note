## 00:03 - dbt Variables & Environments: Adapting Logic for Dev, Staging, and Prod

One of dbt's most underused power features is the ability to write models that behave differently depending on the environment they run in. Two mechanisms make this possible: **project variables** (`var()`) and **target context** (`target`). Together they let you build a single codebase that safely handles dev vs. production without brittle hardcoded values.

### Project Variables with `var()`

Variables are defined in `dbt_project.yml` and accessed in any model or macro with `{{ var('name') }}`. You can supply a default so the project doesn't break when the variable is absent:

```sql
-- models/marts/orders.sql
select *
from {{ ref('stg_orders') }}
where created_at >= '{{ var("start_date", "2020-01-01") }}'
```

Override at runtime without touching files:

```bash
dbt run --vars '{"start_date": "2024-01-01"}'
```

This is ideal for backfills, date-windowed incremental loads, and feature flags.

### Target Context for Environment Branching

`target` is a built-in Jinja object populated from your active profile. Its most useful keys:

| Key | Example value |
|---|---|
| `target.name` | `dev`, `prod`, `ci` |
| `target.schema` | `analytics_dev` |
| `target.type` | `snowflake`, `bigquery` |

Use `target.name` to limit row counts in development — your most effective guard against accidentally processing billions of rows locally:

```sql
-- models/marts/fct_events.sql
select *
from {{ ref('stg_events') }}

{% if target.name != 'prod' %}
  where event_date >= dateadd(day, -7, current_date)
{% endif %}
```

### Combining Both for Incremental Lookback

A common pattern is to vary the incremental lookback window so dev runs are fast while production captures a safe overlap:

```sql
{% set lookback_days = var('lookback_days', 3 if target.name == 'prod' else 1) %}

select *
from {{ ref('stg_sessions') }}
where
  {{ incremental_where_clause('session_start', lookback_days) }}
```

### Exposing Variables in `dbt_project.yml`

Document your vars with defaults so collaborators know what's available:

```yaml
# dbt_project.yml
vars:
  start_date: "2020-01-01"
  lookback_days: 3
  enable_pii_masking: true
```

### Best Practices

- **Default to safe values**: a missing `start_date` var should produce *some* result, not a compilation error.
- **Never branch on `target.schema`**: schema names are fragile. Use `target.name` instead.
- **Avoid `target`-based branching in marts**: keep environment logic in staging or intermediate models so mart SQL stays clean and testable.
- **Use `env_var()` for secrets**: pull sensitive config from environment variables with `{{ env_var('DBT_API_KEY') }}` rather than hardcoding or committing values to `dbt_project.yml`.

Variables and target context are the backbone of a reusable, environment-aware dbt project — master them before reaching for more complex orchestration solutions.

## 05:02 - dbt Model Contracts: Enforcing Schemas at Compile Time

dbt model contracts (introduced in dbt Core 1.5) let you declare the exact columns, data types, and constraints a model **must** produce. When a contract is defined, dbt validates the model's output schema before it ever runs in production — catching breaking changes at CI time rather than downstream.

### Why Contracts Matter

Without contracts, a renamed column or dropped field silently breaks every downstream dashboard and model that depends on it. Contracts turn that silent failure into a loud, early error that surfaces in the pull request.

### Defining a Contract

Contracts live in the model's `.yml` definition file alongside tests and descriptions:

```yaml
# models/marts/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true          # activates the contract
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: unique
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_total_usd
        data_type: numeric
      - name: ordered_at
        data_type: timestamp
```

The `data_type` values must match your warehouse's native types (e.g. `TEXT` for Snowflake, `STRING` for BigQuery, `VARCHAR` for Redshift).

### Supported Constraint Types

| Constraint | Enforced by dbt | Enforced by warehouse |
|---|---|---|
| `not_null` | Yes (compile-time) | Depends on adapter |
| `unique` | No (use dbt tests) | Depends on adapter |
| `primary_key` | No (informational) | Depends on adapter |
| `foreign_key` | No (informational) | Depends on adapter |
| `check` | No (informational) | Depends on adapter |

dbt validates column presence and types at compile time; warehouse-enforced constraints are adapter-specific (Snowflake supports them for tables, BigQuery does not).

### Versioned Models + Contracts

Contracts pair naturally with **model versions**. When you need to make a breaking change, bump the version rather than editing in place:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1
        config:
          contract:
            enforced: true
      - v: 2
        config:
          contract:
            enforced: true
```

Downstream models can pin to `ref('fct_orders', version=1)` while consumers migrate to v2 at their own pace.

### Best Practices

- **Enable contracts only on published interfaces** (marts, exposures) — not on staging or intermediate models where iteration is frequent.
- **Use `dbt compile` in CI** to catch contract violations before merge; no warehouse connection required for the schema check.
- **Pair with `access: public`** in `dbt_project.yml` to signal that a model is a stable, cross-team interface.
- **Start with `not_null` constraints** on primary keys — they provide immediate value with minimal overhead.

Model contracts are dbt's answer to API versioning for data: they make your most critical models tamper-evident and give downstream consumers a guarantee they can build on.

## 10:02 - dbt Unit Tests: Testing SQL Logic with Inline Fixtures

Introduced in dbt Core 1.8, **unit tests** let you validate a model's SQL logic using small, inline datasets — no warehouse rows required. Unlike dbt's data tests (which assert properties of actual data), unit tests assert that your *SQL logic* is correct for a known set of inputs, making them fast, deterministic, and CI-friendly.

### Why Unit Tests?

Before dbt 1.8, the only way to verify that, say, a `CASE` statement handled edge cases correctly was to load real data and check the output. Unit tests remove that dependency: you define input fixtures inline and declare exactly what the output should look like.

### Anatomy of a Unit Test

Unit tests live in `.yml` files alongside your models:

```yaml
# models/marts/fct_order_status.yml
unit_tests:
  - name: test_status_escalation_logic
    model: fct_order_status
    given:
      - input: ref('stg_orders')
        rows:
          - {order_id: 1, days_overdue: 0,  status: 'pending'}
          - {order_id: 2, days_overdue: 5,  status: 'pending'}
          - {order_id: 3, days_overdue: 15, status: 'pending'}
    expect:
      rows:
        - {order_id: 1, risk_tier: 'low'}
        - {order_id: 2, risk_tier: 'medium'}
        - {order_id: 3, risk_tier: 'high'}
```

dbt replaces the `ref('stg_orders')` call with your fixture rows, compiles the model SQL, runs it, and diffs the result against `expect`. No staging data needed.

### Running Unit Tests

```bash
# run all unit tests
dbt test --select "test_type:unit"

# run tests for a specific model
dbt test --select fct_order_status --test-type unit
```

### Mocking Sources and Seeds

You can also mock `source()` calls and seeds:

```yaml
given:
  - input: source('raw', 'payments')
    rows:
      - {payment_id: 99, amount: 150.00, status: 'settled'}
```

### Best Practices

- **Cover boundary conditions**: unit tests shine for edge cases like `NULL` inputs, zero-division guards, `CASE` fallthrough, and date boundary logic.
- **Keep fixtures minimal**: 3–5 rows per test is usually enough. Fixtures are documentation — more rows obscure the intent.
- **Pair with data tests**: unit tests verify logic correctness; data tests (`not_null`, `unique`, `accepted_values`) verify data quality. Use both.
- **Test macros via their models**: dbt doesn't unit-test macros directly — test the model that calls the macro instead.

Unit tests fill the gap between "the SQL compiles" and "the SQL is correct," giving analytics engineers the same confidence that software engineers get from function-level tests.
