## 00:03 - dbt Variables & Environments: Adapting Logic for Dev, Staging, and Prod

One of dbt's most underused power features is the ability to write models that behave differently depending on the environment they run in. Two mechanisms make this possible: **project variables** (`var()`) and **target context** (`target`). Together they let you build a single codebase that safely handles dev vs. production without brittle hardcoded values.

### Project Variables with `var()`

Variables are defined in `dbt_project.yml` and accessed in any model or macro with `{{ var('name') }}`. You can supply a default so the project doesn't break when the variable is absent:

```sql
-- models/marts/orders.sql
select *
from {{ ref('stg_orders') }}
where created_at >= '{{ var("start_date", "2020-01-01") }}'
```

Override at runtime without touching files:

```bash
dbt run --vars '{"start_date": "2024-01-01"}'
```

This is ideal for backfills, date-windowed incremental loads, and feature flags.

### Target Context for Environment Branching

`target` is a built-in Jinja object populated from your active profile. Its most useful keys:

| Key | Example value |
|---|---|
| `target.name` | `dev`, `prod`, `ci` |
| `target.schema` | `analytics_dev` |
| `target.type` | `snowflake`, `bigquery` |

Use `target.name` to limit row counts in development — your most effective guard against accidentally processing billions of rows locally:

```sql
-- models/marts/fct_events.sql
select *
from {{ ref('stg_events') }}

{% if target.name != 'prod' %}
  where event_date >= dateadd(day, -7, current_date)
{% endif %}
```

### Combining Both for Incremental Lookback

A common pattern is to vary the incremental lookback window so dev runs are fast while production captures a safe overlap:

```sql
{% set lookback_days = var('lookback_days', 3 if target.name == 'prod' else 1) %}

select *
from {{ ref('stg_sessions') }}
where
  {{ incremental_where_clause('session_start', lookback_days) }}
```

### Exposing Variables in `dbt_project.yml`

Document your vars with defaults so collaborators know what's available:

```yaml
# dbt_project.yml
vars:
  start_date: "2020-01-01"
  lookback_days: 3
  enable_pii_masking: true
```

### Best Practices

- **Default to safe values**: a missing `start_date` var should produce *some* result, not a compilation error.
- **Never branch on `target.schema`**: schema names are fragile. Use `target.name` instead.
- **Avoid `target`-based branching in marts**: keep environment logic in staging or intermediate models so mart SQL stays clean and testable.
- **Use `env_var()` for secrets**: pull sensitive config from environment variables with `{{ env_var('DBT_API_KEY') }}` rather than hardcoding or committing values to `dbt_project.yml`.

Variables and target context are the backbone of a reusable, environment-aware dbt project — master them before reaching for more complex orchestration solutions.
