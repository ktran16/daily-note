## 00:03 - dbt Snapshots: Capturing Historical Changes with Type-2 SCDs

### What Are Snapshots?

Most source systems only keep the **current state** of a record — when a customer changes their plan or an order status updates, the old value is simply overwritten. dbt **snapshots** solve this by periodically capturing the state of a mutable table and building a full history of every change, implementing a **Type-2 Slowly Changing Dimension (SCD)** automatically.

Snapshots live in the `snapshots/` directory and are materialized as tables in your warehouse. Each time you run `dbt snapshot`, dbt compares the current source data against the snapshot table and inserts new rows for any changed records — preserving the full audit trail.

### Snapshot Strategies

dbt supports two strategies for detecting changes:

**1. `timestamp` strategy** — recommended when the source has a reliable `updated_at` column:

```sql
-- snapshots/snap_orders.sql
{% snapshot snap_orders %}

{{
  config(
    target_schema='snapshots',
    unique_key='order_id',
    strategy='timestamp',
    updated_at='updated_at',
    invalidate_hard_deletes=True
  )
}}

select
    order_id,
    customer_id,
    status,
    total_amount,
    updated_at
from {{ source('shopify', 'orders') }}

{% endsnapshot %}
```

**2. `check` strategy** — used when there is no `updated_at`; dbt hashes a list of columns to detect changes:

```sql
{% snapshot snap_customers %}

{{
  config(
    target_schema='snapshots',
    unique_key='customer_id',
    strategy='check',
    check_cols=['email', 'plan', 'status']
  )
}}

select * from {{ source('crm', 'customers') }}

{% endsnapshot %}
```

### Snapshot Metadata Columns

dbt automatically adds four columns to every snapshot table:

| Column | Description |
|---|---|
| `dbt_scd_id` | Unique surrogate key for each snapshot row |
| `dbt_updated_at` | Timestamp when this snapshot row was written |
| `dbt_valid_from` | When this version of the record became active |
| `dbt_valid_to` | When it was superseded (`NULL` = currently active) |

### Querying the Current State vs. History

```sql
-- Current active records only
select *
from {{ ref('snap_orders') }}
where dbt_valid_to is null;

-- Point-in-time: what did orders look like on 2025-01-01?
select *
from {{ ref('snap_orders') }}
where dbt_valid_from <= '2025-01-01'
  and (dbt_valid_to > '2025-01-01' or dbt_valid_to is null);

-- Full status history for a specific order
select
    order_id,
    status,
    dbt_valid_from,
    dbt_valid_to
from {{ ref('snap_orders') }}
where order_id = 12345
order by dbt_valid_from;
```

### Handling Hard Deletes

By default, if a row disappears from the source, the snapshot keeps the last known row with `dbt_valid_to = NULL` — the delete is invisible. Setting `invalidate_hard_deletes=True` closes the record by setting `dbt_valid_to` to the current snapshot timestamp, so deletes are reflected in history.

### Running Snapshots in Practice

```bash
# Run all snapshots
dbt snapshot

# Run a specific snapshot
dbt snapshot --select snap_orders

# Snapshots run separately from models — include both in CI/CD
dbt snapshot && dbt run && dbt test
```

### Best Practices

1. **Snapshot raw source data** — run snapshots against `source()` tables, not staging models, so you capture the true original values before any transformation.
2. **Snapshot frequently** — run `dbt snapshot` as often as your SLA demands. Snapshots only record state at the moment they execute; a change that comes and goes between runs will be missed.
3. **Use `timestamp` over `check` when possible** — the `check` strategy performs a full row comparison and is significantly slower on large tables.
4. **Limit `check_cols`** — if you must use the `check` strategy, specify only the columns that represent meaningful business changes, not every column (avoids false positives from technical metadata fields).
5. **Don't transform in snapshots** — keep snapshot SQL simple (close to `select *`). Apply business logic in downstream staging or mart models that reference the snapshot via `ref()`.
6. **Partition or cluster snapshot tables** — snapshot tables grow without bound. On BigQuery/Snowflake, partition on `dbt_valid_from` and cluster on `unique_key` to keep point-in-time queries fast.

## 05:02 - dbt Macros & Jinja: Writing Reusable SQL Logic

### Why Macros?

Jinja templating turns dbt SQL files into programs. **Macros** are reusable Jinja functions stored in the `macros/` directory — they let you eliminate copy-paste SQL, enforce standards across models, and build abstractions that adapt to different targets (Snowflake, BigQuery, Redshift) without changing model code.

### Anatomy of a Macro

```sql
-- macros/cents_to_dollars.sql
{% macro cents_to_dollars(column_name, scale=2) %}
    round({{ column_name }} / 100.0, {{ scale }})
{% endmacro %}
```

Call it in any model:

```sql
-- models/marts/fct_orders.sql
select
    order_id,
    {{ cents_to_dollars('amount_cents') }}        as amount_dollars,
    {{ cents_to_dollars('shipping_cents', 4) }}   as shipping_dollars
from {{ ref('stg_orders') }}
```

dbt compiles this before sending it to the warehouse — the warehouse only ever sees plain SQL.

### Control Flow: if / for

Jinja's control structures make macros dynamic:

```sql
{% macro map_status(column_name) %}
    case {{ column_name }}
        {% for status, label in {
            'P': 'Pending',
            'A': 'Active',
            'C': 'Cancelled'
        }.items() %}
            when '{{ status }}' then '{{ label }}'
        {% endfor %}
        else 'Unknown'
    end
{% endmacro %}
```

### Adapter-Aware Macros with `target`

Use `target.type` or `adapter.dispatch()` to write warehouse-agnostic logic:

```sql
-- macros/safe_divide.sql
{% macro safe_divide(numerator, denominator) %}
    {% if target.type == 'bigquery' %}
        safe_divide({{ numerator }}, {{ denominator }})
    {% else %}
        case when {{ denominator }} = 0 then null
             else {{ numerator }} / {{ denominator }}
        end
    {% endif %}
{% endmacro %}
```

`adapter.dispatch()` is the more scalable version — it automatically routes to an adapter-specific implementation if one exists, falling back to a default.

### `run_query` and Dynamic Pivots

Macros can execute SQL at **compile time** to drive logic — useful for dynamic pivots or reading config from a table:

```sql
{% macro get_active_currencies() %}
    {% set results = run_query("select distinct currency_code from raw.exchange_rates") %}
    {% if execute %}
        {% set currencies = results.columns[0].values() %}
    {% else %}
        {% set currencies = [] %}
    {% endif %}
    {{ return(currencies) }}
{% endmacro %}
```

Then pivot dynamically in a model:

```sql
select
    order_id,
    {% for currency in get_active_currencies() %}
        sum(case when currency = '{{ currency }}' then amount end) as amount_{{ currency | lower }}
        {% if not loop.last %},{% endif %}
    {% endfor %}
from {{ ref('stg_payments') }}
group by 1
```

### Calling dbt-utils Macros

The `dbt-utils` package ships production-ready macros you'd otherwise write yourself. After adding it to `packages.yml` and running `dbt deps`:

```sql
-- Generate a surrogate key from multiple columns
select
    {{ dbt_utils.generate_surrogate_key(['order_id', 'line_item_id']) }} as sk,
    order_id,
    line_item_id,
    quantity
from {{ ref('stg_order_lines') }}
```

```sql
-- Date spine: generate a row for every date in a range
{{ dbt_utils.date_spine(
    datepart="day",
    start_date="cast('2024-01-01' as date)",
    end_date="cast('2025-01-01' as date)"
) }}
```

### Best Practices

1. **Prefer `dbt-utils` before writing your own** — the package already has `surrogate_key`, `pivot`, `unpivot`, `date_spine`, `safe_divide`, and more.
2. **Guard with `{% if execute %}`** — any Jinja that runs a query (`run_query`, `log`) must be wrapped so it only fires during execution, not during parsing/compilation.
3. **Keep macros small and single-purpose** — a macro that does one thing is easy to test and reuse; a macro that does five things becomes a maintenance burden.
4. **Document macros in `schema.yml`** — add `args` descriptions just like column docs; dbt generates macro documentation visible in the catalog.
5. **Use `return()` explicitly** — macros that produce a value should use `{{ return(value) }}` rather than relying on implicit output to avoid whitespace surprises.
6. **Test macro output with `dbt compile`** — run `dbt compile -s my_model` to inspect the rendered SQL before executing it; this is the fastest way to debug Jinja logic.

## 10:02 - dbt Model Contracts: Enforcing Schemas Like a Type System

### What Are Model Contracts?

Introduced in **dbt v1.5**, model contracts let you declare an authoritative schema for a model — specific columns with exact data types — and have dbt enforce it at build time. If the model's query produces columns that don't match the contract, the run fails before any data lands in the warehouse. Think of it as a compile-time type check for your data pipeline.

This shifts schema problems from "discovered by a downstream analyst three days later" to "caught in CI before merge."

### Enabling a Contract

Set `contract.enforced: true` in the model's config and declare every column with a `data_type`:

```yaml
# models/marts/schema.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: status
        data_type: varchar
      - name: order_total_usd
        data_type: numeric
      - name: ordered_at
        data_type: timestamp
```

When `dbt run` executes `fct_orders`, dbt introspects the compiled SQL result set and compares column names and types against the declared contract. A mismatch — a renamed column, a `float` where `numeric` was declared, a missing column — aborts the run with a clear error.

### Constraints

Beyond data types, contracts support **column-level** and **model-level constraints** that map to native database constraints:

```yaml
models:
  - name: dim_customers
    config:
      contract:
        enforced: true
    constraints:
      - type: primary_key
        columns: [customer_id]    # model-level composite PK
    columns:
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: email
        data_type: varchar
        constraints:
          - type: not_null
          - type: unique
      - name: region
        data_type: varchar
        constraints:
          - type: check
            expression: "region in ('APAC', 'EMEA', 'AMER')"
```

> **Note:** Constraint enforcement varies by warehouse. Snowflake and BigQuery treat most constraints as *informational* (metadata only, not enforced at insert time); Postgres and Redshift do enforce `not_null` and `unique`. dbt will still validate column types and names regardless of the warehouse.

### Contracts and `access` — the Public Interface Pattern

Contracts pair naturally with **model access levels** (`private` / `protected` / `public`). A `public` model is one other teams can depend on via `ref()` — making it `public` without a contract is risky because there's nothing preventing a schema change from silently breaking downstream consumers.

```yaml
models:
  - name: fct_orders
    access: public          # other projects can ref() this
    config:
      contract:
        enforced: true      # schema is guaranteed stable
```

This combination forms a **stable data API**: the owning team promises the shape, and dbt enforces it on every deploy.

### Versioned Models

When a contract change is unavoidable (adding a column is safe; removing or renaming one is breaking), dbt's **model versioning** lets you run `fct_orders.v1` and `fct_orders.v2` side by side during a migration window:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1
      - v: 2
```

Consumers still call `{{ ref('fct_orders') }}` and automatically resolve to the latest version, or pin with `{{ ref('fct_orders', version=1) }}` while migrating.

### When to Use Model Contracts

| Scenario | Use contract? |
|---|---|
| Core fact/dim tables shared across teams | **Yes** |
| Staging models consumed only within one project | Usually no |
| Models exposed to BI tools or external systems | **Yes** |
| Exploratory / ad hoc models | No |

### Best Practices

1. **Contract public models first** — enforce contracts on any model with `access: public` before other teams start depending on it.
2. **Declare all columns explicitly** — the contract requires every output column to be named; this forces intentional schema design and discourages `select *`.
3. **Use `dbt compile` to catch type mismatches early** — contracts are checked at run time, but reviewing compiled SQL first surfaces obvious problems before a full run.
4. **Treat contract changes like API versioning** — additive changes (new columns) are non-breaking; removals and renames require a new model version or coordinated migration.
5. **Combine with `not_null` tests** — constraints declare intent; generic tests (`not_null`, `unique`) provide runtime validation that the data actually satisfies those constraints, not just the schema.

## 15:03 - dbt Sources & Seeds: Ingesting Raw Data the Right Way

### Sources: Declaring Your Raw Tables

A **source** in dbt is a declaration that tells dbt about a table that lives outside your dbt project — raw data loaded by Fivetran, Airbyte, Stitch, or a custom pipeline. Declaring sources lets you use `{{ source() }}` in models instead of hard-coded schema/table strings, and unlocks **source freshness** monitoring.

Sources are defined in `schema.yml` (or any YAML file in your `models/` directory):

```yaml
# models/sources/shopify.yml
version: 2

sources:
  - name: shopify
    database: raw              # optional: override default database
    schema: shopify_raw        # schema where raw tables live
    loader: fivetran
    loaded_at_field: _fivetran_synced   # column used for freshness checks

    freshness:
      warn_after: {count: 12, period: hour}
      error_after: {count: 24, period: hour}

    tables:
      - name: orders
        description: "Raw orders from Shopify webhook."
        columns:
          - name: id
            description: "Shopify order ID."
          - name: email
          - name: created_at
          - name: updated_at
            meta:
              loaded_at_field: updated_at   # override freshness column per-table

      - name: customers
        freshness: null          # opt out of freshness for this table
```

Reference in a staging model:

```sql
-- models/staging/stg_orders.sql
select
    id                      as order_id,
    email                   as customer_email,
    created_at              as ordered_at,
    financial_status        as payment_status
from {{ source('shopify', 'orders') }}
```

### Source Freshness

Run `dbt source freshness` to check whether raw tables have been updated within the declared thresholds. This is the equivalent of an SLA monitor for your ingestion layer — integrate it into CI/CD or schedule it to fire before your dbt run:

```bash
# Check freshness for all declared sources
dbt source freshness

# Check a specific source table
dbt source freshness --select source:shopify.orders

# Fail the pipeline if any source is stale before running models
dbt source freshness && dbt run
```

dbt produces a `sources.json` artifact (alongside `manifest.json`) with pass/warn/error status per table, which dbt Cloud surfaces visually on the Sources page.

### Seeds: Versioned Reference Data

**Seeds** are CSV files committed to your repo (`seeds/` directory) and loaded into the warehouse as tables via `dbt seed`. They are ideal for small, slowly-changing lookup tables that don't come from a source system:

```
seeds/
  country_codes.csv
  payment_method_mapping.csv
  marketing_channel_categories.csv
```

```csv
# seeds/payment_method_mapping.csv
raw_code,display_name,is_digital
CARD,Credit / Debit Card,true
PYPL,PayPal,true
WIRE,Wire Transfer,false
CHQ,Cheque,false
```

Configure column types and schema in `dbt_project.yml` or a seed-level YAML:

```yaml
# dbt_project.yml
seeds:
  my_project:
    payment_method_mapping:
      +schema: reference          # lands in <target_schema>_reference
      +column_types:
        is_digital: boolean
```

Reference a seed exactly like a model:

```sql
select
    o.order_id,
    p.display_name  as payment_method,
    p.is_digital
from {{ ref('stg_orders') }} o
left join {{ ref('payment_method_mapping') }} p
    on o.raw_payment_code = p.raw_code
```

### Sources vs. Seeds vs. Models — Decision Guide

| Data lives... | Use |
|---|---|
| Loaded by an external pipeline into the warehouse | `source()` |
| Maintained as a CSV in your repo (small ref data) | `seed` |
| Produced by a dbt model | `ref()` |

### Best Practices

1. **One source YAML per upstream system** — group all Shopify tables in `shopify.yml`, all Salesforce tables in `salesforce.yml`. This mirrors team ownership and makes freshness monitoring easier to scope.
2. **Always alias raw tables via staging models** — never reference `{{ source() }}` from a mart model directly. Stage → Mart keeps transformations layered and testable.
3. **Set freshness thresholds conservatively** — start with `warn_after` at 1.5× your expected load frequency. If Fivetran syncs every 6 hours, warn at 9 hours, error at 24.
4. **Keep seeds small** — seeds are version-controlled CSVs; they are not designed for tables with thousands of rows. If a lookup table exceeds ~500 rows or changes frequently, load it as a source instead.
5. **Test seeds like models** — add `not_null` and `accepted_values` tests to seed YAML files. Seeds can contain typos; tests catch them before they corrupt downstream joins.
6. **Use `--full-refresh` for seeds after schema changes** — if you add or rename a column in a CSV, run `dbt seed --full-refresh` to recreate the table; otherwise dbt will not alter the existing schema.
