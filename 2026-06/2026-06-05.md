## 00:03 - dbt Incremental Models: Processing Only What Changed

One of the most impactful performance optimizations in a modern data warehouse is avoiding full table rebuilds on every pipeline run. dbt's **incremental models** solve this by processing only new or updated rows since the last run, dramatically cutting compute costs and latency at scale.

### How It Works

An incremental model is a standard SQL `SELECT` that dbt wraps with logic to `INSERT` or `MERGE` into an existing table rather than recreating it from scratch. You declare the strategy in a config block at the top of your model file:

```sql
{{ config(
    materialized='incremental',
    unique_key='event_id',
    incremental_strategy='merge'
) }}

SELECT
    event_id,
    user_id,
    event_type,
    created_at
FROM {{ source('app', 'events') }}
{% if is_incremental() %}
    WHERE created_at > (SELECT MAX(created_at) FROM {{ this }})
{% endif %}
```

The `is_incremental()` macro is `false` on the first run (full load) and `true` on subsequent runs (incremental load). The `{{ this }}` reference points to the target table itself, letting you filter for only records newer than the latest timestamp already stored.

### Incremental Strategies

| Strategy | Behavior | Best For |
|---|---|---|
| `append` | INSERT new rows only | Immutable event streams |
| `merge` | UPSERT by unique key | Slowly-changing records |
| `delete+insert` | Delete matched partitions, re-insert | Partitioned tables |
| `insert_overwrite` | Replace matched partitions | BigQuery / Spark workloads |

### Key Pitfalls to Avoid

- **Late-arriving data**: Records with a `created_at` in the past will be silently missed if your filter only looks forward. Add a lookback buffer (e.g., subtract 3 hours) or use an `updated_at` column if available.
- **Schema changes**: Adding a column to your SQL won't automatically add it to an incremental table — you must run with `--full-refresh` or alter the table manually.
- **Missing `unique_key`**: Without it, the `merge` strategy can't deduplicate, so repeated runs produce duplicate rows.

### When to Use Full Refresh Instead

If your source data is small (< a few million rows), or your transformation logic changes frequently, the overhead of managing incremental state may not be worth it. Start with `materialized='table'` and only switch to incremental once pipeline runtime becomes a real cost.

Incremental models are one of the highest-leverage tools in a data engineer's dbt toolkit — but only when applied to the right problem.

## 05:03 - Vector Databases: The Engine Behind Semantic Search and RAG

Traditional databases answer the question "does this row match my filter?" Vector databases answer a fundamentally different question: "which rows are *most similar* to this input?" That shift unlocks a new class of applications — semantic search, recommendation engines, and Retrieval-Augmented Generation (RAG) for LLMs.

### What Is a Vector Embedding?

A vector embedding is a list of floating-point numbers that encodes the *meaning* of some content — a sentence, image, or document — in high-dimensional space. Items with similar meaning cluster nearby in that space. For example, the sentence "the cat sat on the mat" and "a feline rested on a rug" will have embeddings that are geometrically close, even though they share no keywords.

Embedding models (like `text-embedding-3-small` from OpenAI or open-source alternatives like `nomic-embed-text`) convert raw content into these vectors at query time and at indexing time.

### How Vector Search Works

1. **Index**: Each document is converted to an embedding and stored in a vector index (e.g., pgvector, Pinecone, Weaviate, Chroma, Qdrant).
2. **Query**: The user's question is also embedded using the same model.
3. **ANN Search**: The database performs **Approximate Nearest Neighbor (ANN)** search — algorithms like HNSW or IVF-Flat find the top-K most similar vectors by cosine similarity or L2 distance in milliseconds, even across millions of vectors.

```python
# Example: semantic search with pgvector + psycopg2
query_embedding = embed("best practices for database indexing")
results = db.execute(
    "SELECT content FROM docs ORDER BY embedding <=> %s LIMIT 5",
    (query_embedding,)
)
```

The `<=>` operator is pgvector's cosine distance operator — the row with the smallest distance is the best semantic match.

### RAG: Grounding LLMs in Your Data

The dominant use case today is RAG: instead of fine-tuning a model on your private data, you retrieve the most relevant chunks at query time and inject them into the LLM's prompt as context. This keeps the model up to date without retraining and gives you full control over the source material.

**RAG pipeline in three steps:**
1. Chunk and embed your documents → store in a vector DB.
2. Embed the user's question → retrieve top-K chunks.
3. Pass retrieved chunks + question to the LLM → generate a grounded answer.

### Choosing a Vector Store

| Option | Best For |
|---|---|
| **pgvector** | Teams already on PostgreSQL; avoids a new infra dependency |
| **Chroma** | Local dev and prototyping; zero-config |
| **Qdrant / Weaviate** | Production scale with filtering and hybrid search |
| **Pinecone** | Fully managed, serverless; highest operational simplicity |

The choice rarely matters at prototype scale — pick pgvector if you have Postgres, Chroma otherwise. Optimize later when you have real traffic data to inform the decision.
