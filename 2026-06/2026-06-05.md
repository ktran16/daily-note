## 00:03 - dbt Incremental Models: Processing Only What Changed

One of the most impactful performance optimizations in a modern data warehouse is avoiding full table rebuilds on every pipeline run. dbt's **incremental models** solve this by processing only new or updated rows since the last run, dramatically cutting compute costs and latency at scale.

### How It Works

An incremental model is a standard SQL `SELECT` that dbt wraps with logic to `INSERT` or `MERGE` into an existing table rather than recreating it from scratch. You declare the strategy in a config block at the top of your model file:

```sql
{{ config(
    materialized='incremental',
    unique_key='event_id',
    incremental_strategy='merge'
) }}

SELECT
    event_id,
    user_id,
    event_type,
    created_at
FROM {{ source('app', 'events') }}
{% if is_incremental() %}
    WHERE created_at > (SELECT MAX(created_at) FROM {{ this }})
{% endif %}
```

The `is_incremental()` macro is `false` on the first run (full load) and `true` on subsequent runs (incremental load). The `{{ this }}` reference points to the target table itself, letting you filter for only records newer than the latest timestamp already stored.

### Incremental Strategies

| Strategy | Behavior | Best For |
|---|---|---|
| `append` | INSERT new rows only | Immutable event streams |
| `merge` | UPSERT by unique key | Slowly-changing records |
| `delete+insert` | Delete matched partitions, re-insert | Partitioned tables |
| `insert_overwrite` | Replace matched partitions | BigQuery / Spark workloads |

### Key Pitfalls to Avoid

- **Late-arriving data**: Records with a `created_at` in the past will be silently missed if your filter only looks forward. Add a lookback buffer (e.g., subtract 3 hours) or use an `updated_at` column if available.
- **Schema changes**: Adding a column to your SQL won't automatically add it to an incremental table — you must run with `--full-refresh` or alter the table manually.
- **Missing `unique_key`**: Without it, the `merge` strategy can't deduplicate, so repeated runs produce duplicate rows.

### When to Use Full Refresh Instead

If your source data is small (< a few million rows), or your transformation logic changes frequently, the overhead of managing incremental state may not be worth it. Start with `materialized='table'` and only switch to incremental once pipeline runtime becomes a real cost.

Incremental models are one of the highest-leverage tools in a data engineer's dbt toolkit — but only when applied to the right problem.
