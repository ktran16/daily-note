## 00:04 - dbt Graph Operators & Node Selection Syntax

dbt's node selection language lets you run precise subsets of your DAG instead of rebuilding everything. Mastering it is essential for fast CI pipelines, targeted debugging, and safe deploys.

### Core Selectors

```bash
# Run a single model
dbt run --select my_model

# Run a model and all its upstream dependencies (the + prefix)
dbt run --select +my_model

# Run a model and all its downstream dependents (the + suffix)
dbt run --select my_model+

# Run both upstream and downstream
dbt run --select +my_model+

# Limit traversal depth: only 2 hops upstream
dbt run --select 2+my_model
```

### Tag, Path, and Config Selectors

```bash
# All models tagged "finance"
dbt run --select tag:finance

# All models in a directory
dbt run --select models/marts/

# All incremental models
dbt run --select config.materialized:incremental

# Models with a specific package
dbt run --select package:dbt_utils
```

### The @ Operator (Include Upstream of Dependents)

The `@` operator is particularly useful for integration testing — it selects a model, all its ancestors, and all models that depend on it plus **their** ancestors:

```bash
# Selects my_model + its entire upstream + all models that use it + their upstream
dbt run --select @my_model
```

This is the "rebuild everything that touches this model" selector, ideal when you want a clean test of a node in isolation with all dependencies satisfied.

### Set Operations: Union, Intersection, Exclusion

```bash
# Union: run both sets
dbt run --select model_a model_b

# Intersection: models tagged "finance" AND in the marts folder
dbt run --select tag:finance,models/marts/

# Exclusion: everything except staging models
dbt run --select models/ --exclude models/staging/

# State-based: only modified models since last production run
dbt run --select state:modified+ --state ./prod-artifacts
```

### Practical CI Pattern

A common slim CI pattern uses state-based selection to only test what changed:

```bash
# In CI: fetch prod manifest, run only modified + downstream
dbt run --select state:modified+ --state s3://my-bucket/prod/
dbt test --select state:modified+ --state s3://my-bucket/prod/
```

This can reduce a 2-hour full run to under 5 minutes by skipping unchanged models entirely.

### Result-Based Selectors

After a failed run, re-run only what failed or was skipped:

```bash
dbt run --select result:error --state ./
dbt run --select result:skipped --state ./
```

Graph operators compound — `result:error+` re-runs failed models **and their dependents**, preventing stale downstream data.

## 05:03 - dbt Hooks: Running SQL Before and After Your Models

dbt hooks let you execute arbitrary SQL at specific points in a model's lifecycle — before it runs (`pre-hook`) or after it completes (`post-hook`). They're the escape hatch for everything dbt doesn't handle declaratively out of the box.

### Where to Define Hooks

Hooks can be set at three scopes, from broadest to narrowest:

**`dbt_project.yml` (project-wide or folder-level):**
```yaml
models:
  my_project:
    +post-hook:
      - "GRANT SELECT ON {{ this }} TO ROLE reporter"

    marts:
      +post-hook:
        - "GRANT SELECT ON {{ this }} TO ROLE analyst"
        - "ALTER TABLE {{ this }} CLUSTER BY (created_date)"
```

**Model config block (single model):**
```sql
{{ config(
    materialized='table',
    pre_hook="BEGIN TRANSACTION",
    post_hook="COMMIT"
) }}

SELECT ...
```

**`schema.yml` config key:**
```yaml
models:
  - name: fct_orders
    config:
      post_hook: "ANALYZE {{ this }}"
```

### The `{{ this }}` Context Variable

Inside a hook, `{{ this }}` resolves to the fully qualified relation being built — `database.schema.model_name`. This makes hooks portable without hardcoding names.

```sql
-- post-hook that logs run metadata to an audit table
INSERT INTO audit.model_runs (model_name, run_at, row_count)
SELECT
    '{{ this.name }}',
    CURRENT_TIMESTAMP,
    (SELECT COUNT(*) FROM {{ this }})
```

### Common Patterns

**Auto-grant permissions after every build:**
```yaml
# dbt_project.yml
models:
  +post-hook: "GRANT SELECT ON {{ this }} TO ROLE bi_tool"
```

**Run VACUUM/ANALYZE on Redshift after large loads:**
```yaml
models:
  marts:
    +post-hook: "VACUUM {{ this }}; ANALYZE {{ this }}"
```

**Create a clone for testing before replacing a table:**
```yaml
models:
  +pre-hook: >
    CREATE TABLE IF NOT EXISTS {{ this }}_backup
    AS SELECT * FROM {{ this }}
```

### Hook Execution Order

- Multiple hooks in a list run **in order**.
- `pre-hook` fires before the model SQL executes (after any temp relation is dropped).
- `post-hook` fires after the model relation is swapped into place.
- Hooks do **not** run on a `dbt compile` — only on `dbt run`, `dbt build`, or `dbt seed`.

### Gotcha: Hooks and `--full-refresh`

Hooks always fire regardless of `--full-refresh`. If your `pre-hook` drops a backup table, it will drop it on every full refresh too. Guard with Jinja conditionals when needed:

```sql
{% if not flags.FULL_REFRESH %}
  INSERT INTO audit.incremental_log VALUES (CURRENT_TIMESTAMP, '{{ this.name }}')
{% endif %}
```

Hooks keep dbt declarative for the 95% case while giving you a surgical escape valve for the rest.
