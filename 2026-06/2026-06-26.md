## 00:04 - dbt Graph Operators & Node Selection Syntax

dbt's node selection language lets you run precise subsets of your DAG instead of rebuilding everything. Mastering it is essential for fast CI pipelines, targeted debugging, and safe deploys.

### Core Selectors

```bash
# Run a single model
dbt run --select my_model

# Run a model and all its upstream dependencies (the + prefix)
dbt run --select +my_model

# Run a model and all its downstream dependents (the + suffix)
dbt run --select my_model+

# Run both upstream and downstream
dbt run --select +my_model+

# Limit traversal depth: only 2 hops upstream
dbt run --select 2+my_model
```

### Tag, Path, and Config Selectors

```bash
# All models tagged "finance"
dbt run --select tag:finance

# All models in a directory
dbt run --select models/marts/

# All incremental models
dbt run --select config.materialized:incremental

# Models with a specific package
dbt run --select package:dbt_utils
```

### The @ Operator (Include Upstream of Dependents)

The `@` operator is particularly useful for integration testing — it selects a model, all its ancestors, and all models that depend on it plus **their** ancestors:

```bash
# Selects my_model + its entire upstream + all models that use it + their upstream
dbt run --select @my_model
```

This is the "rebuild everything that touches this model" selector, ideal when you want a clean test of a node in isolation with all dependencies satisfied.

### Set Operations: Union, Intersection, Exclusion

```bash
# Union: run both sets
dbt run --select model_a model_b

# Intersection: models tagged "finance" AND in the marts folder
dbt run --select tag:finance,models/marts/

# Exclusion: everything except staging models
dbt run --select models/ --exclude models/staging/

# State-based: only modified models since last production run
dbt run --select state:modified+ --state ./prod-artifacts
```

### Practical CI Pattern

A common slim CI pattern uses state-based selection to only test what changed:

```bash
# In CI: fetch prod manifest, run only modified + downstream
dbt run --select state:modified+ --state s3://my-bucket/prod/
dbt test --select state:modified+ --state s3://my-bucket/prod/
```

This can reduce a 2-hour full run to under 5 minutes by skipping unchanged models entirely.

### Result-Based Selectors

After a failed run, re-run only what failed or was skipped:

```bash
dbt run --select result:error --state ./
dbt run --select result:skipped --state ./
```

Graph operators compound — `result:error+` re-runs failed models **and their dependents**, preventing stale downstream data.

## 05:03 - dbt Hooks: Running SQL Before and After Your Models

dbt hooks let you execute arbitrary SQL at specific points in a model's lifecycle — before it runs (`pre-hook`) or after it completes (`post-hook`). They're the escape hatch for everything dbt doesn't handle declaratively out of the box.

### Where to Define Hooks

Hooks can be set at three scopes, from broadest to narrowest:

**`dbt_project.yml` (project-wide or folder-level):**
```yaml
models:
  my_project:
    +post-hook:
      - "GRANT SELECT ON {{ this }} TO ROLE reporter"

    marts:
      +post-hook:
        - "GRANT SELECT ON {{ this }} TO ROLE analyst"
        - "ALTER TABLE {{ this }} CLUSTER BY (created_date)"
```

**Model config block (single model):**
```sql
{{ config(
    materialized='table',
    pre_hook="BEGIN TRANSACTION",
    post_hook="COMMIT"
) }}

SELECT ...
```

**`schema.yml` config key:**
```yaml
models:
  - name: fct_orders
    config:
      post_hook: "ANALYZE {{ this }}"
```

### The `{{ this }}` Context Variable

Inside a hook, `{{ this }}` resolves to the fully qualified relation being built — `database.schema.model_name`. This makes hooks portable without hardcoding names.

```sql
-- post-hook that logs run metadata to an audit table
INSERT INTO audit.model_runs (model_name, run_at, row_count)
SELECT
    '{{ this.name }}',
    CURRENT_TIMESTAMP,
    (SELECT COUNT(*) FROM {{ this }})
```

### Common Patterns

**Auto-grant permissions after every build:**
```yaml
# dbt_project.yml
models:
  +post-hook: "GRANT SELECT ON {{ this }} TO ROLE bi_tool"
```

**Run VACUUM/ANALYZE on Redshift after large loads:**
```yaml
models:
  marts:
    +post-hook: "VACUUM {{ this }}; ANALYZE {{ this }}"
```

**Create a clone for testing before replacing a table:**
```yaml
models:
  +pre-hook: >
    CREATE TABLE IF NOT EXISTS {{ this }}_backup
    AS SELECT * FROM {{ this }}
```

### Hook Execution Order

- Multiple hooks in a list run **in order**.
- `pre-hook` fires before the model SQL executes (after any temp relation is dropped).
- `post-hook` fires after the model relation is swapped into place.
- Hooks do **not** run on a `dbt compile` — only on `dbt run`, `dbt build`, or `dbt seed`.

### Gotcha: Hooks and `--full-refresh`

Hooks always fire regardless of `--full-refresh`. If your `pre-hook` drops a backup table, it will drop it on every full refresh too. Guard with Jinja conditionals when needed:

```sql
{% if not flags.FULL_REFRESH %}
  INSERT INTO audit.incremental_log VALUES (CURRENT_TIMESTAMP, '{{ this.name }}')
{% endif %}
```

Hooks keep dbt declarative for the 95% case while giving you a surgical escape valve for the rest.

## 10:03 - Using `--defer` to Develop Against Production Data

dbt's `--defer` flag lets you reference production-built models during local development instead of rebuilding your entire dependency graph. It's one of the biggest developer productivity features in dbt Core 1.0+ and is essential for large projects where a full run takes hours.

### The Problem It Solves

Suppose you're working on `fct_orders`, which depends on `stg_orders`, `stg_customers`, and `dim_products`. Without defer, you'd need to build all three upstream models locally just to test your change. With defer, you tell dbt: "use the production versions of anything I haven't built locally."

### How It Works

Defer requires a production **manifest** (`manifest.json`) — the artifact dbt generates after every run that describes the full project graph and where each relation lives.

```bash
# Step 1: Pull the production manifest (from dbt Cloud, S3, CI artifact, etc.)
dbt ls --state ./prod-artifacts  # verify the manifest is readable

# Step 2: Run only your changed model, deferring everything else to prod
dbt run --select fct_orders --defer --state ./prod-artifacts

# Step 3: Test it
dbt test --select fct_orders --defer --state ./prod-artifacts
```

dbt resolves `ref('stg_orders')` to the production schema's `stg_orders` table instead of your dev schema. Your model runs against real data without touching upstream models.

### Combining Defer with State Selection

The real power comes from pairing `--defer` with `state:modified`:

```bash
# Build only models you've changed + their downstream dependents,
# using prod for everything else
dbt build \
  --select state:modified+ \
  --defer \
  --state ./prod-artifacts
```

This is the "slim CI" pattern applied to local development — you work at the speed of your change, not the speed of your full DAG.

### Getting the Production Manifest

**dbt Cloud:** Download via the API or point to the "Production" environment artifact URL.

```bash
# Example: fetch from dbt Cloud's metadata API
curl -H "Authorization: Token $DBT_CLOUD_API_KEY" \
  "https://cloud.getdbt.com/api/v2/accounts/$ACCOUNT_ID/runs/$RUN_ID/artifacts/manifest.json" \
  -o prod-artifacts/manifest.json
```

**Self-hosted CI:** Upload `target/manifest.json` after each production run to S3/GCS, then download it in dev:

```bash
aws s3 cp s3://my-dbt-artifacts/prod/manifest.json ./prod-artifacts/manifest.json
```

### The `--favor-state` Flag

By default, if a model exists in both your local target and the prod manifest, dbt uses the **local** version. Add `--favor-state` to always prefer the production relation — useful when your local schema is partially built and you want to guarantee prod data is used:

```bash
dbt run --select fct_orders --defer --favor-state --state ./prod-artifacts
```

### Practical Workflow

```bash
# In your shell profile or Makefile:
alias dbt-dev='dbt build --defer --state ./prod-artifacts --select state:modified+'

# Morning routine: pull latest prod artifact, then develop
make pull-prod-manifest
dbt-dev
```

Defer effectively gives every developer a free, always-current staging environment — the production warehouse — without the cost of duplicating it.

## 15:04 - dbt Model Contracts: Enforcing Schema Guarantees

dbt model contracts, introduced in dbt Core v1.5, let you declare a binding schema for a model — column names, data types, and constraints — that dbt verifies at build time. They shift schema drift from a silent runtime surprise to a loud, fail-fast compile error.

### Why Contracts Matter

In a typical dbt project, a model's columns are implied by its SQL. Downstream consumers depend on that schema, but nothing stops a developer from accidentally renaming a column or changing its type. Contracts make the schema an explicit, versioned contract between the model and its consumers.

### Enabling a Contract

Contracts are configured in `schema.yml`:

```yaml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true   # dbt will validate column defs at build time
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_total
        data_type: numeric
      - name: created_at
        data_type: timestamp
```

When `enforced: true`, dbt compares the declared columns against what the model's SQL actually produces. If a column is missing or has a mismatched type, the run fails before any data is written.

### Constraints

dbt supports several constraint types, though adapter support varies:

| Constraint | Supported adapters |
|---|---|
| `not_null` | All |
| `unique` | All (as a test; native DDL on some) |
| `primary_key` | Snowflake, BigQuery, Databricks |
| `foreign_key` | Snowflake, BigQuery |
| `check` | Postgres, Snowflake |

```yaml
columns:
  - name: status
    data_type: varchar
    constraints:
      - type: check
        expression: "status IN ('pending', 'shipped', 'delivered')"
```

### Contracts and Model Versioning

Contracts pair naturally with dbt's **model versioning** feature. When you bump a model version, you can maintain the old contract in `v1` while introducing breaking schema changes in `v2`:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1          # legacy consumers pin to v1
        config:
          contract:
            enforced: true
      - v: 2          # new schema, new contract
        config:
          contract:
            enforced: true
```

Consumers use `ref('fct_orders', version=1)` to stay pinned until they're ready to migrate.

### What Contracts Don't Do

- They don't enforce constraints at the *database* level unless the adapter supports native DDL constraints.
- They don't validate row-level data quality — that's still the job of dbt tests (`not_null`, `unique`, `accepted_values`).
- Column *order* must match between the contract and the SELECT for most adapters.

### Best Practice

Enable contracts on your **mart-layer models** — the ones exposed to BI tools, data science, or external teams. Staging and intermediate models change too frequently to benefit from strict contracts, but your `fct_` and `dim_` tables are exactly where downstream consumers feel the pain of unannounced schema changes.

```bash
# Check that contracts pass without running the model
dbt compile --select fct_orders
```

Contracts turn an implicit handshake into an explicit guarantee — the right trade-off for any model you'd have to file a ticket to change.
