## 00:03 - dbt Hooks & Operations: Running SQL Before, After, and Outside Models

dbt **hooks** let you execute arbitrary SQL at specific points in a model's lifecycle. Combined with **operations** (`dbt run-operation`), they give you a clean escape hatch for administrative tasks without polluting your models with side-effect logic.

### The Four Hook Types

| Hook | Fires |
|------|-------|
| `pre-hook` | Before a model's `CREATE` statement runs |
| `post-hook` | After a model finishes materializing |
| `on-run-start` | Once at the very start of `dbt run` / `dbt test` |
| `on-run-end` | Once at the very end of `dbt run` / `dbt test` |

### Common Use Cases

**1. Granting permissions after every build**

```sql
-- dbt_project.yml
models:
  my_project:
    +post-hook:
      - "GRANT SELECT ON {{ this }} TO ROLE reporter"
```

**2. Analyzing a table for query planning (Redshift / Postgres)**

```sql
-- models/marts/fct_orders.sql
{{ config(
    materialized='table',
    post_hook="ANALYZE {{ this }}"
) }}

select ...
```

**3. Running a data audit after a critical model**

```sql
{{ config(
    post_hook=[
      "GRANT SELECT ON {{ this }} TO ROLE analyst",
      "{{ log('Built ' ~ this ~ ' at ' ~ run_started_at, info=True) }}"
    ]
) }}
```

Multiple hooks are passed as a list and run in order.

### dbt run-operation: Macros as CLI Commands

For tasks that don't belong inside any model — creating schemas, dropping stale tables, vacuuming — write a macro and invoke it directly:

```sql
-- macros/create_schema.sql
{% macro create_schema(schema_name) %}
  CREATE SCHEMA IF NOT EXISTS {{ schema_name }};
{% endmacro %}
```

```bash
dbt run-operation create_schema --args '{"schema_name": "staging"}'
```

This is useful in CI pipelines to set up environments before `dbt run` fires.

### Key Gotchas

- **`{{ this }}`** is only available inside model-level hooks, not `on-run-start` / `on-run-end`.
- Hooks run inside the **same transaction** as the model (for warehouses that support transactions), so a hook failure will roll back the model too.
- Prefer `on-run-end` over per-model `post-hook` for permission grants — it batches the work and avoids hammering the warehouse with one `GRANT` per model.
- Hooks are **not retried** if they fail; build retry logic into the macro itself if needed.

### Practical Pattern: Bulk Permission Grant at Run End

```yaml
# dbt_project.yml
on-run-end:
  - "{% for rel in results | selectattr('status', 'equalto', 'success') | map(attribute='node.relation_name') %}
       GRANT SELECT ON {{ rel }} TO ROLE reporter;
     {% endfor %}"
```

This iterates over `results` — a special variable available in `on-run-end` — and grants access only to models that actually succeeded, skipping failed or skipped nodes.

## 05:02 - dbt vars, env_var(), and Environment-Based Configuration

One of dbt's most powerful (and underused) features is the ability to **parameterize your project** so the same codebase behaves differently in dev, staging, and production — without branching or duplicating models.

### `var()`: Project-Level Variables

Declare variables in `dbt_project.yml` and reference them in any model or macro:

```yaml
# dbt_project.yml
vars:
  start_date: '2023-01-01'
  payment_methods: ['credit_card', 'bank_transfer', 'gift_card']
  is_production: false
```

```sql
-- models/staging/stg_orders.sql
select *
from {{ source('raw', 'orders') }}
where order_date >= '{{ var("start_date") }}'
  and payment_method in (
    {{ var("payment_methods") | join("', '") | surround("'") }}
  )
```

Override at runtime without touching any file:

```bash
dbt run --vars '{"start_date": "2024-01-01", "is_production": true}'
```

### `env_var()`: Reading System Environment Variables

Use `env_var()` to pull secrets and infra values from the OS environment — ideal for credentials and environment names:

```yaml
# profiles.yml
my_project:
  target: "{{ env_var('DBT_TARGET', 'dev') }}"
  outputs:
    prod:
      type: snowflake
      account: "{{ env_var('SNOWFLAKE_ACCOUNT') }}"
      password: "{{ env_var('SNOWFLAKE_PASSWORD') }}"
      database: PROD_DB
      schema: analytics
```

The second argument to `env_var()` is an optional default — useful so dev machines don't need every variable set.

### Target-Based Logic: `target.name`

`target.name` is the currently active dbt target from `profiles.yml`. Use it to swap behavior between environments:

```sql
-- models/marts/fct_events.sql
{{ config(
    materialized = 'incremental' if target.name == 'prod' else 'view'
) }}

select *
from {{ ref('stg_events') }}

{% if is_incremental() %}
  where event_ts > (select max(event_ts) from {{ this }})
{% endif %}
```

In dev, this materializes as a fast `view`; in prod it runs as a proper incremental table — no manual changes between environments.

### Practical Pattern: Row Limiting in Dev

Avoid full table scans locally by capping rows based on target:

```sql
-- macros/limit_in_dev.sql
{% macro limit_in_dev(n=10000) %}
  {% if target.name != 'prod' %}
    limit {{ n }}
  {% endif %}
{% endmacro %}
```

```sql
-- models/staging/stg_events.sql
select * from {{ source('raw', 'events') }}
{{ limit_in_dev(5000) }}
```

### Quick Reference

| Mechanism | Best For |
|---|---|
| `var()` | Business logic toggles, date ranges, lists of values |
| `env_var()` | Secrets, infra credentials, dynamic target names |
| `target.name` | Environment-specific materializations and behavior |
| `target.schema` | Constructing custom schema names per developer |

Keep secrets in `env_var()`, never hardcoded. Use `var()` for anything a business user might want to override at the CLI. Use `target.name` for structural differences like materialization strategy or row limits.
