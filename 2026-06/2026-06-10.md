## 00:06 - Feature Stores: The Missing Infrastructure Layer in ML Systems

Feature stores are one of the most impactful yet underappreciated components of production machine learning. They solve a fundamental problem: the same features computed for model training need to be reproduced, at low latency, during inference — often weeks or months later.

**The Core Problem**

Without a feature store, teams commonly fall into the "training-serving skew" trap. A data scientist computes `rolling_7d_avg_spend` in a Spark job during training; an engineer reimplements it in Python during serving. Small differences in null handling, timezone logic, or aggregation windows compound into silent model degradation. The model evaluates fine offline, then quietly underperforms in production.

**What a Feature Store Actually Does**

A feature store has two storage layers with very different roles:

- **Offline store** (e.g., Parquet on S3, BigQuery): Stores historical feature values for training and backfills. Optimized for throughput, not latency.
- **Online store** (e.g., Redis, DynamoDB, Cassandra): Stores the latest feature values for real-time inference. Optimized for sub-millisecond point lookups.

A materialization pipeline keeps them in sync: it reads from the offline store and writes current feature values into the online store on a schedule.

**Point-in-Time Correctness**

The other critical guarantee a feature store provides is point-in-time correct joins during training data generation. When building a training dataset, each row should use only features that were available *at that moment in time*, not features that were later updated. Naive joins leak future information into training, inflating offline metrics while the model fails in production.

**When to Adopt One**

Feature stores add real overhead. For a single model with a small team, they're often overkill. They pay off when:
- Multiple models reuse overlapping features (avoiding duplicate compute)
- Low-latency online inference requires precomputed features
- Regulatory requirements demand reproducibility of model predictions

Popular options include Feast (open-source), Tecton, Hopsworks, and Vertex AI Feature Store. Start with Feast if you want to self-host with minimal vendor lock-in.

The real ROI is organizational: when features become shared, versioned assets rather than bespoke per-model logic, iteration speed across the team increases substantially.

## 05:04 - Delta Lake: Bringing ACID Transactions to Your Data Lake

Modern data lakes are built on cheap object storage (S3, GCS, ADLS), but raw object storage has a fundamental limitation: it offers no transactional guarantees. Delta Lake, the open-source storage layer developed by Databricks, addresses this gap by adding an ACID transaction log on top of Parquet files.

**How the Transaction Log Works**

Delta Lake stores data as Parquet files alongside a `_delta_log/` directory. Every write operation — whether an insert, update, delete, or schema change — appends a JSON commit entry to this log. The log records exactly which files were added or removed. Reading a Delta table means first consulting the log to determine the current snapshot of valid files, then reading only those files.

This design gives you:
- **Atomicity**: A write either fully commits or leaves the table untouched — no partial states from failed jobs.
- **Serializable isolation**: Concurrent writers use optimistic concurrency control. Delta detects conflicts (two jobs modifying the same file range) and retries or fails gracefully.
- **Time travel**: Because old log entries and data files are retained, you can query any previous snapshot with `VERSION AS OF n` or `TIMESTAMP AS OF '2026-01-01'`.

**Practical Upserts with MERGE**

One of Delta's most useful features is `MERGE INTO`, which implements upsert semantics efficiently:

```sql
MERGE INTO orders AS target
USING updates AS source
ON target.order_id = source.order_id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;
```

Without Delta, implementing upserts on a data lake requires full partition rewrites or complicated deduplication logic. MERGE handles it in a single, atomic operation.

**Schema Enforcement and Evolution**

Delta enforces schema on write by default — mismatched columns raise an error rather than silently writing nulls or corrupt data. When you legitimately need to add columns, `ALTER TABLE` or `mergeSchema` options handle it gracefully, recording the schema change in the transaction log.

**When to Use It**

Delta Lake is the right default for any Spark-based data lake workload that involves incremental updates, CDC pipelines, or regulatory requirements for data correction. The storage overhead of the transaction log is negligible, and the operational safety it provides — especially the ability to roll back a bad write with `RESTORE TABLE` — pays for itself on the first production incident.
