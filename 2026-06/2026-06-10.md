## 00:06 - Feature Stores: The Missing Infrastructure Layer in ML Systems

Feature stores are one of the most impactful yet underappreciated components of production machine learning. They solve a fundamental problem: the same features computed for model training need to be reproduced, at low latency, during inference — often weeks or months later.

**The Core Problem**

Without a feature store, teams commonly fall into the "training-serving skew" trap. A data scientist computes `rolling_7d_avg_spend` in a Spark job during training; an engineer reimplements it in Python during serving. Small differences in null handling, timezone logic, or aggregation windows compound into silent model degradation. The model evaluates fine offline, then quietly underperforms in production.

**What a Feature Store Actually Does**

A feature store has two storage layers with very different roles:

- **Offline store** (e.g., Parquet on S3, BigQuery): Stores historical feature values for training and backfills. Optimized for throughput, not latency.
- **Online store** (e.g., Redis, DynamoDB, Cassandra): Stores the latest feature values for real-time inference. Optimized for sub-millisecond point lookups.

A materialization pipeline keeps them in sync: it reads from the offline store and writes current feature values into the online store on a schedule.

**Point-in-Time Correctness**

The other critical guarantee a feature store provides is point-in-time correct joins during training data generation. When building a training dataset, each row should use only features that were available *at that moment in time*, not features that were later updated. Naive joins leak future information into training, inflating offline metrics while the model fails in production.

**When to Adopt One**

Feature stores add real overhead. For a single model with a small team, they're often overkill. They pay off when:
- Multiple models reuse overlapping features (avoiding duplicate compute)
- Low-latency online inference requires precomputed features
- Regulatory requirements demand reproducibility of model predictions

Popular options include Feast (open-source), Tecton, Hopsworks, and Vertex AI Feature Store. Start with Feast if you want to self-host with minimal vendor lock-in.

The real ROI is organizational: when features become shared, versioned assets rather than bespoke per-model logic, iteration speed across the team increases substantially.

## 05:04 - Delta Lake: Bringing ACID Transactions to Your Data Lake

Modern data lakes are built on cheap object storage (S3, GCS, ADLS), but raw object storage has a fundamental limitation: it offers no transactional guarantees. Delta Lake, the open-source storage layer developed by Databricks, addresses this gap by adding an ACID transaction log on top of Parquet files.

**How the Transaction Log Works**

Delta Lake stores data as Parquet files alongside a `_delta_log/` directory. Every write operation — whether an insert, update, delete, or schema change — appends a JSON commit entry to this log. The log records exactly which files were added or removed. Reading a Delta table means first consulting the log to determine the current snapshot of valid files, then reading only those files.

This design gives you:
- **Atomicity**: A write either fully commits or leaves the table untouched — no partial states from failed jobs.
- **Serializable isolation**: Concurrent writers use optimistic concurrency control. Delta detects conflicts (two jobs modifying the same file range) and retries or fails gracefully.
- **Time travel**: Because old log entries and data files are retained, you can query any previous snapshot with `VERSION AS OF n` or `TIMESTAMP AS OF '2026-01-01'`.

**Practical Upserts with MERGE**

One of Delta's most useful features is `MERGE INTO`, which implements upsert semantics efficiently:

```sql
MERGE INTO orders AS target
USING updates AS source
ON target.order_id = source.order_id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;
```

Without Delta, implementing upserts on a data lake requires full partition rewrites or complicated deduplication logic. MERGE handles it in a single, atomic operation.

**Schema Enforcement and Evolution**

Delta enforces schema on write by default — mismatched columns raise an error rather than silently writing nulls or corrupt data. When you legitimately need to add columns, `ALTER TABLE` or `mergeSchema` options handle it gracefully, recording the schema change in the transaction log.

**When to Use It**

Delta Lake is the right default for any Spark-based data lake workload that involves incremental updates, CDC pipelines, or regulatory requirements for data correction. The storage overhead of the transaction log is negligible, and the operational safety it provides — especially the ability to roll back a bad write with `RESTORE TABLE` — pays for itself on the first production incident.

## 10:04 - Retrieval-Augmented Generation: Grounding LLMs in Real Data

Large language models are trained on static snapshots of text. Once training ends, they know nothing about events that happened afterward, and they have no access to your private documents, databases, or real-time feeds. Retrieval-Augmented Generation (RAG) solves this by connecting the LLM to a live knowledge source at query time.

**How RAG Works**

At its core, RAG has two phases:

1. **Retrieval**: The user's query is embedded into a vector and used to search a vector database (Pinecone, Weaviate, pgvector, Chroma). The top-k most semantically similar document chunks are returned.
2. **Generation**: Those chunks are injected into the LLM's context window as grounding material. The model answers based on the retrieved evidence, not solely from its training weights.

The result is a system that can answer questions about documents it has never been trained on, with citations traceable to the exact source.

**Chunking Strategy Matters More Than the Model**

A common RAG mistake is treating chunking as an afterthought. If chunks are too large, they dilute relevance. If too small, they lose surrounding context. A practical starting point: 512-token chunks with a 10% overlap, respecting sentence and paragraph boundaries. For technical docs, recursive character splitting works well; for structured data like PDFs with headers, hierarchical chunking that preserves section context significantly improves retrieval quality.

**Hybrid Search: Dense + Sparse**

Pure vector (dense) search can miss exact keyword matches that are semantically distant from the query. Hybrid search combines vector similarity with BM25 keyword scoring, then merges results using Reciprocal Rank Fusion (RRF). This is now the production standard — retrieval recall improves measurably on precise queries like product codes, names, or domain jargon.

**When RAG Beats Fine-Tuning**

Fine-tuning bakes knowledge into model weights — expensive, slow to update, and opaque. RAG keeps knowledge in an index that can be updated in minutes. Choose RAG when:
- Your knowledge base changes frequently (internal wikis, product catalogs)
- You need citations and source traceability
- You want to avoid the cost and risk of retraining

RAG and fine-tuning are complementary: fine-tune for style and task format, use RAG for domain knowledge retrieval.

## 15:06 - The Bitcoin Lightning Network: Payment Channels and Layer-2 Scaling

Bitcoin's base layer processes roughly 7 transactions per second — far too slow for global payments. The Lightning Network is a Layer-2 protocol that routes payments off-chain, achieving near-instant finality at negligible fees, while settling ultimately on the Bitcoin blockchain.

**Payment Channels: The Core Primitive**

Two parties open a channel by locking Bitcoin into a 2-of-2 multisig address on-chain. This single opening transaction creates a private ledger between them. They can then exchange thousands of payments by simply swapping signed balance updates — none of which touch the blockchain. When they're done, they broadcast the final state on-chain to settle. Two on-chain transactions (open + close) replace thousands, compressing fees dramatically.

**Routing Across the Network**

Lightning's power multiplies because channels can be chained. If Alice has a channel with Bob, and Bob has a channel with Carol, Alice can pay Carol without a direct channel — Bob routes the payment and earns a small routing fee. This creates a mesh network where payments hop across multiple nodes.

The routing protocol uses **Hash Time-Locked Contracts (HTLCs)** to ensure atomicity: either every hop in the route succeeds, or the entire payment fails and funds are returned. Nobody in the chain can steal funds mid-route because each hop is cryptographically locked to the same secret preimage.

**Practical Limitations**

Lightning isn't a silver bullet:
- **Inbound liquidity**: You can only receive payments up to the amount others have committed toward you in a channel. Bootstrapping receiving capacity is a real friction point for new nodes.
- **Channel management**: Long-lived routing nodes must actively rebalance channels to keep liquidity distributed.
- **Offline payments**: Both parties must be online to complete a payment, which complicates cold-storage wallet use.

**Why It Matters for Crypto Infrastructure**

Lightning demonstrates a pattern increasingly adopted across blockchains: move high-frequency, low-value interactions off-chain, use the base layer only for dispute resolution and final settlement. Ethereum's rollup-centric roadmap follows the same philosophy. Understanding Lightning's design — channels, HTLCs, routing incentives — provides a mental model for evaluating any Layer-2 scaling solution.
