## 00:04 - dbt Sources & Seeds: Declaring Raw Data and Loading Static Files

Two foundational dbt features — **sources** and **seeds** — solve the problem of connecting your transformation layer to the outside world cleanly and traceably.

### Sources: Documenting and Referencing Raw Tables

A source is a declaration that a raw table exists in your warehouse and that dbt models should depend on it. Instead of hardcoding database/schema/table names throughout your SQL, you define them once in a YAML file and reference them with the `source()` function.

```yaml
# models/staging/_sources.yml
version: 2

sources:
  - name: stripe
    database: raw
    schema: stripe
    tables:
      - name: payments
        description: "Raw payment events from the Stripe API"
        loaded_at_field: created_at   # used for freshness checks
        freshness:
          warn_after: {count: 6, period: hour}
          error_after: {count: 24, period: hour}
        columns:
          - name: id
            tests:
              - unique
              - not_null
```

Reference it in a model with `{{ source('stripe', 'payments') }}`:

```sql
-- models/staging/stg_payments.sql
select
    id          as payment_id,
    amount,
    currency,
    created_at
from {{ source('stripe', 'payments') }}
```

This gives you:
- **Lineage** — the DAG tracks raw tables as root nodes.
- **Freshness checks** — `dbt source freshness` alerts you when upstream data is stale before you even run transforms.
- **Documentation** — column descriptions and tests live alongside the source declaration.

### Seeds: Loading Static CSV Files as Tables

Seeds let you commit small, static reference datasets as CSV files and load them into your warehouse with `dbt seed`. They're ideal for things that change rarely: country codes, account tier mappings, cost center lists.

```
seeds/
  account_tiers.csv
```

```csv
tier_id,tier_name,monthly_limit
1,Starter,1000
2,Growth,10000
3,Enterprise,999999
```

Reference it just like any other model:

```sql
-- models/marts/orders_enriched.sql
select
    o.*,
    t.tier_name
from {{ ref('orders') }} o
left join {{ ref('account_tiers') }} t
    on o.account_tier_id = t.tier_id
```

Configure seed behavior in `dbt_project.yml`:

```yaml
seeds:
  my_project:
    account_tiers:
      +schema: reference_data   # load into a specific schema
      +quote_columns: true
```

### Key Distinction

| | Sources | Seeds |
|---|---|---|
| Data lives in | Warehouse (loaded externally) | Git repo (CSV files) |
| Purpose | Reference existing raw tables | Load small static datasets |
| Command | `dbt source freshness` | `dbt seed` |
| Reference | `{{ source('name', 'table') }}` | `{{ ref('filename') }}` |

Use sources for any table loaded by an ELT tool (Fivetran, Airbyte, custom pipelines). Use seeds for lookup tables that belong in version control alongside your code.

## 05:02 - dbt Model Contracts: Enforcing Column-Level Guarantees

**Model contracts** (introduced in dbt 1.5) let you formally declare the structure a model must produce — column names, data types, and constraints — and have dbt verify them at build time. Think of it as a compile-time schema check between model authors and downstream consumers.

### Why Contracts Matter

Without contracts, downstream models silently break when an upstream author renames a column or changes a type. Contracts make that change a build-time failure instead of a production incident.

### Declaring a Contract

Add `contract: {enforced: true}` to a model's config, then list every column with its `data_type`:

```yaml
# models/marts/fct_orders.yml
version: 2

models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key   # logical only on most adapters
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_total
        data_type: numeric
      - name: status
        data_type: varchar
      - name: created_at
        data_type: timestamp
```

When `dbt run` executes this model, it compares the query's output columns against the declared schema. If a column is missing, has the wrong type, or an extra undeclared column is returned, the run fails immediately.

### Adapter-Level Constraints

Constraints like `not_null`, `unique`, and `foreign_key` can be pushed down to the warehouse DDL (for `table` and `incremental` materializations) if the adapter supports them:

```yaml
columns:
  - name: order_id
    data_type: bigint
    constraints:
      - type: not_null
      - type: unique
  - name: customer_id
    data_type: bigint
    constraints:
      - type: foreign_key
        to: ref('dim_customers')
        to_columns: [customer_id]
```

On Snowflake and BigQuery these generate actual DDL constraints. On most warehouses `primary_key` and `foreign_key` are logical-only — they document intent but aren't enforced by the engine.

### Contracts + Versioning

Contracts pair naturally with **model versioning**. When you need to change a contracted model's schema, bump the version instead of breaking consumers:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1     # kept for existing consumers
      - v: 2     # new schema with contract
        config:
          contract:
            enforced: true
```

Consumers reference `ref('fct_orders', v=1)` or `ref('fct_orders')` (resolves to latest). Old consumers keep working until explicitly migrated.

### Best Practices

- **Only enforce on public interface models** — marts and models shared across teams. Staging models change frequently; adding contracts there creates friction without much value.
- **Start with `data_type` only** — add constraints progressively as you gain confidence in the data.
- **Pair with `access: public`** — the combination of contract + public access signals "this model is stable and safe to depend on."

Model contracts shift the failure mode for schema changes from "silent downstream breakage discovered in production" to "immediate build failure the author sees first."

## 10:02 - dbt Snapshots: Tracking Historical Changes with SCD Type 2

**Snapshots** are dbt's built-in mechanism for capturing how a mutable source table changes over time. They implement a **Slowly Changing Dimension Type 2 (SCD2)** pattern: each change to a row produces a new history record rather than overwriting the old one.

### Why Snapshots Exist

Most operational tables (CRM contacts, order statuses, user attributes) only store the current state. Without snapshots, you lose the ability to answer questions like "What was this customer's plan tier last quarter?" Snapshots solve this by periodically diffing the source and appending change records.

### Defining a Snapshot

Snapshots live in the `snapshots/` directory and use a special `snapshot` block:

```sql
-- snapshots/orders_snapshot.sql
{% snapshot orders_snapshot %}

{{
    config(
      target_schema='snapshots',
      unique_key='order_id',
      strategy='timestamp',
      updated_at='updated_at'
    )
}}

select * from {{ source('app_db', 'orders') }}

{% endsnapshot %}
```

Run with `dbt snapshot`. On subsequent runs dbt:
1. Queries the source for all current rows.
2. Compares against the snapshot table using `unique_key`.
3. Closes changed rows by setting `dbt_valid_to = current_timestamp`.
4. Inserts new rows with `dbt_valid_to = NULL` (the "current" record).

### The Two Strategies

| Strategy | Use when | Config |
|---|---|---|
| `timestamp` | Source has a reliable `updated_at` column | `updated_at='updated_at'` |
| `check` | No reliable timestamp; compare column values | `check_cols=['status', 'plan']` or `check_cols='all'` |

`timestamp` is faster and preferred. `check` is a full row scan — limit `check_cols` to only the columns that matter.

### Snapshot Output Columns

dbt automatically adds four metadata columns to the snapshot table:

| Column | Meaning |
|---|---|
| `dbt_scd_id` | Surrogate key unique to each snapshot row |
| `dbt_updated_at` | When dbt last evaluated this record |
| `dbt_valid_from` | When this version of the row became active |
| `dbt_valid_to` | When it was superseded (`NULL` = current) |

### Querying Snapshots

To get the current state of all orders:

```sql
select * from {{ ref('orders_snapshot') }}
where dbt_valid_to is null
```

To reconstruct a point-in-time view (e.g., all orders as of 2026-01-01):

```sql
select * from {{ ref('orders_snapshot') }}
where dbt_valid_from <= '2026-01-01'
  and (dbt_valid_to > '2026-01-01' or dbt_valid_to is null)
```

### Practical Tips

- **Run snapshots before models** in your pipeline (`dbt snapshot && dbt run`) so the snapshot captures the source state before any transformations overwrite it.
- **Never change `unique_key` or `strategy`** on an existing snapshot without rebuilding — it will corrupt the history.
- **Snapshots are append-heavy**: for very large tables, filter the source query to only rows that could have changed recently (using `updated_at > dateadd(day, -3, current_date)`) to keep runtime manageable.
- **Hard deletes aren't tracked by default** — a row deleted from the source stays open (`dbt_valid_to = NULL`) in the snapshot. Use the `invalidate_hard_deletes` config to auto-close those rows.

```sql
{{
    config(
      strategy='timestamp',
      updated_at='updated_at',
      invalidate_hard_deletes=true
    )
}}
```

Snapshots are one of dbt's most powerful features for analytics engineering — they turn ephemeral operational data into a durable, queryable audit trail.

## 15:03 - dbt Macros & Jinja: Writing Reusable SQL Logic

**Macros** are dbt's mechanism for code reuse. They're Jinja2 functions defined in `.sql` files under the `macros/` directory that can be called from any model, test, or other macro. Mastering macros eliminates copy-pasted SQL and makes your project dramatically easier to maintain.

### The Basics: Defining and Calling a Macro

```sql
-- macros/cents_to_dollars.sql
{% macro cents_to_dollars(column_name) %}
    ({{ column_name }} / 100.0)::numeric(10, 2)
{% endmacro %}
```

Call it in any model with `{{ cents_to_dollars('amount') }}`:

```sql
-- models/staging/stg_payments.sql
select
    id           as payment_id,
    {{ cents_to_dollars('amount') }} as amount_usd,
    currency,
    created_at
from {{ source('stripe', 'payments') }}
```

dbt compiles this to plain SQL before executing, so the warehouse never sees Jinja.

### Macros with Logic: Conditional SQL Generation

Macros can contain full Jinja control flow — `if`, `for`, `set` — to generate SQL dynamically:

```sql
-- macros/generate_surrogate_key.sql
{% macro generate_surrogate_key(field_list) %}
    {{ dbt_utils.generate_surrogate_key(field_list) }}
{% endmacro %}
```

A more custom example — building a `CASE WHEN` block from a dict:

```sql
{% macro map_status_label(column_name, mapping) %}
    case {{ column_name }}
    {% for raw, label in mapping.items() %}
        when '{{ raw }}' then '{{ label }}'
    {% endfor %}
        else 'unknown'
    end
{% endmacro %}
```

```sql
-- usage in a model
select
    order_id,
    {{ map_status_label('raw_status', {
        'P': 'pending',
        'C': 'complete',
        'X': 'cancelled'
    }) }} as status_label
from {{ ref('stg_orders') }}
```

### Adapter-Aware Macros with `target` and `run_query`

Macros can branch on the target adapter, making cross-warehouse projects cleaner:

```sql
{% macro current_timestamp() %}
    {% if target.type == 'bigquery' %}
        current_timestamp()
    {% elif target.type == 'snowflake' %}
        convert_timezone('UTC', current_timestamp())
    {% else %}
        now()
    {% endif %}
{% endmacro %}
```

### `run_query`: Fetching Data at Compile Time

`run_query` executes a SQL statement during compilation and returns the result as a relation object — useful for dynamically generating column lists:

```sql
{% macro get_column_values(table, column) %}
    {% set query %}
        select distinct {{ column }} from {{ table }} order by 1
    {% endset %}
    {% set results = run_query(query) %}
    {% if execute %}
        {{ return(results.columns[0].values()) }}
    {% else %}
        {{ return([]) }}
    {% endif %}
{% endmacro %}
```

### The `execute` Flag

Always guard `run_query` calls with `{% if execute %}`. dbt parses macros twice: once to build the DAG (no execution) and once to run. Without the guard, the query fires during graph compilation when no connection may exist.

### dbt-utils: The Standard Macro Library

The [`dbt-utils`](https://hub.getdbt.com/dbt-labs/dbt_utils/latest/) package ships dozens of production-ready macros. Most-used:

| Macro | Purpose |
|---|---|
| `dbt_utils.generate_surrogate_key(['col1','col2'])` | MD5-based surrogate key from multiple columns |
| `dbt_utils.star(from=ref('model'), except=['col'])` | `SELECT *` minus excluded columns |
| `dbt_utils.pivot(column, values, agg, then_value)` | Dynamic pivot without hardcoding column names |
| `dbt_utils.date_spine(...)` | Generate a complete date-range table |
| `dbt_utils.get_column_values(table, column)` | Fetch distinct values for dynamic SQL |

Install via `packages.yml`:

```yaml
packages:
  - package: dbt-labs/dbt_utils
    version: [">=1.0.0", "<2.0.0"]
```

Then run `dbt deps`.

### Best Practices

- **Namespace macros by project** — prefix project-specific macros (`myproject_cents_to_dollars`) to avoid collisions with package macros.
- **Use `is_incremental()` in macros** rather than duplicating incremental filter logic across models.
- **Avoid business logic in macros** — macros are for structural/SQL-generation patterns. Domain logic belongs in models where it's visible in the DAG.
- **Test compiled output** — run `dbt compile` and inspect `target/compiled/` to verify your macro generates what you expect before running.
