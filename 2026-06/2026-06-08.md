# Daily Notes - 2026-06-08

## 00:06 - Apache Iceberg: The Open Table Format Transforming Data Lakes

Apache Iceberg has emerged as the de facto standard open table format for large-scale analytical workloads, solving long-standing pain points in data lake architectures that formats like Hive tables couldn't address.

### The Core Problem Iceberg Solves

Traditional Hive-style partitioned tables store metadata in a central metastore, causing bottlenecks when engines need to plan queries over thousands of partitions. Listing files in S3 or HDFS becomes slow and expensive. Concurrent writes often produce inconsistent reads, and schema evolution is fragile — renaming or reordering columns can silently break downstream pipelines.

### How Iceberg Works

Iceberg tracks table state through a tree of metadata files:

1. **Catalog** → points to the current metadata file
2. **Metadata file** (JSON) → describes schema, partition spec, and current snapshot
3. **Manifest list** → lists all manifest files for a snapshot
4. **Manifest files** → list individual data files with per-file stats (min/max values, null counts)

This layered approach enables **predicate pushdown at the manifest level** — query engines skip entire manifest files without reading a single data file, making partition pruning orders of magnitude faster than listing directories.

### Key Practical Benefits

**Hidden partitioning**: Iceberg handles partition transforms (bucket, truncate, year/month/day) internally. Writers don't pass partition values; Iceberg derives them from column values. Changing partition strategy doesn't break existing queries — Iceberg can read old and new partitions simultaneously.

**ACID transactions via optimistic concurrency**: Each commit writes a new snapshot atomically. Readers always see a consistent snapshot, and conflicting concurrent writes are detected and retried rather than silently corrupting data.

**Schema evolution without rewrites**: Adding, dropping, renaming, or reordering columns is a metadata-only operation. Iceberg tracks columns by ID rather than name, so renaming a column doesn't break readers that reference it by its old name.

**Time travel and rollback**: Every snapshot is retained (until expired via maintenance procedures), enabling `SELECT * FROM table FOR TIMESTAMP AS OF '2026-01-01'` style queries and one-command rollback to any prior state.

### When to Use It

Iceberg shines in multi-engine environments (Spark for ingestion, Trino/Athena for ad-hoc queries, Flink for streaming upserts) where consistent reads across engines and safe concurrent writes are non-negotiable. For small, single-engine datasets with infrequent writes, the metadata overhead may not justify the complexity.
