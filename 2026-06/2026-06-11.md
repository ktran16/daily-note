# Daily Notes - 2026-06-11

## 00:06 - Apache Iceberg: The Open Table Format Reshaping Data Lakes

Modern data lakes traditionally struggled with problems that relational databases long ago solved: you couldn't reliably update or delete rows, concurrent writes caused corruption, and querying historical data required custom tooling. Apache Iceberg was created at Netflix to fix exactly these pain points, and it has since become the dominant open table format in the industry.

### What Iceberg Actually Does

Iceberg sits as a metadata layer on top of ordinary Parquet, ORC, or Avro files in object storage (S3, GCS, ADLS). Instead of treating a table as a flat directory of files, Iceberg maintains a tree of metadata:

- **Snapshot** — a point-in-time view of the table, created on every write
- **Manifest list** — lists all manifest files belonging to a snapshot
- **Manifest files** — track data files along with column-level statistics (min/max, null counts)
- **Data files** — the actual Parquet/ORC files holding rows

This hierarchy makes ACID transactions possible without a centralized lock manager, because a commit is just an atomic pointer swap to a new snapshot.

### Key Capabilities

**Time travel & rollback** — because every write creates a new snapshot, you can query any prior state with `AS OF TIMESTAMP` or `AS OF VERSION` syntax. Rolling back a bad migration is a metadata operation, not a full rewrite.

**Schema evolution** — Iceberg tracks columns by ID rather than position, so adding, dropping, or renaming a column doesn't break existing data files or downstream queries.

**Hidden partitioning** — partition transforms (bucket, truncate, year/month/day) are defined in table metadata, not baked into file paths. Queries no longer need to filter on partition columns explicitly, eliminating a major foot-gun of Hive-style tables.

**Row-level deletes** — deletion vectors or equality-delete files let you mark rows as deleted without rewriting entire data files, enabling efficient CDC and GDPR-compliant record removal.

### Why It Matters for Data Engineers

Iceberg is now supported natively by Spark, Flink, Trino, Snowflake, BigQuery, and Athena. Adopting it means your data lake gains warehouse-grade reliability while remaining fully open and vendor-neutral. Pair it with a REST catalog (like Polaris or Unity Catalog) and you get cross-engine governance with no single-vendor lock-in — the closest the industry has come to a truly interoperable lakehouse standard.
