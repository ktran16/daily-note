## 00:03 - dbt Semantic Layer & MetricFlow: Defining Metrics Once, Querying Everywhere

The **dbt Semantic Layer** (powered by MetricFlow) lets you define business metrics in YAML once and expose them consistently across every BI tool, notebook, and API — eliminating the "metric sprawl" that plagues data teams when each dashboard redefines `revenue` slightly differently.

### Core Concepts

| Concept | Role |
|---|---|
| **Semantic Model** | Maps a dbt model to reusable entities, dimensions, and measures |
| **Measure** | An aggregatable column (`SUM`, `COUNT`, `AVG`, etc.) |
| **Dimension** | A groupable attribute (time, category, ID) |
| **Metric** | A named business calculation built from measures |

### Defining a Semantic Model

```yaml
# models/marts/semantic_models/orders.yml
semantic_models:
  - name: orders
    model: ref('fct_orders')
    entities:
      - name: order
        type: primary
        expr: order_id
      - name: customer
        type: foreign
        expr: customer_id
    measures:
      - name: total_revenue
        agg: sum
        expr: amount
      - name: order_count
        agg: count_distinct
        expr: order_id
    dimensions:
      - name: status
        type: categorical
      - name: ordered_at
        type: time
        type_params:
          time_granularity: day
```

### Defining Metrics on Top

```yaml
metrics:
  - name: revenue
    label: "Total Revenue"
    type: simple
    type_params:
      measure: total_revenue

  - name: revenue_per_order
    label: "Revenue per Order"
    type: ratio
    type_params:
      numerator: total_revenue
      denominator: order_count
```

### Querying via the CLI

```bash
# Query revenue by month
dbt sl query \
  --metrics revenue \
  --group-by ordered_at__month \
  --limit 12

# Slice by dimension
dbt sl query \
  --metrics revenue,revenue_per_order \
  --group-by ordered_at__week,status
```

### Why It Matters

1. **Single source of truth** — change the `revenue` definition in one YAML file and every downstream consumer gets the update automatically.
2. **BI tool agnostic** — Tableau, Looker, Hex, Mode, and custom APIs all query the same governed layer via the Semantic Layer API.
3. **Reduced join errors** — MetricFlow handles joins between semantic models using declared entity relationships, not ad-hoc SQL.
4. **Cumulative & derived metrics** — You can define `cumulative` (running totals) and `derived` (formula across other metrics) metric types without duplicating SQL.

### Practical Tips

- Keep semantic models **one-to-one with mart models** — don't attach them to staging or intermediate layers.
- Use `fill_nulls_with: 0` on measures to avoid gaps in time-series queries.
- The Semantic Layer requires **dbt Cloud** with the Discovery API for production use; local `dbt sl query` works for development against any adapter.

## 05:02 - dbt Model Contracts: Enforcing Schema Guarantees

### What Are Model Contracts?

Introduced in **dbt v1.5**, model contracts let you declare a formal, enforceable schema for your dbt models. Instead of trusting that downstream consumers get the right columns and types, a contract forces dbt to validate (and in some adapters, enforce) the structure at build time — before any query runs against the model.

Contracts are particularly valuable for **public models** that are shared across teams or consumed by BI tools and APIs.

### Defining a Contract

Add `contract: {enforced: true}` in your model's YAML config, then enumerate every column with its `data_type`:

```yaml
# models/marts/finance/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key   # informational on most adapters
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric(18,2)
      - name: status
        data_type: varchar(50)
```

### What dbt Enforces

| Check | Enforced at compile time | Enforced at run time |
|---|---|---|
| Column names match | ✅ | ✅ |
| Column order matches | ✅ (some adapters) | — |
| Data types match | — | ✅ (adapter-level DDL) |
| NOT NULL constraint | — | ✅ (adapter-level DDL) |

When a contract is violated (e.g., your SQL `SELECT` returns a `varchar` where `date` is declared), dbt raises a **contract error** and halts the run rather than silently writing bad data.

### Constraints Supported

```yaml
constraints:
  - type: not_null
  - type: unique
  - type: primary_key
  - type: foreign_key
    to: ref('dim_customers')
    to_columns: [customer_id]
  - type: check
    expression: "total_amount >= 0"
```

> **Adapter support varies.** Snowflake and Postgres support most constraint types natively; BigQuery enforces `not_null` but treats others as informational.

### Combining Contracts with Model Versions

Contracts pair naturally with **model versioning** (`versions:` block in YAML). When you bump a version, the old contract remains stable for existing consumers while the new version introduces breaking schema changes safely:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1   # legacy shape still honoured by old consumers
      - v: 2   # new shape with contract enforced
        config:
          contract:
            enforced: true
```

### Best Practices

- Enable contracts on **mart-layer and public models only** — staging and intermediate models change too frequently.
- Treat the contract YAML as the **source of truth** for downstream documentation; it replaces separate schema docs for those columns.
- Run `dbt compile` in CI to catch contract mismatches before they reach production.
- Use `--select state:modified+` in CI so contract checks only re-run on changed models, keeping build times short.
