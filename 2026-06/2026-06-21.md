## 00:03 - dbt Semantic Layer & MetricFlow: Defining Metrics Once, Querying Everywhere

The **dbt Semantic Layer** (powered by MetricFlow) lets you define business metrics in YAML once and expose them consistently across every BI tool, notebook, and API — eliminating the "metric sprawl" that plagues data teams when each dashboard redefines `revenue` slightly differently.

### Core Concepts

| Concept | Role |
|---|---|
| **Semantic Model** | Maps a dbt model to reusable entities, dimensions, and measures |
| **Measure** | An aggregatable column (`SUM`, `COUNT`, `AVG`, etc.) |
| **Dimension** | A groupable attribute (time, category, ID) |
| **Metric** | A named business calculation built from measures |

### Defining a Semantic Model

```yaml
# models/marts/semantic_models/orders.yml
semantic_models:
  - name: orders
    model: ref('fct_orders')
    entities:
      - name: order
        type: primary
        expr: order_id
      - name: customer
        type: foreign
        expr: customer_id
    measures:
      - name: total_revenue
        agg: sum
        expr: amount
      - name: order_count
        agg: count_distinct
        expr: order_id
    dimensions:
      - name: status
        type: categorical
      - name: ordered_at
        type: time
        type_params:
          time_granularity: day
```

### Defining Metrics on Top

```yaml
metrics:
  - name: revenue
    label: "Total Revenue"
    type: simple
    type_params:
      measure: total_revenue

  - name: revenue_per_order
    label: "Revenue per Order"
    type: ratio
    type_params:
      numerator: total_revenue
      denominator: order_count
```

### Querying via the CLI

```bash
# Query revenue by month
dbt sl query \
  --metrics revenue \
  --group-by ordered_at__month \
  --limit 12

# Slice by dimension
dbt sl query \
  --metrics revenue,revenue_per_order \
  --group-by ordered_at__week,status
```

### Why It Matters

1. **Single source of truth** — change the `revenue` definition in one YAML file and every downstream consumer gets the update automatically.
2. **BI tool agnostic** — Tableau, Looker, Hex, Mode, and custom APIs all query the same governed layer via the Semantic Layer API.
3. **Reduced join errors** — MetricFlow handles joins between semantic models using declared entity relationships, not ad-hoc SQL.
4. **Cumulative & derived metrics** — You can define `cumulative` (running totals) and `derived` (formula across other metrics) metric types without duplicating SQL.

### Practical Tips

- Keep semantic models **one-to-one with mart models** — don't attach them to staging or intermediate layers.
- Use `fill_nulls_with: 0` on measures to avoid gaps in time-series queries.
- The Semantic Layer requires **dbt Cloud** with the Discovery API for production use; local `dbt sl query` works for development against any adapter.

## 05:02 - dbt Model Contracts: Enforcing Schema Guarantees

### What Are Model Contracts?

Introduced in **dbt v1.5**, model contracts let you declare a formal, enforceable schema for your dbt models. Instead of trusting that downstream consumers get the right columns and types, a contract forces dbt to validate (and in some adapters, enforce) the structure at build time — before any query runs against the model.

Contracts are particularly valuable for **public models** that are shared across teams or consumed by BI tools and APIs.

### Defining a Contract

Add `contract: {enforced: true}` in your model's YAML config, then enumerate every column with its `data_type`:

```yaml
# models/marts/finance/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key   # informational on most adapters
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric(18,2)
      - name: status
        data_type: varchar(50)
```

### What dbt Enforces

| Check | Enforced at compile time | Enforced at run time |
|---|---|---|
| Column names match | ✅ | ✅ |
| Column order matches | ✅ (some adapters) | — |
| Data types match | — | ✅ (adapter-level DDL) |
| NOT NULL constraint | — | ✅ (adapter-level DDL) |

When a contract is violated (e.g., your SQL `SELECT` returns a `varchar` where `date` is declared), dbt raises a **contract error** and halts the run rather than silently writing bad data.

### Constraints Supported

```yaml
constraints:
  - type: not_null
  - type: unique
  - type: primary_key
  - type: foreign_key
    to: ref('dim_customers')
    to_columns: [customer_id]
  - type: check
    expression: "total_amount >= 0"
```

> **Adapter support varies.** Snowflake and Postgres support most constraint types natively; BigQuery enforces `not_null` but treats others as informational.

### Combining Contracts with Model Versions

Contracts pair naturally with **model versioning** (`versions:` block in YAML). When you bump a version, the old contract remains stable for existing consumers while the new version introduces breaking schema changes safely:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1   # legacy shape still honoured by old consumers
      - v: 2   # new shape with contract enforced
        config:
          contract:
            enforced: true
```

### Best Practices

- Enable contracts on **mart-layer and public models only** — staging and intermediate models change too frequently.
- Treat the contract YAML as the **source of truth** for downstream documentation; it replaces separate schema docs for those columns.
- Run `dbt compile` in CI to catch contract mismatches before they reach production.
- Use `--select state:modified+` in CI so contract checks only re-run on changed models, keeping build times short.

## 10:02 - dbt Project Structure & DAG Layering: Staging, Intermediate, and Mart Models

### The Three-Layer Pattern

A well-structured dbt project organizes models into three distinct layers, each with a clear responsibility. This pattern (popularized by dbt Labs) prevents the "spaghetti DAG" anti-pattern where raw source joins bleed into BI-facing models.

```
models/
├── staging/          # 1:1 with source tables — rename, cast, light cleaning only
│   ├── stripe/
│   │   ├── stg_stripe__payments.sql
│   │   └── _stripe__sources.yml
│   └── shopify/
│       ├── stg_shopify__orders.sql
│       └── _shopify__sources.yml
├── intermediate/     # Business logic assembly — joins, pivots, aggregations
│   ├── int_orders_with_payments.sql
│   └── int_customer_lifetime_value.sql
└── marts/            # BI-facing, stable, team-owned
    ├── finance/
    │   └── fct_orders.sql
    └── marketing/
        └── dim_customers.sql
```

### Layer Responsibilities

| Layer | Materialization | Naming | What It Does |
|---|---|---|---|
| **Staging** | `view` | `stg_<source>__<entity>` | Rename columns, cast types, basic deduplication |
| **Intermediate** | `ephemeral` or `view` | `int_<description>` | Join staging models, apply business logic |
| **Mart** | `table` or `incremental` | `fct_*` / `dim_*` | Deliver polished, BI-ready grain |

### Staging: The Raw-to-Consistent Bridge

Staging models are the **only place** that references `source()`. They enforce consistent column naming and data types so every downstream model operates on clean inputs.

```sql
-- models/staging/stripe/stg_stripe__payments.sql
with source as (
    select * from {{ source('stripe', 'payments') }}
),

renamed as (
    select
        id                          as payment_id,
        customer_id,
        amount / 100.0              as amount,           -- cents → dollars
        currency,
        created::timestamp          as created_at,
        status
    from source
    where _fivetran_deleted = false
)

select * from renamed
```

### Intermediate: Assembly Zone

Intermediate models exist to keep mart models readable. If a mart needs a complex join or business calculation, extract it here.

```sql
-- models/intermediate/int_orders_with_payments.sql
with orders as (
    select * from {{ ref('stg_shopify__orders') }}
),

payments as (
    select * from {{ ref('stg_stripe__payments') }}
),

joined as (
    select
        o.order_id,
        o.customer_id,
        o.created_at as order_date,
        p.amount     as paid_amount,
        p.status     as payment_status
    from orders o
    left join payments p using (order_id)
)

select * from joined
```

### Mart: The Contract with Consumers

Marts are wide, denormalized, and optimized for querying. Facts capture events at a specific grain; dimensions describe entities.

```sql
-- models/marts/finance/fct_orders.sql
{{
  config(
    materialized='incremental',
    unique_key='order_id',
    cluster_by=['order_date']
  )
}}

with orders as (
    select * from {{ ref('int_orders_with_payments') }}
    {% if is_incremental() %}
    where order_date >= (select max(order_date) from {{ this }})
    {% endif %}
)

select * from orders
```

### Configuration by Layer via `dbt_project.yml`

Apply materializations and tags to entire folders — no need to set configs in each file:

```yaml
# dbt_project.yml
models:
  my_project:
    staging:
      +materialized: view
      +tags: ["staging"]
    intermediate:
      +materialized: ephemeral
    marts:
      +materialized: table
      finance:
        +tags: ["finance", "pii"]
```

### Key Rules to Follow

1. **Only staging models use `source()`** — all other layers use `ref()`.
2. **Marts never join raw sources directly** — always go through staging.
3. **Name collisions are a code smell** — if two marts need the same complex logic, extract it to `intermediate/`.
4. **Keep staging 1:1 with source tables** — resist the urge to join in staging; that's intermediate's job.

## 15:02 - dbt Testing: Generic Tests, Singular Tests & Test Severity

### Why Testing Matters in dbt

dbt tests are SQL queries that return rows when something is **wrong**. Zero rows = pass; any rows = fail. They run via `dbt test` and integrate cleanly into CI/CD so data quality problems are caught before bad data reaches dashboards.

### Generic Tests (Built-in + Custom)

Generic tests are reusable and applied in YAML. dbt ships with four out of the box:

```yaml
# models/staging/stripe/_stripe__models.yml
models:
  - name: stg_stripe__payments
    columns:
      - name: payment_id
        tests:
          - unique
          - not_null
      - name: status
        tests:
          - accepted_values:
              values: ['succeeded', 'pending', 'failed', 'refunded']
      - name: customer_id
        tests:
          - relationships:
              to: ref('stg_stripe__customers')
              field: customer_id
```

| Built-in Test | Checks |
|---|---|
| `unique` | No duplicate values in the column |
| `not_null` | No `NULL` values |
| `accepted_values` | All values are in an allowed set |
| `relationships` | Every FK value exists in the referenced model |

### Singular Tests

Singular tests are plain SQL files in `tests/`. If the query returns any rows, the test fails. Use them for business-logic assertions that don't generalise across models:

```sql
-- tests/assert_payment_amounts_positive.sql
-- Fail if any completed payment has a non-positive amount
select
    payment_id,
    amount
from {{ ref('stg_stripe__payments') }}
where status = 'succeeded'
  and amount <= 0
```

```sql
-- tests/assert_orders_have_payments.sql
-- Every shipped order must have a succeeded payment
select o.order_id
from {{ ref('fct_orders') }} o
left join {{ ref('stg_stripe__payments') }} p
    on o.order_id = p.order_id
   and p.status = 'succeeded'
where o.status = 'shipped'
  and p.payment_id is null
```

### Custom Generic Tests (Reusable Macros)

Write a macro in `macros/` and reference it by name in YAML — exactly like the built-ins:

```sql
-- macros/test_not_negative.sql
{% test not_negative(model, column_name) %}
select {{ column_name }}
from {{ model }}
where {{ column_name }} < 0
{% endtest %}
```

```yaml
# Apply it anywhere
- name: total_amount
  tests:
    - not_negative
```

### Test Severity & Thresholds

Not every failure warrants stopping the pipeline. Use `severity` and `error_if` / `warn_if` to tune:

```yaml
- name: status
  tests:
    - accepted_values:
        values: ['succeeded', 'pending', 'failed']
        severity: warn          # log a warning, don't fail the run
        warn_if: ">= 1"
        error_if: ">= 100"     # hard fail only if 100+ bad rows
```

### Storing Test Failures for Inspection

Redirect failing rows into a table so you can query them later:

```yaml
# dbt_project.yml
tests:
  my_project:
    +store_failures: true
    +schema: test_failures   # writes to <database>.test_failures.<test_name>
```

Or per-test:
```yaml
- name: payment_id
  tests:
    - unique:
        store_failures: true
```

### dbt-utils Extra Tests

The `dbt-utils` package ships additional generic tests that cover common patterns:

```yaml
# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0
```

```yaml
- name: fct_orders
  tests:
    - dbt_utils.unique_combination_of_columns:
        combination_of_columns: [order_id, line_item_id]
    - dbt_utils.expression_is_true:
        expression: "total_amount >= 0"
    - dbt_utils.recency:
        datepart: day
        field: order_date
        interval: 3             # fail if no rows in last 3 days
```

### Best Practices

1. **Test sources as well as models** — add `tests:` blocks inside `sources:` YAML so raw data quality issues surface before they propagate.
2. **Run tests in CI with `--select state:modified+`** — only re-test models touched by the current PR.
3. **Prefer `warn` severity in development**, `error` in production pipelines so iterating locally isn't blocked by known data quirks.
4. **Name singular tests after what they assert** (`assert_*`), not what they query — makes failures self-documenting in the run log.

## 19:02 - dbt Macros & Jinja: Writing Reusable SQL Logic

### Why Macros?

dbt models are compiled Jinja templates, meaning you can use Python-like control flow directly inside SQL. **Macros** take this further — they are named, callable functions defined in `.sql` files under `macros/`. Use them to eliminate copy-paste logic, enforce consistent patterns, and build utilities your whole project shares.

### Basic Macro Anatomy

```sql
-- macros/cents_to_dollars.sql
{% macro cents_to_dollars(column_name, precision=2) %}
    round({{ column_name }} / 100.0, {{ precision }})
{% endmacro %}
```

Call it anywhere in a model:

```sql
-- models/staging/stripe/stg_stripe__payments.sql
select
    payment_id,
    {{ cents_to_dollars('amount') }}        as amount,
    {{ cents_to_dollars('refund_amount', 4) }} as refund_amount
from {{ source('stripe', 'payments') }}
```

dbt compiles this to plain SQL before execution — no runtime overhead.

### Jinja Control Flow in Models

**`if` / `else`** — great for adapter-specific or environment-specific SQL:

```sql
-- models/marts/fct_events.sql
select
    event_id,
    user_id,
    {% if target.type == 'bigquery' %}
        TIMESTAMP_TRUNC(occurred_at, DAY)
    {% elif target.type == 'snowflake' %}
        DATE_TRUNC('day', occurred_at)
    {% else %}
        occurred_at::date
    {% endif %}                             as event_date
from {{ ref('stg_events') }}
```

**`for` loops** — generate repetitive SQL from a list:

```sql
{% set payment_methods = ['credit_card', 'paypal', 'bank_transfer'] %}

select
    order_id,
    {% for method in payment_methods %}
    sum(case when payment_method = '{{ method }}' then amount else 0 end)
        as {{ method }}_amount
    {%- if not loop.last %},{% endif %}
    {% endfor %}
from {{ ref('stg_stripe__payments') }}
group by 1
```

### The `run_query` Macro: Dynamic SQL

Execute a query at **compile time** and use the results inside another query — useful for dynamically building column lists:

```sql
{% macro get_active_segments() %}
    {% set query %}
        select distinct segment_name
        from {{ ref('dim_segments') }}
        where is_active = true
        order by 1
    {% endset %}

    {% set results = run_query(query) %}
    {% if execute %}
        {{ return(results.columns[0].values()) }}
    {% else %}
        {{ return([]) }}
    {% endif %}
{% endmacro %}
```

```sql
-- Use it in a model
{% set segments = get_active_segments() %}
select
    user_id,
    {% for seg in segments %}
    sum(case when segment = '{{ seg }}' then revenue else 0 end) as {{ seg }}_revenue
    {%- if not loop.last %},{% endif %}
    {% endfor %}
from {{ ref('fct_orders') }}
group by 1
```

### `execute` Flag: Compile vs. Run Time

dbt parses Jinja twice: once at **compile time** (building the DAG) and once at **run time** (executing SQL). The `execute` variable is `False` during compilation and `True` during execution. Always guard `run_query` calls with `{% if execute %}` to avoid errors during graph parsing.

### Macro Dispatch: Overriding Package Macros

The `dispatch` system lets you override macros from packages (like `dbt-utils`) with adapter-specific or project-specific versions:

```sql
-- macros/default__generate_surrogate_key.sql
-- Override dbt_utils surrogate key to always use SHA256 on Snowflake
{% macro snowflake__generate_surrogate_key(field_list) %}
    sha2(concat_ws('|', {% for field in field_list %}
        coalesce(cast({{ field }} as varchar), '')
        {%- if not loop.last %}, {% endif %}
    {% endfor %}), 256)
{% endmacro %}
```

dbt resolves macros in priority order: **project → adapter-specific package → default package** — so your override wins automatically.

### Useful Built-in Macros

| Macro | What It Does |
|---|---|
| `ref('model')` | Resolves to the relation and tracks DAG dependencies |
| `source('src', 'table')` | References a raw source table |
| `config(...)` | Sets model-level config from inside the SQL file |
| `this` | The current model's relation — used in incremental logic |
| `is_incremental()` | Returns `true` when the model runs in incremental mode |
| `log('msg', info=true)` | Prints to the dbt console during a run |
| `dbt_utils.star(ref('model'))` | Selects all columns except specified exclusions |

### Best Practices

1. **Keep macros small and single-purpose** — a macro that does one thing is easy to test; a 200-line macro is a maintenance trap.
2. **Document macros with `arguments:` in a YAML schema** — dbt 1.7+ supports macro documentation alongside model docs.
3. **Avoid side effects at compile time** — don't write DML inside a macro body that runs during `dbt compile`.
4. **Prefix project macros** to avoid collisions with packages: `my_project__clean_string()` vs `clean_string()`.
5. **Test macro output** with `dbt run-operation my_macro --args '{arg: value}'` to inspect compiled SQL before embedding in models.
