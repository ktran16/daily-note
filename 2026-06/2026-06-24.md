## 00:04 - dbt Docs: Generating a Self-Updating Data Catalog from Your Project

dbt's built-in documentation system turns your YAML descriptions into a browsable, searchable data catalog — complete with an auto-generated lineage graph. It costs almost nothing to maintain because it lives right next to the SQL it describes.

### Two commands

```bash
dbt docs generate   # compiles catalog.json + manifest.json into ./target/
dbt docs serve      # starts a local server at localhost:8080
```

In dbt Cloud the catalog is generated automatically on every job run and published to a hosted URL with no extra setup.

### Writing good descriptions

Descriptions belong in `schema.yml` files. They support plain text or Markdown:

```yaml
models:
  - name: fct_orders
    description: |
      One row per confirmed order. Excludes test orders (`is_test = true`)
      and orders cancelled within 30 minutes of placement.
    columns:
      - name: order_id
        description: Surrogate key — join target for `fct_order_items`.
      - name: status
        description: >
          Current fulfillment status. Values: `placed`, `shipped`, `delivered`,
          `returned`. Source of truth is the `orders` raw table; downstream
          systems should read this field rather than querying source directly.
```

### `doc()` blocks for shared descriptions

Repeat yourself across many models? Extract prose into a `docs/` block file:

```sql
-- docs/descriptions.md
{% docs order_status %}
Current fulfillment status. Values: `placed`, `shipped`, `delivered`, `returned`.
{% enddocs %}
```

Then reference it anywhere:

```yaml
columns:
  - name: status
    description: "{{ doc('order_status') }}"
```

Changes to the `doc()` block propagate everywhere it's referenced automatically.

### What the catalog actually shows

- Column names, types, and descriptions for every model, source, seed, and snapshot
- Row counts and byte sizes (where the adapter supports it)
- Full DAG lineage — click any node to see its upstream sources and downstream consumers
- Test coverage per model

### Practical tips

- Document columns that are **non-obvious** first: surrogate keys, status enums, nullable flags. Skip `id bigint — primary key` noise.
- Add `meta` tags to flag ownership: `meta: {owner: analytics-eng, tier: gold}` — they appear in the catalog and are queryable via `dbt ls`.
- In CI, run `dbt docs generate` and diff `target/catalog.json` to catch undocumented columns before they reach production.

Treating the catalog as a living contract rather than an afterthought makes onboarding new analysts dramatically faster.

## 05:03 - dbt Unit Tests: Testing SQL Logic with Mocked Inputs

Introduced in **dbt 1.8**, unit tests let you assert that a model's SQL transformation produces the right output for a given set of inputs — without touching any real warehouse data. This closes a gap that generic tests couldn't fill: verifying business logic like case expressions, conditional joins, or complex date arithmetic.

### How they work

You define a unit test in a YAML file alongside your model. You provide mocked rows for each upstream dependency, specify the expected output rows, and dbt runs the assertion at compile time against a lightweight in-memory fixture.

```yaml
# models/schema.yml
unit_tests:
  - name: test_order_status_logic
    model: fct_orders
    given:
      - input: ref('stg_orders')
        rows:
          - {order_id: 1, raw_status: 'complete',  placed_at: '2024-01-10', shipped_at: '2024-01-11'}
          - {order_id: 2, raw_status: 'cancelled', placed_at: '2024-01-10', shipped_at: null}
          - {order_id: 3, raw_status: 'complete',  placed_at: '2024-01-10', shipped_at: null}
    expect:
      rows:
        - {order_id: 1, status: 'shipped'}
        - {order_id: 2, status: 'cancelled'}
        - {order_id: 3, status: 'pending'}   # complete but not yet shipped
```

Run them with:

```bash
dbt test --select fct_orders,test_type:unit
# or run all unit tests in the project
dbt test --select test_type:unit
```

### Overriding `now()` and other non-deterministic calls

Unit tests support a `overrides` block so you can freeze timestamps and variables:

```yaml
unit_tests:
  - name: test_is_stale_flag
    model: dim_products
    overrides:
      macros:
        dbt_date.now: "'2024-06-01 00:00:00'"
    given:
      - input: ref('stg_products')
        rows:
          - {product_id: 'A', updated_at: '2024-05-01'}  # 31 days ago → stale
          - {product_id: 'B', updated_at: '2024-05-31'}  # 1 day ago  → fresh
    expect:
      rows:
        - {product_id: 'A', is_stale: true}
        - {product_id: 'B', is_stale: false}
```

### When to write unit tests vs. generic tests

| Situation | Use |
|---|---|
| Check column is unique / not null | Generic test (`unique`, `not_null`) |
| Validate a CASE expression maps correctly | **Unit test** |
| Assert a join doesn't fan out | **Unit test** with controlled fixture |
| Verify freshness of source data | Source freshness check |
| Confirm referential integrity | Generic test (`relationships`) |

### Practical tips

- Keep fixture rows minimal — 3-5 rows that each exercise a different branch of logic. More rows increase maintenance without increasing confidence.
- Name tests after the rule being enforced (`test_cancelled_order_never_ships`), not the model (`test_fct_orders`). The name appears in CI output and becomes searchable documentation.
- Unit tests run before the model materializes, so they catch logic errors early and cheaply — ideal for dbt Slim CI pipelines where you want fast feedback on pull requests.

Unit tests are the missing piece between "my SQL compiles" and "my SQL does what I think it does."
