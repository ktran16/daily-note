## 00:04 - dbt Model Contracts: Enforcing Schema Guarantees in Production

### What Are Model Contracts?

Introduced in **dbt 1.5**, model contracts let you declare the schema a model *must* produce — column names, data types, and constraints — and have dbt enforce it at build time. If the model's output doesn't match the contract, the run fails before any downstream models consume bad data.

Without contracts, a stray `SELECT *` or a renamed column silently breaks downstream consumers. Contracts make schema drift a build error, not a silent data quality issue.

### Defining a Contract

Contracts live in the model's YAML file under `config: contract: enforced: true`:

```yaml
# models/marts/fct_orders.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: customer_id
        data_type: bigint
        constraints:
          - type: not_null
      - name: order_date
        data_type: date
      - name: total_amount
        data_type: numeric
      - name: status
        data_type: varchar
```

### How Enforcement Works

When `enforced: true`, dbt:
1. **Compares** the columns and types declared in YAML against what the SQL `SELECT` actually returns.
2. **Fails the run** if any declared column is missing or has an incompatible type.
3. **Injects constraints** (e.g., `NOT NULL`, `PRIMARY KEY`) into the DDL for warehouses that support them (Snowflake, Redshift, BigQuery, Databricks).

### Contracts + Versioned Models

Contracts are most powerful alongside **model versions**. When you need to make a breaking change to a public model, you bump the version instead of editing in place — giving consumers time to migrate:

```yaml
models:
  - name: fct_orders
    latest_version: 2
    versions:
      - v: 1        # kept for backwards compatibility
      - v: 2        # new contract with breaking column changes
        config:
          contract:
            enforced: true
```

Downstream models reference `ref('fct_orders', v=1)` or `ref('fct_orders', v=2)` explicitly, so nothing breaks silently.

### Supported Constraint Types

| Constraint | Snowflake | BigQuery | Redshift | Databricks |
|---|---|---|---|---|
| `not_null` | enforced | enforced | enforced | enforced |
| `primary_key` | informational | — | informational | informational |
| `unique` | informational | — | informational | — |
| `foreign_key` | informational | — | informational | — |
| `check` | informational | — | — | — |

"Informational" means dbt declares the constraint in DDL but the warehouse does not enforce it at write time — it still communicates intent to BI tools and query optimizers.

### Best Practices

- **Enable contracts on public, cross-team models first.** Internal staging models don't need this overhead.
- **Pair with `access: public`** in `dbt_project.yml` to signal that the model is a stable interface.
- **Use `data_type` consistently** with your warehouse's native type names (e.g., `bigint` not `int64` on Snowflake).
- **Don't over-constrain.** Declare columns that matter to consumers; you don't need every intermediate column listed.
- Pin contracts in CI so schema regressions fail in pull requests, not in production.

## 05:03 - dbt Snapshots: Tracking Slowly Changing Dimensions (Type 2 SCD)

### What Are Snapshots?

dbt **snapshots** capture the state of a mutable source table over time, automatically building a Type 2 Slowly Changing Dimension (SCD). Every time a row changes, dbt closes the old record (sets `dbt_valid_to`) and inserts a new one, giving you a full history of how data looked at any point in time.

This is ideal for tracking changes to things like customer status, order state, pricing tiers, or employee records.

### Snapshot File Structure

Snapshots live in the `snapshots/` directory and use a `{% snapshot %}` block:

```sql
-- snapshots/snp_customers.sql
{% snapshot snp_customers %}

{{
  config(
    target_schema='snapshots',
    unique_key='customer_id',
    strategy='timestamp',
    updated_at='updated_at'
  )
}}

SELECT
  customer_id,
  email,
  plan_tier,
  status,
  updated_at
FROM {{ source('app_db', 'customers') }}

{% endsnapshot %}
```

Run with: `dbt snapshot`

### Snapshot Strategies

**Timestamp strategy** — uses a column that tracks when the row was last modified. Most reliable when your source has a reliable `updated_at` field:

```yaml
strategy: timestamp
updated_at: updated_at
```

**Check strategy** — dbt computes a hash of specified columns and detects changes by comparing hashes. Use when no `updated_at` column exists:

```yaml
strategy: check
check_cols:
  - email
  - plan_tier
  - status
# or check_cols: all  (hashes every column)
```

### Metadata Columns dbt Adds

| Column | Description |
|---|---|
| `dbt_scd_id` | Unique surrogate key for each snapshot row |
| `dbt_updated_at` | When dbt last processed this row |
| `dbt_valid_from` | When this version of the record became active |
| `dbt_valid_to` | When it was superseded (`NULL` = current record) |

### Querying Snapshot History

```sql
-- Current state of all customers
SELECT * FROM snapshots.snp_customers
WHERE dbt_valid_to IS NULL;

-- What did customer 42 look like on 2025-01-15?
SELECT * FROM snapshots.snp_customers
WHERE customer_id = 42
  AND dbt_valid_from <= '2025-01-15'
  AND (dbt_valid_to > '2025-01-15' OR dbt_valid_to IS NULL);
```

### Best Practices

- **Snapshot raw/source data**, not transformed models — you want the unmodified history.
- **Run snapshots before other dbt models** in your DAG (`dbt snapshot && dbt run`) so downstream models always have fresh history.
- **Use timestamp strategy when possible** — it's more efficient than check (no full-table hash scan).
- **Add `invalidate_hard_deletes: true`** in config to close out records when source rows are deleted, rather than leaving them as perpetually "current".
- Keep snapshot targets in a **dedicated schema** (e.g., `snapshots`) to separate them from mart tables and make permissions management easier.

## 10:03 - dbt Packages & dbt-utils: Extending dbt with Reusable Macros

### What Are dbt Packages?

dbt **packages** are bundles of reusable macros, models, and tests that you install into your project — think of them like pip packages for your dbt project. The most widely used is **dbt-utils** from dbt Labs, which ships dozens of helper macros that would otherwise require you to rewrite the same boilerplate SQL across every project.

Packages are defined in `packages.yml` at the project root and installed with `dbt deps`.

### Installing Packages

```yaml
# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.2.0
  - package: calogica/dbt_expectations  # Great Expectations-style tests for dbt
    version: 0.10.3
  - package: dbt-labs/audit_helper      # Diff two models for validation during refactors
    version: 0.9.0
```

```bash
dbt deps   # installs all packages into dbt_packages/
```

Add `dbt_packages/` to `.gitignore` — treat it like `node_modules`.

### Key dbt-utils Macros

**`generate_surrogate_key`** — hashes a list of columns into a deterministic surrogate key, handling NULLs consistently across warehouses:

```sql
SELECT
  {{ dbt_utils.generate_surrogate_key(['order_id', 'line_item_id']) }} AS sk,
  order_id,
  line_item_id,
  product_id
FROM {{ ref('stg_order_items') }}
```

**`date_spine`** — generates a contiguous sequence of dates between two bounds, invaluable for filling gaps in time-series data:

```sql
WITH date_spine AS (
  {{ dbt_utils.date_spine(
      datepart="day",
      start_date="cast('2024-01-01' as date)",
      end_date="cast('2025-01-01' as date)"
  ) }}
)
SELECT date_day FROM date_spine
```

**`pivot`** — dynamically pivots a column of values into multiple columns without hardcoding each value:

```sql
SELECT
  customer_id,
  {{ dbt_utils.pivot('channel', dbt_utils.get_column_values(ref('stg_orders'), 'channel')) }}
FROM {{ ref('stg_orders') }}
GROUP BY 1
```

**`union_relations`** — unions a list of `ref()` or `source()` relations with schema-matching flexibility:

```sql
{{ dbt_utils.union_relations(
    relations=[ref('stg_orders_us'), ref('stg_orders_eu'), ref('stg_orders_apac')]
) }}
```

### Generic Tests from dbt-utils

dbt-utils extends dbt's built-in tests (`unique`, `not_null`) with more powerful checks:

```yaml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - dbt_utils.not_empty_string
          - dbt_utils.expression_is_true:
              expression: "total_amount >= 0"
      - name: order_date
        tests:
          - dbt_utils.not_null_proportion:
              at_least: 0.99   # 99% of rows must be non-null
```

### dbt_expectations for Data Quality

`calogica/dbt_expectations` ports Great Expectations checks into dbt YAML:

```yaml
- name: revenue
  tests:
    - dbt_expectations.expect_column_values_to_be_between:
        min_value: 0
        max_value: 1000000
    - dbt_expectations.expect_column_mean_to_be_between:
        min_value: 150
        max_value: 500
```

### Best Practices

- **Pin exact package versions** in `packages.yml` to prevent surprise upgrades from breaking your project.
- **Run `dbt deps` in CI** before `dbt build` — never commit the `dbt_packages/` folder.
- **Prefer `generate_surrogate_key` over manual `MD5`** — it handles NULL coalescing and cross-warehouse compatibility automatically.
- **Use `audit_helper`** when migrating or refactoring models: `compare_relations` diffs row counts and values between old and new, making it easy to verify a refactor produces identical results.
- Don't pull in packages for a single macro you could write in 5 lines — keep your dependency count lean.

## 16:17 - dbt Exposures: Documenting Downstream Consumers of Your Data Models

### What Are Exposures?

**Exposures** let you declare the downstream consumers of your dbt models — dashboards, ML models, applications, reverse ETL jobs — directly inside your dbt project. They appear in the lineage DAG, making it immediately clear which external tools depend on which models, and who owns them.

Without exposures, the impact of changing or deprecating a model is a guess. With exposures, dbt shows you exactly what breaks downstream before you make the change.

### Defining an Exposure

Exposures are declared in YAML files, typically in a `models/exposures/` folder or alongside the relevant mart models:

```yaml
# models/exposures/marketing_exposures.yml
version: 2

exposures:
  - name: marketing_revenue_dashboard
    label: "Marketing Revenue Dashboard"
    type: dashboard
    maturity: high          # low | medium | high
    url: https://looker.example.com/dashboards/42
    description: >
      Primary revenue dashboard used by Marketing leadership.
      Shows daily and weekly revenue by channel, campaign, and region.
    depends_on:
      - ref('fct_orders')
      - ref('fct_campaigns')
      - ref('dim_customers')
    owner:
      name: "Marketing Analytics"
      email: marketing-analytics@example.com

  - name: churn_prediction_model
    label: "Churn Prediction ML Model"
    type: ml
    maturity: medium
    description: >
      Scikit-learn logistic regression model trained weekly.
      Pulls features from the customer feature store.
    depends_on:
      - ref('fct_customer_activity')
      - ref('dim_customers')
    owner:
      name: "Data Science"
      email: data-science@example.com
```

### Supported Exposure Types

| Type | Use Case |
|---|---|
| `dashboard` | BI dashboards (Looker, Tableau, Metabase, etc.) |
| `notebook` | Jupyter/Python notebooks for analysis |
| `analysis` | Ad-hoc SQL analysis or reports |
| `ml` | Machine learning models consuming dbt features |
| `application` | Product features or APIs backed by dbt models |

### What Exposures Unlock

**Lineage visibility**: In `dbt docs`, exposures appear as terminal nodes in the DAG. You can trace from raw source → staging → mart → dashboard in one view.

**Impact analysis**: Before renaming or dropping a column, run `dbt ls --select fct_orders+` to see all models *and* exposures that depend on it. Any exposure in that list is a consumer you need to notify.

```bash
# List everything downstream of fct_orders, including exposures
dbt ls --select fct_orders+ --resource-type exposure

# Build fct_orders and everything it feeds, up to and including exposures
dbt build --select fct_orders+
```

**Freshness SLAs**: Pair exposures with source freshness checks and model-level `meta` tags to document the expected refresh cadence expected by each dashboard.

### Practical Pattern: Source-to-Dashboard Governance

```yaml
exposures:
  - name: finance_p_and_l_report
    type: dashboard
    maturity: high
    url: https://tableau.example.com/views/PnL
    description: Board-level P&L report. Must refresh before 7:00 AM UTC daily.
    depends_on:
      - ref('fct_revenue')
      - ref('fct_costs')
      - ref('dim_departments')
    meta:
      sla_refresh_by: "07:00 UTC"
      priority: critical
    owner:
      name: "Finance Analytics"
      email: finance-analytics@example.com
```

### Best Practices

- **Create an exposure for every production dashboard or ML model** — the governance value compounds as the project grows.
- **Use `maturity: high`** only for dashboards that are actively maintained and have an owner on-call.
- **Add the `url` field** wherever possible so engineers can open the consumer in one click from dbt docs.
- **Include exposures in CI**: `dbt build --select state:modified+` will catch downstream exposures affected by model changes.
- **Review exposures in code review** when deleting or renaming models — if an exposure's `depends_on` list contains the changed model, the PR needs a migration plan.

## 19:02 - dbt Incremental Models: Strategies, Filters, and Performance Patterns

### Why Incremental Models?

Full-refresh models rebuild their entire table on every `dbt run`. For large fact tables (billions of rows, multi-TB), that's prohibitively slow and expensive. **Incremental models** instead process only new or changed rows, appending or merging them into the existing table. The first run always does a full load; subsequent runs only touch rows that pass the `is_incremental()` filter.

### The Basic Pattern

```sql
-- models/fct_events.sql
{{
  config(
    materialized='incremental',
    unique_key='event_id',
    incremental_strategy='merge'
  )
}}

SELECT
  event_id,
  user_id,
  event_type,
  occurred_at,
  properties
FROM {{ source('app', 'events') }}

{% if is_incremental() %}
  -- Only process rows newer than the most recent row already in the table
  WHERE occurred_at > (SELECT MAX(occurred_at) FROM {{ this }})
{% endif %}
```

`{{ this }}` refers to the existing incremental table, letting you compare against its current state in the filter.

### Incremental Strategies Compared

| Strategy | How It Works | Best For |
|---|---|---|
| `append` | INSERT new rows only — no deduplication | Immutable event streams with no late-arriving updates |
| `merge` | UPSERT: update matching rows, insert new ones | Most fact tables; requires `unique_key` |
| `delete+insert` | DELETE matching rows then INSERT — avoids UPDATE locks | Warehouses with expensive UPDATEs (e.g., BigQuery) |
| `insert_overwrite` | Replace entire partitions atomically | Partitioned tables (BigQuery, Spark); very efficient for partition-aligned loads |

```yaml
# models/schema.yml — strategy configured inline or in dbt_project.yml
models:
  - name: fct_events
    config:
      materialized: incremental
      incremental_strategy: merge
      unique_key: event_id
      # For BigQuery partitioned tables:
      partition_by:
        field: occurred_at
        data_type: timestamp
        granularity: day
      cluster_by: [user_id, event_type]
```

### Handling Late-Arriving Data

A naive `WHERE occurred_at > MAX(occurred_at)` filter misses late-arriving rows — records that arrive after the run but with an `occurred_at` in the past. The fix is a **lookback window**:

```sql
{% if is_incremental() %}
  WHERE occurred_at >= (
    SELECT DATEADD('day', -3, MAX(occurred_at))  -- reprocess last 3 days
    FROM {{ this }}
  )
{% endif %}
```

Pair this with `incremental_strategy='merge'` and a `unique_key` so that rows reprocessed within the lookback window are upserted rather than duplicated.

### The `on_schema_change` Option

By default, dbt fails if your model's SELECT adds or removes columns after the table already exists. Control this with `on_schema_change`:

```python
config(
  materialized='incremental',
  on_schema_change='sync_all_columns'  # append new columns, remove dropped ones
  # Other options: 'fail' (default), 'ignore', 'append_new_columns'
)
```

Use `sync_all_columns` carefully — dropping a column from a large table can be an expensive DDL operation on some warehouses.

### Full Refresh Override

When you need to rebuild from scratch (schema change too complex, backfill required), run:

```bash
dbt run --select fct_events --full-refresh
```

This drops and recreates the table, bypassing `is_incremental()`. You can also gate on the `--full-refresh` flag inside the model:

```sql
{% if is_incremental() and not flags.FULL_REFRESH %}
  WHERE occurred_at > (SELECT MAX(occurred_at) FROM {{ this }})
{% endif %}
```

### Best Practices

- **Always test with `--full-refresh` first** on a dev schema to confirm your SQL is correct before relying on incremental logic.
- **Use `merge` + `unique_key` as the default** — it's the safest strategy for most fact tables and handles late arrivals when combined with a lookback window.
- **Prefer `insert_overwrite` on BigQuery** for large partitioned tables — it's significantly faster and cheaper than row-level MERGE operations.
- **Monitor incremental filter selectivity**: if the filter returns >30% of source rows, a full refresh might actually be cheaper.
- **Document the lookback window** in a comment or model description so future engineers know why 3 days (or whatever) was chosen.
