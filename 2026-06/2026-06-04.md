## 00:03 - dbt Incremental Models: Building Efficient Data Pipelines

### What Are dbt Incremental Models?

In data engineering, **dbt (data build tool)** is a transformation framework that lets analysts write modular SQL. One of its most powerful features is **incremental models** — a strategy for processing only new or changed data rather than rebuilding entire tables on every run.

### The Problem They Solve

Full-refresh models re-process every row in a source table each time they run. For tables with millions or billions of rows, this is prohibitively slow and expensive, particularly in cloud data warehouses billed by compute or bytes scanned (BigQuery, Snowflake, Redshift).

### How Incremental Models Work

An incremental model wraps standard SQL with a conditional filter:

```sql
{{ config(materialized='incremental', unique_key='event_id') }}

SELECT
    event_id,
    user_id,
    event_type,
    created_at
FROM {{ source('raw', 'events') }}

{% if is_incremental() %}
  WHERE created_at > (SELECT MAX(created_at) FROM {{ this }})
{% endif %}
```

On the **first run**, dbt ignores the `is_incremental()` block and loads the full dataset. On **subsequent runs**, only rows newer than the current maximum `created_at` are fetched and merged/inserted.

### Key Configuration Options

| Strategy | Behavior |
|---|---|
| `append` | Inserts new rows only; no deduplication |
| `merge` | Upserts rows matched on `unique_key` |
| `delete+insert` | Deletes matching rows then reinserts |
| `insert_overwrite` | Replaces entire partitions (BigQuery/Spark) |

### Practical Tips

- **Always define a `unique_key`** when using `merge` to avoid duplicate rows during reruns or backfills.
- **Use event timestamps** (e.g., `created_at`, `updated_at`) as your incremental filter — avoid surrogate keys which don't signal recency.
- **Run `dbt build --full-refresh`** periodically or after schema changes to resync the full table state.
- **Partition your target table** by the timestamp column to make incremental scans dramatically cheaper in BigQuery and Snowflake.

### When Not to Use Incremental Models

Avoid incremental models when source data is frequently updated in-place (late-arriving facts, corrections) without a reliable `updated_at` column — in those cases, a full-refresh or snapshot strategy is safer and avoids silent data drift.

Incremental models are a cornerstone of cost-efficient, production-grade data pipelines and are worth mastering early in any data engineering practice.

## 05:03 - RAG vs. Fine-Tuning: Choosing the Right LLM Adaptation Strategy

### The Core Question

When a base large language model (LLM) doesn't know your domain — your internal docs, product catalog, or proprietary data — you have two primary options: **Retrieval-Augmented Generation (RAG)** or **fine-tuning**. They solve different problems, and confusing them leads to expensive mistakes.

### Retrieval-Augmented Generation (RAG)

RAG keeps the base model frozen and injects relevant context at inference time. A retriever (typically a vector database like Pinecone, Weaviate, or pgvector) finds the top-k most relevant chunks from your knowledge base and stuffs them into the prompt alongside the user's query.

**When RAG wins:**
- Your knowledge base changes frequently (product prices, support tickets, live docs)
- You need source citations for auditability
- Your data volume is too large to fit in a fine-tuning dataset
- You want to avoid retraining costs when information updates

**RAG's weakness:** It fails when the model lacks the *reasoning pattern* or *output format* your task requires — it can't teach a model a new writing style or domain-specific logic through context alone.

### Fine-Tuning

Fine-tuning updates the model's weights on curated examples of inputs and desired outputs. The model internalizes patterns, tone, and domain-specific reasoning.

**When fine-tuning wins:**
- You need a consistent response style or format (structured JSON output, medical report templates)
- The task requires reasoning the base model consistently gets wrong
- Latency matters and you can't afford large context windows at inference time
- Your training data is stable and unlikely to change often

**Fine-tuning's weakness:** It can't inject facts the training data didn't cover. A fine-tuned model will still hallucinate on questions outside its training distribution.

### The Practical Heuristic

| Need | Solution |
|---|---|
| Up-to-date facts | RAG |
| Consistent format / style | Fine-tuning |
| Both | RAG + fine-tuning together |
| Neither helps | Rethink the task decomposition |

Start with RAG — it's faster to iterate and cheaper to maintain. Graduate to fine-tuning only once you've confirmed the bottleneck is model behavior, not missing context.
