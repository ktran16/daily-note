## 00:04 - dbt: The Analytics Engineering Tool That Brought Software Practices to SQL

dbt (data build tool) has fundamentally changed how data teams write, test, and deploy SQL-based transformations. Before dbt, analytics code lived in ad-hoc scripts, BI tool proprietary logic, or stored procedures with no version control, no testing, and no documentation. dbt treats SQL transformations as software ‚Äî giving them the same rigor as application code.

### What dbt Actually Does

dbt sits in the **T** of ELT (Extract, Load, Transform). It does not move raw data; your ingestion tools (Fivetran, Airbyte, custom pipelines) load data into your warehouse first. dbt then defines a DAG of SQL transformations that build clean, tested, documented models on top of that raw data.

Each dbt **model** is a plain `.sql` file containing a `SELECT` statement. dbt wraps that `SELECT` in a `CREATE TABLE AS` or `CREATE VIEW AS` statement and executes it in your warehouse (Snowflake, BigQuery, Redshift, DuckDB, and others). Your job is to write the transformation logic; dbt handles the DDL.

```sql
-- models/marts/finance/fct_orders.sql
with orders as (
    select * from {{ ref('stg_orders') }}
),
customers as (
    select * from {{ ref('stg_customers') }}
)
select
    o.order_id,
    o.ordered_at,
    c.customer_name,
    o.total_amount
from orders o
join customers c on o.customer_id = c.customer_id
```

The `{{ ref('stg_orders') }}` macro is the key abstraction ‚Äî it resolves to the correct schema-qualified table name in any environment and automatically builds the dependency graph.

### The Layered Architecture Convention

The most widely adopted dbt project structure uses three logical layers:

| Layer | Prefix | Purpose |
|---|---|---|
| Staging | `stg_` | 1-to-1 with source tables; rename, cast, light cleaning only |
| Intermediate | `int_` | Business logic joins and aggregations; not exposed to end users |
| Marts | `fct_`, `dim_` | Fact and dimension tables for consumption by BI tools |

This separation keeps raw source logic isolated from business logic, making refactors safe and auditable.

### Testing Is Built In

dbt ships with four generic tests out of the box: `not_null`, `unique`, `accepted_values`, and `relationships` (referential integrity). You declare them in a YAML file alongside your models:

```yaml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - not_null
          - unique
      - name: status
        tests:
          - accepted_values:
              values: ['placed', 'shipped', 'returned', 'cancelled']
```

Running `dbt test` executes these as SQL `SELECT` statements against your warehouse. Any rows returned indicate a failure. The dbt-expectations and dbt-utils packages extend this with dozens of additional tests (distribution checks, regex patterns, freshness assertions).

### Materializations: Controlling How Models Land in the Warehouse

dbt gives you four core materializations:

- **`view`** ‚Äî Cheapest; no data stored. Best for simple staging models read infrequently.
- **`table`** ‚Äî Full rebuild on every `dbt run`. Best for marts consumed by BI with expensive upstream joins.
- **`incremental`** ‚Äî On the first run, builds a full table. On subsequent runs, only processes new or updated rows. Critical for fact tables with billions of rows.
- **`ephemeral`** ‚Äî A CTE injected into downstream models; never materialized in the warehouse. Good for intermediate logic you don't need to query directly.

The incremental pattern is where most complexity lives:

```sql
{{ config(materialized='incremental', unique_key='order_id') }}

select * from {{ ref('stg_orders') }}
{% if is_incremental() %}
  where ordered_at > (select max(ordered_at) from {{ this }})
{% endif %}
```

### Why It Matters for Data Teams

dbt's real contribution is cultural as much as technical. By making SQL transformations version-controlled, testable, and documentable, it aligns data teams with the engineering practices that make software systems reliable. `dbt docs generate` builds a browsable data catalog from your models and YAML descriptions ‚Äî including a rendered lineage graph showing exactly which models depend on which sources. For data quality and organizational trust in analytics, that lineage is often more valuable than any individual transformation.

## 05:03 - dbt Snapshots: Capturing Slowly Changing Dimensions

Most dbt models answer "what does this row look like *right now*?" Snapshots answer a different question: "what did this row look like *over time*?" They solve the classic slowly changing dimension (SCD Type 2) problem ‚Äî tracking history for mutable source records like a `customers` table where `email` or `subscription_tier` can change, but your warehouse only ever sees the latest state on each extract.

### How It Works

A snapshot is defined in a `.sql` file under the `snapshots/` directory, not `models/`. Instead of a plain `select`, it's wrapped in a `snapshot` block with a config specifying a strategy:

```sql
-- snapshots/customers_snapshot.sql
{% snapshot customers_snapshot %}

{{
    config(
        target_schema='snapshots',
        unique_key='customer_id',
        strategy='timestamp',
        updated_at='updated_at',
    )
}}

select * from {{ source('app', 'customers') }}

{% endsnapshot %}
```

Running `dbt snapshot` checks each row's `unique_key` against what's already stored. If the row is new, it inserts it. If an existing row's `updated_at` value has changed, dbt closes out the old record (sets `dbt_valid_to`) and inserts a new one with `dbt_valid_from` set to now ‚Äî classic SCD Type 2 versioning, fully automated.

### Two Strategies

- **`timestamp`** ‚Äî Compares a reliable `updated_at` column. Preferred when the source system maintains one accurately.
- **`check`** ‚Äî Compares a list of specific columns (or `check_cols='all'`) when no trustworthy timestamp exists:

```sql
{{
    config(
        strategy='check',
        check_cols=['email', 'subscription_tier'],
    )
}}
```

### Practical Notes

- Snapshots should run on a schedule (e.g., as a step before `dbt run` in your production job) ‚Äî if you skip a run, you permanently lose the ability to detect changes that happened in between.
- Querying historical state is simple: `select * from snapshots.customers_snapshot where customer_id = 123 and '2026-03-01' between dbt_valid_from and coalesce(dbt_valid_to, '9999-12-31')`.
- Don't snapshot append-only event tables ‚Äî they already have full history. Snapshots exist specifically for tables that get overwritten in place at the source.

## 10:04 - dbt Model Contracts: Enforcing Schema Stability

By default, a dbt model's output schema is whatever the `SELECT` statement happens to produce ‚Äî column names, order, and types can silently shift when someone edits the SQL. For models consumed by downstream teams, BI dashboards, or reverse-ETL jobs, an unannounced type change (e.g., `order_id` flipping from `integer` to `varchar`) can break consumers without dbt ever raising an error. **Model contracts**, introduced in dbt Core 1.5, close this gap by making the schema an explicit, enforced part of the model definition.

### Defining a Contract

You declare a contract in the model's YAML, specifying every column's name and data type:

```yaml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: int
        constraints:
          - type: not_null
          - type: primary_key
      - name: status
        data_type: varchar
      - name: total_amount
        data_type: numeric(18,2)
```

When `contract.enforced: true` is set, dbt generates the model's `CREATE TABLE` statement with an explicit DDL column list *before* running your `SELECT`, rather than inferring the schema from the query result. If your model's actual output doesn't match ‚Äî a missing column, wrong order, or incompatible type ‚Äî `dbt run` fails immediately at build time instead of silently shipping a schema drift downstream.

### Practical Notes

- Contracts only work with `table` and `incremental` materializations (not `view` or `ephemeral`), since those are the materializations where dbt controls DDL.
- Pair contracts with `constraints` (`not_null`, `primary_key`, `foreign_key`) on warehouses that support them (e.g., Snowflake, Postgres) to get real enforced referential integrity, not just documentation.
- Use contracts on stable, widely-consumed marts (`fct_`, `dim_` models) where breaking changes are costly ‚Äî they add friction, so they're not worth it for fast-iterating staging models.
- Combine with `dbt run --select state:modified+` in CI so any contract violation is caught on a pull request, before it ever reaches production.

## 15:04 - dbt Macros & Jinja: Writing DRY, Reusable SQL Logic

dbt models are SQL files, but they're compiled through the **Jinja** templating engine before being sent to the warehouse. This is what lets you write `{{ ref('stg_orders') }}` instead of hardcoded table names ‚Äî and it's also a full templating language you can use to eliminate repeated SQL patterns across your project via **macros**.

### Anatomy of a Macro

A macro is a parameterized chunk of SQL (or pure Jinja logic) defined once and called from any model:

```sql
-- macros/cents_to_dollars.sql
{% macro cents_to_dollars(column_name, decimal_places=2) %}
    round({{ column_name }} / 100.0, {{ decimal_places }})
{% endmacro %}
```

Used in a model:

```sql
select
    order_id,
    {{ cents_to_dollars('amount_cents') }} as amount_usd
from {{ ref('stg_orders') }}
```

At compile time, dbt substitutes the macro call with the rendered SQL ‚Äî run `dbt compile` and inspect `target/compiled/` to see exactly what gets sent to the warehouse. This is the single best debugging habit for anything Jinja-heavy.

### Where Macros Earn Their Keep

- **Repeated business logic**: a `pivot_status_codes()` macro used across five marts instead of copy-pasted `CASE WHEN` blocks.
- **Cross-database compatibility**: wrap warehouse-specific syntax (e.g., date truncation differs between Snowflake, BigQuery, and Postgres) behind one macro so models stay portable.
- **Dynamic column generation**: loop over a list to generate repetitive `SELECT` columns:

```sql
{% set payment_methods = ['credit_card', 'paypal', 'bank_transfer'] %}

select
    order_id,
    {% for method in payment_methods %}
    sum(case when payment_method = '{{ method }}' then amount end) as {{ method }}_total
    {%- if not loop.last %},{% endif %}
    {% endfor %}
from {{ ref('stg_payments') }}
group by 1
```

### Practical Notes

- Don't reach for a macro until you've copy-pasted the same SQL pattern three times ‚Äî premature abstraction makes models harder to read for the next analyst.
- `dbt-utils` (now partly folded into dbt Core as `dbt.*` cross-database macros) already solves common cases like `date_spine`, `pivot`, `surrogate_key`, and `star` ‚Äî check there before writing your own.
- Macros that generate SQL dynamically (loops, conditionals) are powerful but harder to debug; always check the compiled output when something breaks, since Jinja errors and SQL errors look identical in the CLI but have very different causes.
- Avoid using macros to bypass `ref()`/`source()` ‚Äî even inside a macro, always reference other models through these functions so dbt's dependency graph stays accurate.

## 19:03 - dbt Semantic Layer: Defining Metrics Once, Querying Them Everywhere

A recurring problem in analytics teams: "revenue" gets defined three different ways across three BI dashboards, because each analyst writes their own `SUM(amount)` with slightly different filters. The **dbt Semantic Layer** (powered by MetricFlow, merged into dbt Core after dbt Labs acquired Transform) fixes this by letting you define a metric's logic once, on top of your existing models, and have every consumer ‚Äî Hex, Tableau, a Python notebook, or the `dbt sl` CLI ‚Äî query the same governed definition instead of writing raw SQL.

### Semantic Models and Metrics

You build on top of normal dbt models by declaring a **semantic model** that maps columns to dimensions, entities, and measures:

```yaml
# models/marts/orders.yml
semantic_models:
  - name: orders
    model: ref('fct_orders')
    entities:
      - name: order_id
        type: primary
      - name: customer_id
        type: foreign
    dimensions:
      - name: ordered_at
        type: time
        type_params:
          time_granularity: day
      - name: status
        type: categorical
    measures:
      - name: order_total
        agg: sum
        expr: total_amount

metrics:
  - name: revenue
    type: simple
    type_params:
      measure: order_total
    filter: "{{ Dimension('order__status') }} = 'completed'"
```

### Querying the Metric

Instead of every consumer re-deriving the `completed` filter, they query the metric directly:

```bash
dbt sl query --metrics revenue --group-by metric_time__month,customer__region
```

MetricFlow compiles this into the correct SQL against your warehouse, joining whatever dimension tables are needed automatically.

### Practical Notes

- Semantic models sit on top of marts (`fct_`/`dim_`), not staging models ‚Äî define metrics one layer up from where the joins already happen.
- `type: ratio` and `type: derived` metrics let you compose new metrics (e.g., `average_order_value = revenue / order_count`) without duplicating measure logic.
- The Semantic Layer requires dbt Cloud (or a self-hosted MetricFlow server) to actually serve queries to BI tools; dbt Core alone can validate and compile the YAML but won't run a query API.
- Start small: convert your 3-5 most-disputed metrics (revenue, active users, churn) before trying to migrate every dashboard calculation.
