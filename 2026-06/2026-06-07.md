## 00:06 - Vector Databases: The Memory Layer for Modern AI

As large language models and embedding-based applications have gone mainstream, vector databases have emerged as the essential infrastructure for storing and retrieving high-dimensional numerical representations — called *embeddings* — of text, images, audio, and more.

### What Is an Embedding?

An embedding is a dense numerical vector (e.g., 1,536 floats for OpenAI's `text-embedding-3-small`) that encodes semantic meaning. Two pieces of text with similar meaning will have vectors that are close together in high-dimensional space. This geometric property is what makes semantic search, recommendation engines, and retrieval-augmented generation (RAG) possible.

### Why Not a Regular Database?

Traditional databases excel at exact lookups (`WHERE id = 42`) and range queries on scalars. They cannot efficiently answer *approximate nearest-neighbor* (ANN) queries like "find the 10 vectors most similar to this query vector" over millions of rows. Vector databases are purpose-built for exactly this workload.

### Key Indexing Algorithms

| Algorithm | How It Works | Trade-off |
|---|---|---|
| **HNSW** (Hierarchical Navigable Small World) | Layered graph; navigates from coarse to fine levels | Fast queries, high memory usage |
| **IVF** (Inverted File Index) | Clusters vectors; searches only relevant clusters | Lower memory, slightly less accurate |
| **PQ** (Product Quantization) | Compresses vectors into compact codes | Very low memory, recall loss |

Most production systems combine these — e.g., IVF + PQ — to balance speed, accuracy, and memory.

### Popular Vector Databases

- **Pinecone** — Fully managed, serverless, easy to start with; no infrastructure to run.
- **Weaviate** — Open-source, supports hybrid search (BM25 + vector), strong schema support.
- **Qdrant** — Written in Rust for high throughput; supports payload filtering alongside ANN.
- **pgvector** — PostgreSQL extension; ideal if you're already on Postgres and want to avoid a new system.
- **Chroma** — Lightweight, embedded-first; popular for local RAG prototypes.

### Practical Pattern: Retrieval-Augmented Generation (RAG)

```
User query
    │
    ▼
Embed query → query_vec
    │
    ▼
ANN search in vector DB → top-k relevant chunks
    │
    ▼
Inject chunks into LLM prompt → grounded, accurate answer
```

RAG dramatically reduces hallucinations by giving the LLM factual context retrieved at inference time, without retraining the model.

### Choosing the Right Tool

If your dataset fits in memory and you need a quick prototype, `pgvector` or `Chroma` will get you running in minutes. For production workloads with millions of vectors, dedicated databases like Qdrant or Weaviate offer better ANN performance, filtering, and horizontal scaling. Managed services like Pinecone remove operational overhead at the cost of vendor lock-in.

The key metric to track is **recall@k**: of the true top-k nearest neighbors, what fraction does the index return? A recall of 0.95 at k=10 is typically acceptable; below 0.90 usually signals a need to retune index parameters.
