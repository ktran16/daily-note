## 00:06 - Vector Databases: The Memory Layer for Modern AI

As large language models and embedding-based applications have gone mainstream, vector databases have emerged as the essential infrastructure for storing and retrieving high-dimensional numerical representations — called *embeddings* — of text, images, audio, and more.

### What Is an Embedding?

An embedding is a dense numerical vector (e.g., 1,536 floats for OpenAI's `text-embedding-3-small`) that encodes semantic meaning. Two pieces of text with similar meaning will have vectors that are close together in high-dimensional space. This geometric property is what makes semantic search, recommendation engines, and retrieval-augmented generation (RAG) possible.

### Why Not a Regular Database?

Traditional databases excel at exact lookups (`WHERE id = 42`) and range queries on scalars. They cannot efficiently answer *approximate nearest-neighbor* (ANN) queries like "find the 10 vectors most similar to this query vector" over millions of rows. Vector databases are purpose-built for exactly this workload.

### Key Indexing Algorithms

| Algorithm | How It Works | Trade-off |
|---|---|---|
| **HNSW** (Hierarchical Navigable Small World) | Layered graph; navigates from coarse to fine levels | Fast queries, high memory usage |
| **IVF** (Inverted File Index) | Clusters vectors; searches only relevant clusters | Lower memory, slightly less accurate |
| **PQ** (Product Quantization) | Compresses vectors into compact codes | Very low memory, recall loss |

Most production systems combine these — e.g., IVF + PQ — to balance speed, accuracy, and memory.

### Popular Vector Databases

- **Pinecone** — Fully managed, serverless, easy to start with; no infrastructure to run.
- **Weaviate** — Open-source, supports hybrid search (BM25 + vector), strong schema support.
- **Qdrant** — Written in Rust for high throughput; supports payload filtering alongside ANN.
- **pgvector** — PostgreSQL extension; ideal if you're already on Postgres and want to avoid a new system.
- **Chroma** — Lightweight, embedded-first; popular for local RAG prototypes.

### Practical Pattern: Retrieval-Augmented Generation (RAG)

```
User query
    │
    ▼
Embed query → query_vec
    │
    ▼
ANN search in vector DB → top-k relevant chunks
    │
    ▼
Inject chunks into LLM prompt → grounded, accurate answer
```

RAG dramatically reduces hallucinations by giving the LLM factual context retrieved at inference time, without retraining the model.

### Choosing the Right Tool

If your dataset fits in memory and you need a quick prototype, `pgvector` or `Chroma` will get you running in minutes. For production workloads with millions of vectors, dedicated databases like Qdrant or Weaviate offer better ANN performance, filtering, and horizontal scaling. Managed services like Pinecone remove operational overhead at the cost of vendor lock-in.

The key metric to track is **recall@k**: of the true top-k nearest neighbors, what fraction does the index return? A recall of 0.95 at k=10 is typically acceptable; below 0.90 usually signals a need to retune index parameters.

## 05:04 - Kafka Log Compaction: Keeping Only What Matters

Apache Kafka is widely known as a distributed message queue, but one of its most underused features is **log compaction** — a retention mode that transforms Kafka from a transient event stream into a durable, queryable key-value store.

### How Standard Retention Works

By default, Kafka retains messages for a configured time window (e.g., 7 days) or until a size limit is reached. After that, old segments are deleted entirely. This is fine for event streams where history only needs to go back a fixed window.

### What Log Compaction Does Differently

With log compaction enabled (`cleanup.policy=compact`), Kafka guarantees that **for each unique key, at least the most recent message is always retained** — regardless of time. Older messages with the same key are periodically removed by a background compaction thread.

```
Before compaction:               After compaction:
offset 0: key=user-42 v=A        (removed)
offset 1: key=user-99 v=X        offset 1: key=user-99 v=X
offset 2: key=user-42 v=B        (removed)
offset 3: key=user-42 v=C        offset 3: key=user-42 v=C
offset 4: key=user-99 v=Y        offset 4: key=user-99 v=Y
```

### Tombstone Records for Deletion

To delete a key entirely, produce a message with that key and a **null value** (a "tombstone"). The compaction process eventually purges both the tombstone and all earlier records for that key.

### Practical Use Cases

| Use Case | Why Compaction Fits |
|---|---|
| **Changelog topics** (Kafka Streams, Flink) | State stores rebuild from the compacted log on restart |
| **CDC (Change Data Capture)** | Downstream consumers get current row state, not full history |
| **Config / feature flags** | Latest value per config key always available to new consumers |
| **User profile updates** | Consumers joining late still get the current profile snapshot |

### Key Configuration Parameters

- `min.cleanable.dirty.ratio` (default `0.5`) — compaction starts when >50% of log bytes are "dirty" (not yet compacted). Lower values mean more aggressive compaction at higher I/O cost.
- `segment.ms` — compaction only runs on closed segments; tune this to control how fresh the compacted view is.
- `delete.retention.ms` — how long tombstones are kept before permanent deletion, giving lagging consumers time to see the delete.

### Combining Compaction with Time-Based Retention

Set `cleanup.policy=compact,delete` to apply both policies: compaction keeps the latest value per key, while the time/size limit still eventually purges very old data. This hybrid is common for CDC pipelines where infinite retention is impractical but consumers still need a recent snapshot.

Log compaction is one of the features that sets Kafka apart from simple message brokers — used correctly, it eliminates the need for a separate key-value store alongside your streaming pipeline.

## 10:04 - Model Quantization: Making LLMs Lean and Fast

Large language models are powerful, but their sheer size makes deployment expensive. A 70B-parameter model in 32-bit floats (FP32) consumes ~280 GB of GPU memory — far beyond most hardware. **Quantization** is the technique that shrinks models by representing weights and activations in lower-precision numeric formats, dramatically reducing memory and speeding up inference with minimal accuracy loss.

### The Precision Spectrum

| Format | Bits per weight | Memory for 7B model | Typical accuracy loss |
|---|---|---|---|
| FP32 | 32 | ~28 GB | Baseline |
| BF16 / FP16 | 16 | ~14 GB | Negligible |
| INT8 | 8 | ~7 GB | Very small |
| INT4 | 4 | ~3.5 GB | Small–moderate |
| INT2/3 | 2–3 | ~1.75 GB | Noticeable |

FP16/BF16 is nearly lossless and widely used in training. INT4 is the sweet spot for LLM inference — it fits models that previously required multi-GPU rigs onto a single consumer GPU.

### Two Major Quantization Approaches

**Post-Training Quantization (PTQ)** — Applied after the model is already trained. No retraining required; just calibrate on a small dataset to determine optimal scaling factors. Tools: `bitsandbytes`, `GPTQ`, `AWQ`, `llama.cpp`.

**Quantization-Aware Training (QAT)** — Quantization is simulated during fine-tuning so the model learns to compensate for precision loss. Higher accuracy than PTQ, but requires training compute.

### GPTQ vs AWQ: The Two Dominant PTQ Algorithms

- **GPTQ** (Generative Pre-Trained Transformer Quantization): Applies layer-by-layer quantization using second-order Hessian information to minimize reconstruction error. Very accurate but slower to quantize.
- **AWQ** (Activation-Aware Weight Quantization): Identifies salient weight channels using activation magnitudes and protects them from aggressive quantization. Faster to run than GPTQ and often comparable in quality.

### Practical Takeaways

```python
# Load a 4-bit quantized model with bitsandbytes
from transformers import AutoModelForCausalLM, BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_compute_dtype="bfloat16")
model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3-8B", quantization_config=bnb_config)
```

- **INT4 + BF16 compute** is a common combo: weights stored at 4 bits, matrix multiplications done in BF16 for accuracy.
- **Perplexity** is the standard metric to track quality degradation — a jump of >1–2 points signals meaningful accuracy loss.
- For CPU-only inference, `llama.cpp` with GGUF format supports Q4_K_M and Q5_K_M quantizations that run efficiently on MacBooks and commodity servers.

Quantization has democratized access to large models: a quantized Llama 3 70B runs on a single 48 GB GPU that previously couldn't fit even a 30B model in full precision.

## 15:04 - Apache Iceberg: The Open Table Format Winning the Lakehouse Wars

For years, data lakehouses struggled with a fundamental problem: files on object storage (S3, GCS, ADLS) are cheap and scalable, but they lack the ACID transactions, schema evolution, and efficient querying that data warehouses offer. **Apache Iceberg** is the open table format that closes this gap — and it has become the de facto standard for modern data platforms.

### What Is a Table Format?

A table format is a layer of metadata that sits on top of raw Parquet (or ORC) files and tells query engines what data exists, where it lives, and how it's organized. Without one, a "table" on S3 is just a flat directory of files with no transactional guarantees.

Iceberg defines a tree of metadata files:

```
catalog
  └── table metadata (JSON)
        └── snapshot → manifest list
                          └── manifest files
                                └── data files (Parquet)
```

Each write creates a new **snapshot** — an immutable, atomic view of the table. Old snapshots are retained, enabling time travel.

### Key Features That Matter in Practice

**ACID Transactions** — Concurrent writers don't corrupt data. Iceberg uses optimistic concurrency with commit retries, making it safe to run multiple Spark or Flink jobs simultaneously.

**Hidden Partitioning** — In Hive-style partitioning, queries must filter on the partition column explicitly (`WHERE dt = '2026-06-07'`). Iceberg computes partition values automatically from transform functions (e.g., `days(event_time)`), so queries just filter on the raw column and Iceberg prunes irrelevant files transparently.

**Schema & Partition Evolution** — Add, rename, or drop columns without rewriting data. Change partition schemes without migrating old files. Old data keeps its original layout; new data uses the new scheme.

**Time Travel & Rollback**

```sql
-- Query as of a specific snapshot
SELECT * FROM orders VERSION AS OF 5432109876;

-- Roll back to yesterday's state
ALTER TABLE orders ROLLBACK TO SNAPSHOT 5432109876;
```

### Engine Compatibility

| Engine | Support Level |
|---|---|
| Apache Spark | Native via `iceberg-spark` |
| Trino / Presto | Native connector |
| Apache Flink | Streaming reads and writes |
| DuckDB | Read support via `iceberg` extension |
| Snowflake / BigQuery | External Iceberg table support |

### Iceberg vs Delta Lake vs Hudi

All three solve the same core problem. **Delta Lake** (Databricks-originated) has the richest Spark integration and mature tooling. **Apache Hudi** (Uber-originated) excels at upsert-heavy CDC workloads with its record-level index. **Iceberg** wins on engine neutrality — no single vendor controls it, and the widest range of engines support it natively, making it the safest long-term bet for multi-engine architectures.

### Getting Started

```python
# Create an Iceberg table with PySpark
spark.sql("""
  CREATE TABLE catalog.db.events (
    event_id BIGINT,
    user_id  BIGINT,
    ts       TIMESTAMP,
    payload  STRING
  ) USING iceberg
  PARTITIONED BY (days(ts))
""")
```

For teams choosing a storage format today, Iceberg's vendor-neutral governance (Apache Software Foundation), hidden partitioning, and broad engine support make it the lowest-risk, highest-flexibility choice for a production lakehouse.
