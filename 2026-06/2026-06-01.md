## 00:03 - Apache Iceberg: The Open Table Format Transforming Data Lakehouses

Apache Iceberg has emerged as the dominant open table format for large-scale analytic datasets, solving fundamental limitations that plagued traditional data lakes built on raw Parquet or ORC files.

### Why Iceberg Matters

Traditional data lakes stored files in object storage (S3, GCS, ADLS) with no transactional guarantees. This led to painful problems: readers could see partial writes, schema changes broke downstream pipelines, and time-travel queries were impossible. Iceberg solves all of these.

### Core Architecture

Iceberg introduces a **metadata layer** that sits above the raw data files:

- **Catalog** — tracks the current metadata pointer for each table (can be Hive Metastore, AWS Glue, Nessie, or REST-based).
- **Metadata files** — JSON files describing table schema, partition specs, and snapshots.
- **Manifest lists** — track which manifest files belong to a given snapshot.
- **Manifest files** — list the actual data files along with column-level statistics (min/max, null counts).
- **Data files** — the underlying Parquet, ORC, or Avro files.

### Key Capabilities

**ACID transactions** — Iceberg uses optimistic concurrency control. Writers atomically swap the metadata pointer, so readers always see a consistent snapshot.

**Schema evolution** — you can add, rename, reorder, or drop columns safely. Column IDs (not names) are used internally, so renaming a column doesn't invalidate existing data files.

**Partition evolution** — you can change how a table is partitioned (e.g., from monthly to daily) without rewriting existing data. New data uses the new spec; old data retains its original spec.

**Time travel** — query any past snapshot with `AS OF` syntax. This is invaluable for auditing, debugging bad pipelines, and reproducible ML training sets.

**Hidden partitioning** — Iceberg applies partition transforms (bucket, truncate, year/month/day/hour) automatically, so queries don't need partition predicates to be performant. The engine prunes manifests using the column statistics.

### Practical Tip

When compacting small files (a common Iceberg maintenance task), use the `rewrite_data_files` procedure with a `target-file-size-bytes` of 128–512 MB. Pair it with `expire_snapshots` to reclaim storage from stale metadata. Running these on a schedule (e.g., daily via Spark or Flink) keeps read performance high without manual intervention.

Iceberg is now supported natively by Spark, Flink, Trino, Dremio, Snowflake (via Iceberg tables), and BigQuery Omni — making it the pragmatic choice for any modern data lakehouse.

## 05:03 - Retrieval-Augmented Generation (RAG): Making LLMs Actually Know Things

Large language models have a well-known limitation: their knowledge is frozen at the training cutoff. Retrieval-Augmented Generation (RAG) is the practical engineering pattern that overcomes this by dynamically injecting relevant, up-to-date context into the model's prompt at inference time.

### The Core Idea

Instead of relying solely on what the model memorized during training, RAG splits the problem into two steps:

1. **Retrieve** — find the most relevant chunks of text from an external knowledge base.
2. **Generate** — pass those chunks as context in the prompt, letting the model reason over fresh information.

This turns an LLM into a reading-comprehension engine rather than a memory-recall engine — a fundamentally more reliable architecture for knowledge-intensive tasks.

### Key Components

**Chunking strategy** — Documents are split into overlapping chunks (typically 256–512 tokens with a 10–20% overlap). Chunk size is a critical tuning knob: too small loses context, too large dilutes relevance signals.

**Embedding model** — Each chunk is encoded into a dense vector using a text embedding model (e.g., OpenAI `text-embedding-3-small`, Cohere Embed v3, or open-source `bge-large`). The same model must embed queries at retrieval time.

**Vector store** — Embeddings are indexed in a vector database (Pinecone, Weaviate, pgvector, Chroma) that supports Approximate Nearest Neighbor (ANN) search. At query time, the top-k most similar chunks are retrieved via cosine or dot-product similarity.

**Reranking** — A cross-encoder reranker (e.g., Cohere Rerank, BGE Reranker) re-scores the top-k candidates with higher fidelity. This two-stage retrieve-then-rerank pattern significantly improves answer quality over raw ANN alone.

### Practical Pitfalls

**Retrieval failures are silent** — if the right chunk isn't retrieved, the model will either hallucinate or say "I don't know." Add retrieval evaluation (e.g., measuring recall@k on a labeled set) as a first-class metric alongside generation quality.

**Hybrid search beats pure semantic search** — combine vector similarity with BM25 keyword search using Reciprocal Rank Fusion (RRF). This catches exact-match cases (product SKUs, names, IDs) that embeddings handle poorly.

**Metadata filtering** — tag chunks with metadata (date, source, department) and apply pre-filters before ANN search. This dramatically reduces the search space and prevents stale or irrelevant documents from polluting results.

RAG has become the default architecture for enterprise AI assistants, internal knowledge bots, and any application where accuracy and up-to-date information matter more than raw generation creativity.

## 10:03 - Gradient Boosting: Why XGBoost and LightGBM Dominate Tabular Data

Despite the explosion of deep learning, gradient boosted decision trees (GBDTs) remain the go-to algorithm for structured/tabular data. XGBoost and LightGBM consistently win Kaggle competitions and production ML pipelines where data lives in rows and columns.

### How Gradient Boosting Works

GBDTs build an ensemble of shallow trees sequentially. Each new tree learns to correct the **residual errors** of all previous trees. The final prediction is the weighted sum of all trees in the ensemble. "Gradient" refers to the fact that each tree is fit to the negative gradient of the loss function — making it a general framework that works for regression, classification, ranking, and custom objectives.

### XGBoost vs. LightGBM

Both implement GBDT but with different tree-growing strategies:

- **XGBoost** grows trees **level-wise** (breadth-first). All nodes at depth *d* are split before moving to depth *d+1*. More conservative, easier to tune.
- **LightGBM** grows trees **leaf-wise** (best-first). It always splits the leaf with the highest gain, regardless of depth. This produces asymmetric trees that converge faster and use less memory — especially on large datasets (>100K rows).

In practice: LightGBM trains 5–10× faster than XGBoost on large data. XGBoost can be more robust with small datasets and is less prone to overfitting on noisy features.

### Critical Hyperparameters

| Parameter | What it controls | Typical starting range |
|---|---|---|
| `n_estimators` | Number of trees | 300–3000 (use early stopping) |
| `learning_rate` | Shrinkage per tree | 0.01–0.1 |
| `max_depth` | Tree depth (XGBoost) | 3–8 |
| `num_leaves` | Max leaves (LightGBM) | 20–300 |
| `subsample` | Row sampling per tree | 0.6–0.9 |
| `colsample_bytree` | Feature sampling per tree | 0.5–0.9 |
| `min_child_weight` | Minimum sum of hessian in a leaf | 1–10 |

### Practical Tips

**Always use early stopping.** Set `n_estimators` high (e.g., 5000) and use a validation set to stop when the eval metric stops improving for 50–100 rounds. This automatically finds the optimal number of trees without grid searching.

**Handle categoricals natively.** LightGBM's `categorical_feature` parameter uses an optimal split algorithm for categoricals — far better than one-hot encoding for high-cardinality features (e.g., user IDs, zip codes).

**Feature importance is a starting point, not the truth.** SHAP values (available via the `shap` library with both frameworks) give per-sample, directional feature attributions — use these for debugging and stakeholder explanations instead of raw gain/split importance.

GBDTs are not outdated — they are the pragmatic default for any new tabular ML project, requiring less preprocessing than neural networks and delivering interpretable, production-ready models quickly.
