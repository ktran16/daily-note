## 00:04 - dbt Artifacts: Mining manifest.json, run_results.json & catalog.json

Every `dbt run`, `dbt test`, and `dbt docs generate` command silently drops a set of JSON files into the `target/` directory. These **artifacts** are the backbone of dbt's most powerful features — slim CI, state-based selection, automated documentation, and third-party observability tools like Elementary or re_data. Understanding them unlocks a new layer of control over your dbt project.

### The Three Key Artifacts

**`manifest.json`** — Generated on every `dbt compile` or `dbt run`. It is the complete, static snapshot of your project graph: every node (model, test, snapshot, seed, source), its SQL, config, dependencies, and metadata. Slim CI uses this to compare states (`--state` flag).

**`run_results.json`** — Written after execution. Contains the timing and status (`success`, `error`, `warn`, `skip`) of every node that ran. Invaluable for building dashboards over run history or alerting on model failures.

**`catalog.json`** — Produced by `dbt docs generate`. Describes the actual warehouse schema at query time: column names, data types, row counts, and byte sizes pulled from `information_schema`.

### Practical Use Cases

**1. Slim CI — only test what changed:**
```bash
# On a PR branch, compare against production manifest
dbt run --select state:modified+ --defer --state ./prod-artifacts/
dbt test --select state:modified+ --defer --state ./prod-artifacts/
```

**2. Parse run results in CI to fail the pipeline on errors:**
```python
import json, sys

with open("target/run_results.json") as f:
    results = json.load(f)

failures = [r for r in results["results"] if r["status"] == "error"]
if failures:
    for f in failures:
        print(f"FAILED: {f['unique_id']}")
    sys.exit(1)
```

**3. Extract model metadata from the manifest:**
```python
import json

with open("target/manifest.json") as f:
    manifest = json.load(f)

# List all models with their materialization config
for uid, node in manifest["nodes"].items():
    if node["resource_type"] == "model":
        mat = node["config"]["materialized"]
        print(f"{node['name']}: {mat}")
```

### Persisting Artifacts for State Comparison

In dbt Cloud, artifacts are stored and versioned automatically. In dbt Core, you must persist them yourself — typically by uploading `target/manifest.json` to cloud storage (S3, GCS) after each production run and downloading it before the next CI run:

```yaml
# GitHub Actions example
- name: Download production manifest
  run: aws s3 cp s3://my-bucket/dbt/manifest.json ./prod-artifacts/manifest.json

- name: Run slim CI
  run: dbt run --select state:modified+ --defer --state ./prod-artifacts/
```

### Key Takeaway

Treat artifacts as a first-class output of your dbt pipeline. Storing and comparing them enables incremental CI (cutting run times by 80%+ on large projects), powers observability tooling, and gives you a queryable audit trail of every model's execution history.

## 05:02 - dbt Versioned Models: Managing Breaking Changes Without Blocking Downstream Teams

In large data platforms, a "core" model like `dim_customers` may be referenced by dozens of downstream models and BI dashboards. Changing a column name or removing a field is a breaking change — without coordination, it silently breaks everything that depends on it. **dbt Versioned Models** (introduced in dbt v1.6) solve this by letting you maintain multiple concurrent versions of a model, giving downstream consumers time to migrate on their own schedule.

### Defining a Versioned Model

Declare versions in the model's `schema.yml` entry. Each version can be a separate SQL file or inherit from a shared definition:

```yaml
# models/schema.yml
models:
  - name: dim_customers
    latest_version: 2
    versions:
      - v: 1
        deprecation_date: "2026-09-01"   # warn downstream refs after this date
      - v: 2
        defined_in: dim_customers_v2     # points to models/dim_customers_v2.sql
```

**`dim_customers.sql`** (v1 — original schema):
```sql
select
    customer_id,
    first_name || ' ' || last_name  as customer_name,  -- soon deprecated
    email,
    created_at
from {{ ref('stg_customers') }}
```

**`dim_customers_v2.sql`** (v2 — split name columns):
```sql
select
    customer_id,
    first_name,
    last_name,
    email,
    created_at
from {{ ref('stg_customers') }}
```

### Referencing Versioned Models Downstream

```sql
-- Explicitly pin to v1 (gets a deprecation warning in dbt logs after 2026-09-01)
select * from {{ ref('dim_customers', v=1) }}

-- Reference the latest version (always resolves to v2 now)
select * from {{ ref('dim_customers') }}
```

Unpinned `ref()` calls always resolve to `latest_version`. Pinned calls (`v=1`) continue to work but emit deprecation warnings once `deprecation_date` passes, making it easy to audit who still hasn't migrated:

```bash
dbt run --select dim_customers
# WARNING: Model 'fct_orders' references deprecated version 1 of 'dim_customers'
# (deprecated as of 2026-09-01). Please upgrade to version 2.
```

### Running & Selecting Versioned Models

```bash
# Run a specific version
dbt run --select dim_customers.v1
dbt run --select dim_customers.v2

# Run all versions
dbt run --select dim_customers

# Run only the latest version
dbt run --select dim_customers.v2+
```

### When to Use Model Versions

- Renaming or removing columns used by external consumers
- Changing grain (row-level semantics) of a widely-used model
- Gradual migrations between source systems
- Multi-team projects where you cannot coordinate deploys

### Key Takeaway

Model versioning turns a destructive schema change into a controlled migration. Keep `v1` alive with a `deprecation_date`, ship `v2` as `latest_version`, and let downstream teams migrate at their pace. This pattern is especially powerful in **dbt Mesh** environments where projects across teams share public models via cross-project `ref()`.
