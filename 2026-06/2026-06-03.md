# Daily Notes - 2026-06-03

## 00:03 - Retrieval-Augmented Generation (RAG): Architecture Patterns and Production Pitfalls

Retrieval-Augmented Generation (RAG) has become the dominant pattern for grounding large language model (LLM) outputs in factual, up-to-date, or domain-specific knowledge. Rather than relying solely on what a model learned during training, RAG dynamically retrieves relevant context at inference time and injects it into the prompt.

**The Core Architecture**

A RAG pipeline has three main stages:

1. **Ingestion** — Documents are chunked, embedded using a model like `text-embedding-3-small`, and stored in a vector database (Pinecone, pgvector, Weaviate, Chroma, etc.).
2. **Retrieval** — At query time, the user's question is embedded and a similarity search (cosine or dot-product) returns the top-k most relevant chunks.
3. **Generation** — The retrieved chunks are prepended to the prompt as context, and the LLM generates a grounded response.

**Chunking Strategy Matters More Than Most Expect**

Fixed-size chunking (e.g., 512 tokens with 50-token overlap) is the default, but it often splits logical units mid-sentence. Better alternatives include:
- **Semantic chunking** — split on natural paragraph/section boundaries.
- **Hierarchical chunking** — store both summary-level and detail-level chunks, retrieve at the right granularity.

**Common Production Pitfalls**

- **Retrieval without reranking**: Top-k cosine similarity is a blunt instrument. Adding a cross-encoder reranker (e.g., Cohere Rerank, BGE-Reranker) dramatically improves relevance at the cost of ~50ms latency.
- **Ignoring metadata filters**: Searching all chunks for a query that applies only to one product or time range causes noise. Always filter by metadata (date, source, category) before or during vector search.
- **Naive evaluation**: Many teams ship RAG with no eval loop. Use frameworks like RAGAS or LangSmith to measure retrieval recall, context precision, and answer faithfulness automatically.
- **Stale embeddings**: If you update documents but not their embeddings, retrieved context diverges from ground truth. Build incremental re-indexing into your ingestion pipeline.

**When RAG Beats Fine-Tuning**

RAG is preferred when knowledge changes frequently, when source attribution is required, or when the domain corpus is too large to fit in context. Fine-tuning wins when you need to change model *behavior* (tone, format, reasoning style) rather than inject new knowledge.

Understanding these trade-offs early saves significant re-architecture cost in production systems.
