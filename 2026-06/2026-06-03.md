## 00:03 - Retrieval-Augmented Generation (RAG): Architecture Patterns and Production Pitfalls

Retrieval-Augmented Generation (RAG) has become the dominant pattern for grounding large language model (LLM) outputs in factual, up-to-date, or domain-specific knowledge. Rather than relying solely on what a model learned during training, RAG dynamically retrieves relevant context at inference time and injects it into the prompt.

**The Core Architecture**

A RAG pipeline has three main stages:

1. **Ingestion** — Documents are chunked, embedded using a model like `text-embedding-3-small`, and stored in a vector database (Pinecone, pgvector, Weaviate, Chroma, etc.).
2. **Retrieval** — At query time, the user's question is embedded and a similarity search (cosine or dot-product) returns the top-k most relevant chunks.
3. **Generation** — The retrieved chunks are prepended to the prompt as context, and the LLM generates a grounded response.

**Chunking Strategy Matters More Than Most Expect**

Fixed-size chunking (e.g., 512 tokens with 50-token overlap) is the default, but it often splits logical units mid-sentence. Better alternatives include:
- **Semantic chunking** — split on natural paragraph/section boundaries.
- **Hierarchical chunking** — store both summary-level and detail-level chunks, retrieve at the right granularity.

**Common Production Pitfalls**

- **Retrieval without reranking**: Top-k cosine similarity is a blunt instrument. Adding a cross-encoder reranker (e.g., Cohere Rerank, BGE-Reranker) dramatically improves relevance at the cost of ~50ms latency.
- **Ignoring metadata filters**: Searching all chunks for a query that applies only to one product or time range causes noise. Always filter by metadata (date, source, category) before or during vector search.
- **Naive evaluation**: Many teams ship RAG with no eval loop. Use frameworks like RAGAS or LangSmith to measure retrieval recall, context precision, and answer faithfulness automatically.
- **Stale embeddings**: If you update documents but not their embeddings, retrieved context diverges from ground truth. Build incremental re-indexing into your ingestion pipeline.

**When RAG Beats Fine-Tuning**

RAG is preferred when knowledge changes frequently, when source attribution is required, or when the domain corpus is too large to fit in context. Fine-tuning wins when you need to change model *behavior* (tone, format, reasoning style) rather than inject new knowledge.

Understanding these trade-offs early saves significant re-architecture cost in production systems.

## 05:03 - DeFi Liquidity Pools and Impermanent Loss Explained

Decentralized exchanges (DEXs) like Uniswap, Curve, and SushiSwap rely on **Automated Market Makers (AMMs)** rather than traditional order books. The engine behind AMMs is the **liquidity pool** — and understanding how they work is essential before committing capital as a liquidity provider (LP).

**How Liquidity Pools Work**

A liquidity pool holds two (or more) tokens in a smart contract. The most common model uses the constant product formula:

```
x * y = k
```

where `x` and `y` are the reserves of each token and `k` is a constant. When a trader swaps token A for token B, the pool adjusts both reserves to maintain `k`. The price is derived from the ratio of reserves, not from an order book.

LPs deposit equal value of both tokens, receive LP tokens representing their share, and earn a fraction of every swap fee (typically 0.05%–1% per trade, depending on the pool).

**Impermanent Loss: The Hidden Risk**

Impermanent loss (IL) occurs when the price ratio of your deposited tokens changes after deposit. The AMM rebalances the pool mechanically, which means you end up holding more of the depreciating asset and less of the appreciating one compared to simply holding both tokens.

| Price change of one asset | Impermanent loss |
|---------------------------|-----------------|
| +25%                      | ~0.6%           |
| +50%                      | ~2.0%           |
| +100% (2×)                | ~5.7%           |
| +400% (5×)                | ~25.5%          |

IL is "impermanent" because it reverses if the price ratio returns to the original. It becomes permanent when you withdraw.

**When LP Positions Are Profitable**

- **High-volume, low-volatility pairs** (e.g., USDC/USDT stablecoin pools): fees accumulate fast with near-zero IL.
- **Correlated asset pairs** (e.g., ETH/stETH): price ratio stays tight, minimizing rebalancing losses.
- **Concentrated liquidity** (Uniswap v3): LPs set a price range, earning amplified fees within that range — but face full IL if price exits the range.

**Key Takeaway**

Fee APY must exceed expected IL for an LP position to outperform a simple hold. Always model IL against projected trading volume before providing liquidity, especially for volatile, uncorrelated pairs.

## 10:03 - dbt and the Analytics Engineering Workflow

**dbt (data build tool)** has become the de facto standard for transforming data inside a warehouse. It sits between raw ingested data and the analytics layer, letting data teams apply software engineering practices — version control, testing, documentation, modular design — to SQL transformations.

**The Core Mental Model**

dbt operates on a simple contract: you write `SELECT` statements, dbt handles the materialization. A model is just a `.sql` file:

```sql
-- models/orders/fct_orders.sql
select
    order_id,
    customer_id,
    sum(line_total) as revenue,
    count(*) as item_count
from {{ ref('stg_orders') }}
group by 1, 2
```

`{{ ref('stg_orders') }}` is a Jinja macro that resolves to the correct schema-qualified table name and registers a **DAG dependency**. Running `dbt run` executes models in dependency order; running `dbt docs generate` produces a full lineage graph.

**Materialization Strategies**

| Strategy | When to Use |
|---|---|
| `view` | Exploratory or lightweight models; no storage cost, always fresh |
| `table` | Stable, frequently queried aggregates |
| `incremental` | Large fact tables — only processes new/changed rows |
| `ephemeral` | CTEs you want modularized without creating a warehouse object |

Incremental models are the biggest performance lever. A `unique_key` + `is_incremental()` macro tells dbt to merge only new records:

```sql
{{ config(materialized='incremental', unique_key='event_id') }}
select * from {{ ref('stg_events') }}
{% if is_incremental() %}
  where event_ts > (select max(event_ts) from {{ this }})
{% endif %}
```

**Testing Is Built In**

dbt ships with four generic tests — `not_null`, `unique`, `accepted_values`, `relationships` — applied in `schema.yml`:

```yaml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests: [not_null, unique]
      - name: status
        tests:
          - accepted_values:
              values: [pending, shipped, delivered, cancelled]
```

Run `dbt test` to validate your entire transformation layer before promoting to production. Combined with `dbt source freshness`, you catch upstream data delays before dashboards surface stale numbers.

**The Analytics Engineering Layer**

The convention is to organize models into three layers: **staging** (one-to-one with raw sources, light cleaning), **intermediate** (joins and business logic), and **marts** (metric-level aggregates for BI tools). This separation makes models reusable across reports and prevents logic from diverging across ad-hoc queries.

dbt's value compound over time: every transformation is auditable, reproducible, and self-documenting — qualities that raw SQL scripts in spreadsheets or BI tools can never provide.

## 19:02 - ML Model Drift: Detection, Types, and Production Remediation

Deploying a machine learning model is not the finish line — it's the starting gun for a continuous maintenance problem called **model drift**. In production, the statistical properties of incoming data shift over time, causing model performance to degrade silently unless you are actively measuring it.

**Two Distinct Drift Problems**

**Data drift** (also called covariate shift) occurs when the distribution of input features changes, even if the relationship between features and the target stays the same. For example, a fraud detection model trained in 2023 may never have seen the transaction patterns introduced by a new payment platform in 2025. The model isn't wrong about what it learned — it's just seeing a world it was never shown.

**Concept drift** is more severe: the underlying relationship between inputs and outputs changes. A credit risk model trained before a recession may assign low risk scores to profiles that are genuinely high risk in a post-recession economy. No amount of re-training on old data fixes this — you need new labeled data that reflects the new reality.

**Detecting Drift Without Ground Truth Labels**

Ground truth labels often arrive days or weeks after prediction (e.g., a loan default is only known months later). Waiting for labels means drift goes undetected far too long. Practical early-warning strategies:

- **Population Stability Index (PSI)**: Compares feature distributions between training and production windows. PSI > 0.2 signals significant drift.
- **Kolmogorov-Smirnov (KS) test**: Statistical test for whether two samples come from the same distribution. Run it per feature, on a rolling window.
- **Prediction distribution monitoring**: If your model's output score distribution shifts (e.g., average predicted probability rising), that's a proxy signal for upstream drift even before labels arrive.
- **Business KPI proxy metrics**: Track downstream outcomes like click-through rate, conversion rate, or churn as leading indicators that the model's decisions are degrading.

**Remediation Strategies**

| Approach | When to Use |
|---|---|
| **Retraining on recent data** | Data drift; target relationship stable |
| **Fine-tuning / warm-starting** | Concept drift with limited new labels |
| **Champion/challenger routing** | A/B test updated model before full rollout |
| **Feature engineering updates** | New data sources capture the shifted signal |
| **Model ensemble with recency weighting** | Smooths transition during gradual drift |

**The Monitoring Minimum Viable Setup**

At minimum, every production ML system should track: (1) feature distribution stats per day/week, (2) model output distribution, and (3) delayed label feedback looped into an evaluation dataset. Tools like Evidently AI, WhyLogs, and Arize make this instrumentation straightforward to add without rebuilding the inference pipeline.

Catching drift early is orders of magnitude cheaper than debugging a business incident caused by a silently degraded model.
