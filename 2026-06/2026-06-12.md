## 00:05 - Model Quantization: Running Large Language Models on a Budget

Large language models are notoriously expensive to serve. A 70-billion-parameter model in 32-bit float precision occupies roughly 280 GB of GPU memory — far beyond a single consumer or even enterprise GPU. Model quantization is the primary technique that makes these models deployable without a data center budget.

### What Quantization Does

Quantization reduces the numerical precision of a model's weights. Instead of storing each weight as a 32-bit float (FP32) or 16-bit float (FP16), you store it in 8-bit integers (INT8), 4-bit integers (INT4), or even lower. A 70B model in INT4 drops to roughly 35 GB — feasible on two consumer GPUs or a single high-end workstation card.

The trade-off is accuracy. Lower bit-widths mean fewer possible values per weight, so quantized models introduce rounding error. The art of quantization is minimizing that error.

### Key Approaches in Practice

**Post-Training Quantization (PTQ)** quantizes a model after it has already been trained. No retraining is required, making it fast to apply. Tools like `llama.cpp` (GGUF format) and `bitsandbytes` use PTQ. Quality degrades gradually — INT8 is nearly lossless, INT4 is acceptable for most tasks, INT2 is noticeably worse.

**GPTQ** (Generative Pre-Trained Transformer Quantization) improves PTQ by using a small calibration dataset to minimize per-layer quantization error, achieving better INT4 quality than naive rounding. It's widely used with Hugging Face's `AutoGPTQ` library.

**AWQ** (Activation-Aware Weight Quantization) takes a different approach: it identifies the small fraction of weights that are most salient (those activated by large input values) and protects them from aggressive quantization. AWQ often outperforms GPTQ at the same bit-width.

**QLoRA** combines quantization with fine-tuning: the base model is loaded in 4-bit, and a small set of trainable adapter weights (LoRA) are trained in 16-bit. This allows fine-tuning a 65B model on a single 48 GB GPU — something impossible without quantization.

### Practical Guidance

- For local inference on consumer hardware, GGUF Q4_K_M (via `llama.cpp` or Ollama) is the sweet spot — strong quality, wide tool support, CPU-offload capable.
- For GPU serving with fast inference, AWQ or GPTQ INT4 on a quantization-aware runtime (vLLM, TGI) is preferred.
- Always benchmark on your task. Quantization effects are domain-specific: a model that scores 98% of full-precision on general benchmarks may degrade more on specialized reasoning tasks.

Quantization is now a standard tool in any ML engineer's deployment kit — understanding its mechanics helps you choose the right format for your latency, memory, and accuracy requirements.

## 05:03 - Change Data Capture: Streaming Database Changes in Real Time

Change Data Capture (CDC) is a data engineering pattern that tracks row-level changes in a database — inserts, updates, and deletes — and streams them to downstream systems the moment they happen. It solves a fundamental problem: how do you keep a data warehouse, search index, or cache in sync with an operational database without hammering it with expensive full-table scans?

### How CDC Works

Most production CDC implementations read from the database's **transaction log** (the WAL in PostgreSQL, the binlog in MySQL, the redo log in Oracle). Because every committed change is already recorded there for replication purposes, CDC is essentially a zero-overhead read of data the database was writing anyway. The CDC tool reads the log, decodes the change events, and publishes them to a message bus like Apache Kafka.

Popular open-source tools:
- **Debezium** — the de facto standard; connectors for PostgreSQL, MySQL, MongoDB, SQL Server, and more. Runs as a Kafka Connect plugin.
- **Maxwell's Daemon** — MySQL-focused, simpler to operate, outputs JSON.
- **PG Logical / pgoutput** — PostgreSQL-native logical replication, usable directly without Debezium.

### A Typical CDC Pipeline

```
PostgreSQL (WAL) → Debezium (Kafka Connect) → Kafka topic → consumers
                                                              ├─ Flink/Spark (analytics)
                                                              ├─ Elasticsearch (search)
                                                              └─ Redis (cache invalidation)
```

Each change event carries the **before** and **after** state of the row, plus metadata: timestamp, transaction ID, operation type. This makes it straightforward for consumers to apply the same change or derive computed fields.

### Key Practical Considerations

**Schema evolution** is the hardest operational challenge. If a source table gains a new column, downstream consumers must handle it gracefully. Using a schema registry (Confluent Schema Registry with Avro or Protobuf) enforces compatibility rules before a bad schema reaches consumers.

**Initial snapshot**: when you first configure CDC on an existing table, you need a consistent baseline of all existing rows before the streaming starts. Debezium handles this with a snapshot phase that takes a consistent read lock, dumps all rows, then transitions to log-based streaming.

**At-least-once delivery** is the norm — a crash mid-offset can replay events. Consumers must be idempotent (e.g., upsert on primary key rather than blind insert).

CDC is the foundation of the modern lakehouse real-time layer. Mastering it means your analytics can reflect production state in seconds rather than overnight batch windows.

## 10:04 - Impermanent Loss: The Hidden Cost of DeFi Liquidity Provision

Decentralized exchanges (DEXs) like Uniswap, Curve, and Balancer don't use order books — they use **Automated Market Makers (AMMs)**. Instead of matching buyers and sellers, AMMs hold liquidity in smart-contract pools and let an algorithm set prices. Anyone can deposit tokens into a pool and earn a share of trading fees. But there's a catch that surprises many newcomers: **impermanent loss**.

### How AMM Pools Work

The most common AMM formula is the constant-product invariant: `x * y = k`, where `x` and `y` are the reserves of two tokens and `k` is a constant. When a trader swaps token A for token B, the pool's ratio shifts, and the price adjusts automatically to maintain `k`. Liquidity providers (LPs) deposit both tokens in the current ratio and receive LP tokens representing their share of the pool.

### What Is Impermanent Loss?

Impermanent loss (IL) occurs when the price ratio of your deposited tokens changes relative to when you deposited them. Because the AMM continuously rebalances the pool to satisfy `x * y = k`, it sells the appreciating token and buys the depreciating one — the opposite of what a buy-and-hold investor would want.

**Example:** You deposit 1 ETH and 2,000 USDC when ETH = $2,000. If ETH rises to $4,000, arbitrageurs drain ETH from the pool and add USDC until the ratio reflects the new price. When you withdraw, you get less ETH (≈0.707 ETH) and more USDC (≈2,828 USDC) — worth $5,656 total. But holding 1 ETH + 2,000 USDC outright would be worth $6,000. The $344 gap is the impermanent loss (~5.7%).

The loss is "impermanent" only if prices revert. If prices stay diverged, it becomes permanent upon withdrawal.

### IL by Price Ratio Change

| Price Change | IL |
|---|---|
| 1.25× | ~0.6% |
| 1.5× | ~2.0% |
| 2× | ~5.7% |
| 4× | ~20.0% |
| 10× | ~42.5% |

### Mitigating IL in Practice

- **Stablecoin pools** (USDC/USDT, DAI/USDC) have near-zero IL because prices don't diverge. Curve Finance specializes in these and can offer very tight spreads.
- **Correlated asset pools** (e.g., stETH/ETH) behave similarly and reduce IL significantly.
- **Concentrated liquidity** (Uniswap v3) lets LPs provide liquidity within a chosen price range, earning higher fees per dollar deposited — but IL can be sharper if the price moves outside the range.
- **Fee income** offsets IL. High-volume pools with 0.3–1% fees can overcome moderate IL for patient LPs.

Understanding impermanent loss is essential before committing capital to any AMM pool. Always model expected fee revenue against worst-case IL for your chosen token pair before depositing.

## 15:04 - A/B Testing: Rigorous Experimentation for Data-Driven Decisions

A/B testing is the backbone of data-driven product development. It sounds straightforward — show version A to some users, version B to others, compare outcomes — but running a test that produces trustworthy results requires careful attention to statistical fundamentals that are routinely violated in practice.

### The Core Statistical Framework

An A/B test is a randomized controlled experiment. Users are randomly assigned to a control group (A) or treatment group (B). You measure a primary metric (e.g., conversion rate, revenue per user, session length) and use a hypothesis test to determine whether any observed difference is likely real or due to random chance.

The two key parameters to set before running any test:

- **Significance level (α)**: The probability of a false positive — declaring a winner when there is none. Standard is α = 0.05 (5%).
- **Statistical power (1 − β)**: The probability of detecting a real effect when it exists. Standard target is 80% or higher.

From these parameters and your expected minimum detectable effect (MDE), you calculate the required sample size. Running a test without this calculation is one of the most common errors — stopping early when results look good inflates false positives dramatically.

### Sample Size and Duration

Use the formula for comparing two proportions (for conversion rates):

```
n ≈ 2 × (z_α/2 + z_β)² × p(1−p) / δ²
```

Where `p` is the baseline conversion rate, `δ` is your MDE, and the z-values come from your α and power targets. For a baseline rate of 5%, an MDE of 0.5 percentage points, α=0.05, and 80% power, you need roughly **75,000 users per variant**. Underestimating this is how teams end up running week-long tests on metrics that needed six weeks of data.

Run the test for at least one full business cycle (usually 1–2 weeks) to avoid day-of-week bias. Users who visit Monday have different behavior from those who visit Saturday.

### Common Pitfalls

**Peeking and early stopping**: Checking results daily and stopping when p < 0.05 dramatically inflates the false positive rate — in simulations, it can push the actual false positive rate above 25% for a nominal 5% test. Use sequential testing methods (e.g., always-valid p-values, group sequential designs) if you must check early.

**Multiple metrics and multiple comparisons**: Testing ten metrics simultaneously means roughly a 40% chance of at least one false positive. Use a primary metric designated before the test starts. Apply Bonferroni correction or the Benjamini-Hochberg procedure if secondary metrics matter.

**Network effects and interference**: If users can interact (social features, shared inventory, marketplace pricing), the independence assumption breaks. A user assigned to control may be influenced by their friend in the treatment group. Cluster-based randomization (randomize by household, geography, or friend group) is the remedy.

**Novelty effects**: A new UI may get higher engagement simply because it's unfamiliar. Always let experiments run long enough for novelty to wear off.

### Beyond Binary Outcomes

For continuous metrics (revenue, latency, session time), the t-test applies — but revenue distributions are typically heavy-tailed. Use bootstrapped confidence intervals or the Mann-Whitney U test for robustness, or apply variance reduction techniques like CUPED (Controlled-experiment Using Pre-Experiment Data), which conditions on pre-experiment behavior to reduce noise and shrink required sample sizes by 30–50%.

A rigorous A/B test is not just a statistical exercise — it is the organizational mechanism that separates intuition from evidence. Getting it right means shipping changes you can trust.

## 19:03 - Vector Databases: The Storage Layer Behind Modern AI Applications

As embedding models have become the default way to represent unstructured data — text, images, audio, code — a new class of storage system has emerged to serve them: **vector databases**. Understanding how they work and where they fit is now a core competency for anyone building AI-powered applications.

### What a Vector Database Does

An embedding model converts a piece of content (a sentence, a document chunk, an image) into a high-dimensional numerical vector — typically 768 to 3,072 floats. Semantically similar items end up geometrically close in this space. A vector database stores these vectors and answers **approximate nearest-neighbor (ANN) queries**: given a query vector, find the k most similar vectors in the collection.

This is the backbone of Retrieval-Augmented Generation (RAG), semantic search, recommendation systems, and duplicate detection.

### The Indexing Algorithm: HNSW

The dominant indexing algorithm in production systems is **Hierarchical Navigable Small World (HNSW)**. It builds a multi-layer graph where each node is a vector and edges connect nearby neighbors. During search, the algorithm enters at the top (sparse) layer, greedily traverses toward the query, then descends to denser layers for progressively finer search. This achieves sub-millisecond query latency on millions of vectors with recall above 95%.

HNSW has two key parameters: `M` (edges per node — higher improves recall but increases memory and build time) and `ef_construction` (beam width during index build — higher means better recall at higher build cost). At query time, `ef_search` controls the recall/latency trade-off.

### Dedicated vs. Integrated Options

**Dedicated vector databases**: Pinecone (managed, serverless), Weaviate (open-source, hybrid search built-in), Qdrant (open-source, Rust-based, strong filtering performance), Milvus (open-source, Kubernetes-native for billion-scale).

**Integrated vector search**: PostgreSQL + pgvector adds vector columns and HNSW/IVFFlat indexes to an existing Postgres table — often the best starting point for teams already on Postgres who need vector search without a new infrastructure dependency.

### Filtering: The Practical Bottleneck

The hardest problem in production vector search is **filtered ANN** — find the nearest neighbors among vectors that match a metadata predicate (e.g., `user_id = 123` or `published_date > 2025-01-01`). Naive post-filtering (ANN then filter) misses results when the filter is selective. Pre-filtering (filter then brute-force search) is slow. State-of-the-art solutions (Qdrant's payload indexes, Weaviate's ACORN algorithm) interleave filtering into the graph traversal itself.

When selecting a vector database, benchmark on your actual filter selectivity — it matters more than raw ANN throughput for most real-world workloads.
